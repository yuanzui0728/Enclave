# 隐界前台多语言适配深度规划

日期：2026-04-25

## 目标

把 `apps/app` 前台做成真正可持续的多语言体系：

- 默认语言始终是简体中文。
- 用户可切换 English / 日本語 / 한국어，切换过程不卡死、不混排、不残留旧语言。
- 后续新功能默认沿用同一套多语言机制，不需要每次上线后再单独“补一次多语言”。
- UI 文案、格式化、导航、错误提示、状态提示、静态运营配置都纳入治理。
- 用户自己写的聊天、朋友圈、广场内容不被 UI 语言强行翻译；系统 UI 和系统提示才本地化。

## 当前状态

已存在的基础能力：

- `apps/app/src/main.tsx` 已用 `AppLocaleProvider surface="app"` 包住主 App。
- `packages/i18n` 已支持 `zh-CN`、`en-US`、`ja-JP`、`ko-KR` 四种语言。
- `packages/i18n/src/runtime/catalog-loaders.ts` 已按 surface 动态加载 `shared + app` catalog，日语/韩语有英文 fallback。
- `apps/app/src/routes/profile-settings-page.tsx` 已有 `LanguageSwitcher` 入口。
- `packages/i18n/src/runtime/formatters.ts` 已提供 `formatDateTime`、`formatNumber`、`compareByLocale`、`resolveSpeechRecognitionLocale`。

当前问题：

- 前台 catalog 有 `1257` 个 app msgid，但 `en-US / ja-JP / ko-KR` 各还有 `88` 个空翻译，集中在：
  - `apps/app/src/routes/desktop-feedback-page.tsx`
  - `apps/app/src/features/chat/chat-message-search-panel.tsx`
- `pnpm i18n:hardcode:json --scope apps/app/src` 扫描到前台仍有大量裸文案风险：
  - `377` 个源文件被扫描。
  - 总计 `6941` 条硬编码风险。
  - `jsx-literal-prop`: `955`
  - `object-ui-string`: `530`
  - `jsx-cjk-text`: `50`
  - `setter-message-string`: `69`
  - `literal-error-message`: `72`
  - `intl-locale-literal`: `7`
  - `locale-compare-literal`: `5`
  - `raw-cjk-line`: `5252`
- 最高风险文件集中在主体验入口：
  - `apps/app/src/components/chat-message-list.tsx`
  - `apps/app/src/components/chat-composer.tsx`
  - `apps/app/src/features/mini-programs/mini-programs-data.ts`
  - `apps/app/src/features/games/game-center-data.ts`
  - `apps/app/src/features/search/desktop-search-workspace.tsx`
  - `apps/app/src/routes/profile-settings-page.tsx`
  - `apps/app/src/routes/games-page.tsx`
  - `apps/app/src/routes/discover-page.tsx`
  - `apps/app/src/routes/channels-page.tsx`
  - `apps/app/src/routes/contacts-page.tsx`
- 前台切换语言当前存在结构性隐患：
  - 很多组件用 `translateRuntimeMessage` 做命令式翻译，但组件本身没有订阅 `useAppLocale()` 或 `useLingui()`，语言切换后不一定重新渲染。
  - 静态数组在模块顶层保存中文字符串，例如移动底部 tab、桌面 nav、游戏数据、小程序数据、搜索快捷入口。
  - `LanguageSwitcher` 说明里写“偏好会保存”，但 `resolveInitialLocale()` 当前只读 `?locale` 和默认中文，没有读本地持久化偏好。
  - 多处格式化仍硬编码 `zh-CN`，例如 `Intl.DateTimeFormat("zh-CN")`、`localeCompare(..., "zh-CN")`、`toLocaleDateString("zh-CN")`。

## 原则

### 1. UI locale 和 content locale 分离

`uiLocale` 控制按钮、导航、标题、占位符、错误提示、系统状态、日期数字格式。

`contentLocale` 控制 AI 生成内容、系统生成动态、默认推荐内容的语言倾向。用户聊天内容、朋友圈正文、广场正文默认保留原文，不跟随 UI 语言强翻。

第一阶段只做 `uiLocale`。后续如果要让 AI 生成内容跟随语言，需要在 API / prompt 层显式传 `contentLocale`，不能靠前端 DOM 翻译。

### 2. 新功能自动适配靠“类型约束 + 扫描门禁”

不依赖人工记忆。新功能代码进入仓库时，应该被这些机制拦住：

- UI 配置必须保存 `MessageDescriptor` 或 `LocalizedText`，不能直接保存中文 string。
- 页面组件用 `<Trans>`、`msg`、`useLingui()` 或统一 hook 解析文案。
- `i18n:hardcode:changed` / `i18n:missing:changed` 在改动范围内失败即不能合入。
- 当前存量问题用 baseline ratchet 管住，逐步把 baseline 降到 0。

### 3. 前台不采用后台那种 DOM 自动翻译作为主方案

后台可以用自动翻译边界兜底，是因为后台偏运维、数据密度高、文案变更快。

前台是用户体验核心，不能依赖 MutationObserver 去翻 DOM。前台应以 Lingui descriptor、typed copy、localized data 为主。可以保留极小范围 fallback，但不能作为主路径。

## 总体架构

### Locale Runtime

目标文件：

- `packages/i18n/src/locales.ts`
- `packages/i18n/src/runtime/app-locale-provider.tsx`
- `packages/i18n/src/runtime/language-switcher.tsx`
- `packages/i18n/src/runtime/formatters.ts`

规划：

- 保持默认 `DEFAULT_LOCALE = "zh-CN"`。
- `resolveInitialLocale(surface)` 改为：
  - 优先 `?locale=...`
  - 其次用户显式保存的 surface 偏好
  - 最后 `zh-CN`
- `setLocale()` 只持久化用户显式切换的语言，不因为浏览器语言自动改默认。
- 保持“requestedLocale / active locale”分离，避免切换过程中半语言状态。
- 增加 `isSwitchingLocale`，让切换器可以显示轻量 loading，但页面继续使用旧语言直到新 catalog 激活完成。
- 对 app / admin / cloud-console / site 分 surface 保存偏好，互不干扰。
- 加 `useRuntimeTranslator()`：
  - 内部消费 `useAppLocale().activationVersion` 或 `useLingui()`。
  - 返回 `t(messageDescriptor)`。
  - 禁止新代码在 React 组件中直接用模块级 `translateRuntimeMessage` 作为唯一翻译入口。

### Static UI Copy

目标：

- 所有静态 UI 文案进入 Lingui extraction。
- 所有配置数组存 descriptor，不存已翻译字符串。

推荐类型：

```ts
import type { MessageDescriptor } from "@lingui/core";

type AppCopy = MessageDescriptor;

type AppNavItem = {
  label: AppCopy;
  shortLabel: AppCopy;
  ariaLabel?: AppCopy;
};
```

推荐使用：

```ts
const tabs = [
  { to: "/tabs/chat", label: msg`消息`, shortLabel: msg`消息` },
  { to: "/tabs/contacts", label: msg`通讯录`, shortLabel: msg`通讯录` },
] as const;
```

组件里：

```tsx
const t = useRuntimeTranslator();
<span>{t(tab.shortLabel)}</span>
```

### Localized Data

一些前台数据不是纯 UI，也不是用户内容，而是产品配置：

- `features/mini-programs/mini-programs-data.ts`
- `features/games/game-center-data.ts`
- discover 入口配置
- search quick links
- chat background catalog
- sticker / color / category labels
- legal / onboarding 本地结构化文案

这些不适合散落在 JSX 里。规划统一引入：

```ts
type LocalizedText = Partial<Record<SupportedLocale, string>> & {
  "zh-CN": string;
};
```

并提供：

```ts
resolveLocalizedText(value, locale, fallbackLocale = "zh-CN")
```

使用规则：

- UI 组件配置：优先 `MessageDescriptor`。
- 产品静态数据：用 `LocalizedText`。
- 后端返回的系统配置：长期也迁移为 `LocalizedText` 或返回 stable key + params。
- 用户生成内容：普通 string，不自动本地化。

### Runtime Errors

当前前台大量直接显示 `error.message`。规划拆分：

- API / contracts 层返回 `code + params + fallbackMessage`。
- 前台统一 `resolveApiErrorMessage(error)`：
  - 已知 code 用 `msg` 翻译。
  - 未知 code 显示本地化兜底，如“操作失败，请稍后再试。”
  - 调试详情只在开发模式或诊断页展示。

这能避免后端中文错误透到英文/日文/韩文 UI 中。

### Formatting

替换所有硬编码 locale：

- `new Intl.DateTimeFormat("zh-CN", ...)`
- `.toLocaleDateString("zh-CN", ...)`
- `.toLocaleTimeString("zh-CN", ...)`
- `.localeCompare(..., "zh-CN")`

统一改为：

- `formatDateTime(value, options)`
- `formatNumber(value, options)`
- `compareByLocale(left, right, options)`
- 对联系人排序如需中文拼音优先，可新增 `compareContacts(left, right, locale)`，中文用 pinyin collator，其他语言用 locale collator。

### Responsive Layout

多语言后最容易坏的 UI：

- 移动底部 tab。
- 桌面左侧窄导航。
- 顶栏、设置页行项目。
- 聊天输入工具栏和上下文菜单。
- 卡片 badge、pill、segmented controls。
- 游戏/小程序/搜索卡片里的标题与短描述。

布局规则：

- 导航使用 `shortLabel`，详细说明放 `aria-label` 或 tooltip。
- `flex` 子项默认加 `min-w-0`。
- 操作按钮允许换行或使用图标优先。
- badge / pill 使用 `max-w-full`、`break-words`，不要固定中文宽度。
- 固定格式控件先给稳定尺寸，避免语言切换时跳动。
- 移动底部 tab 最多 2 行文字，超过用 short label，不用挤压 icon。

## 新功能开发规则

新增页面或组件时：

- JSX 文案用 `<Trans>`。
- JS/TS 逻辑文案用 `msg` descriptor。
- 配置数组里的 label、title、description 用 descriptor，不用 string。
- 静态业务数据用 `LocalizedText`。
- 用户生成内容和角色名保持 string。
- 日期、数字、排序全部走 `@yinjie/i18n` helper。
- 错误提示不要直接显示 `error.message`，先过 `resolveApiErrorMessage()`。
- 新增路由必须加入前台 i18n browser matrix。

禁止模式：

```ts
label: "消息"
placeholder="搜索联系人"
new Intl.DateTimeFormat("zh-CN")
name.localeCompare(other, "zh-CN")
setError("保存失败")
```

允许模式：

```ts
label: msg`消息`
placeholder={t(msg`搜索联系人`)}
formatDateTime(date, options)
compareByLocale(left, right)
setError(t(msg`保存失败，请稍后再试。`))
```

## 推荐落地顺序

1. 先做 Phase 0 + Phase 1，解决切换入口、默认中文、偏好保存、导航和门禁。
2. 然后做 Phase 2，优先聊天主链路。
3. 再按页面使用频率做 Phase 3 / 4 / 5。
4. 最后做 Phase 6，把前后端系统文案和内容语言打通。

这样做的好处是：

- 第一阶段后，新功能已经会被门禁约束。
- 后面每完成一个业务域，就能把该域加入浏览器矩阵。
- 不需要一次性翻完 377 个文件，风险可控。
- 默认中文和跨语言切换稳定性先被固定住。
