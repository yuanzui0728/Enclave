# 朋友圈发图发视频微信化深度规划

## 任务目标
- 让隐界朋友圈从当前“纯文本发帖”升级为接近微信当前处理方式的媒体发布能力。
- 首期必须支持：
  - 文本朋友圈
  - 1~9 张图片朋友圈
  - 1 条视频朋友圈
  - 为后续实况照片保留协议与渲染位
- 规划要同时覆盖：
  - `api/` 后端实体、接口、存储
  - `packages/contracts/` 合同
  - `apps/app/` 的移动端与桌面端发帖、预览、时间线渲染
  - Android / iOS 壳层媒体选择能力

## 结论先行

### 产品结论
- 朋友圈发帖类型收口为 4 种：
  - `text`
  - `image_album`
  - `video`
  - `live_photo`
- 首期不做“普通图片 + 普通视频混发”。
- 首期按微信主流心智做成：
  - 图片帖：最多 9 张，宫格展示
  - 视频帖：仅 1 条，封面 + 播放入口
  - 文本可与图片/视频共存
- 移动端发布入口要改成独立发布页，不再继续把发帖表单内嵌在朋友圈时间线顶部。
- 桌面端保留发布面板，但要升级为可选图/视频的完整编辑器，不再只支持文本。

### 工程结论
- 不建议直接复用 `/api/chat/attachments` 作为朋友圈对外协议。
- 应新增 `moments media` 专属上传接口，但底层可复用聊天附件已有的文件落盘能力。
- 不建议首期引入 ffmpeg / 服务端转码链路。
- 首期改为：
  - 图片元数据、压缩、预览由客户端处理
  - 视频时长、尺寸、封面图由客户端先提取
  - 服务端只做 MIME / 大小 / 数量校验、落盘、公开 URL 返回
- 角色自动点赞 / 评论逻辑先兼容媒体帖，但不在首期强行让 AI 真看图。

## 二、微信当前能力基线

### 1. 已确认的当前能力
- `2025-03-11`，IT 之家转引“微信派”确认：朋友圈支持发布 `5 分钟视频`，发朋友圈时从相册选择超过 `30 秒` 的视频，不再被强制裁到 `30 秒`，并且可以“添加表情、文字、配乐”等处理。
- `2024-09-19`，“微信派”官宣朋友圈实况照片正式推出，先覆盖 iOS `8.0.51+`，并支持静态照片与实况照片混合发布。
- `2025-04-24`，IT 之家确认安卓微信 `8.0.58`：安卓朋友圈可发实况图。
- `2025-04-01`，微信 PC `4.0.3` 转正，PC 端可发表朋友圈、可查看朋友的朋友圈相册，且 IT 之家描述“界面布局和手机端如出一辙，操作亦一模一样”。

### 2. 由这些口径可以稳定抄到的行为
- 图片帖与视频帖都已经是微信当前正式产品能力，不是概念验证。
- 视频帖的主约束已放宽到 `5 分钟`，这应该成为隐界的目标上限。
- 实况照片已经不是 iOS 独占能力，协议上不能只按“静态图片”来建模。
- PC 端发朋友圈不应是“桌面阉割版纯文本发帖”，而应尽量贴近手机端发布体验。

### 3. 当前没有可靠核实到的能力
- 我没有找到足够可靠的当前官方口径，来确认“微信普通图片与普通视频是否已经支持同一条朋友圈混发”。
- 这点不能靠印象做产品决策。

### 4. 因此本次规划的保守决定
- 这是基于现有资料做出的推断：
  - 首期不要支持“普通图片 + 普通视频混发”
  - 继续保持“图集帖”和“视频帖”互斥
  - 但协议设计要允许未来扩展，不把自己锁死在单一媒体结构里

## 三、目标产品模型

### 1. 朋友圈帖子类型
```ts
type MomentContentType =
  | "text"
  | "image_album"
  | "video"
  | "live_photo";
```

### 2. 首期发布规则
- `text`
  - 仅文本
- `image_album`
  - 1~9 张图片
  - 可附带文本
- `video`
  - 仅 1 个视频
  - 可附带文本
- `live_photo`
  - 逻辑上归属于图片帖的一种扩展形态
  - 首期协议保留，渲染可先按静图退化

### 3. 微信式交互边界
- 图片与视频是两个不同的发布路径，不在同一条帖里混搭。
- 图片帖按宫格流渲染，不自动转视频、不做视频式全屏沉浸播放。
- 视频帖不在朋友圈主流里自动连播，不复用视频号体验。
- 文案与媒体是并列内容，不是“只能二选一”。

## 四、协议与数据结构设计

### 1. Moment 合同需要升级
- 目标文件：
  - `packages/contracts/src/moments.ts`
  - `packages/contracts/src/client.ts`

### 2. 建议新增的媒体结构
```ts
export interface MomentImageAsset {
  id: string;
  kind: "image";
  url: string;
  thumbnailUrl?: string;
  mimeType: string;
  fileName: string;
  size: number;
  width?: number;
  height?: number;
  livePhoto?: {
    enabled: boolean;
    motionUrl?: string;
  };
}

export interface MomentVideoAsset {
  id: string;
  kind: "video";
  url: string;
  posterUrl?: string;
  mimeType: string;
  fileName: string;
  size: number;
  width?: number;
  height?: number;
  durationMs?: number;
}

export type MomentMediaAsset = MomentImageAsset | MomentVideoAsset;
```

### 3. Moment 主体建议扩展为
```ts
export interface Moment {
  id: string;
  authorId: string;
  authorName: string;
  authorAvatar: string;
  authorType: "user" | "character";
  text: string;
  location?: string;
  contentType: MomentContentType;
  media: MomentMediaAsset[];
  postedAt: string;
  likeCount: number;
  commentCount: number;
  likes: MomentLike[];
  comments: MomentComment[];
  interactions: MomentInteraction[];
}
```

### 4. CreateUserMomentRequest 建议扩展为
```ts
export interface CreateUserMomentRequest {
  text: string;
  location?: string;
  contentType?: MomentContentType;
  media?: MomentMediaAsset[];
}
```

### 5. 为什么不用单个 `mediaUrl`
- 朋友圈图片不是单图场景，必须支持多张。
- 视频需要 `durationMs`、`posterUrl` 之类的派生元数据。
- 实况照片未来需要静图与动态资源关联。
- 所以 Moment 必须从“单媒体字段”升级为“媒体数组 + 内容类型”。

## 五、数据库与后端存储设计

### 1. 不新增表，先扩展 `moment_posts`
- 当前 `api/` 使用 TypeORM `synchronize: true`，且项目一直在旧表上做兼容扩展。
- 这次优先延续现有做法，不新建单独媒体表。

### 2. `MomentPostEntity` 建议新增字段
```ts
@Column({ default: "text" })
contentType: string;

@Column("text", { nullable: true })
mediaPayload?: string;
```

### 3. 为什么先用 `mediaPayload`
- SQLite + TypeORM 下最省改造成本。
- 朋友圈媒体是帖子的内聚内容，不需要独立生命周期时，不值得先建关系表。
- 后续如果真的出现：
  - 视频转码任务
  - 审核状态
  - 草稿资产复用
  - CDN / 对象存储迁移
  再把媒体抽表也来得及。

### 4. 服务端运行时的转换规则
- 数据库存储：
  - `contentType`
  - `mediaPayload: JSON.stringify(media)`
- 对外返回：
  - `mediaPayload -> MomentMediaAsset[]`
- 数据兼容：
  - 旧帖默认 `contentType = "text"`
  - `media = []`

## 六、媒体上传接口设计

### 1. 建议新增独立 moments media 路由
- `POST /api/moments/media`
- `GET /api/moments/media/:fileName`

### 2. 不建议继续直接暴露 `/api/chat/attachments`
- 聊天附件和朋友圈媒体是两套产品语义：
  - 生命周期不同
  - 命名不同
  - 未来审计与清理策略不同
- 但底层存储 helper 可以共用。

### 3. 建议新增的底层实现文件
- `api/src/modules/moments/moment-media.storage.ts`
- `api/src/modules/moments/moment-media.types.ts`
- `api/src/modules/moments/moment-media.utils.ts`

### 4. 上传接口的首期规则
- 图片：
  - 单次上传 1 张
  - MIME 仅允许常见图片格式
  - 客户端循环上传后组装为图集
- 视频：
  - 单次上传 1 个视频
  - 可选附带 `poster` 元数据或单独封面文件
  - 首期允许客户端先提取封面，再一起提交

### 5. 服务端校验建议
- 图片：
  - 单图大小上限：建议 `20 MB`
  - 单帖数量上限：`9`
- 视频：
  - 单视频大小上限：建议 `200 MB`
  - 单帖数量上限：`1`
  - 时长上限：`300000 ms`
- 兜底限制：
  - 文本长度继续沿用 `600`
  - 同时提交图片和视频直接拒绝

### 6. 存储目录建议
- 新目录：`data/moments-media`
- 对外公开路径：`/api/moments/media/:fileName`

## 七、媒体处理策略

### 1. 图片处理策略
- 首期走客户端压缩与尺寸读取，服务端不做再处理。
- Web / 桌面：
  - 读取图片尺寸
  - 需要时在浏览器做缩边压缩
- Android / iOS 壳：
  - 原生桥接只负责选图 / 拍照
  - 压缩仍可由 Web 层完成

### 2. 视频处理策略
- 首期不上服务端转码。
- 客户端负责：
  - 读取 `durationMs`
  - 读取宽高
  - 抓取首帧作为 `poster`
- 服务端负责：
  - 校验时长 / MIME / 大小
  - 原样落盘
- 这样做的原因：
  - 仓内当前没有 ffmpeg / ffprobe / 媒体转码链
  - 首期若强上转码，会把需求从“朋友圈媒体发送”拖成“多媒体基础设施改造”

### 3. 第二阶段再考虑的增强
- 服务端转码到统一 H.264 / AAC
- 视频多码率
- 服务端抽帧
- 视频审核
- 上传中断续传

## 八、移动端产品改造

### 1. 当前问题
- `apps/app/src/routes/moments-page.tsx` 现在把发帖表单直接放在时间线顶部。
- 这不接近微信手机端的朋友圈发布流程。

### 2. 目标形态
- 移动端改为独立发布页：
  - 建议新增路由：`/discover/moments/compose`
- 入口：
  - 朋友圈页右上角发布按钮
  - 进入独立编辑页，而不是滚动到顶部输入框

### 3. 独立发布页建议结构
- 顶部：
  - 取消
  - 标题“朋友圈”
  - 发布
- 内容区：
  - 文本输入区
  - 已选媒体预览区
  - 添加图片 / 添加视频入口
  - 可见范围提示“朋友可见”

### 4. 微信式图片帖交互
- 已选图片显示为 3 列宫格。
- 最后一个格子保留“继续添加”。
- 支持：
  - 删除单张
  - 调整顺序
  - 点击预览
- 数量达到 9 张后隐藏继续添加入口。

### 5. 微信式视频帖交互
- 选择视频后显示：
  - 封面
  - 时长
  - 删除按钮
- 同时隐藏图片入口，避免图视频混发。
- 如果视频超过 5 分钟，直接在编辑页阻断发布。

### 6. 发布时序
1. 本地校验文字、数量、时长、大小。
2. 先上传图片或视频资产。
3. 收到标准化 `MomentMediaAsset`。
4. 再调用 `createUserMoment(...)`。
5. 成功后回到时间线并插到顶部。

## 九、桌面端产品改造

### 1. 当前问题
- `desktop-moment-compose-panel.tsx` 仍是纯文本说明牌。
- 和微信 PC `4.0.3` 已能发朋友圈的现状不匹配。

### 2. 目标形态
- 保留右侧发布面板这一交互壳，但内容升级为完整媒体编辑器。
- 首期不必为了“更像微信”强行改成新窗口，只要能力与信息结构到位即可。

### 3. 桌面端编辑器结构
- 头部：
  - 发朋友圈
  - 关闭
- 主区：
  - 当前用户头像与昵称
  - 文本输入
  - 已选媒体预览
  - “添加图片” / “添加视频”
- 底部：
  - 朋友可见
  - 字数统计
  - 发布按钮

### 4. 桌面端文件选择
- 图片：
  - `<input type="file" accept="image/*" multiple />`
- 视频：
  - `<input type="file" accept="video/*" />`
- 与移动端保持相同约束：
  - 图集与视频互斥
  - 图集最多 9 张
  - 视频仅 1 条

## 十一、时间线渲染设计

### 1. 移动端朋友圈卡片
- 文案在前，媒体在后，时间 / 地点 / 操作在底部。
- 图片帖：
  - `1 张`：大图
  - `2~4 张`：2 列
  - `5~9 张`：3 列
- 视频帖：
  - 封面图 + 播放按钮 + 时长角标

### 2. 桌面端朋友圈卡片
- `desktop-moment-row.tsx` 新增媒体区块：
  - 文本下方
  - 点赞评论区上方
- 视频在桌面端不自动播放，只展示封面，点击后进入详情或播放器。

### 3. 全屏预览
- 图片：
  - 移动端做轻量全屏滑动预览
  - 桌面端做 modal viewer
- 视频：
  - 移动端点击进入播放器态
  - 桌面端详情面板内可播放，必要时再补全屏

### 4. 实况照片渲染策略
- Phase 1：
  - 若有 `livePhoto.enabled`，卡片显示“实况”角标
  - 不支持动态播放的端退化为静图
- Phase 2：
  - 点击或长按播放 motion 资源

## 十二、角色互动与 AI 兼容

### 1. 当前问题
- `moments.service.ts` 里的角色评论 prompt 只引用 `post.text`。
- 如果用户发“只有图 / 只有视频”的朋友圈，角色评论会失真。

### 2. 首期兼容方案
- 新增 `buildMomentPromptSummary(post)`：
  - 文本为空时，至少生成：
    - `用户发了一条包含 3 张图片的朋友圈`
    - `用户发了一条时长 42 秒的视频朋友圈`
- AI 角色自动评论先依据“媒体类型 + 数量 + 文案”生成，不要求模型首期真看图。

### 3. 后续增强
- 若后端后续已经具备稳定多模态能力，可把朋友圈媒体 URL 喂给模型。
- 但这不应阻塞本轮媒体发布能力落地。

## 十五、验收标准
- 用户可以在桌面端发布：
  - 文本
  - 1~9 张图片
  - 1 条视频
- 用户可以在移动端发布：
  - 文本
  - 1~9 张图片
  - 1 条视频
- 朋友圈时间线能够正确渲染：
  - 图片宫格
  - 视频封面
  - 点击预览
- 图集与视频不能混发。
- 视频超过 `5 分钟` 时，发布前就被明确阻断。
- 没有媒体的旧朋友圈仍然正常显示。
- 角色自动评论在纯媒体帖下不会退化成明显错误的文案。

## 十六、风险与取舍

### 1. 最大风险
- 视频媒体链路如果首期强上服务端转码，会显著放大工作量与稳定性风险。

### 2. 本次规划的取舍
- 先把“能发、能看、像微信”做出来。
- 不把第一轮目标扩成：
  - CDN
  - ffmpeg 转码
  - 审核服务
  - 混合媒体帖
  - 视频自动播放算法

### 3. 明确不纳入首期
- 普通图片 + 普通视频同贴混发
- 朋友圈评论区图片回复
- AI 角色自动生成带图朋友圈
- 服务端视频转码与多码率
