# Web 端聊天页背景图功能规划

## Summary
目标是在 `apps/app` 的 Web 手机版和桌面版聊天页，为聊天内容区补齐“微信式聊天背景图”能力，支持两层配置：

- 默认背景图：作为当前世界的全局聊天背景，作用于所有未单独设置的会话
- 好友专属背景图：仅单聊可设置，优先级高于默认背景图

首版建议同时支持：

- 预置背景图
- 本地上传自定义背景图
- 默认背景图与好友专属背景图的预览、保存、清除
- 移动端与桌面端共用同一套背景配置模型

首版不建议做：

- 群聊专属背景图入口
- 背景图跟随时间自动切换
- 云端背景图库管理、收藏夹、历史上传列表
- 背景图同步到原生壳层系统壁纸

## 1. 对齐微信的产品逻辑

### 1.1 配置层级
建议对齐微信的核心心智，明确三层优先级：

1. 当前好友专属背景图
2. 默认背景图
3. 系统兜底背景

也就是：

- 某个好友设置了专属背景图时，只影响这一个单聊
- 当前单聊未设置专属背景图时，回落到默认背景图
- 默认背景图也没有设置时，回落到系统内置背景

### 1.2 配置行为
- “设置默认背景图”是全局行为，保存后立即影响所有未设置专属背景的会话
- “设置好友专属背景图”是局部行为，不反向修改默认背景图
- “清除好友专属背景图”后，该会话自动恢复使用默认背景图
- “清除默认背景图”后，所有未设置专属背景的会话恢复系统背景

### 1.3 数据语义
- 背景图是展示偏好，不属于聊天消息
- 不进入 AI prompt，不影响会话排序、未读数、置顶、清空聊天记录
- 清空聊天记录、隐藏会话、重新打开会话时，背景设置应保留

## 2. 当前现状结论

### 2.1 前端现状
- `apps/app/src/features/chat/conversation-thread-panel.tsx`
  - 是移动端和桌面端共用的消息线程主容器
  - 当前聊天区背景是硬编码渐变，最适合承接“统一背景层”
- `apps/app/src/routes/chat-room-page.tsx`
  - 已根据 `useDesktopLayout()` 分流移动/桌面壳层
- `apps/app/src/features/desktop/chat/desktop-chat-workspace.tsx`
  - 桌面端已有右侧 `inspector` 区，适合作为“聊天背景”桌面入口
- `apps/app/src/components/chat-message-list.tsx`
  - 消息气泡与消息列表已相对稳定，适合只在容器层叠加背景图，不改消息协议
- `apps/app/src/components/chat-composer.tsx`
  - 已有移动端和桌面端两套输入区壳层，可配合背景图改成半透明面板

### 2.2 后端与契约现状
- `api/src/modules/chat/conversation.entity.ts`
  - 目前只有 `pin / hide / clear / activity` 等会话状态，没有背景配置字段
- `api/src/modules/auth/user.entity.ts`
  - 当前保存世界主人资料与专属 API Key，可承接“默认背景图”这类 owner 级偏好
- `api/src/modules/world/world.controller.ts`
  - 已暴露 `GET /api/world/owner` 与 `PATCH /api/world/owner`
  - 默认背景图可以沿着这条 owner 配置链路扩展
- `api/src/modules/chat/chat.controller.ts`
  - 当前会话接口边界清晰，适合增加 `conversation background` 相关接口
- `api/src/app.module.ts`
  - `TypeORM synchronize: true`，SQLite 场景下新增字段落地成本较低

### 2.3 资源链路现状
- 当前仓库没有现成的图片上传、背景资源管理、静态资源映射
- 但 `api/src/modules/ai/ai.controller.ts` 已经在用 `FileInterceptor`
- 结论：
  - 自定义背景图上传不是零基础问题
  - 仍需单独补一条“背景图上传 + 资源访问”链路

## 3. 产品范围

### 3.1 首版必须有
- Web 手机版聊天页支持查看与应用背景图
- Web 桌面版聊天页支持查看与应用背景图
- 默认背景图设置
- 单聊好友专属背景图设置
- 预置背景图库
- 本地上传图片并裁成聊天背景可用资源
- 清除默认背景图
- 清除好友专属背景图
- 聊天页即时预览生效

### 3.2 首版不做
- 群聊专属背景图入口
- 多张背景轮播
- 背景图分享给其他实例
- 按好友标签批量设置背景
- 复杂裁剪编辑器、滤镜、贴纸

## 4. 推荐产品方案

### 4.1 入口设计

#### Web 手机版
建议从当前聊天页右上角 `更多` 进入一个独立的“聊天背景”设置页。

原因：
- 小屏下不适合在聊天页内直接铺满配置面板
- 更接近微信“当前聊天信息 -> 聊天背景”的操作路径
- 可以复用全屏预览和上传流程

建议入口链路：
- 聊天页右上角 `更多`
- 操作项里增加 `聊天背景`
- 进入 `/chat/$conversationId/background` 独立页面

页面内分两块：
- 默认背景图
- 当前好友专属背景图

如果当前会话不是单聊：
- 只展示默认背景图
- “当前好友专属背景图”区域显示禁用提示：`群聊暂不支持专属背景，当前使用默认背景`

#### Web 桌面版
建议复用 `DesktopChatWorkspace` 右侧 `inspector`，新增“聊天背景”分区。

建议入口链路：
- 右上角 `更多`
- 打开右侧 `inspector`
- 增加“聊天背景”卡片
- 点击后在右侧展开设置面板，或直接在 inspector 中完成选择

原因：
- 桌面端已经有稳定的右侧信息区，不必额外造新页面
- 右侧面板可以直接预览“当前会话效果”

### 4.2 设置页/面板结构
建议统一成两个 section：

- `默认背景图`
  - 说明：应用到所有未设置专属背景的聊天
  - 当前状态预览
  - 操作：从预置选择 / 上传图片 / 恢复系统背景
- `当前好友专属背景图`
  - 说明：仅当前单聊生效，优先级高于默认背景图
  - 当前状态预览
  - 操作：跟随默认 / 从预置选择 / 上传图片 / 清除专属背景

这样用户不需要理解复杂设置层级，只需要看见：
- 默认是“全局”
- 当前好友是“覆盖”

### 4.3 预览方式
- 进入设置页后，顶部始终保留一张聊天场景预览卡
- 切换背景图时预览即时变化，但保存前不写入后端
- 点击“保存”后再提交

不建议首版做“所见即保存”：
- 背景图包含上传和压缩，失败回滚会让体验很乱
- 默认背景图会影响所有会话，最好显式保存

## 5. 视觉与交互细节

### 5.1 背景应用范围
建议首版把背景图应用在“消息滚动区”为主，头部和输入区采用半透明覆盖层。

也就是：
- `ConversationThreadPanel` 根容器保留当前结构
- 消息滚动区域改成“背景图 + 遮罩层”
- Header / Composer 改成带透明度的浅色面板

这样最稳：
- 不会破坏现有消息阅读层级
- 更接近微信消息区铺底图的视觉逻辑
- 移动端和桌面端都容易复用

### 5.2 可读性策略
- 背景图上必须加一层浅色遮罩，避免花图影响阅读
- 消息时间、系统消息、对方气泡、输入框都保持现有高对比风格
- 不建议首版允许用户手动调节模糊度和遮罩强度

推荐默认参数：
- `background-size: cover`
- `background-position: center`
- 遮罩透明度约 `0.16 ~ 0.24`
- 移动端不要使用 `background-attachment: fixed`

### 5.3 上传图片限制
为了避免移动端卡顿，建议首版做前端压缩：

- 最大边长压缩到 `1600px`
- 导出 `webp`
- 单张图片上传后目标体积控制在 `1.2MB` 左右

如果压缩失败，再退回原图上传，但后端应限制最大文件体积，例如 `5MB`。

## 6. 推荐数据模型

## 6.1 系统兜底背景
系统兜底背景不进数据库，继续使用当前聊天页已有渐变即可。

这样没有任何配置时：
- 行为保持现状
- 不需要迁移历史数据

## 6.2 默认背景图
推荐挂在 `WorldOwner` 语义下，而不是单独新建一张表。

原因：
- 当前实例只有一个世界主人
- 这本质上是 owner 级 UI 偏好
- `GET /api/world/owner` 已经是前端拿 owner 配置的标准入口

推荐新增到 `UserEntity` 的字段：

- `defaultChatBackgroundPayload?: string | null`

其中 payload 建议存 JSON：

```ts
type ChatBackgroundPayload = {
  source: "preset" | "upload";
  assetId: string;
  url: string;
  thumbnailUrl?: string;
  width?: number;
  height?: number;
  label?: string;
};
```

## 6.3 好友专属背景图
推荐挂在 `ConversationEntity` 上。

建议新增字段：

- `chatBackgroundMode: "inherit" | "custom"`
- `chatBackgroundPayload?: string | null`

说明：
- `inherit` 表示跟随默认背景图
- `custom` 表示当前会话覆盖默认背景图

这样后续如果要扩展到群聊专属背景图，后端结构不需要重做，只需要开放 UI。

## 6.4 为什么首版不建议上独立资产表
如果这次直接引入 `ChatBackgroundAssetEntity`，会多出：
- 资产表
- 引用关系
- 清理策略
- 复用策略

对当前仓库来说偏重。

首版更适合：
- 上传后生成一个文件
- 返回 `assetId + url`
- 把引用元数据直接写进 `UserEntity` / `ConversationEntity`

如果后续真的需要“背景图库、复用、回收站、历史上传”，再把资源抽成实体表。

## 7. 接口规划

### 7.1 默认背景图接口
建议放在 `world owner` 语义下：

- `PATCH /api/world/owner/chat-background`
- `DELETE /api/world/owner/chat-background`

也建议把返回的 `WorldOwner` 扩展为：

```ts
interface WorldOwner {
  id: string;
  username: string;
  onboardingCompleted: boolean;
  avatar?: string;
  signature?: string;
  hasCustomApiKey: boolean;
  customApiBase?: string | null;
  defaultChatBackground?: ChatBackgroundPayload | null;
  createdAt: string;
}
```

这样前端拿 owner 配置时，顺手就能拿到默认背景图。

### 7.2 当前会话背景接口
建议放在会话接口下：

- `GET /api/conversations/:id/background`
- `PATCH /api/conversations/:id/background`
- `DELETE /api/conversations/:id/background`

推荐返回结构：

```ts
type ConversationBackgroundSettings = {
  mode: "inherit" | "custom";
  conversationBackground?: ChatBackgroundPayload | null;
  defaultBackground?: ChatBackgroundPayload | null;
  effectiveBackground?: ChatBackgroundPayload | null;
};
```

这样聊天页和设置页都不用自己拼优先级。

### 7.3 图片上传接口
建议单独提供上传接口：

- `POST /api/conversations/background-assets`

请求：
- `multipart/form-data`
- `file`

响应：

```ts
type UploadedChatBackground = {
  assetId: string;
  url: string;
  thumbnailUrl?: string;
  width: number;
  height: number;
  label: string;
  source: "upload";
};
```

说明：
- 这条接口只负责上传资源，不直接改默认背景或会话背景
- 保存背景设置仍走 `PATCH`

这样“上传资源”和“应用设置”解耦，失败路径更干净。

### 7.4 预置背景图
预置背景图不需要后端管理，建议走前端静态清单：

- `apps/app/public/chat-backgrounds/...`
- `apps/app/src/features/chat/backgrounds/background-catalog.ts`

预置图只要能稳定拿到 `presetId + url + thumbnailUrl` 即可。

## 8. 资源存储方案

### 8.1 推荐方案
建议服务端把上传后的背景图存到实例本地磁盘，例如：

- `api/storage/chat-backgrounds/<ownerId>/<assetId>.webp`

访问方式建议通过 API 暴露，而不是直接裸静态目录：

- `GET /api/conversations/background-assets/:assetId`

原因：
- 便于后续做鉴权、清理、格式迁移
- 不把底层文件结构直接暴露给前端

### 8.2 清理策略
首版建议只做最小清理：

- 用户重新上传并替换默认背景图时，删除旧默认背景图文件
- 用户重新上传并替换当前会话专属背景图时，删除该会话旧文件
- 清除背景设置时，如果该资源由当前设置独占，则删除文件

先不做全局资产引用计数。

## 11. 分阶段实施建议

### Phase 1：只做预置背景图 + 默认/好友覆盖逻辑
目标：先把最核心的产品逻辑跑通，不碰上传链路。

包含：
- 数据结构与接口骨架
- 预置背景图清单
- 默认背景图设置
- 好友专属背景图设置
- 移动端设置页
- 桌面端 inspector 入口
- 聊天页背景渲染

价值：
- 可以最快验证“默认 + 专属覆盖”的微信式心智是否顺手
- 风险最小

### Phase 2：补自定义上传
目标：让背景图真正可个性化。

包含：
- 前端图片选择与压缩
- 上传接口
- 服务端资源读写
- 替换时清理旧文件

### Phase 3：体验打磨
包含：
- 预览交互优化
- 加载占位图
- 弱网和上传失败提示
- 桌面端 inspector 实时预览
- 移动端页面转场与返回保留草稿

## 12. 验收标准

- Web 手机版聊天页可以设置默认背景图
- Web 手机版聊天页可以为某个单聊设置专属背景图
- Web 桌面版聊天页可以设置默认背景图
- Web 桌面版聊天页可以为某个单聊设置专属背景图
- 专属背景图优先级高于默认背景图
- 清除专属背景图后，会话正确回退到默认背景图
- 清除默认背景图后，未设置专属背景的会话正确回退到系统背景
- 背景图不会影响消息发送、未读数、置顶、清空记录等现有功能
- 移动端和桌面端看到的是同一份后端配置结果

## 13. 风险与决策

### 13.1 最大风险
- 上传图片链路当前不存在，首版一次性把“预置 + 上传 + 默认 + 专属 + 两端入口”全做完，范围会偏大
- 花纹复杂的图片会破坏可读性，所以遮罩层不能省
- 如果直接把背景图字段塞进 `ConversationListItem`，会增加会话列表无意义载荷，不建议

### 13.2 当前建议的关键决策
- 背景图优先级严格定为：`专属 > 默认 > 系统`
- 默认背景图挂在 `WorldOwner`，好友专属背景图挂在 `Conversation`
- 首版先以“消息区铺背景、头部与输入区半透明”方式落地，不做整页完全沉浸式重绘

- 实施顺序优先做 Phase 1，再补上传
