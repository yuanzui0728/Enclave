# 多语言硬编码根治与软化规划

日期：2026-04-23

## 1. 问题定义

当前多语言切换入口和运行时已经接入，但“切换语言没有效果”的根因不是选择器本身，而是大量界面和业务文案没有进入可翻译消息体系。语言切换只能影响已经通过 catalog、hook、格式化工具渲染的内容；页面里直接写死的中文、英文、状态描述、错误消息、数组配置和提示词不会响应 locale。

这次要根治的问题是：

- 新功能默认进入多语言流程，而不是每次事后补翻译。
- 存量硬编码逐步清零，并且迁移过程中不阻塞业务迭代。
- “硬编码文案”软化成可提取、可配置、可传参、可按 locale fallback 的结构化内容。
- 后端、前端、AI、语音、运营配置不再各自写死语言。

## 2. 当前事实

已具备的基础：

- `packages/i18n` 已有 `AppLocaleProvider`、`LanguageSwitcher`、locale 解析、localStorage 持久化、`document.lang/dir` 同步。
- 支持语言为 `zh-CN`、`en-US`、`ja-JP`、`ko-KR`，默认 `zh-CN`。
- `lingui.config.ts` 已按 `shared`、`app`、`admin`、`cloud-console` 分 catalog。
- `package.json` 已有 `i18n:extract`、`i18n:compile`、`i18n:audit:*`、`i18n:check:*`。
- `scripts/i18n-audit.mjs` 已能扫描新增 diff 里的部分硬编码风险。

当前缺口：

- `i18n-audit` 只看 diff，不看全仓存量，所以历史硬编码不会被系统性暴露。
- audit 规则偏窄，主要抓 JSX 字面量 prop、硬编码 `Intl` locale、语音 lang、throw 字面量，抓不到大量对象数组、toast、空态、按钮文本、页面裸文本。
- catalog 允许缺失翻译，导致切到英文/日文/韩文时继续 fallback 中文，看起来像“切换无效”。
- 很多组件 props 被设计成 `string`，调用方自然会继续传硬编码字符串。
- 后端和 API client 仍然大量依赖 `message` 字符串，前端只能原样展示。
- AI/STT/TTS 相关链路仍有 `zh-CN`、`zh` 等业务默认值，UI locale 和内容 locale 没有拆开。

近期扫描口径下的存量硬编码规模：

- `apps/admin/src`：字面量 UI props 约 `1881` 处，中文源码行约 `6086`，`Trans/msg` 用量约 `162`。
- `apps/app/src`：字面量 UI props 约 `1221` 处，中文源码行约 `7017`，`Trans/msg` 用量约 `26`。
- `apps/cloud-console/src`：字面量 UI props 约 `49` 处。
- `packages/ui/src`：UI prop 硬编码基本没有，但仍有少量中文源码。
- 管理后台 catalog 仍有缺失翻译：`en-US` 约 `49`，`ja-JP` 约 `48`，`ko-KR` 约 `48`。

高风险存量页面集中在：

- 管理后台：`reply-logic-page`、`need-discovery-page`、`cyber-avatar-page`、`action-runtime-page`、`reminder-runtime-page`、`token-usage-page`、`wechat-sync-page`、`real-world-sync-page`、`dashboard-page`、`evals-page`。
- 主 App：`desktop-chat-details-panel`、`chat-message-list`、`group-chat-details-page`、`character-detail-page`、`desktop-mobile-page`、`desktop-search-workspace`、`chat-composer`。

## 3. 根因拆解

### 3.1 运行时已存在，但调用方没有进入运行时

语言切换只改变 `locale` 和已加载 messages。硬编码 JSX、数组配置、函数返回字符串、后端 `message` 都不会重新翻译。

### 3.2 组件契约鼓励硬编码

大量组件接收 `title: string`、`description: string`、`label: string`，使调用方只能先把文案算成字符串。正确方向是让通用组件接受 `ReactNode` 或 `MessageDescriptor`，业务层用 `<Trans>`、`msg`、`i18n._()` 传入。

### 3.3 差异扫描无法治理存量

只查新增 diff 的好处是不会一次性挡住全仓，但坏处是历史页面永远可以保持“不可切换”。需要引入“全量扫描 + baseline 递减”的 ratchet 机制。

### 3.4 缺失翻译没有被当成失败

`extract/compile` 能跑过不代表多语言完整。非源语言缺 `msgstr` 时会 fallback 到中文，必须在发布和被改动 surface 上把缺失翻译当失败。

### 3.5 后端错误协议还是字符串协议

后端直接抛中文，前端直接展示 message，会让所有客户端、管理后台、云控制台都被服务端源码语言绑定。

### 3.6 运营内容、种子数据、AI 提示词没有 locale 结构

游戏目录、角色预设、系统配置说明、AI prompt、语音识别语言如果仍是单一字符串，就算 UI 翻译完整，用户仍会看到混合语言。

## 4. 目标架构

### 4.1 文案入口统一

新增或强化 `packages/i18n` 的职责：

- `useT()`：业务代码统一拿 `i18n._`、当前 locale、格式化工具，避免每个页面直接接 Lingui 细节。
- `defineSurfaceMessages()`：页面级、模块级文案集中声明，适合列表、状态、tab、菜单、表格列。
- `LocalizedText` 工具：读取持久化多语言内容，并按 `当前 locale -> 默认 locale -> 任意可用值 -> fallback` 解析。
- `error copy registry`：客户端根据后端 `code + params` 显示本地化错误。
- `formatters`：所有日期、数字、相对时间、排序、语音语言映射统一从 locale 解析。

建议新增共享类型：

```ts
export type LocalizedText = {
  defaultLocale: SupportedLocale;
  values: Partial<Record<SupportedLocale, string>>;
};

export type ApiErrorPayload = {
  code: string;
  messageKey?: string;
  params?: Record<string, string | number | boolean | null>;
  fallbackMessage?: string;
};
```

### 4.2 UI 文案分三类处理

页面内静态文案：

- 使用 `<Trans>` 或 `t/msg`。
- 禁止裸中文、裸英文按钮、标题、说明直接出现在 TSX。

配置型 UI 文案：

- tabs、cards、filters、status maps、table columns、nav items 统一放到模块 copy registry。
- registry 存 message descriptor，不存最终字符串。

跨组件传递文案：

- 通用组件 props 从 `string` 扩展为 `ReactNode`。
- 必须用于 `aria-label`、`placeholder` 等 DOM 字符串时，由调用方通过 `i18n._(msg...)` 转成字符串。

### 4.3 后端错误软化

后端不再把用户可见文案作为主协议返回：

- service/controller 抛业务错误时使用稳定 `code`。
- `params` 承载动态变量。
- `fallbackMessage` 仅用于日志、兼容旧客户端和调试，不作为前端主展示来源。
- 前端 `contracts/client`、`admin-api` 统一解析 `ApiErrorPayload`，再走 `error copy registry`。

例子：

```ts
throw new BadRequestException({
  code: "CHAT.MESSAGE_RECALL_FORBIDDEN",
  params: { windowMinutes: 2 },
  fallbackMessage: "只能撤回自己发送的消息。",
});
```

客户端显示：

```ts
i18n._(apiErrorMessages.CHAT_MESSAGE_RECALL_FORBIDDEN, params)
```

### 4.4 数据和运营配置软化

需要迁移为 `LocalizedText` 的内容：

- 游戏目录标题、简介、标签、审核说明。
- 角色预设名称、描述、安装说明。
- 系统配置项名称、帮助文本、诊断提示。
- 管理后台规则说明、候选原因、运行记录中需要展示给人的解释。
- 云世界申请状态说明、邮件/短信模板中需要展示的部分。

迁移方式：

- 数据库保留原字段兼容读取。
- 新增 `xxxI18n` 或 `localizedPayload` JSON 字段。
- 读取时统一用 `resolveLocalizedText()`。
- 后台编辑器允许先只填默认语言，后续由自动翻译或人工补齐其他语言。

### 4.5 AI、语音和内容语言拆分

必须区分：

- `uiLocale`：界面语言，控制按钮、导航、错误、格式化。
- `contentLocale`：AI 回复、角色动态、朋友圈生成、系统播报的目标语言。
- `speechLocale`：STT/TTS 语言，可由 `contentLocale` 映射到 provider 需要的代码，比如 `zh-CN -> zh`。

默认策略：

- 新用户：`uiLocale` 由 query/localStorage/browser/default 决定。
- 世界主人：增加 `preferredLocale` 与 `contentLocale` 后，App 内持久化到 owner profile。
- Admin：默认本地存储，也可后续支持实例级 admin locale。
- AI prompt：模板里不要写死“用中文回复”，改为注入 `contentLocale` 和语言名称。

## 5. 自动适配机制

目标是新功能开发时天然落入多语言系统：

1. 页面写 `<Trans>` 或 `msg`，不能写裸字符串。
2. `pnpm i18n:extract` 自动把新增消息提取到对应 catalog。
3. `pnpm i18n:verify-catalogs` 确认 catalog 已提交。
4. `pnpm i18n:compile` 确认运行时语言包可构建。
5. `pnpm i18n:audit:changed` 阻止新增硬编码。
6. `pnpm i18n:missing:changed` 阻止被改动 surface 出现缺失翻译。
7. 全量 hardcode baseline 只允许下降，不允许上升。

后续可以接 AI 草拟翻译，但不把 AI 翻译作为唯一可信来源：

- 新增脚本 `pnpm i18n:translate:draft --surface admin --locale en-US`。
- 只填空 msgstr，不覆盖人工翻译。
- 生成后仍需 `extract/compile/missing` 验证。

## 6. 审计与 CI 设计

### 6.1 新增全量扫描

新增 `scripts/i18n-hardcode-scan.mjs`：

- 支持 `--scope apps/admin/src`、`--scope apps/app/src`、`--all`。
- 支持 `--json` 输出，便于生成 baseline。
- 扫描全文件，而不是只看 diff。
- 支持 allowlist 注释，比如 `// i18n-ignore-next-line: protocol key`。

规则覆盖：

- JSX 文本节点里的 CJK 文案。
- JSX prop：`title`、`label`、`description`、`placeholder`、`aria-label`、`message`、`subtitle`、`helperText`、`emptyText`、`confirmText`。
- 对象字面量字段：`title`、`label`、`name`、`description`、`hint`、`summary`、`cta`、`statusLabel`。
- `toast.*("...")`、`alert("...")`、`confirm("...")`、`prompt("...")`。
- `setError("...")`、`throw new Error("...")`、NestJS `*Exception("...")`。
- `Intl.*("...")`、`localeCompare(..., "...")`、`recognition.lang = "..."`。
- AI prompt 常量中的语言指令：`中文`、`English`、`请用...回复`。

### 6.2 引入 baseline ratchet

新增 `scripts/i18n-hardcode-baseline.json`：

```json
{
  "apps/admin/src": {
    "jsxLiteralProp": 1881,
    "rawCjkLine": 6086
  },
  "apps/app/src": {
    "jsxLiteralProp": 1221,
    "rawCjkLine": 7017
  },
  "apps/cloud-console/src": {
    "jsxLiteralProp": 49
  }
}
```

执行规则：

- 存量可以先保留，但不能增加。
- 每迁移一个页面，更新 baseline 让阈值下降。
- 新增文件默认阈值为 0，必须完全 i18n 化。
- 被改动文件如果触碰 UI 文案，按 changed audit 严格执行。

### 6.3 缺失翻译检查

新增 `scripts/i18n-missing-check.mjs`：

- 检查 `.po` 中非源语言空 `msgstr`。
- 支持 `--surface admin`、`--changed`。
- 发布构建要求目标 surface 无 missing。
- 开发阶段至少要求“本次新增/修改消息无 missing”。

### 6.4 PR 和本地命令口径

短期：

- `pnpm i18n:check:changed`：开发必跑。
- `pnpm i18n:hardcode:ratchet`：CI 必跑，但只防止变差。
- `pnpm i18n:missing:changed`：CI 必跑，防止新增缺译。

长期：

- `pnpm i18n:check` 必须覆盖全量 hardcode 和全量 missing。
- `build` 之前强制 compile catalog。

## 7. 分阶段执行计划

### 阶段 0：把问题变成可度量

交付：

- 新增全量 hardcode scanner。
- 生成 baseline。
- 新增 missing scanner。
- 更新 `package.json` i18n scripts。
- 写清 allowlist 规则，避免协议 key、CSS class、测试 fixture 被误杀。

验收：

- 能输出每个 surface 的硬编码数量、top 文件、规则分布。
- CI 能防止新增硬编码和新增缺译。
- 不要求存量一次清零。

### 阶段 1：先软化共享组件契约

交付：

- 管理后台共享布局组件接受 `ReactNode` 或 message descriptor。
- App 侧通用组件的 `title/description/emptyText/actionLabel` 逐步从 `string` 放宽。
- `packages/ui` 保持无业务文案，必要默认文案走 shared catalog。

优先组件：

- Admin：`AdminPageHero`、`AdminSectionHeader`、`AdminMetricCard`、`AdminEmptyState`、`AdminCallout`、表格列定义、筛选控件。
- App：聊天空态、设置项、列表项、弹窗、composer、desktop shell panel。

验收：

- 新页面不需要先拼字符串再传组件。
- 共享组件不会强迫调用方硬编码。

### 阶段 2：管理后台高风险页面清理

优先级按硬编码密度和用户可见程度排序：

1. `reply-logic-page`
2. `need-discovery-page`
3. `cyber-avatar-page`
4. `action-runtime-page`
5. `reminder-runtime-page`
6. `token-usage-page`
7. `wechat-sync-page`
8. `real-world-sync-page`
9. `dashboard-page`
10. `evals-page`

迁移规则：

- 页面标题、hero、section、按钮、表单、空态、toast 全部进 catalog。
- 状态 map 改为 message descriptor map。
- 表格列 title 改为 descriptor。
- API 错误展示走 error registry。

验收：

- 管理后台切换语言后，shell、导航、hero、主要 section、按钮、空态、toast 都切换。
- `admin` catalog 非源语言 missing 清零。
- `apps/admin/src` hardcode baseline 按页面持续下降。

### 阶段 3：主 App 用户高频路径清理

优先级：

1. `chat-message-list`
2. `chat-composer`
3. `desktop-chat-details-panel`
4. `group-chat-details-page`
5. `character-detail-page`
6. `profile/settings`
7. `contacts/discover/profile` 关键入口
8. 移动端二级页入口和空态

迁移规则：

- 聊天操作、菜单、撤回/删除/收藏提示、发送失败错误必须本地化。
- 相对时间和日期展示走 `formatters`。
- 语音识别语言从 locale runtime 解析。
- AI 通话 UI 文案和错误进入 app catalog。

验收：

- Web、桌面壳、Android 壳、iOS 壳共用的 App 界面切换语言可见变化。
- 聊天核心路径不再出现硬编码中文按钮和错误。

### 阶段 4：后端错误协议改造

交付：

- `packages/contracts` 新增 `ApiErrorPayload` 和错误码类型。
- `api` 新增业务异常 helper，例如 `throwBusinessError(code, params, fallbackMessage)`。
- `apps/cloud-api` 同步使用错误码协议。
- `packages/contracts/src/client.ts`、`apps/admin/src/lib/admin-api.ts` 统一解析错误码。
- `packages/i18n` 新增 error message catalog。

迁移顺序：

- 先迁移用户会直接看到的错误：auth、world owner、chat、groups、favorites、admin inference、admin runtime rules。
- 服务器内部日志、调试异常可以保留英文技术信息，但不能直接展示给用户。

验收：

- 前端不再把任意服务端 `message` 直接显示为主路径。
- 后端新增用户可见错误必须提供 `code`。
- 旧客户端仍可看到 `fallbackMessage`，但新客户端按 locale 渲染。

### 阶段 5：数据、配置、种子内容软化

交付：

- `LocalizedText` 类型进入 `packages/contracts`。
- `resolveLocalizedText()` 进入 `packages/i18n` 或 contracts utility。
- 游戏、角色预设、系统配置说明支持 localized payload。
- 后台编辑器支持默认语言和其他语言字段。
- 种子数据迁移为 `defaultLocale + values`。

验收：

- 运营配置不再只能保存单一中文或英文字符串。
- 新增配置项天然支持多语言字段。
- 缺失语言有明确 fallback，而不是随机显示源码字符串。

### 阶段 6：AI、提示词、语音全链路 locale 化

交付：

- 世界主人增加 `preferredLocale`、`contentLocale`。
- `runtime-config` 或 owner store 暴露 UI/content locale。
- AI prompt 模板接收 `contentLocale` 和本地化语言名称。
- STT/TTS provider 映射从共享配置读取。
- 现实联动、朋友圈生成、提醒、主动跟进等输出语言跟随 `contentLocale`。

验收：

- 用户可以 UI 用英文，但 AI 内容继续中文，或两者都跟随英文。
- 不再有业务文件零散写死 `zh-CN`、`zh` 作为语言策略。

### 阶段 7：收紧到零新增、核心零存量

交付：

- 新增文件 hardcode 阈值为 0。
- 核心 surface hardcode baseline 清零。
- 发布构建要求目标 locale 无 missing。
- `AGENTS.md` 增加 i18n 开发规则。

验收：

- 新功能如果写了裸文案，提交前直接失败。
- 开发者只要按默认组件和 copy registry 写，新功能会自动提取、自动检查、自动进入语言包。

## 8. 代码规范建议

页面静态文案：

```tsx
<Trans>保存设置</Trans>
```

需要字符串的 DOM prop：

```tsx
const { i18n } = useT();

<input placeholder={i18n._(messages.searchPlaceholder)} />
```

配置数组：

```ts
const statusMessages = {
  pending: msg`待处理`,
  approved: msg`已通过`,
  rejected: msg`已拒绝`,
};
```

运行时渲染：

```tsx
<span>{i18n._(statusMessages[item.status])}</span>
```

持久化配置：

```ts
const title = resolveLocalizedText(game.titleI18n, locale, game.title);
```

禁止新增：

```tsx
<button title="保存">保存</button>
```

```ts
const tabs = [{ label: "消息" }, { label: "通讯录" }];
```

```ts
throw new BadRequestException("只能撤回自己发送的消息。");
```

```ts
new Intl.DateTimeFormat("zh-CN");
```

## 9. 验收标准

短期验收：

- 新增 scanner 和 baseline 后，任何新增 UI 硬编码会失败。
- 管理后台 shell、导航、语言切换、核心 dashboard hero 切换有效。
- `admin` catalog 缺失翻译清零。
- `i18n:check:changed` 能发现本次改动中的硬编码和缺译。

中期验收：

- `apps/admin/src` 字面量 UI props 从约 `1881` 降到 `0` 或明确 allowlist。
- `apps/app/src` 字面量 UI props 从约 `1221` 按高频路径持续下降。
- App 聊天、联系人、发现、我的、桌面 shell 核心路径切换语言有效。
- 后端用户可见错误主路径改为错误码。

长期验收：

- 所有新页面 hardcode 阈值为 `0`。
- 所有生产支持语言无 missing。
- 数据配置、角色预设、游戏目录、系统配置说明支持 `LocalizedText`。
- UI locale、content locale、speech locale 可独立控制。

## 10. 风险与处理

风险：一次性迁移所有页面会拖慢业务。

处理：baseline ratchet，不要求存量一次清零，但每次改动不能变差。

风险：AI 自动翻译质量不稳定。

处理：AI 只生成 draft，不覆盖人工翻译；CI 只保证完整性，不保证语义质量。

风险：错误码迁移影响旧客户端。

处理：保留 `fallbackMessage`，新客户端优先按 `code` 渲染。

风险：组件 props 从 `string` 改 `ReactNode` 可能波及范围大。

处理：先改共享组件和高频调用点，兼容 `string | ReactNode`，再逐步收紧。

风险：有些中文不是 UI 文案，例如 prompt、测试 fixture、协议样例。

处理：scanner 支持 allowlist 注释，并要求写明原因。

## 11. 下一步执行顺序

如果进入实现，建议先做这四件事：

1. 新增 `i18n-hardcode-scan.mjs`、baseline、missing check，并接入 scripts。
2. 改造管理后台共享组件契约，让它们接受可翻译节点或 descriptor。
3. 清理管理后台 shell + `dashboard-page` + 一个最高硬编码页面，验证迁移模板。
4. 新增 API 错误码基础类型和客户端 error registry，先不大规模迁移业务错误。

这个顺序的原因是：先建立闸门，后续改页面才不会边清边漏；先改组件契约，页面迁移成本才会下降；先做一个完整页面样板，后续可以批量复制模式。
