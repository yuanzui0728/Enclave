# Web端聊天页表情包功能规划

## 1. 目标
- 为 `apps/app` 的 Web 端移动版与桌面版聊天页补齐“表情包发送、展示、选择”能力。
- 首版优先支持“静态表情包”，不做视频表情、长图裁切、表情商店、云端同步购买等扩展能力。
- 兼容当前单世界模型与现有聊天链路，不引入多用户隔离逻辑。

## 2. 现状结论

### 2.1 前端现状
- `apps/app/src/components/chat-composer.tsx`
  - 已有 `Smile` / `Plus` 图标入口，但未接任何行为。
  - 输入链路仅支持纯文本 `value + onSubmit()`。
- `apps/app/src/components/chat-message-list.tsx`
  - 消息气泡渲染只识别文本/系统消息，未区分媒体消息。
- `apps/app/src/features/chat/use-conversation-thread.ts`
  - Socket 发送 payload 仅支持 `text`。
  - 本地 optimistic message 也只有 `type + text`。
- `apps/app/src/features/chat/conversation-thread-panel.tsx`
  - 桌面版与移动版共用一套消息数据和发送逻辑，适合做统一消息能力、分端交互壳层。

### 2.2 契约与后端现状
- `packages/contracts/src/chat.ts`
  - `MessageType` 当前只有 `text | system | proactive`。
  - `Message` 结构没有 `attachments` / `meta` / `resource`。
- `packages/contracts/src/ws.ts`
  - `SendMessagePayload` 只有 `conversationId + characterId + text`。
- `api/src/modules/chat/message.entity.ts`
  - 物理表当前只有 `type`、`text`，没有媒体元数据字段。
- 结论：
  - 如果要做“真正的表情包消息”，必须扩展消息契约、Socket payload、数据库结构、消息渲染器。
  - 如果只把表情包当文本占位符发送，会很快碰到排序、预览、资源管理、转发兼容问题，不建议作为正式方案。

## 3. 产品范围

### 3.1 首版必须有
- 表情包面板
- 最近使用
- 默认内置表情包包组
- 单条表情包消息发送
- 消息列表内展示静态图片表情包
- 移动版与桌面版统一资源与消息协议
- 发送失败提示与本地 optimistic 占位

### 3.2 首版不做
- GIF 自动播放控制
- 视频表情
- 自定义上传
- 收藏夹、下载中心、表情商店
- 多选连发
- 拖拽发送
- 富文本混排消息气泡

## 4. 方案原则
- 消息语义独立：表情包是独立消息类型，不塞进文本协议。
- 资源先内置：首版资源跟随前端构建产物，避免先引入上传服务。
- 渲染统一：移动版与桌面版共用消息渲染组件，仅在面板交互和布局上区分。
- 渐进扩展：数据结构预留后续自定义上传和云端表情包包组能力。

## 5. 推荐技术方案

### 5.1 消息模型扩展
- 将 `MessageType` 扩展为：
  - `text | system | proactive | sticker`
- 为 `Message` 增加可选元数据字段，推荐结构：

```ts
type MessageAttachment =
  | {
      kind: "sticker";
      packId: string;
      stickerId: string;
      url: string;
      width: number;
      height: number;
      label?: string;
    };

interface Message {
  id: string;
  conversationId: string;
  senderType: MessageSenderType;
  senderId: string;
  senderName: string;
  type: MessageType;
  text: string;
  attachment?: MessageAttachment;
  createdAt: string;
}
```

- `text` 字段继续保留：
  - `text` 对 `sticker` 消息保存可读兜底文案，例如 `[表情包] 赞同`
  - 便于会话列表最后一条预览、日志、搜索兜底、旧端降级展示

### 5.2 数据库存储
- `messages` 表新增字段：
  - `attachmentKind` `varchar nullable`
  - `attachmentPayload` `text nullable`
- SQLite 下首版直接存 JSON 字符串即可，避免过度拆表。
- 不需要新增实体表，先保持 `messages` 单表承载。

### 5.3 Socket / API 协议
- `SendMessagePayload` 扩展为：

```ts
type SendMessagePayload =
  | {
      conversationId: string;
      characterId: string;
      type?: "text";
      text: string;
    }
  | {
      conversationId: string;
      characterId: string;
      type: "sticker";
      text?: string;
      sticker: {
        packId: string;
        stickerId: string;
      };
    };
```

- 服务端收到 `sticker` 后：
  - 根据服务端注册表补齐 `url/width/height/label`
  - 写入消息表
  - 广播标准化后的 `Message`
- 这样可以避免客户端伪造任意资源 URL。

### 5.4 资源管理
- 新增前端静态清单，例如：
  - `apps/app/src/features/chat/stickers/sticker-catalog.ts`
- 推荐结构：

```ts
type StickerPack = {
  id: string;
  title: string;
  coverStickerId: string;
  stickers: Array<{
    id: string;
    src: string;
    width: number;
    height: number;
    label: string;
  }>;
};
```

- 首版资源建议放在：
  - `apps/app/public/stickers/...`
- 原因：
  - 路径稳定，前后端都容易约定
  - 不需要额外文件上传接口

### 5.5 最近使用
- 最近使用仅存在客户端：
  - `localStorage` 键建议 `yinjie.chat.recent-stickers`
- 存储内容：
  - 最近 20 个 `{ packId, stickerId, usedAt }`
- 原因：
  - 首版不需要跨设备同步
  - 避免把偏好数据写进后端，降低范围

## 6. 前端交互规划

### 6.1 移动版 Web
- 入口：
  - 继续使用输入框右侧 `Smile` 图标
- 交互：
  - 点击后在输入区上方弹出底部抽屉式表情面板
  - 面板高度约占屏幕 42% 到 48%
  - 顶部 Tab：`最近` + 各表情包包组
  - 网格 4 列或 5 列，保证单手点按
- 行为：
  - 选择表情包后直接发送
  - 发送后自动收起面板
  - 键盘与表情面板互斥，打开面板时主动收键盘
- 为什么这样做：
  - 贴近微信式移动端心智
  - 不破坏当前输入框布局

### 6.2 桌面版 Web
- 入口：
  - 继续使用 `chat-composer` 左侧 `Smile` 图标
- 交互：
  - 以输入框上方浮层 Popover 打开
  - 左侧包组导航，右侧宫格列表
  - 支持 hover 高亮与点击发送
- 行为：
  - 选择后直接发送，不占据文本输入框
  - 面板打开时不阻塞继续输入文字
  - 点击空白处关闭
- 可选增强：
  - 后续再补键盘方向键导航，首版可不做

### 6.3 消息展示
- `sticker` 消息单独渲染，不使用普通文本气泡。
- 视觉建议：
  - 用户发送：右对齐，保留头像
  - 对方发送：左对齐，保留头像
  - 气泡背景透明或弱底色，主体展示图片
- 尺寸策略：
  - 移动端最大宽高 `132px`
  - 桌面端最大宽高 `160px`
  - 按原始比例缩放
- 加载失败：
  - 回退到 `[表情包]` 占位卡片

### 6.4 会话列表预览
- 会话列表最后一条文案统一显示：
  - `[表情包]`
  - 如果有 label，可显示 `[表情包] 赞同`
- 避免直接展示资源 URL 或 JSON。

## 8. 分阶段实施建议

### Phase 1：协议打底
- 扩展 contracts、entity、service、gateway
- 打通 `sticker` 消息存取与广播
- 先用固定假数据调用验证整条链路

### Phase 2：消息展示
- `chat-message-list` 支持 sticker 渲染
- 会话列表支持 `[表情包]` 预览
- 完成桌面/移动两端消息展示一致性

### Phase 3：表情面板
- 落地静态包组
- 移动端底部抽屉
- 桌面端 Popover
- 接入最近使用

### Phase 4：打磨
- 失败回退
- 图片占位与懒加载
- 交互细节与视觉节奏优化

## 9. 风险与决策点

### 9.1 最大风险
- 当前消息表没有 JSON 元数据字段，必须做结构扩展。
- 现有 optimistic message 去重逻辑基于 `text`，加入 sticker 后需要改成：
  - 优先按本地临时 id 映射
  - 或按 `type + attachment identity + senderId` 去重

### 9.2 兼容性风险
- 旧客户端如果只认识文本，看到 `sticker` 可能无法正确渲染。
- 建议通过 `text` 兜底，并保证 API 返回里始终有可读文案。

### 9.3 产品决策建议
- 首版不要开放“上传自定义表情包”。
- 首版不要让 AI 角色主动发送表情包。
  - 否则 prompt、资源选择、审核与情绪控制会一起膨胀。

## 10. 推荐落地顺序
1. 先做协议与数据库字段，不碰 UI。
2. 再做消息列表 sticker 展示，保证收发闭环。
3. 最后接入移动版与桌面版面板交互。

## 11. 验收标准
- Web 移动版聊天页可打开表情面板并发送表情包。
- Web 桌面版聊天页可打开表情面板并发送表情包。
- 表情包消息在消息流中正确展示、滚动、排序。
- 会话列表最后一条消息预览正确显示 `[表情包]`。
- 刷新页面后历史表情包消息可回显。
- 发送失败时不会留下错误的永久占位消息。

## 12. 结论
- 这项能力适合按“新消息类型”正式实现，不适合做文本占位式临时方案。
- 前端层面桌面版与移动版可以共用消息模型、资源清单、发送逻辑，只分开处理表情面板交互壳层。
- 如果要快速推进，建议先把首版范围锁死为“内置静态表情包 + 最近使用 + 直接发送”，不要在第一轮把上传、自定义包组、AI 主动发包一起做进去。
