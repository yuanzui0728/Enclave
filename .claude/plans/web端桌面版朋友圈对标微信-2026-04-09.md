# Web 端桌面版朋友圈对标微信规划

## Summary
目标是在 `apps/app` 的桌面 Web 形态里，把朋友圈从“宽屏信息流页面”升级成“微信电脑端朋友圈式工作区”。

这次规划只针对桌面 Web / 桌面壳体验，不动移动端朋友圈交互。首版优先把信息架构、浏览节奏、评论点赞手感、桌面布局密度抄到位；真正涉及“多图发帖 / 相册 / 置顶 / 图片评论”这类能力时，再进入第二阶段的数据结构补齐。

## 对标基线
以目前可确认的微信电脑端能力为基线：
- 2025-03-31，微信 Windows / macOS `4.0.3` 正式版已支持：发表朋友圈、查看朋友朋友圈相册、朋友圈置顶内容。
- 2025-11-08，微信 Windows / macOS `4.1.4` 正式版已支持：朋友圈评论发表情和图片。
- 2026-03-05，微信 Windows / macOS `4.1.8.21` 内测版出现“发现页”收口，把朋友圈等入口统一进发现页。

对隐界的落地建议：
- 第一阶段先对齐 `4.0.3` 的桌面主体验。
- 第二阶段补齐相册、多图、置顶。
- 第三阶段再决定是否跟进 `4.1.8` 那种桌面“发现页收口”。

## 当前现状与差距
当前桌面基础已经有一半：
- `apps/app/src/features/shell/desktop-shell.tsx` 已有桌面侧边栏，也已经放了 `朋友圈` 入口。
- `apps/app/src/features/desktop/chat/desktop-chat-workspace.tsx` 已经证明“桌面专属工作区”这条路是成立的。
- `apps/app/src/routes/moments-page.tsx` 里已经区分了桌面 / 移动两套 UI。

但朋友圈桌面态离“微信电脑端”还差三层：
- 现在的桌面朋友圈本质上还是 `AppHeader + 左发帖卡片 + 右侧信息流卡片`，更像响应式营销页，不像桌面社交工作区。
- 互动仍然是移动端逻辑的放大版，没有“桌面阅读优先、操作悬浮、详情抽屉 / 独立查看”的层次。
- 数据结构目前只有文本发帖、文本评论、点赞。`packages/contracts/src/moments.ts` 和 `api/src/modules/moments/moment-post.entity.ts` 都没有媒体、多图、置顶、相册所需字段。

## 目标形态
桌面朋友圈页改成“全屏工作区”，而不是“一个普通内容页”。

建议的信息架构：
- 全局左侧继续复用现有 `DesktopShell` 导航，不另起一套朋友圈导航。
- 朋友圈主区改成 `内容流 + 右侧资料 / 操作栏` 的桌面布局。
- 顶部增加桌面工具条：标题、刷新、发朋友圈、回到顶部、筛选入口。
- 中间主列是连续动态流，默认把阅读体验放在第一优先级。
- 右栏承担个人摘要、我的朋友圈 / 某角色相册入口、筛选条件、置顶内容、待发草稿。
- 点击动态时，不跳整页；优先使用右侧详情态或内容层弹出详情面板。

首版不要做的事：
- 不做多窗口。
- 不做独立原生子窗口。
- 不做复杂的 Masonry / 瀑布流。
- 不把朋友圈并回发现页里一起做。

## 交互方案
### 1. 桌面主页
- 进入 `/tabs/moments` 时直接进入桌面朋友圈工作区。
- 桌面环境下访问 `/discover/moments` 时，建议直接复用同一工作区，或重定向到 `/tabs/moments`，避免两份桌面实现。
- 顶部工具条提供：
  - `刷新`
  - `发朋友圈`
  - `仅看自己 / 仅看角色 / 全部`
  - `最新优先`

### 2. 动态流
- 卡片样式改成更接近微信电脑端的“列表流”，减少当前大圆角卡片和装饰性背景。
- 每条动态保持固定结构：
  - 头像 / 名称 / 身份
  - 正文
  - 位置信息
  - 时间
  - 点赞 / 评论操作
  - 点赞列表
  - 评论列表
- 操作按钮默认弱化，hover 时增强存在感，保持桌面端“看内容优先”。
- 评论输入不要长期占位；默认折叠，点击评论后展开输入。

### 3. 发朋友圈
- 首版采用桌面弹层或右栏抽屉式发帖，不再把发帖框永久钉死在页面左侧。
- 发帖完成后保留轻量成功提示，并将新内容插入动态流顶部。
- 草稿可以先只放前端本地态，不引入新接口。

### 4. 详情态
- 点击评论区、更多评论、作者头像时，优先在当前页内打开详情态。
- 详情态里补齐：
  - 完整评论区
  - 点赞人列表
  - 作者资料摘要
  - 跳转到“TA 的朋友圈 / 相册”的入口

### 5. 相册视角
- 第一阶段先做“按作者筛选后的动态流视图”，作为相册能力的前置形态。
- 第二阶段再补成真正的“朋友圈相册 / 相册网格 / 置顶头图”。

## 分阶段实施
### Phase 1：桌面工作区重构，不改后端
目标：先把桌面交互和布局抄到位，完全复用现有 moments API。

前端改造：
- 把 `apps/app/src/routes/moments-page.tsx` 拆成：
  - 数据容器
  - `DesktopMomentsWorkspace`
  - `MobileMomentsPage`
- 新增桌面组件族：
  - `apps/app/src/features/desktop/moments/desktop-moments-workspace.tsx`
  - `apps/app/src/features/desktop/moments/desktop-moments-toolbar.tsx`
  - `apps/app/src/features/desktop/moments/desktop-moments-feed.tsx`
  - `apps/app/src/features/desktop/moments/desktop-moments-sidebar.tsx`
  - `apps/app/src/features/desktop/moments/desktop-moment-row.tsx`
  - `apps/app/src/features/desktop/moments/desktop-moment-detail-panel.tsx`
- 复用现有：
  - `getMoments`
  - `createUserMoment`
  - `toggleMomentLike`
  - `addMomentComment`

交互落地：
- 工具条 + 主内容流 + 右栏
- 评论折叠 / 展开
- 详情抽屉
- 作者筛选
- 自己 / 角色筛选
- 顶部刷新和回到顶部

这一阶段不做：
- 多图
- 相册网格
- 置顶
- 图片评论 / 表情评论

### Phase 2：补齐微信电脑端朋友圈核心能力
目标：从“桌面像微信”升级到“功能也接近微信”。

需要的数据结构变更：
- `MomentPostEntity` 增加：
  - `mediaItems`
  - `coverImage`
  - `isPinned`
  - `pinnedAt`
- `MomentCommentEntity` 视是否要跟进微信 `4.1.4`，增加：
  - `commentType`
  - `mediaUrl`
  - `emojiCode`
- `packages/contracts/src/moments.ts` 同步补齐类型。
- `api/src/modules/moments/moments.service.ts` 补齐序列化和读写逻辑。

前端能力：
- 发朋友圈支持多图
- 动态卡片支持九宫格 / 横向图组
- 作者相册页
- 置顶朋友圈卡片
- 评论支持图片 / 表情

注意：
- 这一阶段属于结构变更，落地时要立即更新 `AGENTS.md` 中的实体 / 页面口径。

### Phase 3：导航与产品收口
目标：决定是否进一步贴近微信较新的桌面导航方式。

可选项：
- 保持现有 `DesktopShell` 的独立 `朋友圈` 入口
- 或参考 2026-03-05 的微信 `4.1.8.21` 内测，把桌面端 `朋友圈 / 广场动态 / 搜索型发现` 收口成新的桌面发现入口

建议：
- 先不动。等桌面朋友圈和桌面发现都稳定后再决定，否则会把两个正在演进的模块绑死。

## 文件落点建议
- 保持路由不变：
  - `/tabs/moments` 继续作为桌面主入口
  - `/discover/moments` 继续作为移动端发现二级页入口
- 优先新增桌面组件，不直接在 `moments-page.tsx` 里继续堆 if / else
- 现有 `apps/app/src/components/moment-post-card.tsx` 保留给移动端
- 桌面端不要继续复用 `SocialPostCard`，需要独立的桌面 row/card 组件

## 验收标准
- 桌面宽屏进入朋友圈后，不再出现当前“左侧发布大卡 + 右侧大圆角信息流”的移动放大感。
- 桌面端能稳定完成：浏览、点赞、评论、发文本朋友圈。
- 同一份数据源下，移动端朋友圈行为不回归。
- `/tabs/moments` 与 `/discover/moments` 在桌面态不出现两套分裂 UI。
- 1280px、1440px、1680px 三档宽度下都成立，不出现明显空洞。

## 风险与决策
- 最大约束不是 UI，而是 moments 数据模型太薄。只要不加媒体字段，就只能先抄“桌面骨架”，还抄不到“微信完整版朋友圈”。
- 当前仓库已有桌面聊天工作区，朋友圈应沿用同一种“workspace”模式，不要再回到页面级拼卡片。
- 现有桌面侧边栏已经有独立 `朋友圈` 入口，短期不建议同时推动“桌面发现收口”，否则范围会失控。

## 建议执行顺序
1. 先做 Phase 1，把桌面朋友圈从页面改成工作区。
2. 稳定后再补 moments 媒体字段和置顶字段。
3. 最后才讨论是否把桌面朋友圈并回桌面发现入口。
