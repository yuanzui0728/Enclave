# Web 端电脑版朋友圈微信电脑端排版规划

## 目标

把 `apps/app` 的桌面 Web 朋友圈从现在这套“桌面营销页 + 大卡片”的信息流，改成更接近微信电脑端朋友圈的工作区排版。

这次规划只覆盖：

- `apps/app` 的桌面 Web / 桌面壳形态
- 朋友圈主列表、作者入口、发帖入口、详情侧栏
- 现有文本动态、点赞、评论能力的重排与桌面化

这次不覆盖：

- 移动端朋友圈
- 多图、多媒体、置顶、相册网格
- 评论图片 / 表情
- 新测试文件

## 对标依据

外部参考只拿来约束桌面排版和入口结构，不直接照搬视觉资源。

- 2025-04-01，新浪转载的微信 Windows `4.0.3` 正式版信息明确提到：
  - 左侧功能栏有 `朋友圈` 入口
  - 顶部左上角有 `相机` 按钮用于发朋友圈
  - 支持查看朋友的朋友圈相册
  - 支持朋友圈置顶
- 2024-11-29，社群运营百科对 PC 微信 `4.0.1` 的整理提到：
  - 可在左侧 `朋友圈` 入口直接进入朋友圈页
  - 点击好友头像后，可以在名片页看到 `朋友圈` 入口

基于这些公开信息，可以确认桌面版微信朋友圈的排版重点不是“大卡片展示”，而是：

- 入口清晰
- 工具栏很轻
- 内容流连续
- 作者资料与作者时间线入口稳定存在

## 排版目标

### 总体结构

桌面态采用 `全局侧边导航 + 朋友圈主工作区`，不再使用普通页面头图式布局。

朋友圈主工作区分三层：

- 顶部轻工具栏
- 中间主内容流
- 右侧上下文栏

推荐宽度：

- 主内容流：`680px - 760px`
- 右栏：`300px - 340px`
- 两栏间距：`20px - 24px`

### 1. 顶部轻工具栏

微信电脑端的感觉是“顶部有功能，但不抢内容”。这里建议做成单行工具栏，而不是大标题区。

保留：

- 左侧标题：`朋友圈`
- 左上主操作：`相机 / 发朋友圈`
- 右侧轻操作：`刷新`
- 搜索框：搜索作者 / 正文 / 评论

去掉或下沉：

- 大段介绍文案
- 大号 KPI 指标
- 多个品牌色筛选胶囊并排

实现建议：

- `发朋友圈` 从文字主按钮改成 `相机图标 + 简短文案`
- `只看我 / 只看角色 / 全部` 不放在第一行，可收进二级筛选条或右栏

### 2. 主内容流

目标不是做“卡片墙”，而是做“连续朋友圈流”。

每条动态的结构建议固定为：

- 左侧头像
- 右侧正文区
- 作者名 + 作者身份标签
- 时间 / 位置
- 正文
- 底部轻操作区：点赞、评论
- 点赞列表与评论列表收纳在同一个浅灰互动盒中

视觉要求：

- 背景以白底或极浅灰为主
- 圆角明显收小，从现在的 `24-26px` 收到 `12-16px`
- 行间距靠内容撑开，不靠大面积留白
- 尽量减少显式描边和大阴影

交互要求：

- 默认点击正文或评论区域进入右栏详情态
- `查看详情` 独立按钮去掉
- 评论输入默认折叠，点击评论后展开
- 点赞与评论按钮使用轻图标按钮，不用大胶囊主按钮

### 3. 右侧上下文栏

右栏不要再做成“指标面板”，而要做成“上下文容器”。

分两种状态：

- 未选中动态：显示“我的资料摘要 + 发朋友圈快捷入口 + 最近作者列表”
- 选中动态：切换成动态详情态

未选中态建议内容：

- 我的头像、昵称、朋友圈入口说明
- `发朋友圈` 快捷入口
- 最近有动态的作者列表
- 当前筛选状态

选中态建议内容：

- 作者头像、名字、身份
- 完整正文
- 时间、位置
- 点赞列表
- 完整评论区
- 写评论输入框
- `查看 TA 的朋友圈` 入口

### 4. 作者时间线

微信电脑端一个很重要的感受是“从作者进入作者的朋友圈很自然”。

隐界首版不单独做完整相册页，但要先把这条浏览路径做顺：

- 点击头像或作者名
- 右栏切到该作者摘要
- 主流自动筛选为该作者的动态
- 顶部显示一个轻量的作者筛选状态

这一步先替代“真正的朋友圈相册”。

## 实现前深度规划

下面这部分不是再讲“想做成什么样”，而是把第一阶段真正要怎么落地先定死，避免实现过程中反复返工。

### 一、先定首阶段冻结范围

这次实现只做桌面朋友圈工作区重构，明确冻结下面几件事：

- 只动 `apps/app`
- 只动桌面 Web / 桌面壳表现
- 只复用现有朋友圈接口和当前 `Moment` 契约
- 不新增测试文件
- 不改后端实体、接口、共享契约
- 不改移动端朋友圈 UI 结构
- 不把朋友圈和 `发现 / 广场动态` 一起收口

这意味着首阶段交付目标非常明确：

- `moments-page.tsx` 只负责数据获取、突变和桌面/移动分流
- 桌面态全面切到 `DesktopMomentsWorkspace`
- `DesktopMomentsWorkspace` 改成真正的桌面工作区，而不是继续堆临时 UI

### 二、当前代码层面的关键判断

实现前需要先认清当前代码真实状态，而不是只看文件是否存在。

#### 1. `DesktopMomentsWorkspace` 现在还没接入页面入口

当前 [moments-page.tsx](/home/ps/claude/yinjie-app/apps/app/src/routes/moments-page.tsx) 的桌面态仍然走：

- `AppPage`
- `AppHeader`
- 左发帖卡
- 右动态列表

也就是说，现有桌面工作区组件虽然已经写出来，但还没有成为真实入口。

这次实现的第一步应该是：

- 保持 `MomentsPage` 作为数据容器
- 在桌面态直接渲染 `DesktopMomentsWorkspace`
- 移动态继续保留现有页面结构

#### 2. `DesktopMomentsWorkspace` 当前的状态组织还不够稳定

当前 [desktop-moments-workspace.tsx](/home/ps/claude/yinjie-app/apps/app/src/features/desktop/moments/desktop-moments-workspace.tsx) 里混在一起的东西太多：

- 顶栏排版
- 动态列表
- 右栏概览
- 发帖抽屉
- 筛选状态
- 选中态逻辑

问题不只是文件大，而是“状态语义不够清楚”：

- `selectedAuthorId` 同时承担作者过滤和右栏上下文
- `selectedMomentId` 会自动选中第一条，导致右栏很难维持默认摘要态
- `showCompose` 是全局覆盖层，但右栏又有发帖入口，语义容易重复

这次实现时要把状态拆清楚。

#### 3. 右栏默认不应自动进入详情态

从当前规划目标看，右栏应有两种主要状态：

- 默认摘要态
- 动态详情态

所以首阶段不建议继续保留“首屏自动选中第一条动态”的行为。更合理的规则是：

- 初始进入页面：右栏显示默认摘要态
- 点击某条动态正文 / 互动区：右栏切到该动态详情
- 点击作者头像 / 名字：主流切到该作者时间线，右栏显示作者摘要
- 关闭详情后：回到默认摘要态或作者摘要态

这个调整虽然不大，但会直接决定工作区是不是像微信电脑端。

首阶段桌面朋友圈工作区统一收敛成下面这个结构：

```text
DesktopShell

    -> DesktopMomentsWorkspace（状态编排）

      -> Feed

```

布局固定为：

- 顶部：轻工具栏
- 左主列：连续动态流
- 右侧栏：上下文栏
- 右侧覆盖层：发帖面板

其中：

- 顶部工具栏始终存在
- 右栏始终存在
- 发帖面板按需覆盖出现
- 不做整页跳转式详情

### 四、页面状态模型

建议把桌面朋友圈状态拆成下面几层。

#### 1. 数据容器状态：放在 `moments-page.tsx`

只保留和数据读写强相关的状态：

- `text`
- `commentDrafts`
- `successNotice`
- `createMutation`
- `likeMutation`
- `commentMutation`
- `visibleMoments`

这里继续负责：

- 获取朋友圈数据
- 过滤已屏蔽角色
- 处理发布 / 点赞 / 评论
- 处理接口报错和成功提示

这里不要再负责桌面排版细节。

#### 2. 工作区视图状态：放在 `desktop-moments-workspace.tsx`

只保留和桌面浏览上下文有关的状态：

- `activeFilter`
- `searchText`
- `activeAuthorId`
- `selectedMomentId`
- `showCompose`

建议明确三个派生视图状态：

- `sidebarMode = "summary" | "author" | "detail"`
- `filteredMoments`
- `authorSummaries`

推导规则建议固定为：

1. `selectedMomentId` 有值时，右栏进入 `detail`
2. 否则 `activeAuthorId` 有值时，右栏进入 `author`
3. 否则右栏进入 `summary`

这样右栏行为会稳定很多。

#### 3. 行内临时交互状态：保留在行组件

[desktop-moment-row.tsx](/home/ps/claude/yinjie-app/apps/app/src/features/desktop/moments/desktop-moment-row.tsx) 只保留轻量本地状态，例如：

- 评论输入框展开/折叠

不要再让行组件自行决定作者筛选、详情切换、全局布局行为。

### 五、组件职责重新划分

首阶段建议按下面的边界拆组件。

#### 1. `desktop-moments-toolbar.tsx`

#### 3. `desktop-moments-sidebar.tsx`

- `summary`

其中：

- `summary` 展示我的摘要、快捷发帖、最近作者

#### 4. `desktop-moment-compose-panel.tsx`

负责：

- 右侧覆盖式发帖面板
- 头像、提示文案、文本框、发布按钮

继续保留覆盖层方案，不塞回右栏正文区，原因很简单：

- 右栏需要稳定承接上下文
- 发帖是一次性动作
- 覆盖式抽屉更符合桌面“临时打开、完成后关闭”的节奏

### 六、关键交互规则定稿

这部分需要在实现前先定死，否则最容易边写边改。

#### 1. 路由行为

- `/tabs/moments` 继续作为桌面主入口
- `/discover/moments` 在桌面态直接复用同一套 `DesktopMomentsWorkspace`
- 首阶段不强制做桌面自动重定向，先保证两条路由共用同一个桌面实现

这样更稳，能先把 UI 收敛，不引入额外路由副作用。

#### 2. 作者时间线行为

点击作者头像 / 名字时：

- 设置 `activeAuthorId`
- 清空 `selectedMomentId`
- 主流切成该作者动态
- 右栏进入作者摘要态

点击动态本体时：

- 设置 `selectedMomentId`
- 右栏进入详情态
- 不清空 `activeAuthorId`

这样用户可以在“某作者时间线”里继续查看某条详情。

#### 3. 搜索与筛选优先级

筛选顺序建议固定为：

1. 作者筛选 `activeAuthorId`
2. 类型筛选 `activeFilter`
3. 关键词搜索 `searchText`

这样规则最容易解释，也方便后面继续扩展。

#### 4. 评论输入行为

首阶段保持两层评论入口：

- 列表行内可快速评论
- 右栏详情可完整评论

但要统一规则：

- 行内评论默认折叠
- 有草稿时自动保持展开
- 发送成功后收起与否由当前交互位置决定

建议首阶段先这样落地：

- 行内发送成功后自动收起
- 右栏发送成功后保持当前详情态

#### 5. 成功提示与错误提示

`successNotice` 继续由页面容器统一管理，但展示位置收敛到工具栏下方，不要在主列和右栏重复出现。

错误提示也统一收口：

- 列表/详情交互错误显示在工作区顶层
- 发帖错误显示在发帖面板内部
