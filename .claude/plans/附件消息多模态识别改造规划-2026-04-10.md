# 附件消息多模态识别改造规划

## 1. 目标
- 让聊天附件消息不再只以占位文本进入 AI，上下文里保留结构化附件语义。
- 图片消息在 provider / model 支持时，真正以多模态输入喂给模型，而不是仅传 `[图片] xxx.jpg`。
- 文件 / 名片 / 位置消息即使暂时不走“真文件理解”，也要以结构化描述进入 AI，而不是退化成单行占位文本。
- 保持现有消息表、附件上传接口、前端渲染协议兼容，不引入新表，不生成测试文件。

## 2. 现状结论

### 2.1 当前问题不在附件上传，而在 AI 入参建模
- `apps/app/src/features/chat/use-conversation-thread.ts`
  - 图片 / 文件 / 名片 / 位置发送时，前端主动把 `text` 写成 `[图片] xxx.jpg`、`[文件] xxx.pdf`、`[名片] 张三`、`[位置] 某地点`。
- `api/src/modules/chat/chat.service.ts`
  - `normalizeOutgoingMessageInput()` 对附件消息生成 `fallbackText`，并把它同时作为 `text` 和 `promptText`。
  - `sendMessage()` 把 `history.push({ role: 'user', content: normalizedInput.promptText })` 写入 AI 历史。
- `api/src/modules/chat/chat.service.ts`
  - `ensureConversationHistory()` 从数据库回放历史时，只重建 `{ role, content }`，附件结构完全丢失。
- `api/src/modules/chat/group.service.ts`
  - 群聊 `triggerAiReplies()` 直接把最近消息拼成 `[发送者]: 文本`，附件同样只剩占位文本。

### 2.2 现有数据层其实已经保留了附件结构
- `api/src/modules/chat/message.entity.ts` / `group-message.entity.ts`
  - 已有 `attachmentKind` / `attachmentPayload`。
- `api/src/modules/chat/chat.types.ts`
  - 已有 `ImageAttachment` / `FileAttachment` / `ContactCardAttachment` / `LocationCardAttachment`。
- 结论：
  - 数据库和消息协议不是瓶颈。
  - 真正缺的是“AI 历史消息模型”和“provider 多模态适配层”。

### 2.3 provider 配置已经预留了 API 风格，但主链路没有用上
- `api/src/modules/system/system.service.ts`
  - 已保存 `provider_api_style`，支持 `openai-chat-completions` / `openai-responses`。
- `api/src/modules/ai/ai-orchestrator.service.ts`
  - `generateReply()` 仍固定调用 `client.chat.completions.create(...)`。
  - 没有根据 `apiStyle` 分流，也没有构造 `image_url` / `input_image`。
- 结论：
  - 这次改造不只是“补一个附件字段”，而是要把编排层改成“结构化内容 -> provider adapter”。

### 2.4 还有一个隐含前置条件
- `api/src/modules/chat/chat.service.ts`
  - 附件 URL 通过 `PUBLIC_API_BASE_URL` 生成；若未配置则回退到 `http://localhost:${PORT}`。
- 对云端模型来说，`localhost` 图片 URL 不可访问。
- 结论：
  - 图片多模态要么依赖正确的公网 `PUBLIC_API_BASE_URL`，要么增加后端转 base64 / 文件上传到 provider 的兜底策略。

## 3. 方案原则
- 预览文案和 AI 输入分离：`text` 继续承担列表预览 / 旧端兼容，AI 入参走独立结构。
- 先做最小可落地：首版优先打通“图片真看图 + 其他附件结构化理解”，不在第一步就追求所有 provider 的文件原生理解。
- 兼容 provider 差异：统一内部消息结构，最后一跳再分流到 `chat.completions` 或 `responses`。
- 失败可降级：模型不支持多模态、图片地址不可访问时，自动退回结构化文本，而不是直接报错。

## 4. 推荐改造方案

### 4.1 先扩展 AI 内部消息模型，而不是改数据库
- 目标文件：
  - `api/src/modules/ai/ai.types.ts`
  - `api/src/modules/chat/chat.service.ts`
  - `api/src/modules/chat/group.service.ts`
- 把当前只有 `content: string` 的 `ChatMessage` 升级成“多段内容”模型，建议类似：

```ts
type AiMessagePart =
  | { type: "text"; text: string }
  | { type: "image"; imageUrl: string; detail?: "auto" | "low" | "high"; altText?: string }
  | { type: "file_ref"; fileName: string; mimeType: string; url: string; summaryText: string }
  | { type: "contact_card"; name: string; relationship?: string; bio?: string; summaryText: string }
  | { type: "location_card"; title: string; subtitle?: string; summaryText: string };

interface AiChatMessage {
  role: "user" | "assistant" | "system";
  parts: AiMessagePart[];
  characterId?: string;
}
```

- 这样处理后：
  - 历史里可以保留“这是图、图在哪、还有哪些附加字段”。
  - 编排层可以针对不同 provider 决定哪些 part 走原生多模态，哪些转文本。

### 4.2 把“显示用 fallbackText”和“AI 用结构化内容”拆开
- 目标文件：
  - `api/src/modules/chat/chat.service.ts`
  - `api/src/modules/chat/group.service.ts`
- 当前 `normalizeOutgoingMessageInput()` 返回 `{ text, promptText, attachment }`。
- 建议改成返回：

```ts
{
  type: ...,
  text: string,              // UI / 列表预览兜底
  attachment?: MessageAttachment,
  aiParts: AiMessagePart[],  // 真正给模型的内容
}
```

- 行为建议：
  - `image`：
    - `text` 保留 `[图片] xxx.jpg`
    - `aiParts` 为 `text + image`
  - `file`：
    - `text` 保留 `[文件] xxx.pdf`
    - `aiParts` 为结构化文本摘要，首版不强依赖 provider 原生文件理解
  - `contact_card`：
    - `aiParts` 至少包含姓名、关系、简介
  - `location_card`：
    - `aiParts` 至少包含地点标题、副标题

### 4.3 历史回放必须从附件字段重建 AI parts
- 目标文件：
  - `api/src/modules/chat/chat.service.ts`
  - `api/src/modules/chat/group.service.ts`
- 当前问题：
  - 只要消息离开当前请求，下一次从数据库回放就又退化成纯文本。
- 必须新增统一 helper，例如：
  - `buildAiMessageFromConversationMessage(entity)`
  - `buildAiMessageFromGroupMessage(entity)`
- 它们要做的事：
  - 读取 `type + text + attachmentKind + attachmentPayload`
  - 重建 `AiChatMessage.parts`
  - 对角色消息继续做 `sanitizeAiText()`

### 4.4 在 orchestrator 增加 provider adapter，而不是把 provider 判断散落在业务层
- 目标文件：
  - `api/src/modules/ai/ai-orchestrator.service.ts`
- 建议新增两层 helper：
  - `buildChatCompletionMessages(...)`
  - `buildResponsesInput(...)`
- 分流逻辑：
  - `provider_api_style === "openai-chat-completions"`
    - 文本 part -> `{ type: "text", text }`
    - 图片 part -> `{ type: "image_url", image_url: { url, detail: "auto" } }`
  - `provider_api_style === "openai-responses"`
    - 文本 part -> `{ type: "input_text", text }`
    - 图片 part -> `{ type: "input_image", image_url }`
- 输出提取也要统一封装，避免主流程同时关心两种 API 返回结构。

### 4.5 首版对非图片附件采用“结构化文本理解”，不要一开始就追求原生 file input
- 原因：
  - 当前附件消息里只有文件 URL，没有 provider file id。
  - 不同 provider 对 PDF / doc / file input 支持差异更大。
  - 文件原生上传到 provider 会引入额外缓存、权限、生命周期管理。
- 首版建议：
  - `file` 先转成丰富文本，例如文件名、MIME、大小、用户附言。
  - `contact_card` / `location_card` 直接结构化文本即可，收益已经明显高于 `[名片]` / `[位置]`。
- 第二阶段再评估是否要为特定 provider 增加真正的 file input。

### 4.6 图片可访问性必须纳入实现
- 目标文件：
  - `api/src/modules/chat/chat.service.ts`
  - `api/src/modules/ai/ai-orchestrator.service.ts`
  - `api/src/modules/system/system.service.ts`
- 建议增加最少一项保护：
  - 如果 `image` 附件 URL 指向 `localhost` / `127.0.0.1` / 内网地址，则不要把它当多模态图像发给云端 provider，直接降级为文本描述。
- 更稳妥的后续增强：
  - 新增“服务端抓取附件并转 base64”能力，让模型无需直接访问业务图片 URL。
- 如果不做这层保护，很多部署环境会出现“代码支持看图，但线上实际上读不到图”的伪可用状态。

### 4.7 前端只保留 UI 占位文案，不再承担 AI 语义构建
- 目标文件：
  - `apps/app/src/features/chat/use-conversation-thread.ts`
  - `packages/contracts/src/ws.ts`
- 当前前端把 `[图片] xxx.jpg` 直接塞进 payload。
- 建议：
  - 附件消息发送时允许 `text` 为空或仅作 UI 预览用途。
  - 真正的 AI 语义统一由服务端基于 `attachment` 构建。
- 这样可以避免：
  - 前端不同入口重复拼文案
  - 后端未来升级后，前端还继续把旧占位逻辑写死

## 5. 建议实施顺序

### Phase 1：打通直聊图片多模态主链路
- 改 `ai.types.ts`，引入 `AiChatMessage` / `AiMessagePart`
- 改 `chat.service.ts`
  - 发送时产出 `aiParts`
  - 回放历史时从附件字段重建 `aiParts`
- 改 `ai-orchestrator.service.ts`
  - 接收结构化历史
  - 对 `chat.completions` 生成多段 `content`
- 结果：
  - 直聊里的图片消息可以真正走视觉输入
  - 非图片附件至少不再只剩一行占位文本

### Phase 2：补齐 group chat
- 改 `group.service.ts`
  - 最近消息回放改成结构化历史
  - 发给单个角色前保留发言者标签，但不要破坏附件 part
- 结果：
  - 群聊 AI 回复也能看到图片与结构化附件信息

### Phase 3：接入 provider_api_style 分流
- 改 `ai-orchestrator.service.ts`
  - 支持 `openai-chat-completions`
  - 支持 `openai-responses`
  - 统一响应解析
- 结果：
  - 现有系统配置里的 `provider_api_style` 真正生效
  - 为后续兼容 OpenAI 兼容网关 / Responses API 铺平

### Phase 4：前端去冗余与兜底优化
- 改 `use-conversation-thread.ts`
  - 不再把 AI 占位文案当业务真值
  - optimistic UI 仍可显示 `[图片]` / `[文件]`
- 可选补充：
  - 管理后台或系统诊断页提示“当前 provider/model 是否支持图片理解”

## 6. 提交拆分建议
- 提交 1：重构 AI 消息内部类型与直聊历史重建
- 提交 2：直聊图片多模态编排接入
- 提交 3：群聊历史与群聊 AI 回复链路改造
- 提交 4：provider_api_style 分流与兼容降级
- 提交 5：前端去占位文案耦合与细节收口

## 7. 验收标准
- 发送一张图片后，AI 回复内容能引用图片里的实际元素，而不是只复述文件名。
- 发送文件时，AI 至少能理解“这是一个什么文件”，而不是只回复 `[文件] xxx.pdf`。
- 发送名片时，AI 能识别联系人姓名 / 关系等字段。
- 发送位置时，AI 能引用地点标题 / 副标题，而不是只看到 `[位置]`。
- 关闭或切换会话后重新进入，历史回放仍保留附件语义，不会重新退化成纯文本。
- `provider_api_style` 切到 `openai-responses` 时，主聊天链路可正常工作。
- `PUBLIC_API_BASE_URL` 不可被外网访问时，系统能安全降级，不出现不可解释的空回复。

## 8. 风险与注意事项
- 默认模型目前常见是 `deepseek-chat`，即使 API 兼容 OpenAI，也不代表一定支持图片理解；能力检测和降级必须做。
- `gpt-4-vision-preview` 这类旧模型名仍出现在配置列表里，后续最好顺手清理模型清单，但不应阻塞本次改造主线。
- 如果未来要支持真正的文件内容理解，可能要引入“服务端读取文件内容 / OCR / PDF 文本提取 / provider file upload”中的至少一种，不建议和这次改造绑定在同一批提交里。

## 10. 建议执行口径
- 先做“图片真多模态 + 其他附件结构化语义”。
- 不在第一步追求所有附件都做 provider 原生文件输入。
- 不新增表，不新增测试文件，按小步提交推进。
