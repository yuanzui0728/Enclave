# 多模态非数字人缺口彻底收口规划

## 1. 规划范围

本规划只处理“除了数字人以外”的多模态缺口。

明确纳入：

- 单聊 / 群聊里的音频、视频、文档理解
- Web 手机版 AI 语音通话
- 文本 / 语音 / 图片三类回复的真实闭环
- 后台诊断、系统状态、前台可见状态的真实语义

明确排除：

- `api/src/modules/chat/digital-human-calls.service.ts`
- `api/src/modules/chat/digital-human-provider.ts`
- `/api/chat/digital-human-calls/*`
- 数字人视频渲染、外部视频流、播放器 iframe

这里的“彻底收口”口径是：

- 输入真的被系统消费，而不是只靠文件名、提示词补丁或偶发兼容。
- 输出真的由正式链路生成，而不是 mock、占位或旁路补发。
- direct / group 语义一致。
- 失败、降级、重试、取消、诊断都能落到运行态状态上。

## 2. 2026-04-24 的真实基线

基于当前代码和已跑过的运行态诊断，非数字人部分的现状是：

- 普通聊天里的图片理解已经是真实链路。
- 普通聊天里的原生音频理解已经是真实链路，但只对 `wav/mp3` 稳定，任意音频格式还没统一归一化。
- 普通聊天里的语音回复 TTS、图片回复生成已经真实可用。
- 语音通话仍未真实闭环，因为 `VoiceCallsService.createTurn()` 先强依赖 `transcribeAudio()`，见 `api/src/modules/chat/voice-calls.service.ts:23-102`。
- 聊天主回复链路只对 `image` 和 `audio` 做了原生输入处理；`document` 只在 Responses API 下支持 `input_file`，`video` 完全没有原生视觉处理，见 `api/src/modules/ai/ai-orchestrator.service.ts:1274-1676`。
- 附件洞察仍然只有三种窄能力：`audio_transcription`、`video_transcription`、`document_text_extraction`，见 `api/src/modules/chat/media-insight-job.service.ts:346-383`。
- 当前附件洞察在 `sendMessageDetailed()` 里是同步执行的，消息发送会被洞察处理直接阻塞，见 `api/src/modules/chat/chat.service.ts:822-840`。
- 系统状态仍把 `speechReady` 绑定成“转写 + TTS 同时 ready”，见 `api/src/modules/system/system.service.ts:2591-2658`；这会误伤本来已经具备原生音频理解能力的语音通话场景。
- 回复模态规则仍然把“收到音频/视频附件”误判成“应该回语音”，见 `api/src/modules/chat/assistant-reply-modalities.ts:27-39`。

## 3. 剩余缺口的根因判断

### 3.1 语音通话被错误绑死在转写链路上

当前语音通话的执行顺序是：

1. 上传录音
2. 先转写
3. 把转写文本作为普通文本消息发给聊天链路
4. 再做 TTS

这个设计的问题是：

- 它没有复用已经存在的“原生音频输入给模型”的能力。
- 只要独立转写 provider 不可用，整个语音通话就直接失效。
- 用户语音内容、通话文本、普通聊天消息被拆成了两套逻辑，后续维护会继续分叉。

### 3.2 附件理解模型还停在“单一字段兜底”

当前 `MessageAttachment` 里对多模态理解可复用的信息只有：

- `transcriptText`
- `extractedText`

这不够承载：

- 视频关键帧摘要
- 视频 OCR
- 视频时序摘要
- PDF / DOCX 分页或分块引用
- 扫描件 OCR
- 音频归一化产物

更关键的是，当前 `MediaInsightJobService.ensureMessageInsight()` 按 `sourceMessageId` 查单条 job，天然偏向“一个附件只做一个洞察”。如果继续在现有形态上硬加更多视频 / 文档洞察，结构会很快失控。

### 3.3 视频理解现在只有“听”，没有“看”

`chat.service.ts` 和 `group.service.ts` 虽然能把视频附件包装成 `AiMessagePart.video`，但主模型调用根本不会消费这类 part，见：

- `api/src/modules/chat/chat.service.ts:1829-1839`
- `api/src/modules/chat/group.service.ts:940-950`
- `api/src/modules/ai/ai-orchestrator.service.ts:1555-1658`

这意味着当前视频理解本质上还是：

- 有音轨时，靠转写文本
- 没音轨时，只剩文件名和补充说明

它不算真实视频理解。

### 3.4 文档理解只覆盖了纯文本型文件

当前文档路径主要是：

- `txt/md/json/xml/csv/html` 直接按文本抽
- `pdf/doc/docx` 只被 mime/fileName 识别成“文档”，但没有真正解析器
- 扫描 PDF 没有 OCR

所以“文档理解真实存在”的前提还没成立。

### 3.5 诊断语义和真实能力语义还没对齐

现在后台有：

- `text`
- `image_input`
- `audio_input`
- `transcription`
- `tts`
- `image_generation`

但真正用户关心的是：

- 普通聊天音频理解 ready 不 ready
- 语音通话 ready 不 ready
- 视频理解 ready 不 ready
- 文档理解 ready 不 ready

当前状态没有把这些使用场景单独表达出来，导致看板会误导。

## 4. 推荐的总体路线

推荐路线不是“继续堆更多 provider 字段”，而是先把输入理解拆成三层：

1. 原始资产层：用户上传的真实附件
2. 派生洞察层：转写、关键帧、OCR、分块摘要、归一化产物
3. 模型消费层：最终送给 reply model 的 `parts` / `text bundle`

只有这样，才能同时解决：

- provider 能力不一致
- 文件格式不一致
- direct / group / 通话三条链路各自分叉

## 5. 收口目标

本轮非数字人目标建议定义为：

### 5.1 输入侧

- 语音通话在没有独立转写 provider 的情况下仍能工作
- 普通聊天上传 `audio/video/document` 后，角色能真实理解内容
- 视频理解至少能同时处理音轨和关键画面
- PDF / DOCX / 扫描 PDF 能进入真实理解链路

### 5.2 输出侧

- 文本回复继续保持
- 语音回复继续真实可用
- 图片回复继续真实可用
- 回复规划不再把“收到视频/音频附件”误判成“必须回语音”

### 5.3 可观测性

- 后台能看出“原生理解 / 洞察理解 / 降级理解”
- 前台能看出“处理中 / 部分完成 / 失败 / 已完成”
- 系统状态能区分 `audioInputReady`、`voiceCallReady`、`videoUnderstandingReady`、`documentUnderstandingReady`

## 6. 关键架构决策

### 6.1 语音通话改成“原生音频优先，转写辅助”

这是第一优先级，也是收益最大的改造。

推荐把 `VoiceCallsService.createTurn()` 改成：

1. 先保存上传音频到服务端可访问存储
2. 构造 `voice` / `audio` 附件
3. 直接走聊天主回复链路，使用 `input_audio`
4. 再做 TTS
5. 转写改成可选的后台补充，用于字幕、检索、回看，不再是硬 blocker

这样做的原因：

- 当前默认模型 `gpt-4o-all` 已经实测支持原生音频输入。
- 主聊天链路已经能处理音频 part。
- 语音通话不应该再依赖单独转写 provider 才能说话。

对返回契约的建议调整：

- `userTranscript` 改为可选
- 新增 `transcriptStatus: "completed" | "pending" | "failed" | "skipped"`
- 保留 `assistantText`、`assistantAudioUrl`

这样前端 UI 才能真实表达“通话可用，但字幕仍在补跑”。

### 6.2 附件洞察从“窄字段”升级成“复合 bundle”

不建议继续增加大量零散的 insight kind。

更推荐的做法是把洞察升级成三类复合 bundle：

- `audio_understanding`
- `video_understanding`
- `document_understanding`

每个 bundle 里再包含：

- 原始格式信息
- 归一化产物
- 洞察文本
- 质量 / 降级标记
- provider / 耗时 / 失败原因

这样可以避免当前“一条消息一个洞察 job”模型被拆穿后大面积返工。

### 6.3 视频理解先走“抽帧 + 视觉理解”，不要等待原生视频模型

当前最可落地的路线不是接一个新的 native video provider，而是：

1. `ffprobe` 取元数据
2. `ffmpeg` 抽关键帧
3. 如有音轨，抽音频并归一化
4. 关键帧送到现有支持图片输入的主模型
5. 合并音轨转写、OCR、帧级摘要，得到统一 `video_understanding` bundle

这样做的好处：

- 直接复用当前已经真实可用的 `gpt-4o-all` 图片理解能力
- 不被 provider 是否支持 native video input 绑死
- Docker 部署只需补 `ffmpeg`

### 6.4 文档理解先走“本地解析 + OCR 兜底”，原生 `input_file` 作为加分项

不建议把“文档真实理解”完全押在 Responses API 上。

推荐路线：

- `txt/md/csv/json/xml/html`：保留当前文本抽取
- `docx`：`mammoth`
- `pdf`：`pdfjs-dist` 或稳定 PDF 文本解析器
- 扫描 PDF：渲染页面为图片，再走现有图片理解模型做 OCR / 页面摘要

当且仅当某个 provider 真实支持 `input_file` 且 smoke 通过时，再把 native document input 作为增强路径加入。

### 6.5 洞察处理改成“双层时序”

当前 `sendMessageDetailed()` 在保存消息后会同步跑 `ensureConversationMessageInsight()`。

这会让大附件、视频、PDF 的理解成本直接顶在主请求上。

推荐拆成两层：

- 快路径：主请求只做可在 300-800ms 内完成的轻量处理
- 慢路径：复杂视频 / 文档洞察进入后台 job，消息先发出，状态展示为“理解补全中”

建议标准：

- 音频归一化、短音频原生理解可以快路径
- 视频抽帧、PDF OCR 一律慢路径

## 7. 分阶段执行计划

## 阶段 A：先把语音通话从“转写依赖”里解出来

### 目标

- 让 Web 手机版 AI 语音通话在没有独立转写 provider 的情况下仍然真实可用。

### 主要改动

- `api/src/modules/chat/voice-calls.service.ts`
- `api/src/modules/ai/ai-orchestrator.service.ts`
- `packages/contracts/src/speech.ts`
- `apps/app/src/features/chat/use-voice-call-session.ts`
- `apps/app/src/features/chat/mobile-ai-call-screen.tsx`

### 实施点

- 新增“通话输入音频持久化”步骤，生成可访问 URL
- 通话输入不再先转文字，而是先作为 `audio` part 喂给主聊天链路
- `transcribeAudio()` 改成 optional sidecar，而不是 turn 主链路前置
- 返回结构补 `transcriptStatus`
- 前端字幕区域接受 `pending/failed/skipped`

### 验收

- 独立转写 provider 关闭时，语音通话仍能完成一轮对话和 TTS 播报
- 若后台补转写成功，UI 自动出现字幕
- 若补转写失败，UI 只显示“字幕不可用”，不影响整轮通话结果

## 阶段 B：加统一的音频 / 视频预处理层

### 目标

- 让任意音频 / 视频格式都能被主链路稳定消费，而不是只对 `wav/mp3` 运气好。

### 主要改动

- 新增 `media-preprocess` 类服务，放在 `api/src/modules/chat/` 或 `api/src/modules/ai/`
- `api/src/modules/chat/media-insight-job.service.ts`
- `api/src/modules/chat/chat.types.ts`

### 实施点

- 音频统一转成单声道 `wav`
- 视频统一抽音轨为 `wav`
- 保存归一化产物 URL / mime / size / duration
- `collectUsableAudioParts()` 优先使用归一化音频，而不是直接依赖原文件格式

### 基础依赖

- Docker / 运行环境补 `ffmpeg` 和 `ffprobe`

### 验收

- 上传 `webm/ogg/m4a/mp4` 音频或视频，系统都能稳定拿到可用音频产物
- 普通聊天和语音通话共享同一套归一化逻辑

## 阶段 C：把视频理解做成真实能力

### 目标

- 视频不再只是“文件名 + 音轨转写”，而是带画面理解。

### 主要改动

- `api/src/modules/chat/media-insight-job.service.ts`
- `api/src/modules/chat/media-insight-job.entity.ts`
- `api/src/modules/chat/chat.types.ts`
- `api/src/modules/ai/ai-orchestrator.service.ts`
- 前端消息气泡和后台记录页

### 推荐的最小可行 bundle

`video_understanding` 至少包含：

- `transcriptText`
- `frameSummary`
- `ocrText`
- `timelineSummary`
- `durationMs`
- `width`
- `height`
- `normalizedAudioUrl`
- `keyframeUrls`
- `quality: "full" | "partial" | "degraded"`

### 实施点

- 关键帧抽取默认 3-6 帧
- 对关键帧走现有图片理解模型，输出结构化 JSON
- 如有音轨，同时跑转写
- 最终把视觉摘要 + OCR + 音轨转写合并成一份 bundle
- `buildAiParts()` 和主回复 prompt 改为读取 bundle，而不是只读 `transcriptText`

### 验收

- 上传无音轨但有明显画面内容的视频，角色能回答画面是什么
- 上传含字幕或招牌文字的视频，角色能回答视频里写了什么
- 上传同时有音轨和画面的视频，角色能同时引用两类信息

## 阶段 D：把文档理解补齐到 PDF / DOCX / 扫描件

### 目标

- 文档理解不再只覆盖纯文本文件。

### 主要改动

- `api/src/modules/ai/ai-orchestrator.service.ts`
- `api/src/modules/chat/media-insight-job.service.ts`
- `api/src/modules/chat/chat.types.ts`
- 文档上传后的前台 / 后台状态展示

### 推荐能力包

`document_understanding` 至少包含：

- `extractedText`
- `ocrText`
- `chunkSummaries`
- `pageHints`
- `sourceKind: "text" | "pdf" | "docx" | "scanned_pdf"`
- `quality: "full" | "partial" | "degraded"`

### 推荐技术栈

- `docx`：`mammoth`
- `pdf` 文本：稳定 PDF 解析器
- 扫描件 OCR：把页面渲染为图片后交给现有图片理解模型，而不是先引入新的 OCR 基础设施

### 实施点

- 解析结果做分块，不直接把整份文档原样塞进 prompt
- prompt 里保留文件名、页码或块编号
- 若未来某 provider 真正支持 `input_file` 并 smoke 通过，再增开 native path

### 验收

- 上传文本 PDF，角色能回答文档中的具体事实
- 上传 DOCX，角色能给出摘要
- 上传扫描 PDF，角色能给出 OCR 后的关键信息

## 阶段 E：把诊断语义从“能力字段”升级成“场景 readiness”

### 目标

- 后台和系统状态不再误导。

### 主要改动

- `api/src/modules/inference/inference.service.ts`
- `api/src/modules/system/system.service.ts`
- `packages/contracts/src/system.ts`
- `packages/contracts/src/inference.ts`
- `apps/admin/src/routes/inference-page.tsx`

### 推荐新增语义

- `voiceCallReady`
- `voiceCallTranscriptReady`
- `videoUnderstandingReady`
- `documentUnderstandingReady`
- `nativeDocumentInputReady`

### 关键判断

- `voiceCallReady` 不应再等价于 `transcription && tts`
- 在原生音频输入可用且 TTS 可用时，语音通话就应视为 ready
- `documentUnderstandingReady` 不应等价于 `supportsNativeDocumentInput`
- `videoUnderstandingReady` 必须看抽帧 + 视觉理解链路的 smoke

### 验收

- `system/status` 和后台总览能准确说明“通话可用但字幕补跑不可用”“文档理解可用但 native input_file 不可用”这类场景

## 阶段 F：清理回复模态规划误判

### 目标

- 不再因为用户发了语音或视频，就强行触发“回语音”。

### 主要改动

- `api/src/modules/chat/assistant-reply-modalities.ts`
- `api/src/modules/chat/chat.service.ts`
- 群聊对应 planner / artifact 调度路径

### 实施点

- `shouldCreateVoiceReplyFromAttachment()` 删除“收到音频 / 视频附件就自动回语音”的逻辑
- 语音回复只由两类信号触发：
  - 用户明确要求
  - 场景策略明确要求
- 图片回复同理，逐步从 regex 过渡到结构化 planner

### 验收

- 用户发一个视频问“这段在讲什么”，系统默认应该先正常文字回答视频内容，而不是回一段语音

## 8. 推荐实施顺序

按收益 / 风险比，推荐顺序是：

1. 阶段 A：语音通话去转写依赖
2. 阶段 B：统一音频 / 视频预处理
3. 阶段 C：视频真实理解
4. 阶段 D：文档真实理解
5. 阶段 E：诊断与状态语义
6. 阶段 F：回复模态规划清理

原因很明确：

- A 能立刻解决当前最影响体验的真实 blocker
- B 是 C 和 D 的基础
- C、D 解决“理解是否真实存在”
- E、F 解决“看板是否真实”和“回复是否误判”

## 9. 具体验证矩阵

收口时至少要补下面这些 smoke，而不是只跑 build：

- 用 `wav/webm/m4a` 三种录音走 AI 语音通话，验证在无转写 provider 情况下仍能出一轮真实回复
- 上传 `mp3` 音频文件到普通聊天，验证角色引用音频内容
- 上传无音轨但有明显画面信息的 `mp4`，验证角色回答画面内容
- 上传含文字画面的 `mp4`，验证 OCR 信息进入回答
- 上传文本 PDF，验证能回答文档里的具体事实
- 上传 DOCX，验证能总结
- 上传扫描 PDF，验证 OCR 兜底有效
- direct / group 各跑一轮，避免只修单聊

## 10. 风险与约束

### 10.1 不要继续把“转写 provider 不可用”当成语音通话 blocker

当前已经有原生音频输入能力，继续坚持“先转写再通话”属于架构惯性，不是技术必要条件。

### 10.2 视频和文档不要继续堆在同步请求里

如果把抽帧、OCR、PDF 解析继续塞进主请求，最终只会得到：

- 超时
- 429
- 前端卡死
- 偶尔成功、无法稳定复现

### 10.3 不要把“支持 native input_file / native video”作为前置条件

现阶段最现实的目标是：

- 先用已有真实可用的图片理解能力和本地预处理把视频 / 文档变成真实可理解输入
- provider 原生能力只做增强，不做前置依赖

## 11. 本规划的最终落点

如果按上面的顺序收口，非数字人部分最终会变成：

- 普通聊天：文本 / 图片 / 音频 / 视频 / 文档都能真实理解
- 语音通话：不依赖转写 provider 也能工作
- 回复：文本 / 语音 / 图片都是真实闭环
- 后台状态：反映真实场景 readiness，而不是字段是否填写

数字人视频能力之后可以单独接回这套结果：

- 语音通话 turn
- 视频 / 文档理解 bundle
- TTS 结果

但那是下一阶段，不应该再绑在这次非数字人收口上。
