# Web端手机版 AI 语言通话功能规划

## 1. 目标
- 范围只覆盖 `apps/app` 的 Web 端手机版。
- 目标能力不是传统 RTC 真人通话，而是“用户说话 -> AI 转写理解 -> AI 生成回复文本 -> TTS 播放语音”的拟通话体验。
- 首版要解决的是“像打电话一样连续对话”，不是“做一个新的语音消息入口”。
- 首版必须复用现有聊天会话和角色回复链路，避免做第二套 AI 对话系统。

## 2. 产品定位

### 2.1 首版定义
- 首版定义为“半双工 AI 语言通话”：
  - 用户说话时采集麦克风。
  - 说完一段后结束录音并上传。
  - 服务端先做 STT，再走现有聊天回复逻辑生成文本。
  - 最后把 AI 回复转成语音返回前端播放。
- 用户看到的是独立“通话中”界面，但底层仍绑定当前 `conversationId`。

### 2.2 为什么不直接做全双工
- 当前仓库已有稳定能力是：
  - Web 麦克风采集：`apps/app/src/features/chat/use-speech-input.ts`
  - 服务端语音转写：`POST /api/ai/transcriptions`
  - 聊天会话与 AI 回复链路：现有 `chat` + `ai` 模块
- 当前仓库没有：
  - TTS 接口
  - 实时音频流协议
  - WebRTC / SIP / RTC 会话层
  - 通话会话状态管理
- 对移动 Web 来说，首版直接做全双工 RTC 风险过高，尤其是：
  - iOS WebKit 麦克风和自动播放限制
  - 回声消除与打断管理
  - AI TTS 首包时延
  - 浏览器切后台后的连接稳定性
- 因此建议先做“像通话”的语音助手体验，不直接进入 RTC 工程。

## 3. 当前代码现状

### 3.1 已有基础
- `apps/app/src/features/chat/use-speech-input.ts`
  - 已经有 `MediaRecorder` 录音能力。
  - 已经处理权限请求、状态机、录音时长、异常文案。
- `api/src/modules/ai/ai.controller.ts`
  - 已提供 `POST /api/ai/transcriptions`。
- `api/src/modules/ai/ai-orchestrator.service.ts`
  - 已有转写 provider 调用和音频格式校验。
- `apps/app/src/features/chat/conversation-thread-panel.tsx`
  - 手机版聊天头部已有“语音通话”入口，但目前只是 fallback。
- `apps/app/src/routes/character-detail-page.tsx`
  - 好友详情页已有“语音通话”按钮，目前也是占位。
- `apps/app/src/routes/chat-details-page.tsx`
  - 聊天详情页已有“实时通话 -> 语音通话”入口，目前是占位。
- `apps/app/src/components/mobile-chat-plus-panel.tsx`
  - `语音通话` 在加号面板里仍是 disabled 占位入口。

### 3.2 当前真实缺口
- 没有 TTS 能力，无法把 AI 文本回复转成可播放音频。
- 没有“通话态页面”，目前只有聊天页和语音消息输入面板。
- 没有语音通话 session 概念，用户无法连续说多轮并管理中断、播放、挂断。
- 现有 `/api/ai/transcriptions` 只负责转文字，不负责后续生成回复和语音回放。

## 4. 核心产品决策

### 4.1 首版交互形态
- 从单聊页、好友详情页、聊天详情页进入“语音通话页”。
- 页面视觉是通话态：
  - 顶部显示角色头像、名称、当前状态
  - 中部显示最近一轮的“我说了什么 / TA 回复了什么”
  - 底部是大尺寸主按钮：`按住说话` / `正在聆听` / `AI 回复中` / `播放中`
  - 辅助按钮：`静音播放`、`重播本轮`、`挂断`
- 用户每次说一段，系统回复一段，形成连续多轮。

### 4.2 首版技术语义
- 首版不是实时双向音频流，而是“回合制语音对话”。
- 一轮对话包含四步：
  1. 录音
  2. 转写
  3. AI 文本回复
  4. TTS 播放
- 通话页的连续感来自：
  - 独立页面
  - 连续回合
  - 自动播报 AI 回复
  - 不必每轮回到聊天输入框

### 4.3 与现有聊天系统的关系
- 每一轮用户语音都要落成当前会话的一条文本消息或语音摘要消息。
- AI 的每一轮回复仍然走当前角色回复逻辑，并正常进入会话消息流。
- 通话页只是新的交互外壳，不是新的消息域。
- 这样可以直接复用：
  - 当前 `conversationId`
  - 角色 prompt / 记忆 / 关系状态
  - 消息列表预览
  - 后续聊天继续

## 5. 建议实现方案

### Phase 1：补齐服务端 TTS 基础
目标是先让服务端具备“文本 -> 音频”的统一能力。

改造点：
- `api/src/modules/ai/ai.controller.ts`
  - 新增 TTS 接口，建议不要新开模块，先继续挂在 `ai` 模块里。
- `api/src/modules/ai/ai-orchestrator.service.ts`
  - 新增 `synthesizeSpeech()` 能力。
  - 支持按角色或默认配置选择 voice。
- `packages/contracts/src/`
  - 新增 `voice-call` 或扩展 `speech.ts` 契约。
- `packages/contracts/src/client.ts`
  - 新增 TTS / 语言通话相关请求函数。

接口建议：
- `POST /api/ai/speech`
  - 入参：`text`、`voice?`、`conversationId?`、`characterId?`
  - 出参：音频文件流或可下载 URL + 元数据

配置建议：
- 复用 `SystemConfig`，新增配置键即可：
  - `provider_tts_model`
  - `provider_tts_voice`
- 首版不新增表。

交付结果：
- 服务端可以稳定输出单段 AI 语音。

### Phase 2：做语言通话回合接口
目标是把 STT、AI 回复、TTS 串成一个原子回合。

建议不要让前端依次调用三个接口，而是由服务端封一个“单轮语言通话”接口。

接口建议：
- `POST /api/ai/voice-calls/turns`
  - 请求：`conversationId`、`characterId`、`file`
  - 服务端流程：
    1. 转写用户语音
    2. 生成并写入用户消息
    3. 触发 AI 回复文本
    4. 把 AI 回复转为语音
    5. 返回本轮结果
  - 响应：
    - `userTranscript`
    - `assistantText`
    - `assistantAudioUrl`
    - `assistantAudioDurationMs`
    - `messageIds`
    - `providerMeta`

实现原则：
- 首版不新增数据库实体。
- 通话 session 先做前端本地态 + 服务端无状态接口。
- 每轮都显式带 `conversationId`，由会话历史保证连续上下文。

交付结果：
- 前端一次上传即可拿到整轮回复，通话态实现难度明显下降。

### Phase 3：落移动 Web 通话页
目标是把当前的占位入口替换成真页面。

建议新增路由：
- `apps/app/src/routes/chat-voice-call-page.tsx`
- 路由建议：`/chat/$conversationId/voice-call`

建议新增目录：
- `apps/app/src/features/chat/voice-call/`
  - `use-voice-call.ts`
  - `voice-call-session.ts`
  - `voice-call-audio-player.ts`
  - `voice-call-status.ts`

页面能力：
- 进入页后先完成一次用户手势，解锁音频播放。
- 按住说话时录音，松开后自动发起本轮。
- AI 语音播放期间默认禁止再次录音，先做半双工。
- 展示状态机：
  - `idle`
  - `requesting-permission`
  - `recording`
  - `uploading`
  - `thinking`
  - `speaking`
  - `error`
- 展示最近一轮文案：
  - 我：识别结果
  - TA：文本回复摘要

入口替换范围：
- `apps/app/src/features/chat/conversation-thread-panel.tsx`
- `apps/app/src/routes/character-detail-page.tsx`
- `apps/app/src/routes/chat-details-page.tsx`
- `apps/app/src/components/mobile-chat-plus-panel.tsx`

交付结果：
- 所有手机版“语音通话”入口都进入统一通话页，不再显示“暂未开放”。

### Phase 4：角色语音与体验细化
目标是把它从“能说”提升到“像这个角色在说”。

改造点：
- 角色可配置默认 voice preset，但首版可以先做服务端映射，不改数据库结构。
- TTS 请求里带入角色信息：
  - 角色名
  - 语气
  - 说话习惯摘要
- 若 Provider 不支持可控 voice persona，则先退化为：
  - 固定音色
  - 通过 prompt 控制回复风格

UI 细节：
- 播放时加头像呼吸态和波纹态。
- 支持“重播上一句”。
- 支持“切成文字聊天”快速回跳。

交付结果：
- 用户感知从“语音播报器”提升到“和角色通话”。

### Phase 5：异常与可观测性
目标是避免上线后无法定位问题。

需要覆盖的异常：
- 麦克风权限拒绝
- 录音为空
- 转写失败
- AI 回复失败
- TTS 失败
- 音频播放失败
- 浏览器切后台导致播放中断

日志建议：
- `conversationId`
- `characterId`
- 音频大小和 mimeType
- 转写耗时
- 回复耗时
- TTS 耗时
- 整轮耗时
- 失败阶段

交付结果：
- 能区分是录音、STT、模型回复、TTS 还是前端播放出了问题。

## 6. 推荐实现顺序
1. 先补 TTS 接口和契约，不动前端入口。
2. 再封单轮语言通话接口，把 STT -> AI -> TTS 串起来。
3. 再做 `/chat/$conversationId/voice-call` 页面。
4. 最后把现有“语音通话暂未开放”的入口全部切到新页面。

## 7. 验收标准
- Web 手机版单聊页可以进入独立语音通话页。
- 用户按住说一段话后，系统能在同一页面返回 AI 语音回复。
- 用户和 AI 可以连续完成至少 3 轮对话，无需退回文本聊天页。
- 当前会话消息流中可以看到本次语言通话产生的文本消息。
- 任一环节失败时，用户能明确知道失败在：
  - 麦克风
  - 转写
  - AI 回复
  - TTS
  - 播放

## 8. 风险与取舍
- 最大风险不是录音，而是 TTS 首包延迟。首版必须接受“1 到 3 秒思考后再说话”的拟通话节奏。
- 移动 Web 不能指望稳定后台播放和锁屏播放，所以首版只承诺前台使用。
- 自动播放限制要求进入通话页后尽早完成一次用户交互，否则 AI 回复音频可能被浏览器拦截。
- 如果后续真的要做“像微信语音电话一样实时不断句”，那是另一阶段的 RTC 项目，不应和本次 TTS 拟通话混在一起。

## 9. 结论
- 这项需求首版不该理解为“做实时电话系统”，而应该理解为“做一个移动 Web 的 AI 语音对话壳”。
- 当前仓库已经有录音、转写、聊天会话三块基础，缺的主要是：
  - TTS
  - 单轮串联接口
  - 通话态页面
- 按这个方案推进，首版可以不改数据库、不加新模块，在现有 `ai`、`chat`、`apps/app` 结构内完成。
