# 剩余多模态能力彻底真实实现规划

## 1. 规划目标

本规划只解决一件事:

- 把当前还“不彻底”或“不真实”的多模态能力补成真实闭环。

这里的“真实”指:

- 输入模态真的被系统消费，而不是只靠文件名、占位文本或偶然可用的旁路。
- 输出模态真的由主链路编排和交付，而不是 mock 壳、临时 helper 或不可取消的后台补发。
- 关键能力在运行态稳定可验证，不因为 429、provider 差异、页面刷新或会话清空就失真。

## 2. 当前基线

### 2.1 已经真实成立

- 图片理解真实成立。
  - `image -> AiMessagePart.image -> provider image_url/input_image`
- 文本文档理解部分成立。
  - `txt/json/xml/html/md/csv` 可抽取正文并进入模型
- 普通聊天 `text + voice` 回复已成立。
- 普通聊天 `text + image` 回复已成立。
- 群聊 `text + voice` / `text + image` 回复已成立。

### 2.2 还没有彻底成立

- 音频理解不稳定。
  - 当前主要依赖转写
  - 转写通道会命中 429
- 视频理解不成立。
  - 当前基本只有音轨转写
  - 视频画面没有稳定进入理解链路
- 文档理解不完整。
  - `pdf/doc/docx/扫描件` 没有完整抽取和 OCR 兜底
- 多模态回复不完整。
  - 当前只有 `text / voice / image`
  - 没有真实视频回复
  - 数字人仍是 mock / iframe 壳
- 异步补图没有取消与隔离。
  - 清空会话后，旧轮补图仍可能穿透落库

## 3. 最终目标口径

### 3.1 输入侧目标

系统最终必须稳定支持:

- `text`
- `image`
- `audio`
- `video`
- `document`

其中每一类输入都要同时具备三层表示:

1. 原始资产引用
2. 派生洞察
3. 面向 provider 的最终 payload

### 3.2 输出侧目标

系统最终必须稳定支持:

- `text`
- `voice`
- `image`
- `digital_human_video`

并满足:

- 同一轮可生成多条 sibling message
- 所有 sibling message 共享 `responseGroupId`
- 每个非文本结果都有可观测 render job
- 每个 render job 都支持取消、重试、失败原因记录

### 3.3 “彻底实现”的验收口径

满足以下条件才算收口:

- 单聊和群聊都可用
- 清空会话、删除消息、消息撤回不会留下悬挂补发
- provider 拒绝原生模态输入时，系统可自动回退到结构化派生表示
- 后台能看见每轮多模态理解和渲染的真实状态
- 至少完成端到端 smoke，而不只是 build 通过

## 4. 总体架构

### 4.1 新的核心分层

新增四层稳定结构:

1. `Asset Ingest`
   - 负责接收原始附件
   - 负责资产归档、基本元数据、MIME 归一化

2. `Insight Derivation`
   - 负责转写、OCR、抽帧、摘要、正文抽取
   - 产出结构化 `DerivedMediaInsight`

3. `Inference Payload Adapter`
   - 根据 provider capability 决定喂原始模态还是派生表示

4. `Reply Artifact Orchestrator`
   - 根据回复模态计划，生成 `voice/image/video`
   - 负责异步 job、取消、补发、状态事件

### 4.2 建议新增的内部模型

建议引入:

- `MultimodalAttachmentEnvelope`
- `DerivedMediaInsight`
- `ReplyModalityPlan`
- `ReplyArtifactJob`
- `ReplyArtifactResult`

### 4.3 数据结构演进

现有表先兼容扩展，不推翻消息模型。

建议新增字段或实体:

- `Message.responseGroupId`
- `GroupMessage.responseGroupId`
- `Message.generationMetadata`
- `GroupMessage.generationMetadata`
- `Message.derivedInsightPayload`
- `GroupMessage.derivedInsightPayload`

建议新增实体:

- `MediaInsightJob`
- `MediaInsightArtifact`
- `ReplyArtifactJob`
- `DigitalHumanSessionEntity`
- `DigitalHumanRenderJobEntity`

说明:

- `MediaInsightJob` 负责输入理解
- `ReplyArtifactJob` 负责输出渲染
- 数字人会话不能继续只放内存 `Map`

## 5. 必须收口的 5 个缺口

### 5.1 缺口 A: 语音理解不稳定

根因:

- 当前音频理解高度依赖在线转写
- provider 429 会直接让语义理解退化
- 转写失败后系统没有可恢复的 attachment processing 状态

目标:

- 语音消息先进入稳定的转写任务队列
- 转写结果持久化
- 模型消费的是“稳定转写结果 + 音频元数据”
- 主聊天链路只读结果，不在请求内无限阻塞

### 5.2 缺口 B: 视频画面没有真正进入理解

根因:

- 当前只走音轨转写
- `supportsNativeVideoInput = false`
- 即使未来 provider 支持视频，也不能把系统能力完全寄托于 provider

目标:

- 建立 `Video Understanding Bundle`
- 至少包含:
  - 关键帧
  - OCR
  - 音轨转写
  - 时长/分辨率
  - 场景摘要

### 5.3 缺口 C: 文档理解只覆盖文本型文档

根因:

- 当前只对可直接 `utf8` 读取的文档抽取正文
- `pdf/doc/docx/扫描件` 没有完整抽取

目标:

- 文档理解至少覆盖:
  - `txt`
  - `md`
  - `csv`
  - `json`
  - `pdf`
  - `docx`
  - 扫描版 pdf

### 5.4 缺口 D: 数字人视频回复不真实

根因:

- 当前 provider 是 `mock_digital_human`
- 会话状态在内存里
- 只提供播放器 URL 或音频海报壳

目标:

- 数字人会话持久化
- 渲染任务可重试、可回调、可追踪
- 至少接一条真实外部 render adapter
- 若 provider 不可用，明确回退到 `voice + poster`

### 5.5 缺口 E: 异步补发没有取消机制

根因:

- deferred image reply 是 fire-and-forget
- `clearConversationHistory()` 只清内存 history，不取消在途渲染

目标:

- 所有异步补发都必须 job 化
- job 必须绑定 source message / response group
- 会话清空、消息删除、任务失效时必须取消

## 6. 分阶段实施

## Phase 0: 稳定底座

### 目标

- 先补任务化、状态化、能力注册化
- 不先追新模态，先把现有异步链路变成可控系统

### 后端改造

- 把普通聊天 deferred image reply 改成 `ReplyArtifactJob`
- 群聊 deferred media 也统一进 job 表
- 增加 job 状态:
  - `pending`
  - `processing`
  - `completed`
  - `cancelled`
  - `failed`
- 增加 job 关联:
  - `threadType`
  - `threadId`
  - `sourceMessageId`
  - `responseGroupId`
  - `characterId`
  - `artifactType`

### 接口与事件

- 新增实时事件:
  - `reply_artifact_job_created`
  - `reply_artifact_job_updated`
  - `reply_artifact_job_cancelled`
- typing 阶段继续保留，但不再作为唯一状态来源

### 验收

- 清空单聊后，旧轮补图不会再落库
- 群聊取消旧 reply task 后，旧轮媒体不会补发
- 后台可看到每个 artifact job 的状态

## Phase 1: 语音理解稳定化

### 目标

- 把“语音理解”从一次性请求改成稳定处理流程

### 设计

- 上传语音后立即落库 attachment
- 同时创建 `MediaInsightJob(type=audio_transcription)`
- 由 worker 处理转写
- 结果回写:
  - `attachmentPayload.transcriptText`
  - `derivedInsightPayload.audio`

### provider 策略

- 主转写 provider: 当前运行时 provider 的 transcription model
- 失败策略:
  - 限流退避
  - 指数退避重试
  - 超过阈值后标记失败，不阻塞整个线程
- 可选增强:
  - 单独配置 transcription provider account
  - 与主推理 provider 解耦

### 需要改的模块

- `api/src/modules/ai/ai-orchestrator.service.ts`
- `api/src/modules/chat/chat.service.ts`
- `api/src/modules/chat/group.service.ts`
- 新增 `api/src/modules/chat/media-insight.service.ts`

### 关键实现点

- 转写任务从同步聊天流程抽离
- 聊天主链路判断:
  - 若已有转写结果，直接用
  - 若没有结果但用户本轮强依赖语音内容，先提示“正在听这段语音”，等 insight 完成后再触发角色回复
- 为 voice message 增加处理状态:
  - `processing`
  - `ready`
  - `failed`

### 验收

- 同一段语音附件重复发送不重复转写
- 429 时任务重试，不直接把用户问题原样复读
- 语音 smoke 在 5 次连续测试中至少 4 次正确命中暗号

## Phase 2: 视频理解真实化

### 目标

- 视频不再只靠音轨

### 设计

- 新增 `Video Understanding Bundle`
- 处理流程:
  1. 提取基础元数据
  2. 抽关键帧
  3. 对关键帧做 OCR
  4. 提取音轨并转写
  5. 生成镜头级摘要
  6. 聚合为结构化 insight

### 推荐依赖

- `ffmpeg` / `ffprobe`
- OCR 服务或本地 OCR 接口
- 现有 transcription provider

### 结构化 insight 建议

```json
{
  "video": {
    "durationMs": 52341,
    "width": 1080,
    "height": 1920,
    "frames": [
      { "tsMs": 0, "imageUrl": "...", "ocrText": "...", "sceneSummary": "..." }
    ],
    "audioTranscript": "...",
    "summary": "..."
  }
}
```

### provider 投喂策略

- 若 provider 未来支持原生视频:
  - 原视频 + bundle 一起作为高保真输入
- 当前默认:
  - `关键帧图片 + OCR + 音轨转写 + summary`

### 需要改的模块

- 新增 `video-insight.service.ts`
- `chat.service.ts`
- `group.service.ts`
- `ai-orchestrator.service.ts`

### 验收

- 视频里即使没有可听音轨，只靠画面文本也能答题
- 视频里即使画面没字，也能根据音轨答题
- 视频里画面与音轨冲突时，回复能体现区分

## Phase 3: 文档理解扩展到真实常用格式

### 目标

- 文档理解从“文本型文件”升级成常用办公文档能力

### 覆盖范围

- `pdf`
- `docx`
- `txt`
- `md`
- `csv`
- `json`
- 扫描版 `pdf`

### 抽取策略

- `txt/md/csv/json/xml/html`:
  - 继续走当前轻量提取
- `docx`:
  - 新增 docx 正文抽取
- `pdf`:
  - 先正文抽取
  - 抽不到文字时自动 OCR
- 扫描版 pdf:
  - 页级图片 + OCR

### 结构化结果

- 标题
- 页数
- 正文 chunk
- 摘要
- OCR 命中率
- 是否扫描件

### 验收

- 发 `docx/pdf` 文档问答时，不再只靠文件名
- 扫描版 pdf 至少能正确读出短暗号

## Phase 4: Provider 适配层补齐原生 audio/video/document 入口

### 目标

- 内部模型保留统一语义
- provider adapter 按能力做最佳映射

### 设计

- `AiMessagePart.audio` 不再只是挂在类型里
- `ai-orchestrator` 增加:
  - `collectUsableAudioParts`
  - `collectUsableVideoParts`
  - `buildResponsesAudioMessage`
  - `buildResponsesVideoMessage`

### 回退链

1. 原生模态输入
2. 结构化 `document/input_file`
3. 结构化 `summary + transcript + OCR + metadata`
4. 最后才退到普通文本 prompt

### 验收

- capability profile 与实际 adapter 行为一致
- 原生输入失败时会自动回退，不直接整轮报错

## Phase 5: 回复编排器彻底化

### 目标

- 普通聊天和群聊都用统一回复编排器
- 不再散落在 `chat.service`、`group-reply-task.service` 的局部 helper

### 设计

- 新增 `reply-orchestrator.service.ts`
- 输出:
  - `responseGroupId`
  - `primaryText`
  - `artifactJobs[]`

### 支持模态

- `text`
- `voice`
- `image`
- `digital_human_video`

### 规划依据

- 用户明确请求
- 输入模态
- surface
- 角色配置
- provider capability
- 预算策略

### 规则示例

- 收到语音消息且转写 ready:
  - 默认 `text + voice`
- 收到明确画图需求:
  - 默认 `text + image`
- 在数字人视频通话里:
  - 默认 `text + voice + digital_human_video`

### 验收

- 普通聊天、群聊、语音通话、数字人通话全部走统一 planner
- 回复 sibling message 共享 `responseGroupId`

## Phase 6: 数字人视频真实化

### 目标

- 用真实 render job 取代当前 mock 壳

### 设计

- `DigitalHumanSessionEntity`
- `DigitalHumanRenderJobEntity`
- provider adapter 标准化:
  - `createSession`
  - `submitRenderJob`
  - `pollRenderJob`
  - `cancelRenderJob`

### 最低可交付口径

- 接入一个真实可渲染 provider
- provider 不支持实时流也可以，先交付异步可播放视频结果
- 当 provider 完全不可用时:
  - 自动回退为 `voice + poster`
  - 前端明确提示“已回退到语音模式”

### 前端

- 不再把 `player_url` 视为真实视频能力本身
- 区分:
  - `rendering`
  - `ready`
  - `failed`
  - `fallback_to_voice`

### 验收

- 创建数字人 session 后，能看到真实 render job 生命周期
- 至少一轮真实 turn 产出非 mock 视频结果

## Phase 7: 前端与后台可观测性补齐

### 前端

- 输入消息显示 insight 处理中状态
- 对视频/文档显示“解析中 / 已解析 / 失败”
- 对回复 artifact 显示“生成中 / 已送达 / 失败”

### 后台

- 新增多模态调试面板:
  - 原始附件
  - insight 结果
  - provider payload
  - artifact jobs
  - 降级原因

### 验收

- 后台能定位任意一次多模态失败原因
- 前端不会出现“用户看不到系统在做什么”的长时间黑盒等待

## 10. 执行顺序

严格按下面顺序做，不要跳:

1. Phase 0
   - 先把异步 reply job 化和可取消化
2. Phase 1
   - 先稳住语音理解
3. Phase 3
   - 再补文档广覆盖
4. Phase 2
   - 然后做视频 bundle
5. Phase 4
   - 再补 provider adapter 原生音频/视频入口
6. Phase 5
   - 收口统一回复编排器
7. Phase 6
   - 最后接真实数字人 render
8. Phase 7
   - 收尾前端和后台可观测

原因:

- Phase 0 不做，后面所有异步模态都会继续脏穿透
- 语音和文档是最容易真实闭环的收益项
- 视频和数字人最重，必须放在底座稳定之后

## 12. 风险与控制

### 风险 1: provider 能力波动

控制:

- 所有原生多模态都必须有结构化 fallback

### 风险 2: 视频处理过重

控制:

- 限制首期最大时长和分辨率
- 先抽关键帧，不做全量逐帧理解

### 风险 3: 队列堆积

控制:

- insight job 和 artifact job 分队列
- 设置最大并发和超时

### 风险 4: 前端状态复杂度快速膨胀

控制:

- 前端只消费统一 job 状态，不直接理解后端内部步骤

## 13. 完成定义

只有以下条件全部满足，才能对外宣称“已彻底实现多模态理解与多模态回复”:

- 图片、语音、视频、文档输入都能真实进入系统理解
- 普通聊天、群聊、通话场景都能稳定输出对应多模态结果
- 数字人视频不再是 mock 壳
- 所有异步模态结果都可取消、可追踪、可回退
