# 管理后台角色工厂与运行逻辑台实施计划

## 1. 计划目标

- 把“角色制造过程可见且可配置”做成独立产品能力，而不是继续塞在当前角色编辑页和回复逻辑页里。
- 把“角色当前回复逻辑、生活逻辑可见且可配置”做成独立运行控制台。
- 保留现有 `/reply-logic` 作为世界级调试台，不让它继续承担角色工厂职责。
- 规划必须可以直接进入实现，不停留在概念层。

## 2. 当前起点

### 2.1 已有基础能力

- 角色名册已存在：
  - `apps/admin/src/routes/characters-page.tsx`
- 角色编辑页已存在：
  - `apps/admin/src/routes/character-editor-page.tsx`
- 世界级回复逻辑页已存在：
  - `apps/admin/src/routes/reply-logic-page.tsx`
- 后端已有回复快照接口：
  - `GET /api/admin/reply-logic/overview`
  - `GET /api/admin/reply-logic/rules`
  - `PATCH /api/admin/reply-logic/rules`
  - `GET /api/admin/reply-logic/characters/:id`
  - `GET /api/admin/reply-logic/conversations/:id`

### 2.2 已完成的底座修正

- 自定义角色不会再被启动 seed 清掉。
- 回复运行规则已配置化，统一走 `SystemConfig`。
- `Character.onlineMode` / `Character.activityMode` 已加入，用于区分自动调度和人工锁定。

### 2.3 仍然缺失的关键能力

- 没有角色工厂数据模型。
- 没有工厂发布机制。
- 没有角色级运行逻辑页。
- 没有调度器可观测性。
- 现有角色编辑页和回复逻辑页职责重叠，会导致后续边界混乱。

## 3. 最终产品结构

## 3.1 保留世界级控制台

- 路由：`/reply-logic`
- 目标：
  - 面向整个世界实例看回复链路
  - 看全局 provider / 全局规则 / 会话链路
  - 继续承担“真实运行调试台”职责

## 3.2 新增角色工厂

- 路由：`/characters/$characterId/factory`
- 目标：
  - 看角色如何被定义、组装、发布
  - 管理工厂草稿、发布版本、回滚

## 3.3 新增角色运行逻辑台

- 路由：`/characters/$characterId/runtime`
- 目标：
  - 看角色当前如何回复
  - 看角色当前如何生活
  - 管理运行时可改参数和手动锁定状态

## 3.4 收窄旧角色编辑页

- 路由：`/characters/$characterId`
- 调整后只保留：
  - 名称
  - 头像
  - 关系类型
  - 简介
  - 是否模板
- 不再承担完整人格、记忆、生活策略配置。

## 4. 边界定义

## 4.1 哪些配置归角色工厂

- 这是“设计期 / 制造期”的内容：
  - 身份
  - 关系
  - 能力域
  - 语气
  - 行为模式
  - 边界
  - 默认记忆底稿
  - 默认生活策略
- 工厂保存的是“角色配方”。
- 发布时把配方映射到 `Character` 运行实体。

## 4.2 哪些配置归角色运行逻辑台

- 这是“运行期 / 当前态”的内容：
  - 当前在线状态
  - 当前活动状态
  - 在线状态模式：`auto | manual`
  - 当前活动模式：`auto | manual`
  - 当前 recent summary
  - 当前 narrative progress
  - 当前 prompt 快照
  - 当前会话窗口
  - 当前调度与触发结果

## 4.3 哪些配置归世界级控制台

- 这是“实例全局规则”：
  - provider
  - reply runtime rules
  - 全局规则预览
  - 会话分支调试

## 5.5 回填策略

- 每个已有角色第一次进入工厂页时：
  - 若没有 blueprint，则从当前 `Character` 反向生成一个 blueprint
  - 默认：
    - 内置默认角色：`sourceType = default_seed`
    - 其他角色：`sourceType = manual_admin`
- 首次回填后自动创建一条 revision：
  - `changeSource = seed_backfill`

## 6. 后端接口设计

## 6.1 角色工厂接口

- `GET /api/admin/characters/:id/factory`
  - 返回：
    - `character`
    - `blueprint`
    - `publishedRecipe`
    - `draftRecipe`
    - `sourceType`
    - `status`
    - `publishedVersion`
    - `publishedRevisionId`
    - `fieldSources`
    - `diffSummary`

- `PATCH /api/admin/characters/:id/factory`
  - 用途：
    - 保存工厂草稿
  - 行为：
    - 只更新 `draftRecipe`
    - 不改 `Character`

- `POST /api/admin/characters/:id/factory/publish`
  - 用途：
    - 发布草稿到运行实体
  - 行为：
    - 创建 revision
    - 用 recipe 映射更新 `Character`
    - 更新 `publishedRecipe` / `publishedVersion`

- `GET /api/admin/characters/:id/factory/revisions`
  - 列出版本

- `GET /api/admin/characters/:id/factory/revisions/:revisionId`
  - 读取指定版本详情

- `POST /api/admin/characters/:id/factory/revisions/:revisionId/restore`
  - 把旧版本恢复为当前 draft
  - 不自动发布

## 6.2 角色运行逻辑台接口

- `GET /api/admin/characters/:id/runtime`
  - 返回：
    - `character`
    - `runtimeState`
    - `replySnapshot`
    - `lifeSnapshot`
    - `memorySnapshot`
    - `schedulerSnapshot`
    - `sourceSummary`

- `PATCH /api/admin/characters/:id/runtime`
  - 用途：
    - 修改运行期字段
  - 允许修改：
    - `onlineMode`
    - `activityMode`
    - `isOnline`
    - `currentActivity`
    - `profile.memory.recentSummary`
    - `profile.memory.coreMemory`
    - `profile.memory.forgettingCurve`

- `POST /api/admin/characters/:id/runtime/preview`
  - 用途：
    - 输入一条用户消息，预演当前链路
  - 返回：
    - 状态门控结果
    - prompt sections
    - effective prompt
    - history window
    - branch summary
    - provider summary

## 6.3 调度与生活逻辑接口

- `GET /api/admin/runtime/scheduler`
  - 返回：
    - jobs
    - lastRunAt
    - lastOutcome
    - nextWindowHint

- `GET /api/admin/characters/:id/life`
  - 可选
  - 如果后续 runtime 页过重，可拆独立 life snapshot 接口

## 12. 验收标准

## 12.1 角色工厂验收

- 进入角色工厂页时，旧角色会自动回填 blueprint。
- 修改工厂草稿不会立即污染运行中的 `Character`。
- 点击发布后，`Character` 字段按映射更新。
- revision 可见、可恢复。

## 12.2 角色运行逻辑台验收

- 能看到当前角色的真实回复快照。
- 能看到在线/活动是自动还是手动。
- 切到 `manual` 后，调度器不会覆盖该状态。
- 能看到生活逻辑参数和最近一次调度结果。

## 12.3 世界级控制台验收

- `/reply-logic` 仍然正常可用。
- 世界级规则修改后，角色 runtime 和世界 reply snapshot 会同步体现。

## 13. 风险与处理

## 风险 1：三套页面重复编辑同一字段

- 处理：
  - 旧编辑页收窄
  - factory 管设计期
  - runtime 管运行期

## 风险 2：Blueprint 与 Character 不一致

- 处理：
  - 发布时只允许 blueprint -> character 单向映射
  - 非 runtime 字段尽量不允许直接绕过 blueprint 修改

## 风险 3：`synchronize: true` 下 JSON 结构快速变化

- 处理：
  - blueprint 首版用 `simple-json`
  - 少做复杂关系

## 风险 4：runtime 页和 `/reply-logic` 组件重复

- 处理：
  - 提前抽公共组件：
    - `ActorSnapshotCard`
    - `HistoryList`
    - `NarrativeCard`
    - `RuntimeRulesEditorCard`

## 14. 推荐下一个实现顺序

1. 先做 Phase 1 和 Phase 2，把 blueprint 数据层和接口打通。
2. 紧接着做 Factory 只读页，确认回填模型和发布映射没偏差。
3. 再做 Factory 编辑与发布。
4. 然后做 Runtime 页，先复用现有 reply snapshot。
5. 最后补 scheduler 可观测性和旧页收口。

## 15. 本计划对应的直接结论

- 这件事不能只继续扩展当前 `/reply-logic`。
- 应该正式落成三层：
  - 角色名册
  - 角色工厂
  - 角色运行逻辑台
- 当前最值得立刻开工的是：
  - `CharacterBlueprintEntity`
  - `CharacterBlueprintRevisionEntity`
  - `CharacterBlueprintService`
  - `GET/PATCH/PUBLISH factory` 接口
  - `character-factory-page.tsx`
