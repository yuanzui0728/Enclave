# Web 端电脑版好友朋友圈独立页对齐微信深度规划

## 任务目标
- 把桌面版“好友的朋友圈”从当前的“在大朋友圈工作区里切作者筛选态”，升级成微信电脑端那种**独立页面 / 独立界面**。
- 明确拆开两个概念：
  - `/tabs/moments`：全局朋友圈主入口
  - `好友朋友圈独立页`：从好友资料进入，只看某位好友的朋友圈相册 / 时间线
- 避免继续在现有 `DesktopMomentsWorkspace` 上叠加 `authorId` 筛选语义，导致全局页和好友页混成一套。

### 微信电脑端近期开口径
- 2025-03-21，IT 之家转引微信官网信息，提到 Windows / macOS `4.0.3` 测试版支持：
  - 发表朋友圈
  - 查看朋友的朋友圈相册
  - 朋友圈置顶内容
- 2025-04-01，IT 之家在 `4.0.3` 正式版实测里明确写到：
  - 点击任一朋友资料详情页，中间会新增“朋友圈”板块
  - 鼠标移入后右侧出现 `>`，点击后进入**朋友的朋友圈界面**
- 2026-03-05，IT 之家在 `4.1.8.21` 内测版更新详解中提到：
  - PC 端尝试过把朋友圈等入口统一收进“发现页”
  - 但这个“发现页”后来又被移除
  - 各业务入口重新回到独立状态

### 由微信口径推导出的产品结论
- 这是**推断**，不是官方设计文档原文：
  - 微信电脑端现在更接近“独立业务入口 + 好友内容独立界面”的模式
  - 而不是“进入大朋友圈页后，再用作者筛选切到某个好友”
- 所以隐界桌面端如果要抄微信电脑端，应该把“好友朋友圈”做成**独立页面**，而不是把 `authorId` 继续当主路径。

## 当前真实现状

### 已经做了什么
- 桌面端已经有全局朋友圈主入口 `/tabs/moments`。
- 通讯录好友详情、好友资料页、聊天信息侧栏、头像资料卡，已经都能跳到朋友圈。
- 但这些入口目前都还是跳到：
  - `/tabs/moments#authorId=xxx`
- 也就是说，用户进入的仍然是**同一个全局朋友圈工作区**，只是在里面切了作者筛选态。

### 当前结构性问题
- `DesktopMomentsWorkspace` 目前把两个产品语义混在了一起：
  - 全局朋友圈浏览
  - 单个好友的朋友圈浏览
- `activeAuthorId` 本质只是局部 state / hash state，不是一个真正的页面。
- 这会带来四个问题：
  - 路径语义不对：现在还是“全局朋友圈页”，不是“好友朋友圈页”
  - 视觉语义不对：页面里仍有“作者列表 / 查看全部 / 作者摘要侧栏”这类筛选态 UI
  - 交互语义不对：点击作者头像和点击好友资料里的“朋友圈”，都落到同一个筛选逻辑
  - 扩展语义不对：后续想补封面、相册、置顶、作者介绍时，会被全局页结构卡死

## 总判断

### 结论
- 现在这套 `authorId` 方案，只适合当短期过渡，不适合继续演进。
- 用户说得对：如果目标是抄微信电脑端，**好友朋友圈必须是单独一个页面**。

### 这次规划的核心调整
- 不再以“作者筛选”作为好友朋友圈的最终形态。
- 改成“两页模型”：
  - 页 1：全局朋友圈主页
  - 页 2：好友朋友圈独立页

## 目标信息架构

### 一、全局朋友圈主页
- 路径：`/tabs/moments`
- 语义：世界里所有朋友圈内容的桌面主入口
- 角色：
  - 浏览全局动态流
  - 发朋友圈
  - 查看单条详情
  - 从作者头像 / 资料入口跳进好友朋友圈独立页

### 二、好友朋友圈独立页
- 建议路径：`/tabs/moments/friend/$characterId`
- 语义：某个好友的朋友圈相册 / 时间线页
- 角色：
  - 展示该好友的朋友圈内容
  - 展示好友头部资料摘要
  - 展示空态 / 无动态态
  - 从这里继续打开单条朋友圈详情

### 为什么不用 `#authorId=xxx`
- hash 更适合“页内状态”，不适合“独立业务页面”。
- 好友朋友圈在微信电脑端里是一个可明确进入的界面，不是页内临时切片。
- 路径如果仍然是 `/tabs/moments#authorId=xxx`，就会持续误导实现往筛选态走。

## 目标交互

### 微信式主路径
1. 用户在通讯录 / 聊天信息 / 头像资料卡里点击“朋友圈”。
2. 跳到 `好友朋友圈独立页`，不是跳到全局页后再高亮一个作者。
3. 页面顶部出现好友资料头部，而不是“全局朋友圈工具条 + 作者筛选标签”。
4. 主内容区只展示这个好友的朋友圈。
5. 点击返回时，优先回上一页；没有历史时再回：
  - 通讯录来源：`/tabs/contacts`
  - 聊天来源：当前聊天
  - 默认：`/tabs/moments`

### 页面内交互
- 点击朋友圈作者头像：
  - 如果是全局页，进入好友朋友圈独立页
  - 如果已经在好友页，不再触发作者筛选，保持当前页
- 点击某条朋友圈：
  - 在当前好友页里展开详情，不跳回全局页
- 点击“查看全部朋友圈”：
  - 从好友页回到全局页
- 空态：
  - 明确写“TA 还没有发表朋友圈”
  - 不再把用户送回全局页

## UI 方向

### 一、全局朋友圈主页不再承担好友页职责
- 保留全局浏览、发帖、单条详情。
- 移除或弱化当前“作者筛选态”相关结构：
  - `selectedAuthorName`
  - `onClearAuthor`
  - `activeAuthorId` 驱动的作者模式
  - 右侧“作者时间线”模式
- `DesktopMomentsSidebar` 应收口成：
  - 概览
  - 单条详情
  - 发帖入口
- 不再承载“某个作者的时间线页”。

### 二、好友朋友圈独立页的页面气质
- 要更接近微信电脑端的“个人相册 / 时间线页”，不是全局流加一个过滤条件。
- 建议布局：
  - 顶部：返回、好友头像、昵称、关系说明、可选“更多”
  - 中部：封面区域 / 资料头部
  - 下部：按时间倒序的朋友圈时间线
- 第一阶段不必硬抄真实封面图结构，但至少要把**页面分层**拉开：
  - 顶部头部是好友
  - 内容区是好友时间线
  - 不出现全局作者列表

### 三、微信式不要做的东西
- 不要在好友页里继续出现“只看角色 / 只看我 / 全部”。
- 不要出现“查看全部作者”侧栏。
- 不要让好友页和全局页只是标题不同，主体完全一样。
- 不要通过清空筛选的方式“返回好友页外层”。

## 路由与状态设计

### 一、建议新增路由
- 新增：`/tabs/moments/friend/$characterId`

### 二、建议新增 route-state
- 新增：
  - `apps/app/src/features/desktop/moments/desktop-friend-moments-route-state.ts`

### 三、建议状态结构
```ts
export type DesktopFriendMomentsRouteState = {
  momentId?: string;
  source?: "contacts" | "character-detail" | "chat-details" | "avatar-popover" | "moments";
};
```

### 四、状态职责划分
- 路径参数 `characterId`：定义“正在看谁的朋友圈”
- hash 里的 `momentId`：定义“当前页内选中的那条朋友圈”
- hash 里的 `source`：仅用于回退 fallback，不参与数据筛选

### 五、现有 route-state 的处理
- `desktop-moments-route-state.ts` 里的 `authorId` 应降级为兼容字段，不再作为主路径继续扩。
- 过渡期建议：
  - 保留解析
  - 但所有新入口一律跳到 `/tabs/moments/friend/$characterId`
- 最终目标：
  - 全局朋友圈 route-state 只保留页内状态，例如 `momentId`
  - 好友朋友圈独立页自己维护自己的 `momentId`

## 数据与接口策略

### Phase 1：前端先分页面，不改后端
- 复用现有 `getMoments(baseUrl)`。
- 在 `FriendMomentsPage` 数据容器里按 `characterId` 过滤。
- 这是**实现层过渡**，不是产品层筛选。
- 关键区别：
  - 即便暂时复用同一接口，表现层也已经是**独立页面**
  - 不再让全局朋友圈工作区承担好友页职责

### Phase 2：补独立接口
- 新增独立路由，避免重新把 `authorId` 塞回 `GET /api/moments`
- 建议新增：
  - `GET /api/moments/characters/:characterId`

### 返回结构建议
```ts
type CharacterMomentsPagePayload = {
  profile: {
    characterId: string;
    name: string;
    avatar?: string | null;
    remarkName?: string | null;
    signature?: string | null;
    relationship?: string | null;
  };
  timeline: Moment[];
  totalCount: number;
};
```

### 为什么不改 `GET /api/moments?authorId=...`
- 当前项目已经明确把 `GET /api/moments` 收口成“当前世界的朋友圈流”，不再接受 `authorId`。
- 重新塞回 query 参数，会和现有单世界口径冲突。
- 独立接口更清晰，也更利于之后补封面、置顶、相册摘要。

## 分阶段实施

### Phase 1：独立页骨架落地
目标：先把“好友朋友圈是独立页”这件事立起来。

改动范围：
- 新增路由 `/tabs/moments/friend/$characterId`
- 新增 `FriendMomentsPage`
- 新增 `desktop-friend-moments-route-state.ts`
- 所有资料入口改为跳新路由
- 全局朋友圈点击作者改为跳新路由

这一阶段不做：
- 新接口
- 封面图真实上传
- 相册网格
- 置顶内容

### Phase 2：全局页去作者筛选化
目标：让 `/tabs/moments` 和好友页真正分工，不再共享一套作者模式。

改动范围：
- `DesktopMomentsWorkspace` 删除作者模式
- `DesktopMomentsSidebar` 删除作者时间线侧栏
- `DesktopMomentsToolbar` 删除作者筛选 chip
- 清理 `authorId` 相关 route-state 的主流程使用

这一阶段完成后：
- 全局页就是全局页
- 好友页就是好友页

### Phase 3：独立接口与资料头部增强
目标：把好友朋友圈独立页的数据语义补完整。

改动范围：
- 后端新增 `GET /api/moments/characters/:characterId`
- 合同新增 `CharacterMomentsPagePayload`
- 好友页头部增加：
  - 好友资料摘要
  - 动态总数
  - 最近发布

### Phase 4：微信完整版相册能力
目标：继续向微信电脑端“朋友圈相册”靠近。

后续能力：
- 封面图
- 置顶朋友圈
- 相册视图
- 多图 / 视频
- 更接近微信的评论区排布

## 风险与注意点

### 一、不能继续把独立页做成“换了标题的筛选页”
- 这是这次最关键的边界。
- 如果新页只是把 `authorId` 从 hash 挪到 path，但工作区主体不变，那仍然没抄到微信电脑端。

### 二、全局页和好友页一定要拆职责
- 不能再让 `DesktopMomentsWorkspace` 同时承担：
  - 全局动态
  - 作者时间线
  - 作者空态
  - 作者侧栏

### 三、当前已有一份未提交中的聊天信息面板大改
- 实施时要注意和当前工作区里可能存在的联系人资料式重构合并，不要覆盖用户已有改动。
- 特别是：
  - `apps/app/src/features/desktop/chat/desktop-chat-details-panel.tsx`

### 四、落地时要更新 AGENTS.md
- 因为这次会新增路由结构。
- 一旦真正实现了新页面和新路由，需要立即同步更新 `AGENTS.md` 的页面口径。

## 验收标准
- 从通讯录 / 聊天 / 头像资料卡进入好友朋友圈时，URL 进入的是**独立路径**，而不是 `/tabs/moments#authorId=...`
- 好友朋友圈页不再出现全局作者筛选 UI
- 全局朋友圈页点击作者，进入好友独立页
- 好友没有朋友圈时，停留在好友页空态，不跳回全局页
- 浏览器回退符合进入来源预期
- 全局朋友圈和好友朋友圈页面在视觉上有明显角色区分

## 建议执行顺序
1. 先建好友朋友圈独立页和新路由。
2. 再把所有资料入口切过去。
3. 然后把全局朋友圈里的作者筛选模式下掉。
4. 最后再补独立接口和相册增强能力。
