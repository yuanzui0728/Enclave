# 多语言剩余问题治理规划

日期：2026-04-23

## 1. 当前结论

已经落地的能力解决了“继续无感变差”的第一层问题：全量 hardcode baseline、ratchet、changed missing check 都已经可跑。但多语言体系还没有真正完成，当前仍处在“有闸门、有运行时、少量页面接入”的阶段。

剩余问题可以分成 9 类：

1. 存量 UI 硬编码规模很大，语言切换只能覆盖已接入 catalog 的小部分界面。
2. 管理后台 catalog 仍有缺失翻译，切换英文/日文/韩文会 fallback 中文。
3. 组件 props 和 contracts 仍大量使用 `title/label/message: string`，会持续诱导新功能写死字符串。
4. 后端错误仍是 message 字符串协议，前端只能被动展示服务端源码语言。
5. 格式化、排序、语音、AI 默认语言仍散落硬编码 `zh-CN`、`zh`。
6. 游戏、角色预设、系统配置、运行记录等数据内容还没有 `LocalizedText` 结构。
7. 目前 scanner 是正则级别，能做 ratchet，但存在误报、漏报和跨行文案识别不足。
8. `i18n:hardcode:ratchet` 是全局计数，不能完全防止“一个地方减少、另一个新功能增加”的抵消。
9. 目前没有端到端语言切换验收矩阵，无法保证 Web / Desktop / Android / iOS / Admin / Cloud Console 的核心路径都生效。

## 3. 问题一：存量 UI 硬编码

现状：

- App 和 Admin 都有大量 JSX prop、对象数组字段、裸中文节点。
- 已接入的 Provider 和 LanguageSwitcher 只影响少数 catalog 消息。
- 业务页面里 `tabs/cards/status maps/table columns/actions` 大量仍是字符串数组。

影响：

- 用户切换语言后，只有壳层或少量入口变化，页面主体仍是中文。
- 后续新页面如果复制老页面模式，会继续写死字符串。

治理方案：

- 先迁移“共享组件契约”，再迁移页面。
- 页面文案分三层：静态 JSX 用 `<Trans>`，数组配置用 `msg` descriptor，DOM 字符串 prop 用 `i18n._(msg)`。
- 每迁移一个页面后同步下调 baseline，不留下“已修但 baseline 没降”的治理漏洞。

优先级：

1. Admin root/shell 相关缺译补齐。
2. Admin 高密度工作台先做 2 个样板页：`dashboard-page` 和 `reply-logic-page`。
3. App 高频路径先做聊天：`chat-message-list`、`chat-composer`、`desktop-chat-details-panel`。
4. 数据型页面再做游戏、小程序、二维码、搜索。

## 4. 问题二：缺失翻译仍 fallback 中文

现状：

- `admin` catalog 还有 `145` 条空 msgstr。
- 当前 `i18n:missing:changed` 只拦截新增空翻译，不拦截历史空翻译。

影响：

- 后台切到英文、日文、韩文后，部分 root-layout、digital human summary、导航说明仍显示中文。
- 用户会感知为“语言切换没彻底生效”。

治理方案：

- 先补齐 Admin 现有 `145` 条缺失翻译。
- 然后把 `pnpm i18n:missing --surface admin` 纳入 Admin 发布前检查。
- 等 App 页面迁移扩大后，再把 `app`、`shared`、`cloud-console` 也纳入全量 missing 阻断。

验收：

- `node ./scripts/i18n-missing-check.mjs --surface admin` 通过。
- Admin 切换 `en-US/ja-JP/ko-KR` 时 root shell、sidebar、dashboard hero、digital human summary 不再 fallback 中文。

## 5. 问题三：组件契约仍然强迫字符串

现状：

- `packages/ui`、`apps/app/src/components`、`apps/admin/src/components` 里大量 props 是 `title: string`、`description: string`、`label: string`、`message: string`。
- `packages/contracts` 里也有很多 `label/title/description/message: string`，其中一部分是用户可见文案，一部分是业务数据，边界没拆清。

影响：

- 开发者即使用 Lingui，也会被组件类型迫使先转成字符串。
- 可组合的 ReactNode 文案、带组件插值的 `<Trans>`、message descriptor 都不好传递。
- contract 层会把“可翻译 UI 文案”和“业务数据字段”混在一起。

治理方案：

- UI 组件 props 迁移为 `ReactNode`，需要 DOM string 的地方由调用方显式 `i18n._()`。
- 配置数组类型从 `label: string` 改为 `label: MessageDescriptor` 或 `labelNode: ReactNode`。
- contracts 只保留真实业务数据字符串；用户可见枚举标签、说明、状态解释迁移成客户端 copy registry 或 `LocalizedText`。

优先组件：

- `packages/ui/src/components/app-header.tsx`
- `packages/ui/src/components/panel-empty.tsx`
- `packages/ui/src/components/error-block.tsx`
- `packages/ui/src/components/metric-card.tsx`
- `packages/ui/src/components/setup-scaffold.tsx`
- `apps/app/src/components/empty-state.tsx`
- `apps/app/src/features/chat-details/*`
- `apps/admin/src/components/admin-sidebar.tsx`
- Admin 页面内复用卡片/空态/表格标题组件

验收：

- 新增 UI 组件默认不再暴露单纯 `string` 文案 props。
- scanner 中 `object-ui-string` 和 `jsx-literal-prop` 的新增项不能通过 changed check。

## 6. 问题四：后端错误仍是字符串协议

现状：

- `api/src` 有 `239` 个 literal error signal。
- `apps/cloud-api/src` 有 `57` 个 literal error signal。
- `packages/contracts/src/client.ts` 和 `apps/admin/src/lib/admin-api.ts` 仍以 `message` 字符串为主。

影响：

- 后端抛中文，所有前端都显示中文。
- 后端抛英文，中文用户也显示英文。
- 同一个错误在 App/Admin/Cloud Console 无法按界面语言统一展示。

治理方案：

- 新增 `ApiErrorPayload`：`code + params + fallbackMessage`。
- 后端提供 `businessError(code, params, fallbackMessage)` helper。
- 前端 API client 解析错误码后进入 `errorMessages` registry。
- 先迁移用户高频可见错误，不先迁移纯内部调试错误。

优先迁移模块：

1. `chat`、`group`、`moments`、`favorites`，因为 App 用户直接看到。
2. `inference`、`admin`、`system`，因为 Admin 初始化和模型配置直接看到。
3. `cloud-api/auth`、`cloud-api/cloud`，因为云平台登录、验证码、世界唤起直接看到。
4. `action-runtime`、`reminder-runtime`、`games`，作为第二批后台工作台。

验收：

- 新增 user-facing exception 不允许直接传字符串。
- `packages/contracts/src/client.ts` 不再把任意服务端 `message` 当主展示文案。
- 保留 `fallbackMessage` 兼容旧客户端。

## 7. 问题五：语言、日期、排序、语音仍散落硬编码

现状：

- App 仍有多个 `new Intl.DateTimeFormat("zh-CN")`、`localeCompare(..., "zh-CN")`。
- API 仍有 `toLocaleString('zh-CN')`、`language: 'zh'`。
- `real-world-sync`、`cyber-avatar`、`need-discovery` 等默认 Google News/Prompt 语言固定中文。
- `use-speech-input` 已有 resolver，但 provider/STT/TTS 服务端仍有 `zh` 默认值。

影响：

- UI 文案变了，日期、排序、语音识别、AI 输出仍可能中文化。
- 英文/日文/韩文用户看到混合语言，尤其是聊天、新闻、AI 通话。

治理方案：

- `packages/i18n/runtime/formatters.ts` 扩展：`formatRelativeTime`、`formatDateRange`、`getCollator`、`resolveSpeechLocale`、`resolveProviderLanguageCode`。
- App 所有 `Intl` 和 `localeCompare` 改走 hook/formatter。
- 服务端新增 `LocalePolicy` 或共享 map，区分 `uiLocale`、`contentLocale`、`providerLanguageCode`。
- 世界主人 profile 增加 `preferredLocale`、`contentLocale`，不把 UI 语言和 AI 内容语言绑死。

验收：

- scanner 中 `intl-locale-literal`、`locale-compare-literal`、`speech-lang-literal` 降到 0 或 allowlist。
- AI/STT/TTS 默认语言从 owner/content locale 派生。

## 8. 问题六：数据内容没有 LocalizedText

现状：

- `games.ts`、`characters.ts`、`evals.ts`、`reply-logic.ts`、`system.ts` 等 contracts 有大量 `title/description/label/message: string`。
- API 种子数据中角色预设、游戏目录、Prompt/规则说明大多是单语言。

影响：

- 即使 UI 页面都迁移，列表里的游戏名、角色描述、运行记录原因、配置说明仍无法多语言。
- 后续运营后台如果继续保存单字符串，翻译只能靠前端临时映射，无法覆盖动态内容。

治理方案：

- 在 `packages/contracts` 新增：

```ts
export type LocalizedText = {
  defaultLocale: SupportedLocale;
  values: Partial<Record<SupportedLocale, string>>;
};
```

- 在 `packages/i18n` 新增 `resolveLocalizedText(localized, locale, fallback)`。
- 数据库/实体不一次性破坏旧字段，采用 `titleI18n`、`descriptionI18n`、`localizedPayload` 兼容扩展。
- 后台编辑器支持默认语言字段和其他语言折叠字段。
- 种子数据逐步迁移：游戏目录、角色预设、系统配置说明优先。

验收：

- 新增运营配置不再只允许单语言字符串。
- 已迁移数据在目标 locale 有值时展示目标语言，无值时按明确 fallback 展示。

## 9. 问题七：scanner 还是正则级别

现状：

- 当前 scanner 能做总量治理，但它不理解 AST。
- 对跨行 JSX、模板字符串、数组对象、条件表达式、变量引用的识别不完整。
- `raw-cjk-line` 会把 Prompt、测试样本、种子数据、真实 UI 文案混在一起。

影响：

- 误报会让 baseline 看起来过大。
- 漏报会让部分新增硬编码绕过检查。
- 无法精确给出“这个文件应该怎么改”的机器化建议。

治理方案：

- 保留正则 scanner 做快速总量趋势。
- 新增 AST scanner 做 changed/new file 精准拦截。
- AST scanner 用 TypeScript AST 或 Babel parser，识别：
  - JSXText
  - JSXAttribute
  - ObjectLiteralExpression key
  - CallExpression：toast/alert/confirm/setError/throw
  - NewExpression：Intl
  - TaggedTemplateExpression：允许 `msg/t`
- 引入 `i18n-ignore-next-line: reason` 强制要求 reason，避免随意忽略。

验收：

- 新文件有硬编码时，不能靠全局 baseline 抵消通过。
- PR 输出具体规则、文件、行号、建议替代写法。

## 10. 问题八：ratchet 粒度仍太粗

现状：

- baseline 是按 scope + rule 计数。
- 如果一个页面减少 10 个硬编码，另一个新功能新增 8 个，ratchet 仍然通过。

影响：

- 能防止整体变差，但不能严格保证“新功能自动适配多语言”。

治理方案：

- 增加 per-file baseline。
- 新增文件 baseline 默认为 0。
- changed files 使用更严格规则：只要新增硬编码就失败，不允许被其他文件减少抵消。
- 只对历史 untouched files 使用全局 ratchet。

验收：

- `pnpm i18n:hardcode:changed` 能阻断本次改动新增硬编码。
- `pnpm i18n:hardcode:ratchet` 负责存量总量下降。

## 11. 问题九：缺少端到端验收矩阵

现状：

- 目前验证主要是脚本级。
- 没有覆盖“切换语言后页面是否真的变”的 smoke。

影响：

- Provider、catalog、组件、路由壳层其中任何一环坏了，脚本可能仍通过。
- 各端入口可能出现 App 有效、Admin 无效、Cloud Console 无效的问题。

治理方案：

- 建立手工 + 自动 smoke 矩阵。
- 第一阶段不新增重型 E2E，先做轻量 DOM smoke：
  - 启动 app/admin/cloud-console
  - 设置 `?locale=en-US`
  - 检查已迁移的 sentinel 文案不再是中文
  - 切到 `zh-CN` 检查中文恢复
- 对 App 的桌面/移动共用 web 入口先验；Android/iOS/desktop 壳以后只验证 shell 是否正确加载相同 web build。

验收路径：

- Admin：sidebar、topbar、dashboard、inference。
- App mobile：消息、通讯录、发现、我、聊天页。
- App desktop：左侧导航、聊天工作区、联系人、设置。
- Cloud Console：登录、dashboard、world detail。

## 12. 推荐执行顺序

### 第 1 批：让当前体验明显变好

目标：用户马上看到语言切换更完整。

交付：

- 补齐 Admin `145` 条缺失翻译。
- 把 `node ./scripts/i18n-missing-check.mjs --surface admin` 接入 Admin 专项检查。
- 迁移 Admin shell/root-layout/digital-human-summary 中已提取消息。

验证：

- `pnpm i18n:extract`
- `node ./scripts/i18n-missing-check.mjs --surface admin`
- `pnpm i18n:compile`
- `pnpm i18n:check:changed`

### 第 2 批：让新功能更难写错

目标：把“自动适配”从口头规范变成类型和脚本约束。

交付：

- `i18n-hardcode-scan` 增加 per-file baseline。
- 新增 `i18n:hardcode:changed`。
- 改造共享 UI 组件 props：`string` -> `ReactNode` / `MessageDescriptor`。
- 新增 `useT()` 或轻量 `useI18nText()`，统一业务层拿 `i18n._` 和 formatters。

验证：

- 新建一个测试 fixture 页面写硬编码，确认 changed scanner 失败。
- 新建一个使用 `<Trans>`/`msg` 的 fixture，确认通过。

### 第 3 批：迁移高频 UI 页面

目标：覆盖用户最常用路径。

Admin：

- `dashboard-page`
- `reply-logic-page`
- `need-discovery-page`
- `cyber-avatar-page`
- `action-runtime-page`

App：

- `chat-message-list`
- `chat-composer`
- `desktop-chat-details-panel`
- `group-chat-details-page`
- `character-detail-page`
- `profile-settings-page`

每迁移一个页面必须：

- 下调 baseline。
- 补齐 catalog。
- 记录手工切换语言验证点。

### 第 4 批：错误协议

目标：用户可见错误不再受服务端源码语言影响。

交付：

- `ApiErrorPayload`。
- `BusinessErrorCode`。
- `throwBusinessError()` helper。
- `packages/i18n` error registry。
- 迁移 chat/group/moments/admin-api/cloud-api auth 的首批错误。

验证：

- 后端返回错误码。
- 前端按当前 locale 显示错误。
- 旧 fallbackMessage 仍存在。

### 第 5 批：数据和 AI 内容 locale 化

目标：动态数据和 AI 输出也进入多语言策略。

交付：

- `LocalizedText` 类型和 resolver。
- 游戏目录、角色预设、系统配置说明的 localized 字段。
- owner profile 的 `preferredLocale/contentLocale`。
- prompt/STT/TTS 语言从 `contentLocale` 派生。

验证：

- UI 英文、AI 中文的组合可用。
- UI 英文、AI 英文的组合可用。
- 游戏/角色预设在有目标语言时显示目标语言，没有时按 fallback。

## 13. 里程碑验收

M1：后台切换可信

- Admin missing 为 0。
- Admin shell + dashboard + 2 个高频工作台切换有效。
- Admin 新增页面硬编码无法通过 changed check。

M2：App 高频路径可信

- 聊天列表、聊天页、输入框、聊天信息、群详情、个人设置切换有效。
- 日期、排序、语音识别语言不再固定中文。

M3：错误可信

- App/Admin/Cloud Console 不再直接展示服务端任意 `message`。
- 主要业务错误按 locale 展示。

M4：动态内容可信

- 游戏、角色预设、系统配置说明支持 `LocalizedText`。
- AI 内容语言与 UI 语言可独立配置。

M5：治理闭环可信

- 新文件 hardcode 阈值为 0。
- production locale 无 missing。
- baseline 按页面持续下降，并在核心 surface 清零。

## 14. 下一步建议

下一步不建议直接冲全量页面迁移。最优顺序是：

1. 先补 Admin `145` 条缺失翻译，让后台语言切换立刻不再 fallback。
2. 再做 per-file / changed hardcode scanner，避免新功能靠全局抵消漏进来。
3. 同时改共享组件 `string` props，否则页面迁移会反复被组件契约卡住。
4. 然后拿 `dashboard-page`、`reply-logic-page`、`chat-message-list` 做三类样板：后台总览页、后台复杂工作台、App 高频复杂组件。

这个顺序能最快把“用户感知”和“开发约束”同时拉起来。
