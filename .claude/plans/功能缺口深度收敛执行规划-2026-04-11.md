# 功能缺口深度收敛执行规划

## 1. 目标

- 不再继续做泛泛巡检，直接把当前已经确认的功能缺口转成可执行实施序列。
- 优先处理两类问题：
  - 前端已接线但后端缺失，现状会直接失败或 404。
  - 页面已经上线，但实际仍是本地占位链路，和产品口径不一致。
- 规划目标不是“一次性把所有后续能力都做完”，而是先把系统从“有入口但不闭环”收口到“入口真实可用、口径一致、剩余占位边界清楚”。

## 2. 当前审计结论

### 2.1 P0 级缺口

- `Admin /evals` 页面已经按真实后台能力在写，但 Core API 只实现了 `GET /system/evals/overview`，其余：
  - `datasets`
  - `strategies`
  - `prompt-variants`
  - `experiments`
  - `reports`
  - `runs`
  - `compare`
  - `comparisons`
  - `traces`
  - `trace detail`
  目前都没有后端路由承接。
- App 举报链路前端已经接 `createModerationReport()`，contracts 也已有 `/moderation/reports`，但 Core API 不存在 `moderation` 模块与控制器。

### 2.2 P1 级缺口

- 群语音 / 群视频页面当前只是“群消息状态卡片 + 本地工作台状态”，不是真实群通话链路。
- 移动端聊天 `+` 面板里：
  - `语音通话` 入口只接了单聊语音。
  - `视频通话` 入口仍为硬禁用。
- 桌面反馈没有正式提交后端，当前只有：
  - 本地草稿
  - 本地历史
  - 复制反馈包
- 直播伴侣没有真实推流或直播 session 后端，当前是桌面本地工具台。

### 2.3 P2 级缺口

- 游戏中心仍是内容工作区，不是小游戏容器。
- 若干资料页仍保留“后续接入”字段，例如好友详情里的朋友圈字段。
- 个别桌面工具页是“可运营工作台先行”，不是正式线上能力。

## 3. 实施原则

### 3.1 优先级原则

- 第一优先级只做“前端会失败”的真缺口。
- 第二优先级做“对外口径已像正式功能，但实际仍是占位”的能力。
- 第三优先级才处理“明确写了后续接入”的扩展能力。

### 3.2 收口策略

- 能快速闭环的，补成真实功能。
- 暂时不准备补全底层能力的，必须统一：
  - 入口状态
  - 文案口径
  - 页面行为
  - 是否保留入口
- 禁止继续出现“列表里像正式能力，点进去才知道没后端”的状态。

### 3.3 提交策略

- 每个阶段单独提交，不混做。
- 每个提交必须满足“至少打通一条真实链路”或“至少消灭一类假入口”。
- 不在同一提交里同时重做 `evals`、`moderation`、`直播伴侣` 三块，避免回归面失控。

## 4. 建议执行顺序

## 4.1 第一阶段：补齐真死链

### 4.1.1 Moderation 最小闭环

- 目标：
  - 让现有举报入口不再 404。
  - 先把举报能力收口成“可创建、可查询、可查看状态”。
- 最小实现范围：
  - 新增 `ModerationReport` 实体。
  - 新增 `moderation` 模块、service、controller。
  - 提供：
    - `GET /api/moderation/reports`
    - `POST /api/moderation/reports`
  - 先不做复杂审核流，只支持：
    - `open`
    - `reviewed`
    - `resolved`
- 前端范围：
  - 保持现有 App 举报入口可用。
  - 视情况在 Admin 补一个简易举报列表页，或先放到 Dashboard 摘要区。
- 验收：
  - 单聊详情举报可成功提交。
  - 举报记录能从 API 读出来。
  - 前端不再出现 404/接口不存在。

### 4.1.2 Admin Evals 真实化或暂时收口

- 这块不能直接硬上，要先做技术选择。
- 推荐方案：
  - 先做 `Phase 1A`，补齐只读数据接口，让页面至少能打开。
  - 再做 `Phase 1B`，补运行与决策写接口。
- `Phase 1A` 最小范围：
  - `GET /system/evals/datasets`
  - `GET /system/evals/strategies`
  - `GET /system/evals/prompt-variants`
  - `GET /system/evals/experiments`
  - `GET /system/evals/reports`
  - `GET /system/evals/runs`
  - `GET /system/evals/runs/:id`
  - `GET /system/evals/traces`
  - `GET /system/evals/traces/:id`
  - `GET /system/evals/comparisons`
- `Phase 1B` 范围：
  - `POST /system/evals/runs`
  - `POST /system/evals/compare`
  - `POST /system/evals/compare/run`
  - `POST /system/evals/experiments/:id/run`
  - `POST /system/evals/reports/:id/decision`
- 如果后端当前没有评测存储基础：
  - 先把 `/evals` 页面临时降级为“概览 + 明确未开放的区块”。
  - 不允许继续保留现在这种大面积请求不存在接口的状态。
- 验收：
  - 进入 `/evals` 不出现一串接口报错。
  - 概览、列表、详情至少能真实展示。
  - 若写接口未做完，前端必须明确禁用对应动作。

## 4.2 第二阶段：收口占位能力

### 4.2.1 群语音 / 群视频口径收口

- 当前不建议直接做真人群 RTC，工程跨度过大。
- 本阶段目标不是做真 RTC，而是把现有“群通话状态工作台”定义清楚并对齐入口。
- 具体策略：
  - 明确把现有实现定义为“群通话协调面板”。
  - 页面文案、入口、副标题统一，不再暗示真实音视频上传。
  - 若保留摄像头/麦克风 UI，必须明确是本地状态提示，不是媒体链路。
  - 桌面端和移动端入口统一到同一套语义。
- 产出：
  - 一套统一 copy。
  - 一套统一 route state。
  - 一组统一入口开关。
- 验收：
  - 用户不会再误判为真人群视频已打通。
  - 群聊卡片、详情页、群通话页的语义一致。

### 4.2.2 移动端聊天 `+` 面板入口收口

- 当前问题：
  - `video-call` 明明已有单聊视频路由，但入口仍硬禁用。
  - `voice-call` 入口只在一部分链路里可用，表现不统一。
- 本阶段目标：
  - 对单聊场景，把 `语音通话`、`视频通话` 入口统一接到真实路由。
  - 对不支持的场景明确禁用规则，而不是全局硬禁用。
- 建议落地：
  - `MobileChatPlusPanelProps` 增加 `onStartVideoCall`。
  - 按会话类型计算：
    - 单聊：语音 / 视频都开放。
    - 群聊：若仍是群通话协调面板，则按新口径展示。
    - 不支持对象：明确禁用并给出统一 fallback。
- 验收：
  - 单聊里 `+` 面板可直接发起语音和视频。
  - 不再出现“详情页能发起，+ 面板却说待接入”的冲突。

### 4.2.3 桌面反馈能力收口

- 当前不建议直接接复杂工单系统。
- 先做“服务端落档”版本：
  - 新增轻量 `feedback` 模块。
  - 支持提交、查询历史。
  - 保留本地草稿，删除“只有本地历史”的限制。
- 推荐接口：
  - `POST /api/feedback`
  - `GET /api/feedback`
- 记录字段：
  - 分类
  - 优先级
  - 标题
  - 详情
  - 复现步骤
  - 期望结果
  - 系统摘要
  - 提交时间
- 验收：
  - 点击“提交反馈”后能落到服务端。
  - 本地历史切换为服务端历史或本地+服务端混合视图。

## 4.3 第三阶段：桌面工具台真实化

### 4.3.1 直播伴侣

- 这块不建议首轮就做真实推流。
- 推荐拆成两步：
  - `Step A`：把“桌面本地工作台”升级成“服务端会话工作台”。
  - `Step B`：再考虑是否接外部推流 provider。
- `Step A` 范围：
  - 新增 live session 实体与 API。
  - 把草稿、开播状态、历史从 `localStorage` 移到服务端。
  - 保留当前内容准备与状态检查 UI。
- 推荐接口：
  - `GET /api/channels/live-companion/sessions`
  - `POST /api/channels/live-companion/sessions`
  - `PATCH /api/channels/live-companion/sessions/:id`
  - `POST /api/channels/live-companion/sessions/:id/end`
- 验收：
  - 多端刷新后仍能看到同一场直播准备和历史。
  - 页面不再声称“仅本地工作台”。

### 4.3.2 游戏中心 / 小程序 / 其他工具台

- 这些先不作为本轮主线。
- 统一要求：
  - 继续保留内容工作区可以。
  - 但入口、提示、页面标题必须明确是“工作台/容器占位”，不是已接入真实运行时。

## 7. 风险与规避

### 7.1 Evals 风险最高

- 风险：
  - 前端页面已经很大。
  - 后端当前几乎没有现成接口。
- 规避：
  - 不要一口气做完整评测平台。
  - 先把页面从“报错页面”收口到“可读页面”。

### 7.2 群通话容易误入 RTC 大工程

- 风险：
  - 一旦按“真实群视频”实现，会立刻跨到 WebRTC、房间媒体流、端状态同步。
- 规避：
  - 本轮只收口语义和入口，不碰真人群 RTC。

### 7.3 桌面工具台容易继续堆本地存储

- 风险：
  - 做着做着又变成一个 `localStorage` 页面。
- 规避：
  - 从第三阶段开始，新增能力默认优先服务端存储。

## 8. 验收口径

- P0 验收：
  - 举报入口真实可用。
  - `/evals` 不再充满不存在接口请求。
- P1 验收：
  - 单聊 `+` 面板通话入口和详情页入口一致。
  - 群语音 / 群视频不再伪装成真实媒体通话。
  - 桌面反馈不再只是本地历史。
- P2 验收：
  - 直播伴侣不再只是浏览器本地工具台。
  - 其余工作区的“占位态”语义清晰，没有误导性入口。

## 9. 本次推荐的实际开工顺序

1. 先做 `moderation`。
   - 体量小，收益直接，能立刻消灭真 404。
2. 再做 `evals` 第一阶段收口。
   - 决定是补只读接口还是先降级页面。
3. 再做移动端 `+` 面板通话入口统一。
   - 这是最明显的产品冲突点之一。
4. 再做群通话口径收口。
5. 最后推进桌面反馈和直播伴侣服务端化。

- 如果 `evals` 从 `system` 中拆成独立模块，立即补后台路由/接口说明。
