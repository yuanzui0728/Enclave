# Web 端手机版聊天页加号功能规划

## Summary
本次规划聚焦 `apps/app` Web 端手机版“会话页输入框右侧 `+`”能力，对齐微信聊天页附件面板的交互模型。

本次范围不包含：
- 消息列表页右上角 `+`
- `Mic` 语音转文字入口
- `Smile` 表情包入口

上面三者已经是独立入口，`+` 不应重复承接它们，而应承担微信式“更多消息能力/附件能力”的入口。

## 当前现状

### 前端现状
- `apps/app/src/components/chat-composer.tsx`
  - 移动端在输入框为空时展示 `+` 按钮。
  - 按钮当前没有任何点击行为。
  - 已经同时接入：
    - `Mic`：语音转文字输入
    - `Smile`：表情包面板
- `apps/app/src/features/chat/conversation-thread-panel.tsx`
  - 单聊页已通过 `ChatComposer` 接入文本、表情包、语音转文字。
- `apps/app/src/routes/group-chat-page.tsx`
  - 群聊页也复用 `ChatComposer`，但发送链路仍是纯文本。

### 契约与后端现状
- `packages/contracts/src/chat.ts`
  - `MessageType` 目前只有 `text | system | proactive | sticker`
- `packages/contracts/src/sticker-types.ts`
  - `MessageAttachment` 当前实际上等同于 `StickerAttachment`
- `packages/contracts/src/ws.ts`
  - 单聊 Socket 发送仅支持：
    - 文本消息
    - `sticker`
- `packages/contracts/src/chat.ts`
  - `SendGroupMessageRequest` 仍只有 `text`
- `api/src/modules/chat/chat.service.ts`
  - `attachmentKind / attachmentPayload` 已存在，但解析逻辑只识别 `sticker`
- `api/src/main.ts`
  - 当前没有聊天附件的静态资源或下载服务

### 结论
- 现在的 `+` 只是 UI 占位，不是功能缺一点，而是整条“附件消息”链路还没建立。
- 如果要做成接近微信的 `+` 面板，不能只补前端弹层，必须同时规划：
  - 面板交互
  - 附件选择器
  - 附件上传
  - 消息协议
  - 消息渲染
  - 单聊 / 群聊一致性

## 产品目标

### 目标定义
- 让 Web 端手机版聊天页的 `+` 入口在交互上接近微信：
  - 输入为空时显示 `+`
  - 点击后拉起底部附件面板
  - 以宫格方式展示可发送能力
  - 键盘、表情面板、附件面板三者互斥

### 对齐原则
- 对齐微信的是“交互骨架”和“使用心智”，不是机械照搬所有条目。
- 不把支付类能力硬塞进当前产品。
- 结合隐界单世界设定，保留可替换为“角色名片 / 场景卡片”的产品化映射。
- 首期只上线真实可用能力，不展示大量死入口。

## 微信对齐的目标形态

### 1. 入口规则
- 输入框有文本时：
  - `+` 隐藏
  - 右侧显示发送按钮
- 输入框为空时：
  - 显示 `+`
  - 点击 `+` 打开附件面板

### 2. 面板行为
- 面板从底部上滑出现，贴合安全区。
- 打开 `+` 面板时：
  - 主动收起键盘
  - 自动关闭表情包面板
  - 自动结束语音输入临时态
- 点击空白区域、再次点击 `+`、选择附件后，面板关闭。
- 面板高度建议控制在 `248px` 到 `320px`，与微信移动端接近。

### 3. 视觉结构
- 采用微信式浅灰背景附件板，不沿用当前高饱和品牌卡片。
- 每个能力项为：
  - 上方圆角方形图标底板
  - 下方短标签
- 首屏建议 4 列布局，最多 8 项；超过 8 项再做分页或横滑。

## 终态能力矩阵

### 推荐终态菜单
- 相册
- 拍摄
- 视频通话
- 位置
- 名片
- 文件

### 为什么不是完全照抄微信
- `红包 / 转账 / 收付款` 与当前产品语义不匹配，不建议进入聊天 `+`。
- `语音输入` 已经由 `Mic` 独立承接，不应再在 `+` 里重复出现。
- `表情包` 已经由 `Smile` 独立承接，也不应重复。

## 隐界产品映射建议

### A. 相册
- 语义与微信一致。
- 允许从系统相册选择图片。
- 首期建议支持：
  - 单张发送
  - 最多 9 张多选
- 多选后先进入发送前预览，而不是选完立即发。

### B. 拍摄
- 语义与微信一致。
- Web 手机版首期建议仅做“拍照”：
  - 使用 `input[type=file][accept=image/*][capture]`
- 不建议首期就承诺完整视频拍摄，因为 H5 浏览器支持不稳定。

### C. 视频通话
- 终态需要，但当前仓库没有实时音视频链路。
- 首期不要展示死入口。
- Phase 3 再决定是否接入 WebRTC 或壳能力桥接。

### D. 位置
- 不建议直接使用真实 GPS 作为首版核心依赖。
- 更适合映射为“场景卡片 / 当前世界地点卡片”：
  - 咖啡馆
  - 图书馆
  - 公园
  - 健身房
- 文案仍可对齐微信的“位置”，但底层能力更符合当前产品。

### E. 名片
- 映射为“角色名片 / 群聊名片”分享。
- 单聊里可分享：
  - 角色名片
  - 已有群聊
- 这是最适合隐界世界观、同时又贴近微信“名片”语义的能力。

### F. 文件
- 更偏桌面，但移动 Web 也可保留。
- 首期可只允许常见文档与图片外文件。
- 如果移动 Web 兼容性不好，可放到 Phase 2。

## 首期上线建议

### 必须首批上线
- 相册
- 拍摄
- 名片

### 第二批上线
- 位置（场景卡片）
- 文件

### 明确后置
- 视频通话

这样做的原因：
- 首批三项最接近微信主路径，也最能体现 `+` 面板存在价值。
- 位置和文件需要补更多附件协议与选择器。
- 视频通话不具备底层能力，强上只会制造空入口。

## 交互细节规划

### 1. 相册
- 点击后拉起系统文件选择器。
- 选中后进入“发送前预览条”：
  - 缩略图列表
  - 删除单张
  - 可选输入配文
  - `发送` / `取消`
- 和微信一致，不建议选完立刻发。

### 2. 拍摄
- 调起相机或拍照输入。
- 完成后进入同一套预览确认流程。
- 若浏览器不支持 `capture`，回退到普通文件选择器。

### 3. 名片
- 点击后弹出二级选择面板：
  - 角色名片
  - 群聊名片
- 角色名片优先使用：
  - 已建立关系的联系人
  - 或可见角色列表
- 发送后消息列表显示为卡片消息，不是普通文本。

### 4. 位置
- 点击后弹出场景选择面板，而不是立刻请求浏览器定位。
- 选中场景后发送“位置卡片消息”。
- 后续如果产品接入真实地理位置，再升级为“双模式”：
  - 世界场景
  - 真实位置

### 5. 面板状态互斥
- `+` 面板打开时关闭 `StickerPanel`
- `StickerPanel` 打开时关闭 `+` 面板
- 开始语音输入时关闭 `+` 面板
- 输入框重新聚焦时默认关闭 `+` 面板

## 技术方案

### 1. 前端状态分层
建议把当前 `ChatComposer` 中的浮层状态进一步拆分为：
- `composerSurface`: `keyboard | sticker | plus | idle`
- `plusActionDraft`: 当前附件草稿
- `plusActionAvailability`: 当前会话允许的动作集合

这样可以避免 `ChatComposer` 内继续堆多个 `useState`，后面变成状态冲突源。

### 2. 附件消息模型扩展
当前 `MessageAttachment = StickerAttachment` 已经不够用，建议改成真正的联合类型：

```ts
type MessageAttachment =
  | StickerAttachment
  | ImageAttachment
  | ContactCardAttachment
  | LocationCardAttachment
  | FileAttachment;
```

对应 `MessageType` 建议扩展为：
- `text`
- `system`
- `proactive`
- `sticker`
- `image`
- `contact_card`
- `location_card`
- `file`

如果后续坚持把所有非文本都统一进 `attachment`，也可以保留 `type: "attachment"`，但不如显式类型清晰。当前项目已有 `sticker` 先例，更适合继续显式扩展。

### 3. 上传链路建议
不建议把二进制直接塞进 Socket。

推荐两段式：
1. HTTP 上传附件
2. Socket 发送消息元数据

建议新增接口：
- `POST /api/chat/attachments`

职责：
- 接收 `multipart/form-data`
- 校验文件类型、大小、数量
- 落盘到服务端存储
- 返回标准化附件元数据

随后客户端再通过现有 `send_message` Socket 发送：
- `conversationId`
- `characterId`
- `type`
- `attachment`
- `text`（可选兜底文案）

### 4. 服务端存储建议
当前后端没有附件静态服务。

首期最务实的方案：
- 文件落在实例本地，例如 `api/storage/chat-attachments/...`
- 通过 Nest controller 提供读取 URL
- 消息 JSON 中只保存：
  - `kind`
  - `url`
  - `mimeType`
  - `width/height` 或 `size`
  - `title`
  - `preview`

不建议首期就引入对象存储，范围会失控。

### 5. 群聊一致性
`ChatComposer` 已被单聊和群聊共用，所以 `+` 面板不能只做单聊半套。

建议同步扩展：
- `GroupMessage`
- `SendGroupMessageRequest`
- `group-message.entity`
- 群聊消息渲染

否则会出现：
- 单聊能发图
- 群聊只能发文本

这会明显违背微信心智。

## 分阶段实施

### Phase 1：面板壳与状态互斥
目标：
- 先让 `+` 按钮真正可用
- 建立键盘 / 表情 / `+` 三态互斥
- 做出微信式底部宫格面板

包含：
- `ChatComposer` 状态改造
- `mobile-chat-plus-panel`
- 菜单配置化
- 先用 mock action 验证交互

### Phase 2：图片链路
目标：
- 打通最像微信、使用频率最高的主路径

包含：
- 相册
- 拍摄
- 附件上传接口
- 图片消息渲染
- 发送前预览确认
- 单聊 / 群聊统一

### Phase 3：卡片链路
目标：
- 补齐隐界自己的“微信化映射能力”

包含：
- 名片消息
- 位置（场景卡片）消息
- 选择器面板
- 会话列表最后一条文案兜底

### Phase 4：文件与高级能力
包含：
- 文件消息
- 上传异常与重试
- 大文件限制
- 图片多选体验优化

### Phase 5：视频通话决策
目标：
- 决定是否在 Web 端引入真正音视频能力

结论可能有两种：
- 接入真实音视频后再开放入口
- 如果短期不做，就继续不展示该入口

## 验收标准
- Web 手机版聊天页输入为空时显示 `+`，有内容时切换回发送按钮。
- 点击 `+` 后出现微信式底部附件宫格，而不是普通弹窗。
- `+`、表情包、语音输入三者不会互相打架。
- 至少一条完整附件链路可用：
  - 选择
  - 预览
  - 发送
  - 消息列表展示
- 单聊和群聊的 `+` 面板结构保持一致，不出现一边能发、一边不能发的大裂缝。
- 首批上线条目都是真实可用能力，不展示明显死入口。

## 风险与决策
- 最大风险不是 UI，而是附件协议当前仍是 `sticker` 专用结构。
- 第二个风险是群聊链路落后于单聊，如果不一起补齐，体验会非常割裂。
- 第三个风险是 H5 `capture`、多选上传、文件权限在不同浏览器表现不一致，所以“拍摄”首期应限定为拍照优先，不承诺视频。
- 第四个风险是如果把“位置”做成真实 GPS，会把权限、地图、坐标系问题一并引进来，不适合首期。

## 推荐执行顺序
1. 先把 `+` 面板壳和互斥状态做对。
2. 再优先打通“相册 / 拍摄”图片主链路。
3. 然后补“名片 / 位置”这类隐界化卡片能力。
4. 最后再决定文件和视频通话是否进入手机版首屏。
