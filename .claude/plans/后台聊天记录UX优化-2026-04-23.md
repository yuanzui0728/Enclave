# 后台聊天记录 UX 优化规划

## 目标

- 把 `聊天记录` 从“能看消息”的页面提升为“运营可连续处理”的工作台
- 让运营人员形成稳定路径：`找会话 -> 看上下文 -> 标样本 -> 跳联动台`
- 在不改后端接口和契约的前提下，先解决排版、信息层级和操作连续性问题

## 当前问题

1. 页面虽然是三栏，但功能并没有按运营任务分层，筛选、搜索、时间线、复盘、成本都处在同一优先级。
2. 左侧列表缺少“当前看到的是什么”解释，运营很难快速确认筛选口径、排序方式和分页结果。
3. 中间阅读区缺少当前会话上下文头部，进入页面后要自己拼接“这是哪个角色、现在看的是哪段样本、下一步该做什么”。
4. 搜索命中与完整时间线混排，命中结果出现后会把阅读区向下推长，定位和回到主时间线的状态不够清晰。
5. 右侧操作区没有 sticky 语义，长时间线滚动后，复盘状态、标签、备注和联动入口容易丢失。
6. 消息卡片是统一样式，用户消息、角色消息、系统消息没有明显阅读方向，长对话扫读成本偏高。

## 设计原则

- 先上下文后操作：先让运营知道当前在处理哪段会话，再进入搜索、阅读、复盘。
- 分工作流而不是分接口：围绕“导航”“阅读”“沉淀”三件事组织页面。
- 单一焦点优先：搜索命中、时间线过滤、复盘草稿都要显式告诉运营当前处于什么模式。
- 低风险重排：复用现有接口、字段和组件，不引入新的后端依赖。
- 运营语言优先：用中文状态、摘要和建议动作替代“接口数据直出”。

## 信息架构

### 1. 左侧会话导航

- 顶部保留筛选，但补足：
  - 当前筛选摘要
  - 分页结果说明
  - 默认视图说明
- 会话卡片强化：
  - 当前选中态
  - 近 7 天消息量
  - 复盘状态
  - 清空前历史 / 隐藏会话标记

### 2. 中间阅读工作区

- 在搜索区之前增加“当前工作会话”头部：
  - 角色名、关系、最近消息预览
  - 快捷跳转：运行逻辑台 / 回复逻辑 / Token 用量
  - 近 30 天消息、留存消息、成本等关键指标
- 搜索区与时间线工具条整合：
  - 搜索条件
  - 搜索命中状态
  - 导出动作
  - 时间线类型筛选
- 搜索命中单独作为一个“定位区块”，避免和正文时间线互相干扰
- 时间线内加入日期分割和消息方向区分：
  - 用户消息右对齐
  - 角色消息左对齐
  - 系统消息居中

### 3. 右侧运营动作区

- 右栏改成 sticky 工作区，避免长时间线下丢失操作位
- 复盘卡前置，优先于洞察和成本
- 增加草稿状态与快捷标签，降低标样本成本
- 洞察区和成本区保留，但压缩成更适合运营扫读的摘要结构

## 关键交互

### 搜索与定位

- 点击搜索结果后进入“命中上下文模式”
- 页面显式说明当前不是完整时间线，而是围绕命中消息展开
- 提供“返回最新消息”与“清除命中”两个退出路径

### 复盘沉淀

- 状态、标签、备注保留原有保存接口
- 新增快捷标签按钮，减少手工输入
- 当草稿有改动时显式显示 `未保存`

### 会话切换

- 当筛选或分页导致当前选中会话不在结果内时，自动切回当前页第一条
- 避免出现左侧列表与中间/右侧详情不一致的状态

## 本轮实施范围

- 只改 `apps/admin/src/routes/chat-records-page.tsx`
- 新增本规划文档，不改任何接口、数据库实体或后台模块
- 不新增测试文件，使用页面级最小验证
