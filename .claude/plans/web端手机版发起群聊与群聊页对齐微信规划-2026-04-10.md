# Web 端手机版发起群聊与群聊页对齐微信规划

## 任务目标
- 只规划 `apps/app` Web 手机版的两段链路：
  - 发起群聊
  - 群聊主页面与群信息页
- 目标不是补几个入口，而是把“从哪里发起、怎么选人、建群后怎么聊、进群后怎么管理”尽量按微信手机端的路径和操作习惯抄齐。
- 这份规划优先对齐功能和操作顺序，不直接复制微信商标素材、专有图标源文件和品牌文案。

## 当前范围
- 发起群聊入口：
  - `apps/app/src/routes/chat-list-page.tsx`
  - `apps/app/src/routes/group-contacts-page.tsx`
  - `apps/app/src/routes/chat-details-page.tsx`
- 发起群聊页：
  - `apps/app/src/routes/create-group-page.tsx`
- 群聊主页面：
  - `apps/app/src/features/chat/group-chat-thread-panel.tsx`
  - `apps/app/src/components/chat-composer.tsx`
  - `apps/app/src/components/mobile-chat-plus-panel.tsx`
  - `apps/app/src/components/chat-message-list.tsx`
- 群信息相关：
  - `apps/app/src/routes/group-chat-details-page.tsx`
  - `apps/app/src/routes/group-member-picker-page.tsx`

## 先说结论
- 当前项目已经有“能用”的群聊链路，但还不是微信手机端的发起群聊和群聊页。
- 最大差异不在有没有页面，而在三件事：
  - 发起群聊入口和选人流程不够像微信，尤其是“从单聊直接拉人建群”和“选择联系人页”的节奏不对。
  - 群聊主页面已经有回复、`@`、多选、转发、附件，但里面有几项还是本地假动作，输入区也不是微信手机端的主语义。
  - 群信息页字段不少，但缺“微信群管理深度”和“成员管理的搜索/分层/提醒语义”。
- 如果目标是“完全抄微信手机端”，优先级应该是：
  - 先抄入口和选人流程
  - 再抄群聊主线程状态机
  - 最后补齐群信息页和弱业务能力

## 一、当前实现现状

### 1. 发起群聊入口已经有 3 个，但都太平
- 消息页右上角 `+` 菜单已经有“发起群聊”，直接跳 `/group/new`
  - 见 `apps/app/src/routes/chat-list-page.tsx`
- 通讯录的“群聊”页右上角也能跳 `/group/new`
  - 见 `apps/app/src/routes/group-contacts-page.tsx`
- 单聊详情页的成员区也有“发起群聊”
  - 见 `apps/app/src/routes/chat-details-page.tsx`
- 但这三个入口目前都只是“打开同一个建群页”，没有入口语义差异。

### 2. 当前建群页是“表单式创建”，不像微信
- `apps/app/src/routes/create-group-page.tsx` 当前逻辑是：
  - 先显示页面说明
  - 顶部先输群名称
  - 下方整页列出联系人
  - 勾选后直接创建
- 当前移动端建群页虽然能搜联系人、勾选联系人、直接创建，但还缺这些微信感最强的细节：
  - 选中联系人头像条
  - 顶部选中数量联动
  - A-Z 索引和通讯录式列表分组
  - 从单聊进入时自动预选当前对方
  - 建群前默认不强调手填群名

### 3. 群聊主页面已经有不少能力
- `apps/app/src/features/chat/group-chat-thread-panel.tsx` 已支持：
  - 群消息读取
  - 群公告条
  - 回复和部分引用
  - `@成员` 与 `@所有人`
  - 图片 / 文件 / 名片 / 位置 / 表情发送
- `apps/app/src/components/chat-message-list.tsx` 已支持：
  - 长按消息菜单
  - 多选
  - 转发
  - 收藏
  - 删除
  - 撤回
- `apps/app/src/components/chat-composer.tsx` 已支持：
  - 文本输入
  - 表情
  - 加号面板
  - 语音转文字模式

### 4. 群信息页已经不是空壳
- `apps/app/src/routes/group-chat-details-page.tsx` 已支持：
  - 群成员宫格
  - 添加 / 移除成员
  - 群名称
  - 群公告
  - 群二维码
  - 查找聊天记录
  - 消息免打扰
  - `@我 / @所有人 / 群公告` 仍通知
  - 置顶聊天
  - 保存到通讯录
  - 我在本群的昵称
  - 显示群成员昵称
  - 群背景
  - 隐藏聊天
  - 清空聊天记录
  - 删除并退出

## 二、和微信手机端相比，差异在哪里

### P0：发起群聊链路不像微信，必须先改

#### 1. “从单聊发起群聊”没有继承当前会话语义
- 当前：
  - 单聊详情页点“发起群聊”只是跳到 `/group/new`
  - 不会默认勾上当前聊天对象
- 微信常见习惯：
  - 从单聊发起群聊，本质是“把当前这个人带进新群，再继续加人”
- 结论：
  - 这个入口必须带 `seedMemberIds`
  - 进入选人页后默认已选当前对方
  - 返回逻辑也应回到原单聊，而不是丢失上下文

#### 2. 当前建群页过早暴露“群名称”
- 当前：
  - 进页面第一块就是“群名称”
  - 不填则创建“临时群聊”
- 微信常见习惯：
  - 发起群聊第一动作是选人，不是先命名
  - 群名通常在建群后由系统组合显示，之后再去群信息页修改
- 结论：
  - 发起群聊首屏应收口为“选择联系人”
  - 群名称输入移到创建后群信息页
  - 服务端可继续保留默认名，但前端不要把“手填群名”放在第一优先级

#### 3. 当前联系人列表是“平铺勾选”，不是微信通讯录式选人
- 当前：
  - `create-group-page.tsx` 直接渲染所有好友
  - 搜索只是普通过滤
  - 没有首字母分组，没有右侧索引，没有“已选头像条”
- 微信常见习惯：
  - 顶部是搜索和已选成员
  - 中间是通讯录式长列表
  - 选择行为和通讯录体验高度一致
- 结论：
  - 需要把建群页重构成“微信式成员选择器”
  - 最好与群成员添加页共用一套 picker 基础件

#### 4. 当前没有“面对面建群 / 标签选人 / 最近群成员语义”
- 当前：
  - 只有普通好友勾选
- 微信已知稳定能力里，发起群聊通常不只一种路径：
  - 普通选人建群
  - 面对面建群
  - 通过现有联系人体系快速筛人
- 结论：
  - 首版不一定马上做完所有路径
  - 但 UI 结构要预留“更多建群方式”入口，不然之后只能重做

### P0：群聊主页面还不是微信手机端的主线程感

#### 5. 消息同步机制不对
- 当前：
  - `group-chat-thread-panel.tsx` 还是 `3s` 轮询
- 微信用户感知：
  - 群消息应该基本实时出现
- 结论：
  - 群聊线程要切到稳定实时推送
  - 至少让新消息、撤回、成员变更、群公告更新走 socket 实时下发

#### 6. 长按菜单里有真入口，但不少是假语义
- 当前：
  - 长按菜单有 `回复 / 部分引用 / 转发 / 多选 / 提醒 / 收藏 / 复制 / 撤回 / 删除`
  - 但 `提醒` 不是微信群原生消息提醒语义
  - `收藏` 现在是本地收藏
  - `撤回 / 删除` 也有前端本地态成分
- 微信用户感知：
  - 这些操作只要点一下就能知道是不是“真消息动作”
- 结论：
  - 菜单顺序可以先抄
  - 但 `收藏 / 撤回 / 删除` 必须尽快补成服务端真语义
  - `提醒` 要么改成本产品明确语义，要么不要伪装成微信群原生动作

#### 7. 输入区主语义不对
- 当前：
  - 有麦克风，但本质是“按住说话后转成文本”
  - `+` 面板里视频通话、语音通话、收藏、转账还是 disabled
- 微信手机端主语义：
  - 左侧语音/键盘切换
  - 按住说话默认发语音消息
  - 文本为空显示 `+`
  - 文本非空显示绿色发送
- 结论：
  - 现有语音转文字可以保留成增强能力
  - 但不能占住微信式语音消息主入口

#### 8. 群消息类型仍然不够微信
- 当前已支持：
  - 文本
  - 图片
  - 文件
  - 名片
  - 位置
  - 表情
- 仍明显缺：
  - 语音消息
  - 视频消息
  - 链接卡片
  - 小程序卡片
  - 实时位置共享
  - 红包 / 转账 / 群收款 / 接龙类卡片
- 结论：
  - 如果要“完全抄”，语音消息和视频消息至少要进 P1/P2
  - 红包转账这类可后置，但消息契约要预留扩展位

### P1：群信息页还不够微信

#### 9. 成员管理还是“功能有了，操作质感没到”
- 当前：
  - 群信息页成员宫格后面直接挂“添加 / 移除”
  - 添加成员页、移除成员页是普通列表 + 勾选 + 底部确认
- 微信手机端习惯：
  - 成员区、查看更多、搜索、选择状态会更紧凑
  - 选人页和建群页高度同构
- 结论：
  - 建群选人页、加人页、减人页应该统一成同一套微信式成员选择器
  - 只切 mode，不要三套页面三套交互

#### 10. 缺“群管理深度”
- 当前已有：
  - 免打扰
  - 置顶聊天
  - 保存到通讯录
  - 群昵称
  - 显示群成员昵称
- 还缺常见微信群深层能力或语义：
  - 群管理
  - 指定重要群成员提醒
  - 群待办
  - 入群方式/邀请确认
  - 成员权限更细粒度管理
- 结论：
  - 首版可不全做
  - 但群信息页信息架构要预留“群管理”分组，不然后面会越来越乱

#### 11. 群背景、二维码、公告都有，但视觉与顺序还要更贴微信
- 当前：
  - 字段不少，但还是“项目设置页”的 section 风格
- 微信：
  - 更接近轻分组白底列表
  - 分组标题更弱，行间距更稳定
- 结论：
  - 群详情页应做成更强的 `wechat-mobile-details` 皮肤
  - 不建议继续堆当前通用卡片组件的变体

### P1：消息页与发起群聊入口之间缺回流设计

#### 12. 建群成功后的回流不够像微信
- 当前：
  - 创建成功后直接进入新群聊
- 还差：
  - 首条系统消息
  - 默认群名生成规则
  - 创建成功后的轻提示
  - 从原单聊拉起时，对原会话的回退路径保持
- 结论：
  - 需要定义“从单聊建群”“从消息页建群”“从群聊页加人”三种回流语义
  - 否则操作一多，用户会觉得跳转很硬

## 三、对标后建议的目标口径

### 1. 发起群聊应该统一成一个基础选择器
- 新建一个移动端专用群成员选择器基础件，统一服务：
  - `/group/new`
  - `/group/$groupId/members/add`
  - `/group/$groupId/members/remove`
- 这套基础件需要具备：
  - 搜索
  - 已选头像条
  - 已选数量
  - 列表分组
  - 入口预选
  - 顶部确认按钮

### 2. 建群首页改成“先选人，后补资料”
- 移除当前首屏群名称表单
- 默认流程改成：
  - 进入发起群聊
  - 选联系人
  - 完成创建
  - 进入群聊
  - 在群信息页补群名/公告/权限

### 3. 群聊主线程优先抄操作节奏
- 优先做成微信感最强的几项：
  - 实时新消息
  - 语音消息主入口
  - 长按菜单真语义
  - 时间分隔条
  - 新消息/发送失败/发送中状态
- 先不要被“还有红包/转账没做”拖住首版

### 4. 群信息页分两层
- 第一层先抄稳定项：
  - 成员区
  - 群名称
  - 群公告
  - 群二维码
  - 查找聊天记录
  - 免打扰
  - 置顶聊天
  - 保存到通讯录
  - 我在本群昵称
  - 聊天背景
  - 删除并退出
- 第二层再加增强项：
  - 群管理
  - 重要群成员提醒
  - 群待办
  - 入群权限

## 四、实施拆分

### Phase 1：发起群聊入口微信化
- 改造入口：
  - `chat-list-page.tsx`
  - `group-contacts-page.tsx`
  - `chat-details-page.tsx`
- 目标：
  - 从单聊发起群聊时自动带上当前联系人
  - 从消息页或通讯录发起时保持空选择
  - 统一 route state / search params 传递 `seedMemberIds`、`returnTo`

### Phase 2：重做移动端建群选择器
- 改造：
  - `create-group-page.tsx`
  - 抽离 `mobile-group-member-picker`
- 目标：
  - 顶部搜索
  - 已选头像条
  - 通讯录式列表
  - 底部或顶部确认
  - 取消当前默认“先填群名”

### Phase 3：统一加人/减人/建群选择体验
- 改造：
  - `group-member-picker-page.tsx`
  - 复用 Phase 2 基础件
- 目标：
  - `new / add / remove` 三个场景只换 mode 和按钮文案
  - 交互和视觉保持一致

### Phase 4：群聊主线程真微信化
- 改造：
  - `group-chat-thread-panel.tsx`
  - `chat-composer.tsx`
  - `mobile-chat-plus-panel.tsx`
  - `chat-message-list.tsx`
- 目标：
  - 真正实时同步
  - 语音消息主入口
  - 长按菜单真语义
  - 时间分隔条
  - 发送态/失败态

### Phase 5：群信息页收口
- 改造：
  - `group-chat-details-page.tsx`
  - 相关 setting row / member grid 样式基础件
- 目标：
  - 分组结构更微信
  - 成员区与群管理区更清晰
  - 为后续群管理深度功能留位

## 七、落地判断
- 这件事不是“再补一个建群页”就结束。
- 真正要抄的是一整条连续体验：
  - 从聊天或通讯录发起
  - 选人
  - 建群
  - 进入群聊
  - 长按消息
  - 管理群
- 现在项目距离“能用”已经不远，但距离“像微信手机端”最核心还差两刀：
  - 入口和选人流程重做
  - 群聊主线程从“有功能”升级成“微信式状态机”
