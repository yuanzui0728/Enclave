# Web 端电脑版通讯录右侧好友详情微信化规划

## 任务目标
- 将 `apps/app/src/routes/contacts-page.tsx` 桌面态通讯录右侧详情区改造成接近微信电脑端联系人详情页的排版和交互。
- 不再使用当前“资料卡 + 大段介绍 + 若干通用字段块”的泛化面板，而是收敛为微信式联系人资料页。
- 规划分成两层：
  - 第一层：先把桌面排版、视觉层级、主操作路径复刻到位
  - 第二层：补齐微信式联系人字段和设置能力所需的数据模型与接口

## 当前现状

### 现有入口与承载位置
- 桌面通讯录主路由：`apps/app/src/routes/contacts-page.tsx`
- 桌面右侧详情面板：`apps/app/src/features/contacts/contact-detail-pane.tsx`
- 左侧已具备：
  - 搜索框
  - 功能入口列表
  - 好友分组列表
  - 选中态
- 右侧当前仅具备：
  - 头像、昵称、联系人/世界角色标记
  - 简介文案
  - `发消息`
  - `查看资料`
  - 若干摘要字段卡片

### 当前与微信电脑端的主要差距
- 详情区结构不对。
  - 微信电脑端联系人详情更接近“资料页”，不是营销式卡片面板。
- 字段组织不对。
  - 当前展示“在线状态”“最近互动”“擅长领域”等抽象字段，微信联系人详情更强调备注、昵称、地区、来源、标签、朋友圈、共同群聊、更多信息与设置项。
- 交互层级不对。
  - 当前只有“发消息 / 查看资料”两个按钮，没有微信式的联系人管理和会话入口分层。
- 数据契约不够。
  - 目前 `Character` / `Friendship` 能支撑头像、昵称、关系、简介、最近互动，但不够支撑微信式联系人详情完整字段。

## 设计基线

### 对标页面
- 对标微信电脑端“通讯录 > 点击联系人 > 右侧联系人详情页”。
- 本次只规划桌面 Web 宽屏态，不动移动端通讯录详情路径。
- 左侧通讯录列表维持现有实现，改造重点只在右侧详情区。

### 复刻原则
- 排版优先复刻微信电脑端的信息结构和阅读节奏。
- 文案与字段尽量使用微信式命名，但需要保留隐界的角色世界语义。
- 不为了“看起来像”而塞假字段；没有数据的项必须明确标注为二期补齐，不用硬编码假值。

## 目标版式

### 右侧详情区目标结构
建议改成从上到下四段，而不是当前单页卡片流。

#### 1. 顶部资料区
- 左侧大头像
- 右侧核心身份信息
  - 昵称
  - 备注名
  - 微信号风格的唯一标识位
  - 关系/标签摘要
- 右上或同区主操作
  - `发消息`
  - `音视频通话` 先不做，保留布局位

#### 2. 基础资料区
- 类似微信联系人资料行，不做大卡片瀑布布局
- 每行是 `字段名 + 字段值`
- 建议字段顺序：
  - 备注
  - 昵称
  - 隐界号/角色标识
  - 地区
  - 来源
  - 个性签名/状态
  - 标签

#### 3. 内容与关系区
- 朋友圈入口
- 共同群聊入口
- 更多资料入口
- 这部分做成微信式可点击行，而不是说明型文本块

#### 4. 设置与危险操作区
- 置顶聊天
- 消息免打扰
- 加入黑名单
- 删除联系人
- 这部分首版不一定全部可用，但 UI 结构要预留正确

## 字段与功能映射

### 现有数据可直接复用
- `Character.name`
  - 作为昵称基础值
- `Character.avatar`
  - 作为联系人头像
- `Character.relationship`
  - 可映射到关系说明或联系人标签摘要
- `Character.currentStatus`
  - 可映射到个性签名/状态
- `Character.bio`
  - 可作为更多资料中的“简介”
- `Character.id`
  - 可先映射成临时的“隐界号”
- `Friendship.lastInteractedAt`
  - 可作为更多信息里的“最近互动”
- `Friendship.status`
  - 可用于联系人状态判断

### 当前缺失但微信式详情需要补齐
- 备注名 `remarkName`
- 备注图片或备注附加信息
- 地区 `region`
- 添加来源 `source`
- 标签列表 `contactTags`
- 共同群聊统计/列表
- 朋友圈可见性与入口能力
- 消息免打扰 `messageMuted`
- 会话置顶与联系人资料页联动展示
- 是否加入黑名单 `blocked`
- 删除联系人能力

### 不建议继续展示的字段
- 在线状态
  - 当前产品没有稳定在线体系，继续展示会误导
- 亲密度
  - 更适合角色关系系统，不适合微信式联系人详情首屏
- 擅长领域
  - 更适合“查看资料”或角色档案，不适合微信联系人详情主页面

## 交互规划

### 首屏交互
- 点击左侧联系人后，右侧立刻切换到该联系人详情页。
- 顶部主按钮固定为 `发消息`，直达现有会话创建逻辑。
- `查看资料` 不再作为顶部主次双按钮之一。
  - 建议降级为“更多资料”入口，放入中部列表。

### 行级交互
- 基础资料区使用不可编辑的资料行。
- 内容与关系区使用可点击行：
  - 朋友圈
  - 共同群聊
  - 更多资料
- 设置区使用开关或危险操作行：
  - 置顶聊天
  - 消息免打扰
  - 加入黑名单
  - 删除联系人

### 空状态
- 未选中联系人时，右侧显示更接近微信桌面端的浅空态：
  - 中性图标或占位头像
  - 文案只保留“选择联系人后可查看详细资料”
- 不再显示当前解释型长文案。

## 分阶段实施

### Phase 1：桌面详情区重排，不改后端结构
目标：先把页面结构和主流程做对。

前端改造范围：
- `apps/app/src/features/contacts/contact-detail-pane.tsx`
  - 从“卡片面板”改为“联系人资料页”
- `apps/app/src/routes/contacts-page.tsx`
  - 保留当前双栏结构
  - 调整右侧面板 props 组织方式
- 如有必要，新增桌面专用小组件：
  - `desktop-contact-profile-header.tsx`
  - `desktop-contact-profile-section.tsx`
  - `desktop-contact-action-row.tsx`

Phase 1 展示内容建议：
- 顶部：头像、昵称、关系摘要、发消息
- 基础资料：
  - 备注：暂无
  - 昵称：角色名
  - 隐界号：角色 ID 映射值
  - 个性签名：`currentStatus`
- 内容区：
  - 更多资料
- 设置区：
  - 仅展示已接通能力或占位态

Phase 1 已接通能力建议：
- `发消息`
- `加入黑名单`
  - 可复用现有社交屏蔽路由
- `置顶聊天`
  - 如果能定位现有联系人对应会话，可接通；否则先延后到 Phase 2

Phase 1 不做：
- 编辑备注
- 地区
- 来源
- 共同群聊
- 朋友圈权限设置
- 删除联系人

### Phase 2：补齐微信式联系人字段
目标：让右侧详情页的字段更像微信，而不只是版式像微信。

建议新增联系人资料模型，承载联系人层字段，而不是继续塞进 `Character`：
- `remarkName`
- `region`
- `source`
- `tags`
- `momentsPermission`
- `messageMuted`
- `isBlocked`

建议落点：
- 后端：
  - `api/src/modules/social/` 新增联系人资料读写能力
- 共享契约：
  - `packages/contracts/src/social.ts`
  - `packages/contracts/src/client.ts`

注意：
- 这是结构变更。
- 真正进入实施时，需要按仓库规则同步更新 `AGENTS.md`。

### Phase 3：补齐微信式设置项与联动
目标：把联系人详情页从“展示页”升级成“联系人管理页”。

建议补齐：
- 编辑备注
- 标签管理
- 消息免打扰开关
- 置顶聊天开关
- 加入/解除黑名单
- 删除联系人
- 朋友圈入口联动
- 共同群聊入口联动

其中需要复用的现有能力：
- 会话置顶：`POST /api/conversations/:id/pin`
- 社交拉黑：
  - `GET /api/social/blocks`
  - `POST /api/social/block`
  - `POST /api/social/unblock`

## 文件改造建议

### 必改文件
- `apps/app/src/features/contacts/contact-detail-pane.tsx`
  - 当前核心改造位
- `apps/app/src/routes/contacts-page.tsx`
  - 负责桌面选中态和详情面板挂载

### 建议新增文件
- `apps/app/src/features/contacts/desktop-contact-profile-header.tsx`
- `apps/app/src/features/contacts/desktop-contact-profile-info-list.tsx`
- `apps/app/src/features/contacts/desktop-contact-profile-actions.tsx`

原因：
- 右侧详情页如果继续全部堆在 `contact-detail-pane.tsx`，后面接备注、屏蔽、置顶、更多资料时会很快失控。

## 验收标准
- 桌面通讯录点击好友后，右侧第一眼是微信式联系人资料页，而不是卡片式内容面板。
- 详情区首屏只保留微信联系人详情真正需要的字段，不再出现在线状态、亲密度、擅长领域这些跑题信息。
- 用户能在右侧稳定完成“看资料 -> 发消息 -> 管理联系人”主路径。
- 没有真实数据支撑的微信字段，不以假数据占位冒充完成。

## 风险与决策
- 最大风险不是布局，而是联系人资料模型缺失。
  - 如果没有备注、来源、地区、标签，就只能先做“微信式骨架”。
- 当前 `Character` 同时承担“角色档案”和“联系人资料”语义，继续复用会越来越混乱。
  - 二期更合理的方向是给联系人关系单独扩展资料字段。
- `查看资料` 与“联系人详情页”职责重叠。
  - 这次改造后需要重新定义：
    - 右侧页负责微信式联系人信息与操作
    - 独立角色资料页负责更深的角色世界设定、人格档案、长简介

## 推荐实施顺序
1. 先做 Phase 1，把右侧详情页的结构和视觉节奏改对。
2. 然后补联系人资料模型，承接备注、地区、来源、标签。
3. 最后再接朋友圈、共同群聊、删除联系人等微信式管理能力。
