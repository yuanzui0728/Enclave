# 剩余多模态能力彻底真实收口执行规划

## 1. 本次规划解决什么

这份规划只覆盖一件事：

- 把当前还没有“彻底真实成立”的多模态理解与多模态回复能力，按可实施的执行包拆开并收口。

这里的“彻底真实”有明确口径：

- 输入模态真的被系统消费，而不是只靠文件名、补充提示词或偶然成功的旁路。
- 输出模态真的由正式链路编排、渲染、取消、重试和回放，而不是 mock 壳、helper 或临时补发。
- 关键能力在运行态稳定可验证，不因为 provider 429、异步 job、会话清空或页面刷新而失真。

## 2. 截至 2026-04-23 的最新真实基线

### 2.1 已经真实成立

- 图片理解已经真实成立。
- 文本文档理解已经部分成立，`txt/md/json/xml/html/csv` 可进入模型。
- 普通聊天 `text + voice` 回复已经成立。
- 普通聊天 `text + image` 回复已经成立。
- 群聊 `text + voice` / `text + image` 回复已经成立。
- 异步补发已经不再是纯内存 fire-and-forget。
- `ReplyArtifactJob` 已经落地，单聊和群聊的补媒体可以持久化、取消和收口。

### 2.2 还没有彻底成立

- 音频理解仍主要依赖转写，且转写在运行态会被 429 和通道繁忙打断。
- 视频理解仍不真实，当前基本只会处理音轨，画面本身没有稳定进入理解链路。
- 文档理解没有覆盖 `pdf/docx/扫描件` 的完整解析和 OCR 兜底。
- 回复模态仍只覆盖 `text / voice / image`，没有真实的视频回复链路。
- 数字人视频仍是 mock/iframe 壳，不是真实 render provider。
- 回复规划策略还比较粗，音频/视频附件会被错误地自动推成 `voice` 回复。

## 3. 这次收口后的目标形态

### 3.1 输入侧目标

系统最终稳定支持以下输入模态：

- `text`
- `image`
- `audio`
- `video`
- `document`

每个输入模态都必须同时具备三层表达：

- 原始资产引用
- 持久化的派生洞察
- 面向 provider 的最终 payload

### 3.2 输出侧目标

系统最终稳定支持以下回复模态：

- `text`
- `voice`
- `image`
- `digital_human_video`

可选扩展：

- `document_artifact`

说明：

- `document_artifact` 不是“聊天回复闭环”的首要 blocker。
- `digital_human_video` 才是当前多模态回复侧最大的真实缺口。

### 3.3 Definition of Done

只有满足以下条件才算彻底收口：

- 单聊和群聊都可用。
- direct 和 group 共用同一套能力判断与 artifact 编排语义。
- provider 原生支持某输入模态时，系统能走原生路径。
- provider 不支持时，系统能自动回退到派生洞察路径。
- 清空会话、删除消息、撤回消息、新消息覆盖旧轮时，不会再产生悬挂媒体补发。
- 后台能看到每轮输入理解和回复渲染的真实状态、失败原因和耗时。
- 至少完成端到端 smoke，而不只是 build 通过。

## 4. 目前剩余缺口的真实根因

## 4.1 缺口 A：音频理解不稳定

- 当前聊天链路仍依赖 `transcriptText` 才能把音频内容写进 prompt。
- 转写结果不是一个明确的、可重试、可回放的理解资产。
- 转写失败后，主链路直接退化成“我没有收到具体语音内容”。
- 没有多 provider 转写降级和重试预算。

## 4.2 缺口 B：视频画面还没真正进入理解

- 当前没有 `Video Understanding Bundle`。
- 没有关键帧、OCR、视觉摘要、时序事件摘要这类中间表示。
- `supportsNativeVideoInput` 仍然是关闭状态。
- 即使以后接入支持视频的 provider，也不能把系统能力全部寄托在 provider 原生视频输入上。

## 4.3 缺口 C：文档理解只覆盖文本型文件

- 当前本地提取器本质还是 `utf8` 文本读取。
- `pdf/docx` 没有稳定的本地解析兜底。
- 扫描版 PDF 没有 OCR。
- 文档分块、页码引用、抽取质量分级都还没有正式建模。

## 4.4 缺口 D：回复规划器仍然过粗

- 现在的 planner 只会决定要不要附带 `voice` 和 `image`。
- `audio/video` 附件会被硬编码推成 `voice` 回复。
- 没有统一的 `ReplyModalityPlan` 持久化结果。
- 没有视频回复的 artifact 类型。

## 4.5 缺口 E：数字人视频回复仍是 mock

- provider 仍是 `mock_digital_human`。
- `supportsRealtimeStream` 仍为 `false`。
- session 和 render 生命周期没有接入真实外部渲染能力。
- 前端看到的还是播放器 URL 和海报壳，不是真实视频生成或直播流。

## 5. 整体方案原则

- 不把“真实多模态能力”绑定在单一模型或单一 provider 上。
- 输入理解和输出渲染都必须 job 化、状态化、可回放。
- 先补“中间层资产”，再补“模型原生能力”。
- direct/group 共用同一套能力注册与模态规划，不再各写一遍 heuristic。
- 所有降级都必须显式记录，不能只在 prompt 里悄悄退化。

## 8. 执行包 1：补齐输入理解中间层

### 目标

- 先把所有附件统一变成“资产 + 洞察 + provider payload”三层模型。

### 后端改造

- 新增 `MediaInsightJob`，负责音频、视频、文档的派生理解。
- 新增 `MediaInsightArtifact`，落转写、关键帧、OCR、正文抽取、摘要、质量标记。
- 所有附件入库后先写 `MultimodalAttachmentEnvelope`，再异步生成洞察。
- `chat.service` 和 `group` 链路只消费洞察结果，不在主请求内长时间阻塞。

### 验收标准

- 任意 `audio/video/document` 附件都能查到明确的 `insightStatus`。
- insight 失败时有结构化失败原因。
- 模型调用日志中能看到“用的是原始资产”还是“用的是派生洞察”。

## 9. 执行包 2：把音频理解做成稳定能力

### 目标

- 让音频理解脱离“请求期临时转写”的脆弱模式。

### 实施内容

- 新增转写 provider registry，支持主 provider、备 provider、禁用开关和预算。
- `MediaInsightJob(audio)` 引入：
  - `pending`
  - `processing`
  - `completed`
  - `failed`
  - `degraded`
- 持久化音频元数据：
  - 时长
  - 采样率
  - 声道数
  - MIME
  - 文件大小
- provider 支持原生 `audio input` 时：
  - 直接传原始音频
  - 同时附带已有 transcript 作为辅助上下文
- provider 不支持时：
  - 使用 transcript + 音频元数据 + 用户问题组成结构化输入
- 429 时采用队列重试、指数退避和 provider 降级。
- 超过 SLA 后，把消息状态标成“理解未完成”，而不是假装理解成功。

### 依赖

- 需要统一转写 job 模块。
- 需要在后台暴露转写积压、成功率、平均耗时、429 命中率。

### 验收标准

- 同一条语音在 direct 和 group 中都能稳定抽取暗号。
- provider 主通道被限流时，能自动切到备通道或降级等待。
- 用户看不到“伪理解成功”的回复。

## 10. 执行包 3：把视频理解做成真实能力

### 目标

- 视频不仅能“听音轨”，还要真的“看画面”。

### 实施内容

- 新增 `VideoUnderstandingBundle`，至少包含：
  - `keyframes`
  - `frameOcr`
  - `audioTranscript`
  - `durationMs`
  - `width`
  - `height`
  - `sceneSummary`
  - `timelineEvents`
- 技术路径建议：
  - 用 `ffprobe` 取元数据
  - 用 `ffmpeg` 抽关键帧
  - 对关键帧做 OCR
  - 对关键帧做视觉摘要
  - 音轨单独转写
  - 最终聚合成视频洞察对象
- provider 支持原生视频输入时：
  - 走原生视频输入
  - 同时把关键帧 OCR 和音轨摘要作为辅助上下文
- provider 不支持时：
  - 用 `VideoUnderstandingBundle` 作为结构化替代输入

### 关键决策

- 视频理解不能只有“抽一帧”。
- 至少要支持首帧、中段帧、结尾帧和镜头变化帧。
- OCR 和音轨必须保留时间戳，避免只得到一段无时序文本。

### 验收标准

- 无声视频里的画面暗号能被读出来。
- 有音轨视频能同时回答“画面里有什么”和“旁白说了什么”。
- 视频理解结果能被后台逐段检查，而不是只有最终回复文本。

## 11. 执行包 4：补齐完整文档理解

### 目标

- 把文档理解从“纯文本文件支持”提升到“常见办公与扫描文档可用”。

### 实施内容

- 为 `pdf` 增加本地文本提取和页码索引。
- 为 `docx` 增加本地正文解析。
- 为扫描版 `pdf/image-pdf` 增加页面栅格化和 OCR。
- 文档洞察对象至少包含：
  - `plainText`
  - `pageCount`
  - `pageAnchors`
  - `extractionQuality`
  - `ocrApplied`
  - `truncated`
- 对大文档做 chunking，保留 chunk 对应页码。
- provider 支持 `input_file` 时保留原生文件输入。
- provider 拒绝 `input_file` 时自动退回本地抽取文本。

### 技术建议

- `pdf`：优先本地文本解析，再按需 OCR。
- `docx`：优先结构化正文提取。
- 扫描件：先页图像化，再 OCR，再回写页码锚点。

### 验收标准

- `pdf/docx/扫描版 pdf` 都能从同一轮对话中回答具体内容问题。
- 回答带页码或片段来源时，后台可追溯到文档页级锚点。
- provider 对文件输入失效时，系统仍能靠本地抽取得到可用答案。

## 12. 执行包 5：重做回复模态规划器

### 目标

- 把当前“voice/image heuristic”提升成正式的 `ReplyModalityPlan`。

### 实施内容

- 新增 `ReplyModalityPlan` 字段：
  - `primaryTextRequired`
  - `artifacts[]`
  - `plannerReason`
  - `capabilitySnapshot`
  - `budgetSnapshot`
  - `degradeReason`
- direct 和 group 共用同一套 planner。
- 去掉“音频/视频附件自动触发 voice 回复”的硬编码。
- 改成根据以下信号联合判断：
  - 用户显式请求
  - 当前输入模态
  - 角色能力
  - provider 能力
  - 当前预算
  - 当前线程上下文
- `ReplyArtifactJob` 扩到：
  - `voice`
  - `image`
  - `digital_human_video`
  - `document_artifact`

### 前端配套

- typing / sending 状态继续按阶段展示。
- 新增 `video_rendering` 阶段。
- direct/group 都显示 artifact job 的失败和补发状态。

### 验收标准

- 视频理解请求不会再无脑回一条 `voice`。
- 明确要求语音回复、图片回复、视频回复时，planner 会给出可审计决策。
- direct/group 的 planner 行为一致。

## 13. 执行包 6：接入真实数字人视频回复

### 目标

- 把数字人视频从 mock player 壳改成真实 provider adapter。

### 实施内容

- 抽象 `DigitalHumanRenderAdapter`：
  - `createSession`
  - `submitTurn`
  - `pollJob`
  - `cancelJob`
  - `closeSession`
- 会话持久化，不再只放内存。
- 新增 `DigitalHumanRenderJob`，保存：
  - `sessionId`
  - `responseGroupId`
  - `sourceMessageId`
  - `provider`
  - `status`
  - `renderMode`
  - `streamUrl/playbackUrl/fileUrl`
  - `errorCode`
  - `errorDetail`
- 首期可接受两种交付形态：
  - 异步渲染短视频并回传文件 URL
  - 实时流播放器地址
- provider 不可用时明确降级到：
  - `voice + poster`

### 前端配套

- 单聊视频通话页和普通聊天视频回复都使用同一套 render 状态语义。
- 允许用户看见“正在生成视频”“视频生成失败”“已降级为语音”的明确状态。

### 验收标准

- 新建 session 返回的 provider 不再是 `mock_digital_human`。
- `supportsRealtimeStream` 能跟真实 provider 能力同步。
- 至少有一个真实 provider 在运行态完成一轮视频回复。

## 14. 执行包 7：后台可观测与评测闭环

### 目标

- 把“到底有没有真的理解和真的渲染”做成可审计系统，而不是靠人工猜。

### 实施内容

- 后台新增多模态回合详情：
  - 原始附件
  - 洞察资产
  - provider payload 摘要
  - planner 决策
  - reply artifact jobs
  - 降级原因
  - 失败原因
- 新增多模态 smoke 数据集：
  - 图片读数
  - 语音暗号
  - 无声视频画面暗号
  - 视频音画联合问答
  - `pdf/docx/扫描件` 精确抽取
  - 语音/图片/视频回复请求
- 新增核心指标：
  - insight 成功率
  - 音频转写成功率
  - 视频 OCR 成功率
  - 文档抽取成功率
  - artifact 渲染成功率
  - 端到端耗时
  - 降级率

### 验收标准

- 任意失败都能在后台看到确切失败点。
- smoke 可以稳定复现，不靠人工翻聊天记录。

## 15. 推荐实施顺序

### 第一阶段

- 执行包 1：输入理解中间层
- 执行包 2：稳定音频理解

原因：

- 语音理解现在是最明显的真实性缺口。
- 没有统一 insight 层，后面的音频、视频、文档都会重复造轮子。

### 第二阶段

- 执行包 3：真实视频理解
- 执行包 4：完整文档理解

原因：

- 这两块都依赖 insight 层。
- 视频和文档都需要统一的资产派生、分块、OCR 和可追溯表示。

### 第三阶段

- 执行包 5：回复模态规划器重做
- 执行包 6：真实数字人视频回复

原因：

- 先把“能理解”做实，再把“怎么回”做实。
- 数字人视频的复杂度最高，应该在 planner 和 artifact 类型稳定后再接。

### 第四阶段

- 执行包 7：后台可观测与评测闭环

说明：

- 这一包不应该最后才做完。
- 但它的完整交付必须建立在前几包的真实状态数据之上。

## 16. 每阶段必须做的最小 smoke

### 音频

- 语音中只包含一个暗号，系统只回复暗号原文。
- 主转写 provider 人工压限流后，系统能自动重试或降级。

### 视频

- 无声视频只在画面里出现暗号，系统能读出来。
- 有声视频同时问画面内容和旁白内容，系统能分别回答。

### 文档

- `pdf` 精确抽取一段指定内容。
- `docx` 精确抽取一段指定内容。
- 扫描版 PDF 读出页内暗号。

### 回复

- “用语音回复我”稳定返回 `text + voice`。
- “直接发图给我”稳定返回 `text + image`。
- “给我发视频回复”在真实 provider 下返回视频，provider 不可用时明确降级。

## 17. 主要风险与对应策略

### 风险 1：多 provider 差异过大

- 策略：把 provider 能力判断收口到 `capability registry`，不要散落在 chat service。

### 风险 2：媒体处理显著拉长时延

- 策略：主链路只做最小首响，复杂洞察和渲染都走 job，但状态必须透明。

### 风险 3：OCR 和视频抽帧成本高

- 策略：先做分层策略，默认轻量摘要，命中复杂场景时再升级处理深度。

### 风险 4：数字人 provider 不稳定

- 策略：视频 render adapter 必须支持 provider 级降级和 `voice + poster` 回退。

## 18. 最该先开的三个执行包

- 先开 `执行包 1`，补齐统一的 `MediaInsightJob/Artifact` 底座。
- 紧接着做 `执行包 2`，把语音理解从“偶尔成功”收口成稳定能力。
- 然后做 `执行包 3`，让视频画面真正进入理解链路。

## 19. 结论

当前系统已经有了一部分真实多模态能力，但离“彻底真实实现”还差最后四个关键闭环：

- 稳定音频理解
- 真实视频理解
- 完整文档理解
- 真实视频回复

这次不应该再从 UI 或 prompt 小修小补开始，而应该从 `MediaInsight` 和 `ReplyModalityPlan` 两个中间层入手，把理解资产和回复工件都做成正式系统对象。只要这两个中间层补齐，剩下的 provider 接入、文档 OCR、数字人视频渲染就都能按统一框架继续推进。
