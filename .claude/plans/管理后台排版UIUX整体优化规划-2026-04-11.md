# 管理后台排版 UI / UX 整体优化规划

## 1. 目标

- 把管理后台从“开发者拼装页”改成“运营工作台”。
- 把当前顶部 Tab 导航改成左侧主导航，降低切页成本。
- 让运营人员进入页面后先看到状态、对象和可执行动作，而不是先面对大段装饰性 Hero 和长滚动内容。
- 先解决信息架构和高频操作路径，再逐步细化视觉和交互，不做只换皮的改版。

## 2. 当前问题判断

### 2.1 全局壳层问题

- 当前导航在 `apps/admin/src/components/root-layout.tsx`，采用顶部胶囊式 Link。
- 这种导航适合 3 到 5 个轻页面，不适合当前后台这种多工作台、多上下文、多二级路由的产品。
- 角色工厂、角色运行逻辑台、回复逻辑、评测页都属于重工作页，但仍被放在“统一顶部入口 + 页面内自找结构”的模式里，认知负担高。
- 管理密钥输入、全局导航、页面标题都挤在同一块头部里，首屏层级不清晰。

### 2.2 页面结构问题

- 多个页面顶部统一使用大号 `AppHeader` Hero，视觉存在感太强，但对运营任务帮助有限。
- 页面内部普遍是“卡片堆叠 + 长表单 + 长列表”，缺少固定筛选、固定操作、固定上下文区域。
- 页面内部缺少稳定的左右分栏规则，导致列表、详情、编辑区常常混排。
- 高密度页面的“看数据”和“做操作”没有明确主次，容易迷路。

### 2.3 信息架构问题

- 现在一级入口只有“总览 / 角色 / 设置 / 评测 / 回复逻辑”，但实际后台已经存在三类不同任务：
  - 实例运维
  - 角色运营
  - 回复与评测分析
- 角色相关能力分散在名册、编辑页、工厂页、运行逻辑台，缺少统一上下文入口。
- 二级路由没有被明确表达，用户进入深页后容易失去位置感。

### 2.4 运营体验问题

- 当前后台不强调“下一步该做什么”。
- 缺少面向运营的快捷动作区、异常优先级、待处理提示和对象筛选入口。
- 重页面体量过大：
  - `apps/admin/src/routes/reply-logic-page.tsx` 约 4156 行
  - `apps/admin/src/routes/evals-page.tsx` 约 2872 行
  - `apps/admin/src/routes/character-factory-page.tsx` 约 1273 行
  - `apps/admin/src/routes/dashboard-page.tsx` 约 913 行
- 这意味着单页承载过多职责，后续必须通过布局和子模块拆分来降复杂度。

## 3. 改版原则

- 先重构壳层和导航，再改各页内容。
- 先服务高频运营任务，不追求首版把每个页面都做成完整设计稿。
- 统一工作台语言：
  - 左侧是导航和上下文
  - 中间是主内容
  - 右侧是详情、快捷操作或状态摘要
- 减少“装饰型大标题”，增加“任务型头部”。
- 页面首屏必须回答三个问题：
  - 我现在在哪个模块
  - 当前对象是谁
  - 下一步能做什么

## 4. 目标信息架构

### 4.1 一级导航改为左侧主导航

- 建议一级导航分组：
  - 运行总览
  - 实例设置
  - 角色中心
  - 回复逻辑
  - 评测分析

### 4.2 左侧导航形态

- 固定左侧侧栏，桌面宽度建议 `248px - 280px`。
- 侧栏内容分四层：
  - 产品标识与实例状态
  - 主导航分组
  - 当前世界关键信号
  - 管理密钥与环境入口
- 当前激活项不再只是高亮文字，而要有清晰块状选中态。

### 4.3 二级上下文导航

- 角色模块进入后，二级上下文导航放在左侧主导航下方或主内容左列，不再依赖页面内零散按钮。
- 例如进入某个角色后，明确给出：
  - 基础资料
  - 角色工厂
  - 运行逻辑台
- 回复逻辑和评测页内部也要有二级视图切换区，但不应该继续使用裸 `tab` 堆在正文顶端。

## 5. 目标页面框架

### 5.1 全局后台壳层

- 新增统一后台壳层组件，建议替代当前 `RootLayout` 的单块头部结构。
- 目标结构：

```tsx
<AdminShell>
  <AdminSidebar />
  <AdminWorkspace>
    <AdminTopbar />
    <AdminPageLayout>
      <PageMain />
      <PageAside />
    </AdminPageLayout>
  </AdminWorkspace>
</AdminShell>
```

- `AdminTopbar` 不再承担主导航职责，只负责：
  - 当前页标题
  - 面包屑
  - 全局搜索或对象切换
  - 页面级主操作

### 5.2 页面类型统一

- 后台页面统一收口为三种模板：

#### A. 总览型页面

- 典型页面：
  - `/`
  - `/setup`
- 结构：
  - 顶部状态摘要
  - 中部关键指标与异常卡片
  - 底部快捷入口

#### B. 列表 + 详情型页面

- 典型页面：
  - `/characters`
- 结构：
  - 左侧列表
  - 中间详情
  - 右侧快捷动作
- 不再用“卡片瀑布流”承担全部信息。

#### C. 工作台型页面

- 典型页面：
  - `/reply-logic`
  - `/evals`
  - `/characters/$characterId/factory`
  - `/characters/$characterId/runtime`
- 结构：
  - 顶部轻量工具栏
  - 左侧对象切换 / 过滤器
  - 中间主工作区
  - 右侧快照 / 操作 / 状态

## 6. 各模块改造重点

### 6.1 总览页

- 当前问题：
  - 内容太全，像系统状态 dump，而不是运营首页。
- 调整目标：
  - 首屏只展示健康度、异常、下一步动作。
  - 中部展示今日运营关键信号：
    - 世界实例健康
    - Provider 就绪
    - 调度器状态
    - 角色在线情况
    - 最近异常
  - 底部展示跳转卡：
    - 去设置
    - 去角色中心
    - 去回复逻辑
    - 去评测
- 去掉大面积 Hero 感，改成控制台首页。

### 6.2 设置页

- 当前问题：
  - 视觉结构已经比其它页清楚，但仍偏“大卡片展示”，缺少运维步骤感。
- 调整目标：
  - 左侧展示实例健康检查流程。
  - 右侧展示配置表单。
  - 顶部固定当前实例地址、连接状态、最近检查时间。
  - 明确“待完成项”与“已完成项”。

### 6.3 角色中心

- 当前问题：
  - `characters-page.tsx` 目前是角色卡片墙，适合浏览，不适合高频管理。
- 调整目标：
  - 改成“角色列表 + 详情面板 + 快捷动作”结构。
  - 左列显示角色列表，支持：
    - 搜索
    - 在线状态筛选
    - 关系类型筛选
  - 中列显示当前角色摘要：
    - 身份
    - 当前状态
    - 最近运行情况
    - 核心标签
  - 右列显示快捷操作：
    - 编辑基础资料
    - 打开角色工厂
    - 打开运行逻辑台
    - 删除
- 新建角色入口固定在列表头部，不再漂浮在页面大头图里。

### 6.4 角色编辑页

- 当前问题：
  - 页面像大表单，字段块较多，回看成本高。
- 调整目标：
  - 拆成分段式表单：
    - 基础资料
    - 关系与触发
    - 人格与提示词
    - 记忆与边界
  - 左侧增加段落导航，支持快速跳转。
  - 右侧保留保存状态、字段校验和角色摘要。
- 若后续继续收窄此页职责，则页面最终应更像“基础资料编辑器”。

### 6.5 角色工厂页

- 当前问题：
  - 信息量很大，但缺少清晰主线。
- 调整目标：
  - 主线改成四段：
    - 工厂状态
    - 草稿编辑
    - 发布差异
    - 历史版本
  - 右侧固定“当前发布状态”和“发布动作”。
  - AI 辅助制造区域不要继续顶在正文前半段，应放到草稿编辑流中，作为辅助入口。

### 6.6 角色运行逻辑台

- 当前问题：
  - 状态、配置、快照、观察数据都堆在同一页。
- 调整目标：
  - 拆成三个工作区：
    - 当前运行状态
    - 可编辑运行参数
    - 可观测性与最近执行
  - 右侧固定展示：
    - 在线模式
    - 当前活动
    - 最近调度结果
    - 保存状态

### 6.7 回复逻辑页

- 当前问题：
  - 页面承担了对象选择、规则配置、快照查看、调试预览、角色编辑等多重职责。
- 调整目标：
  - 保留世界级调试台定位，但拆成明确的工作流：
    - 选择观察范围
    - 查看上下文快照
    - 调整全局规则
    - 运行预览
  - 页面内部不要再用长滚动承载全部内容，改为多栏工作台。
  - 左列：
    - scope 切换
    - 角色 / 会话对象选择
  - 中列：
    - prompt、actor、状态机、记忆、链路快照
  - 右列：
    - 全局规则编辑
    - 预览输入与执行结果
- 这页优先级高，因为它是当前最重、最难用的后台页。

### 6.8 评测页

- 当前问题：
  - 信息密度极高，筛选器、对比结果、trace、报告都在一页，进入成本太高。
- 调整目标：
  - 拆成四个稳定视图：
    - 总览
    - Runs
    - Compare
    - Trace
  - 视图切换放到左侧上下文导航或页内纵向导航，不再堆在首屏顶端。
  - 每个视图保留独立过滤区和详情区。
- 首版不一定拆成多路由，但在布局上必须先拆分。

## 7. 组件层实施建议

### 7.1 新增后台壳层组件

- 建议新增：
  - `apps/admin/src/components/admin-shell.tsx`
  - `apps/admin/src/components/admin-sidebar.tsx`
  - `apps/admin/src/components/admin-topbar.tsx`
  - `apps/admin/src/components/admin-page-layout.tsx`
  - `apps/admin/src/components/admin-section-nav.tsx`

### 7.2 在 `@yinjie/ui` 抽公共后台组件

- 建议补齐：
  - `WorkbenchHeader`
  - `FilterBar`
  - `InspectorPanel`
  - `StickyActionBar`
  - `ContextList`
  - `EmptySelectionState`
- 目标不是继续堆通用卡片，而是形成“后台工作台组件集”。

### 7.3 设计 token 调整方向

- 当前暖色视觉可保留，但后台需要更强的信息层级。
- 建议加强：
  - 侧栏背景与工作区背景对比
  - 激活项对比度
  - 表格 / 列表 hover 态
  - 分区标题与正文层次
- 视觉方向应更偏“运营控制台”，少一点营销 Hero 感。

## 8. 建议实施顺序

### 阶段 1：壳层重构

- 目标：
  - 左侧导航上线
  - 顶部 Hero 退场
  - 页面有统一工作区骨架
- 涉及文件：
  - `apps/admin/src/components/root-layout.tsx`
  - `apps/admin/src/router.tsx`
  - `apps/admin/src/index.css`
  - 新增后台壳层组件

### 阶段 2：高频入口重排

- 优先改：
  - `dashboard-page.tsx`
  - `setup-page.tsx`
  - `characters-page.tsx`
- 目标：
  - 让运营人员先能顺畅找到对象和入口。

### 阶段 3：角色纵向工作流打通

- 改造：
  - `character-editor-page.tsx`
  - `character-factory-page.tsx`
  - `character-runtime-page.tsx`
- 目标：
  - 进入角色后不迷路，基础资料 / 工厂 / 运行逻辑形成稳定二级结构。

### 阶段 4：重工作台重排

- 改造：
  - `reply-logic-page.tsx`
  - `evals-page.tsx`
- 目标：
  - 把最难用的两个大页改成真正可操作的控制台。

## 9. 交付标准

- 左侧主导航可稳定承载所有一级模块。
- 深层页面进入后，用户能明确知道当前所属模块和对象上下文。
- 总览、设置、角色中心三页首屏在 5 秒内可理解。
- 回复逻辑和评测页不再要求用户通过长滚动寻找功能。
- 页面主操作固定，保存、发布、运行预览等动作不再漂浮在内容流中。

## 10. 风险与注意事项

- 这次改版优先是 IA 和工作流重构，不建议一开始就同步大规模业务逻辑抽离，否则会拖慢节奏。
- `reply-logic-page.tsx` 与 `evals-page.tsx` 体量过大，布局重构时要同步做局部组件拆分，否则后续仍难维护。
- 角色相关页面之间的上下文导航必须统一，否则“左侧主导航上线后，页面内部还是乱”。
- 现有 `users-page.tsx` 基本无实际作用，后续可考虑并入或移除，避免导航噪音。

## 11. 推荐下一步

- 先直接开做阶段 1 和阶段 2。
- 第一批实现建议只追求三件事：
  - 左侧主导航落地
  - 总览页改成控制台首页
  - 角色页改成列表 + 详情 + 快捷动作
- 等壳层稳定后，再处理角色工厂、运行逻辑台、回复逻辑和评测这几个重页面。
