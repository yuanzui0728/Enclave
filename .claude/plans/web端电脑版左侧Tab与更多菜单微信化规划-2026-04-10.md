# Web端电脑版左侧Tab与更多菜单微信化规划

## 任务目标

- 把 `apps/app` 的 Web 桌面版左侧壳层，从现在的 `消息 / 通讯录 / 朋友圈 / 发现 / 我`，调整成更接近微信电脑版的功能导航。
- 一级导航目标顺序按本次需求收口为：
  - `消息`
  - `通讯录`
  - `收藏`
  - `朋友圈`
  - `广场动态`
  - `视频号`
  - `搜一搜`
  - `游戏中心`
  - `小程序面板`
- 左侧底部固定两个按钮：
  - `手机`
  - `更多`
- `更多` 菜单收口这些能力：
  - `视频号直播伴侣`
  - `聊天文件`
  - `聊天记录管理`
  - `加载历史聊天记录`
  - `锁定`
  - `意见反馈`
  - `设置`
- 核心原则是“交互路径、入口层级、桌面工作区组织尽量贴近微信电脑版”，同时保留 `广场动态` 作为隐界特有一级入口。

## 产品判断

### 1. 桌面端不再保留“发现”一级 Tab

如果目标是高度贴近微信电脑版，就不应该继续保留现在这个泛化的 `发现` 一级入口。

建议改成：

- `朋友圈` 独立一级入口
- `广场动态` 独立一级入口
- `视频号` 独立一级入口
- `搜一搜` 独立一级入口
- `游戏中心` 独立一级入口
- `小程序面板` 独立一级入口

现有 `发现` 页保留给移动端，桌面端不再作为 Shell 主导航显示。

### 2. `我` 继续保持非一级入口

微信电脑端不是把“我”做成左侧一级导航，而是通过头像和设置进入。

建议继续沿用当前做法：

- 左上头像点击进入资料/设置语义
- `设置` 放入 `更多` 菜单
- 不把 `我` 放回左侧一级 Tab

### 3. `广场动态` 是隐界专属项，但要放进微信式节奏里

`广场动态` 不应该再藏在 `发现` 里，否则它在桌面端没有稳定入口。

建议：

- 放在 `朋友圈` 后、`视频号` 前
- 语义上明确区分：
  - `朋友圈`：熟人私域
  - `广场动态`：世界居民公开流
  - `视频号`：短视频/直播内容流

### 4. “更多”菜单既要像微信，也要避免空壳感

这次不能把所有菜单项都做成纯文案占位。

建议分两类：

- 可直接落地的，优先做真能力：
  - `聊天文件`
  - `聊天记录管理`
  - `设置`
  - `锁定`
- 需要后端或新域模型的，允许首期做桌面工作区壳：
  - `视频号直播伴侣`
  - `意见反馈`
  - `收藏`
  - `视频号`
  - `游戏中心`
  - `小程序面板`

### 推荐路由

建议首期把桌面主导航收口到下面这些路由：

- `/tabs/chat`
- `/tabs/contacts`

底部能力建议：

- `手机`：`/desktop/mobile`
- `更多`：不是单独一级路由，而是壳层 Popover / Menu

更多菜单落点建议：

- `视频号直播伴侣` -> `/desktop/channels/live-companion`
- `聊天文件` -> `/desktop/chat-files`
- `聊天记录管理` -> `/desktop/chat-history`
- `意见反馈` -> `/desktop/feedback`
- `设置` -> `/desktop/settings`

`加载历史聊天记录` 和 `锁定` 更适合作为动作，而不是普通页面：

- `加载历史聊天记录`
  - 有当前会话时：直接对当前会话触发“向前加载”
  - 无当前会话时：打开聊天记录管理面板，引导选择会话
- `锁定`
  - 直接拉起桌面锁屏层，不建议单独跳页

### 路由命名判断

这里建议把 `广场动态` 提升为 `/tabs/feed`，而不是继续复用 `/tabs/discover`。

原因：

- 这次桌面结构目标是“微信式功能导航”，不是“发现聚合页”
- `广场动态` 已经是独立产品语义
- 后面如果补 `视频号`，再把 `广场动态` 塞回 `发现` 会更乱

注意：

- 路由新增属于结构变更，后续真正实施时要同步更新 `AGENTS.md`

## 壳层实现建议

### 1. 抽离桌面导航注册表

不要继续在 [desktop-shell.tsx](/home/ps/claude/yinjie-app/apps/app/src/features/shell/desktop-shell.tsx) 内硬编码数组。

建议新增：

- `apps/app/src/features/shell/desktop-nav-config.ts`

职责：

- 声明一级导航顺序
- 声明图标、文案、路由、激活规则
- 区分：
  - 主导航项
  - 底部动作项
  - 更多菜单项

这样后续扩展不会继续把壳层文件写成超长条件分支。

### 2. Shell 需要支持三种入口类型

左栏项不再只有“普通链接”一种。

需要支持：

- `route`
  - 例如 `消息 / 通讯录 / 朋友圈 / 广场动态`
- `panel`
  - 例如 `手机`
- `menu`
  - 例如 `更多`

这层抽象能避免后续为了一个底部菜单继续把所有逻辑塞进 `desktop-shell.tsx`。

### 3. 高度不足时允许左栏滚动

从 4 项增加到 9 项后，桌面高度低时左栏必然挤压。

建议：

- 主导航区独立滚动
- 底部 `手机 / 更多` 固定吸底
- 窄宽度下只显示图标，文字改成 Tooltip

### 4. 激活规则统一收口

当前 `isActive()` 只覆盖 `消息 / 通讯录` 的少量路由匹配。

扩展后应改成配置驱动：

- 每个入口自带 `match` 规则
- 支持：
  - 精确路径
  - 前缀路径
  - 多路由归并

否则后面 `更多` 打开的子页面会让高亮状态继续失控。

## 各工作区规划

### 1. 消息

继续以 [desktop-chat-workspace.tsx](/home/ps/claude/yinjie-app/apps/app/src/features/desktop/chat/desktop-chat-workspace.tsx) 为核心。

需要补的不是主结构，而是“微信桌面消息体系的外围工具”：

- 左栏加入特殊系统项的承接能力：
  - 未来可加 `订阅号消息`
  - 未来可加 `文件传输助手`
- 顶部搜索升级为全消息域搜索入口
- 与 `聊天文件 / 聊天记录管理 / 加载历史聊天记录` 互通

### 2. 通讯录

通讯录基础已经足够好，首期主要保持不动，只调整左栏导航入口。

后续只需保证：

- 从 `搜一搜` 可以跳转到通讯录详情
- 从 `更多 -> 设置` 不打断通讯录工作区状态

### 3. 收藏

`收藏` 不能只做前端数组。

它在微信里的核心是跨内容类型统一收口，隐界里至少应覆盖：

- 聊天文本
- 聊天文件
- 公众号文章
- 朋友圈文本
- 广场动态文本

首期建议：

- 先做桌面收藏工作区壳
- 先定义收藏条目统一协议
- 没有真实收藏数据时展示空状态和设计口径

第二阶段建议新增：

- `FavoriteItem` 实体
- `favorites` 模块
- 收藏接口与共享 contracts

### 4. 朋友圈

保持当前独立一级入口，继续沿用现有桌面朋友圈工作区。

这一项已经有较完整基础，后续只要继续按既有规划微信化即可。

### 5. 广场动态

这是本次最应该优先“升格”的现有能力。

建议：

- 把当前 [discover-feed-page.tsx](/home/ps/claude/yinjie-app/apps/app/src/routes/discover-feed-page.tsx) 的桌面承接从“发现二级页”改成“一级导航工作区”
- 新建桌面版 `feed` 工作区，不再让桌面走移动页结构
- 内容语义明确标为“公开动态流”

这里最好不要再复用 `发现` 这个概念。

### 6. 视频号

`视频号` 是微信桌面端的重要一级能力，但当前项目没有对应内容模型。

建议分两步：

- Phase 1
  - 先做桌面工作区壳
  - 支持空状态、推荐位、直播入口位
  - `更多 -> 视频号直播伴侣` 能跳进对应工作区
- Phase 2
  - 新增视频号实体、视频内容流、评论/点赞、直播状态模型

不要拿现有 `feed` 直接冒充 `视频号`，否则后面会很难收口。

### 7. 搜一搜

`搜一搜` 首期不必等专用搜索服务。

可以先做聚合搜索工作区，跨这些已有数据源：

- 会话标题和消息文本
- 联系人
- 公众号
- 朋友圈
- 广场动态

桌面工作区建议三段式：

- 顶部全局搜索框
- 左侧结果分类
- 右侧结果明细 / 跳转预览

第二阶段如果数据量上来，再补独立后端搜索接口。

### 8. 游戏中心

首期建议做“配置化入口面板”，不要一上来就做复杂运行平台。

可包含：

- 最近玩过
- 推荐入口
- 官方活动位

后端不一定需要先做表，可以先从配置或静态清单启动。

### 9. 小程序面板

首期建议定义为“桌面快捷面板”，而不是完整小程序容器。

建议能力边界：

- 收藏的小程序入口
- 最近使用
- 搜索/筛选
- 打开方式统一走新窗口、内嵌页签或业务工作区容器

真正的“小程序运行时”不要和这次左栏导航改造绑死。

### 10. 手机

`手机` 不建议做成普通页面导航，建议做底部设备面板。

首期可以先承接：

- 当前移动设备连接状态
- 提示用户在手机端继续操作
- 后续可扩展到文件互传或通知联动

### 11. 更多

`更多` 是壳级菜单，不是业务频道。

它的设计重点：

- 打开速度快
- 不破坏当前主工作区
- 能一眼区分“页面入口”和“立即执行动作”

建议分组：

- 内容与消息
  - `聊天文件`
  - `聊天记录管理`
  - `加载历史聊天记录`
- 工具
  - `视频号直播伴侣`
  - `锁定`
- 支持
  - `意见反馈`
  - `设置`

## “更多”菜单能力落地判断

### 聊天文件

这项优先级很高，而且可以优先落地。

现有基础：

- 聊天消息已经支持附件元数据
- 服务端已有附件上传与访问能力

建议补充：

- 桌面 `聊天文件` 聚合页
- 按会话、类型、时间筛选
- 点击文件可跳到原消息

必要时新增后端接口：

- `GET /api/chat/files`
- `GET /api/groups/files`

### 聊天记录管理

这项也可以较快落地。

现有基础：

- 单会话已经有 `clearConversationHistory`
- 已有单聊/群聊消息查询

建议工作区包含：

- 会话列表
- 最近活跃时间
- 消息量/附件量
- 清空入口
- 跳转会话入口

必要时新增汇总接口：

- `GET /api/chat/history/summary`
- `POST /api/chat/history/clear-batch`

### 加载历史聊天记录

这项不建议做成独立页面。

建议定义为两个行为：

- 在会话线程内点击时，向前加载更旧消息
- 在 `聊天记录管理` 中点击时，跳转到目标会话并自动加载目标时间段附近消息

这要求后续把消息查询从“只取当前列表”升级为“支持 cursor / before / pageSize”。

### 锁定

这项建议只做客户端能力，不依赖后端。

建议：

- 在桌面壳层加本地锁屏状态
- 进入锁定后需要输入本地密码或系统二次验证
- 不影响服务端世界状态

### 意见反馈

当前仓库没有反馈接口。

建议分两阶段：

- Phase 1
  - 做桌面反馈页壳
  - 允许复制实例信息、版本信息、当前世界地址
- Phase 2
  - 增加反馈提交接口或 webhook 投递

### 设置

不要直接把微信式桌面设置做成继续跳 [profile-settings-page.tsx](/home/ps/claude/yinjie-app/apps/app/src/routes/profile-settings-page.tsx) 的手机版式内容页。

建议：

- 新增 `/desktop/settings`
- 复用现有世界主人资料与 API Key 相关表单
- 以桌面工作区重排

## 最终结论

这次桌面端改造的关键不是“单纯把左侧按钮加多”，而是把桌面信息架构从“当前移动端 Tab 的放大版”升级成“微信电脑版式的功能壳 + 工作区体系”。

优先级建议非常明确：

1. 先改壳层和路由，把 `广场动态` 升格、把 `更多` 做出来。
2. 先兑现 `聊天文件 / 聊天记录管理 / 设置 / 锁定` 这类已有基础的真实能力。
3. 再补 `收藏 / 视频号 / 搜一搜 / 游戏中心 / 小程序面板 / 意见反馈` 的完整域模型和桌面工作区。

这样能最快把桌面端的“微信味”做出来，同时不把后续实现带进一堆假入口和重复返工。
