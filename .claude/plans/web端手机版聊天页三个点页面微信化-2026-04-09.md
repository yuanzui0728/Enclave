# Web端手机版聊天页三个点页面微信化规划

## 任务目标
- 将 `apps/app` 的 Web 移动版聊天页右上角“三个点”入口改造成微信式聊天详情页。
- 目标不是“做一个像设置页的替代品”，而是尽量直接复刻微信移动端的页面结构、列表节奏、开关项和危险操作区。
- 范围同时覆盖：
  - 单聊：`/chat/$conversationId`
  - 群聊：`/group/$groupId`

## 当前现状

### 1. 已有入口
- `apps/app/src/features/chat/conversation-thread-panel.tsx`
  - 移动端单聊头部已经有 `Ellipsis` 按钮，但没有任何行为。
- `apps/app/src/routes/group-chat-page.tsx`
  - 群聊头部同样有 `Ellipsis` 按钮，也没有行为。

### 2. 已有可复用能力
- 会话能力：
  - `getConversations`
  - `getConversationMessages`
  - `setConversationPinned`
  - `hideConversation`
  - `clearConversationHistory`
- 社交能力：
  - `getFriends`
  - `getBlockedCharacters`
  - `blockCharacter`
  - `unblockCharacter`
  - `createModerationReport`
- 群聊能力：
  - `getGroup`
  - `getGroupMembers`
  - `addGroupMember`
  - `getGroupMessages`

### 3. 当前缺口
- 路由树里没有“聊天详情页 / 聊天信息页”。
- 单聊和群聊都没有“查找聊天记录”页面。
- 群聊没有以下后端能力：
  - 消息免打扰
  - 置顶聊天
  - 清空聊天记录
  - 群公告
  - 我在本群的昵称
  - 显示群成员昵称
  - 退出群聊 / 删除并退出
  - 移除成员
- 当前移动聊天头部下方还有一排“语音 / 视频”胶囊按钮，这本身就不像微信。

## 微信式目标口径

### 1. 单聊详情页
- 点击右上角三个点，直接进入独立详情页，不做浮层菜单。
- 页面结构按微信常见单聊详情页拆成 5 段：
  - 成员宫格区：世界主人、对方角色、加号入口
  - 聊天记录区：`查找聊天记录`
  - 开关区：`消息免打扰`、`置顶聊天`、`强提醒`
  - 辅助设置区：`保存到通讯录`、`设置当前聊天背景`
  - 风险操作区：`清空聊天记录`、`投诉`、`加入黑名单`

### 2. 群聊详情页
- 点击右上角三个点，直接进入独立群聊详情页。
- 页面结构按微信常见群聊详情页拆成 7 段：
  - 成员宫格区：成员头像栅格、加号、减号
  - 基础信息区：`群聊名称`、`群公告`
  - 聊天记录区：`查找聊天记录`
  - 开关区：`消息免打扰`、`置顶聊天`、`保存到通讯录`
  - 群内设置区：`我在本群的昵称`、`显示群成员昵称`
  - 辅助设置区：`设置当前聊天背景`
  - 风险操作区：`清空聊天记录`、`删除并退出`

### 3. 视觉要求
- 这一页要主动贴近微信，不沿用当前项目普遍的“大圆角 + 渐变卡片 + 悬浮感”。
- 推荐视觉基调：
  - 纯平浅灰背景
  - 白底列表分组
  - 细分隔线
  - 行高和留白接近微信
  - 开关、危险按钮、成员宫格都按微信阅读节奏排布
- 结论：
  - 这页可以在 `MobileShell` 大框里存在，但页面内部视觉要明显收平，不要继续做成“梦幻卡片页”。

## 建议路由方案

### 1. 新增详情页路由
- 单聊详情页：
  - `/chat/$conversationId/details`
- 群聊详情页：
  - `/group/$groupId/details`

### 2. 新增聊天记录检索页路由
- 单聊记录检索：
  - `/chat/$conversationId/search`
- 群聊记录检索：
  - `/group/$groupId/search`

### 3. 为什么不用抽屉
- 微信这里是整页跳转，不是底部抽屉。
- 当前移动端壳已经有稳定的二级页路由模式，跟 `discover/*`、`profile/settings` 一致，落地成本低。
- 详情页后续会越来越重，整页路由更适合继续叠加搜索、公告、成员管理等二级页。

## 信息架构映射

### A. 单聊详情页映射
- 成员宫格
  - 世界主人头像
  - 角色头像
  - `+` 入口
  - `+` 点击后进入现有 `/group/new`
  - 后续可补“默认预选当前角色”
- 查找聊天记录
  - 新建独立搜索页
  - 直接基于 `getConversationMessages` 本地搜索消息文本
  - 首版不做服务端全文检索
- 消息免打扰
  - 首版建议做真实 UI，但先接前端本地偏好
  - 存储键可按 `conversationId` 维度写入 `localStorage`
- 置顶聊天
  - 直接接现有 `setConversationPinned`
- 强提醒
  - 首版可先走本地偏好
  - 后续若接原生通知桥或服务端提醒策略，再升级为真实能力
- 保存到通讯录
  - 本项目里应解释为“保持为好友 / 已进入通讯录”
  - 如果该角色已经是好友，展示为已保存态
  - 如果尚未是好友，点击后跳现有好友申请或直接触发加好友动作
- 设置当前聊天背景
  - 首版只做入口和空壳页或本地占位，不接真正背景系统
- 清空聊天记录
  - 直接接现有 `clearConversationHistory`
- 投诉
  - 可接现有 `createModerationReport`
- 加入黑名单
  - 直接接现有 `blockCharacter`

### B. 群聊详情页映射
- 成员宫格
  - 展示现有 `getGroupMembers`
  - `+` 入口接后续“添加成员”子页
  - `-` 入口先只做 UI 保留位，正式移除成员需新增后端
- 群聊名称
  - 首版展示只读
  - 后续新增修改接口后再可编辑
- 群公告
  - 首版只能是占位字段或静态文案
  - 如果不想先做假数据，就先做“暂无群公告”
- 查找聊天记录
  - 新建 `/group/$groupId/search`
  - 基于 `getGroupMessages` 本地搜索
- 消息免打扰
  - 首版走本地偏好
- 置顶聊天
  - 当前群聊不在 `Conversation` 主链路里，没有现成后端
  - 首版只能做本地偏好或延后
- 保存到通讯录
  - 当前没有群维度收藏能力
  - 建议首版只做 UI 占位，不启用
- 我在本群的昵称
  - 当前没有 owner 在 group member 上的可编辑昵称接口
  - 需后端新增字段和更新接口
- 显示群成员昵称
  - 首版走前端本地偏好
- 设置当前聊天背景
  - 同单聊，先做占位
- 清空聊天记录
  - 需要新增群聊清空接口
- 删除并退出
  - 需要新增群成员退出接口

## 交互细节建议

### 1. 顶部成员宫格
- 单聊页顶部不要只放一个头像卡片，应该像微信一样做成员栅格。
- 单聊建议顺序：
  - 我
  - 对方
  - `+`
- 群聊建议顺序：
  - 已有成员
  - `+`
  - `-`
- 栅格建议 4 列，头像下方一行名字，超长省略。

### 2. 列表项
- 普通跳转行：
  - 左侧标题
  - 右侧状态文案或箭头
- 开关行：
  - 左侧标题
  - 右侧 switch
- 危险行：
  - 单独分组
  - 红色文字
  - 底部留足安全区

### 3. 查找聊天记录
- 进入独立页后顶部是搜索框，不做弹窗内搜索。
- 搜索结果按消息时间倒序。
- 命中项展示：
  - 发送者名
  - 片段高亮
  - 时间
- 点击结果后返回聊天页并滚到目标消息：
  - 首版如果不做滚动定位，也至少先高亮一条最近命中消息。

## 验收标准
- 点击聊天页右上角三个点后，进入的是微信式整页详情，而不是自定义菜单或卡片弹层。
- 单聊和群聊详情页的信息架构明显区分，不是同一套列表硬复用。
- 单聊页至少有 4 个微信式真功能可用：
  - 查找聊天记录
  - 置顶聊天
  - 清空聊天记录
  - 加入黑名单
- 群聊页顶部成员宫格、群名区、聊天记录搜索入口必须成立。
- 视觉上不再是当前项目卡片风，而是接近微信的扁平白底分组页。

## 风险与决策
- 当前群聊链路不在 `Conversation` 体系里，这是群聊详情页最核心的能力缺口来源。
- 如果不补群聊后端，群聊详情页第一版只能做到“像微信的壳”，做不到“像微信的完整功能”。
- 如果继续保留移动聊天头部第二排语音/视频胶囊，整体会出现“详情页像微信，但聊天页本身不像微信”的割裂。
- 建议实施时优先把“微信式结构”和“能直接落的真功能”先做出来，不要一开始就试图把所有群聊能力补齐。

## 推荐落地顺序
1. 先加详情页和搜索页路由。
2. 再做单聊详情页，因为现有接口最完整。
3. 然后做群聊详情页壳层和成员宫格。
4. 最后补群聊后端能力，把占位项逐步转成真功能。
