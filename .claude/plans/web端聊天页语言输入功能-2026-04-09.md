# Web端聊天页输入框语音转文字规划

## 1. 目标
- 范围只覆盖 `apps/app` 的 Web 端手机版和桌面版聊天页输入框。
- 能力定义是“语音转文字后回填输入框”，不自动发送，不做语音消息气泡，不做语音通话。
- 在现有聊天输入区补齐稳定可用的按钮、状态提示、权限处理和失败兜底。

## 2. 当前代码现状

### 2.1 已经存在的能力
- `apps/app/src/components/chat-composer.tsx`
  - 已经有移动端和桌面端两套 `Mic` 按钮 UI。
  - 移动端已经有 `MobileSpeechInputSheet` 底部面板。
  - 桌面端已经有 `InlineNotice` 状态条，支持“正在聆听 / 正在转写 / 识别完成 / 取消”。
- `apps/app/src/features/chat/use-speech-input.ts`
  - 已实现统一状态机：`idle / requesting-permission / listening / processing / ready / error`。
  - 已实现双通路：
    - 浏览器原生 `SpeechRecognition / webkitSpeechRecognition`
    - `MediaRecorder + /api/ai/transcriptions` 服务端转写兜底
- `apps/app/src/features/chat/conversation-thread-panel.tsx`
  - 已经把语音输入接到主聊天线程页。
  - 只在 `runtimeConfig.appPlatform === "web"` 时启用，符合“只做 Web 端”的边界。
- `packages/contracts/src/client.ts`
  - 已有 `createSpeechTranscription()` 契约调用。
- `api/src/modules/ai/ai.controller.ts`
  - 已有 `POST /api/ai/transcriptions`。
- `api/src/modules/ai/ai-orchestrator.service.ts`
  - 已有音频格式校验、10MB 限制、Provider 调用和错误兜底。
  - 默认转写模型已经走 `gpt-4o-mini-transcribe`。

### 2.2 当前真实缺口
- `apps/app/src/routes/group-chat-page.tsx`
  - 仍在使用 `ChatComposer`，但没有传 `speechInput`。
  - 结果是群聊页仍会看到语音按钮，但点击后只能进入“不支持”错误流，体验是假的。
- `apps/app/src/components/chat-composer.tsx`
  - 当前按钮是否显示，基本只看布局，不看是否真的应该暴露给用户。
  - 非 Web 场景或不支持的场景里，按钮仍可能出现，应该收口。
- 当前语义有一点历史债：
  - 转写接口请求体里用的是 `conversationId`。
  - 兼容 `GroupChatPage` 时，日志字段语义会和 `groupId` 不完全一致。
- 现有实现更像“能力已经接上”，但还没有完全收成一个稳定、统一的产品交互。

## 3. 产品决策

### 3.1 统一原则
- 识别结果永远先进入草稿态，再由用户决定是否插入输入框。
- 插入输入框后，仍沿用现有文本发送按钮，语音输入不改发送链路。
- Web 手机版和桌面版共用同一套转写状态机，不拆两套底层逻辑。
- 非 Web 端不在本轮范围内，不暴露这套入口。

### 3.2 Web 手机版
- 输入框左侧保留 `Mic` 按钮。
- 点击后打开底部语音面板 `MobileSpeechInputSheet`。
- 面板继续采用“三段式”：
  - 状态文案
  - 开始/停止主按钮
  - 取消 / 插入输入框
- 识别结果先显示在面板草稿区，不自动写回输入框。
- 点击“插入”后再合并回当前文本输入。

### 3.3 Web 桌面版
- 输入框右侧保留 `Mic` 按钮。
- 点击后进入内联语音输入状态，不打断当前工作区。
- 继续使用 composer 下方状态条展示：
  - 正在请求权限
  - 正在聆听
  - 正在转写
  - 识别完成
  - 错误/取消
- 识别完成后，由用户点击“插入输入框”。

### 3.4 不支持场景的策略
- `speechInput.enabled === false` 时，直接不显示语音按钮。
- Web 环境但浏览器能力不足时，按钮可以保留但必须是明确禁用态，不能点进去才报错。
- 不再允许“看起来能点，点了才发现没开”的伪可用状态。

## 4. 建议实施方案

### Phase 1：入口收口
目标是先把按钮暴露范围和启用条件收干净。

改造点：
- `apps/app/src/components/chat-composer.tsx`
  - 增加“是否展示语音入口”的显式判断。
  - 区分：
    - `showSpeechEntry`
    - `speechSupported`
    - `speechDisabledReason`
  - 移动端和桌面端按钮都不能再无条件渲染。
- 交互规则调整：
  - 非 Web 或未传 `speechInput` 时，隐藏按钮。
  - Web 且支持时，正常开始录音/识别。
  - Web 但不支持时，展示禁用态和明确文案。

交付结果：
- 语音按钮只会在真正属于本次范围的页面出现。
- 不再出现“群聊页能看到按钮但其实不可用”的状态。

### Phase 2：页面覆盖补齐
目标是把所有 Web 聊天入口统一接上。

改造点：
- `apps/app/src/features/chat/conversation-thread-panel.tsx`
  - 保持现有启用逻辑，继续作为主线程页标准接入方式。
- `apps/app/src/routes/group-chat-page.tsx`
  - 补传 `speechInput`，让旧群聊页和主线程页一致。
  - 首版可以继续把 `groupId` 填进现有 `conversationId` 字段作为日志上下文，不阻塞功能落地。
  - 如果后续要做语义收口，再把接口字段统一改成 `threadId` / `threadType`。

交付结果：
- Web 手机版和桌面版里的单聊、群聊页面都能走同一套语音转文字入口。

### Phase 3：交互打磨
目标是让这套能力从“能跑”变成“可交付”。

改造点：
- 移动端：
  - 打开语音面板时主动处理键盘收起，避免键盘和面板抢空间。
  - 识别完成后保留结果，允许重复插入前先编辑。
- 桌面端：
  - 状态条文案统一，区分“识别中”和“转写中”。
  - `Mic` 和 `Stop` 的图标切换保持一致。
- 通用：
  - 插入输入框后把焦点还给输入框。
  - 权限拒绝、无语音、网络失败、Provider 不可用四类错误文案分开。
  - 保持“不会自动发送”的产品约束。

交付结果：
- 移动端和桌面端交互表面不同，但用户心智一致。

### Phase 4：后端与运维补充
目标是减少上线后不可诊断的问题。

改造点：
- 继续使用现有 `POST /api/ai/transcriptions`，不新增聊天专用接口。
- 补充日志关注点：
  - 页面来源
  - 会话上下文标识
  - 音频类型
  - 文件大小
  - 转写耗时
  - Provider 失败原因
- 评估是否需要把 `provider_transcription_model` 暴露到管理端配置；首版可以先不做，沿用服务端默认模型。

交付结果：
- 首版上线后能定位“浏览器不支持 / 权限问题 / Provider 不支持 / 网络失败”分别占多少。

## 5. 推荐实现顺序
1. 先收口 `ChatComposer` 的按钮显隐与禁用态。
2. 再补 `GroupChatPage` 的 `speechInput` 接入。
3. 再处理移动端和桌面端的交互细节与文案统一。
4. 最后补日志和可观测性。

## 6. 验收标准
- Web 手机版单聊页可以通过 `Mic` 按钮进入语音转文字流程，并把结果插入输入框。
- Web 桌面版单聊页可以在当前聊天工作区内完成语音输入，不跳页、不弹大浮层。
- Web 群聊页不再出现“按钮可见但功能不可用”的状态。
- 浏览器不支持时，入口表现为明确禁用态或隐藏态，不再走错误流假触发。
- 识别结果不会自动发送，用户始终可以编辑后再发。
- 现有文本发送、表情包发送和消息列表不受影响。

## 7. 风险与取舍
- 最大风险不是后端接口，而是入口显隐和浏览器差异导致的“伪可用”体验，所以第一步必须先收入口。
- `GroupChatPage` 仍是旧路由形态，字段命名与 `conversationId` 有轻微语义偏差，但首版不建议为这件事阻塞功能。
- Web 浏览器兼容性本身不稳定，所以必须保留现有“双通路”方案，不能退回只做 `SpeechRecognition`。

## 8. 结论
- 这项需求不需要从零开发。
- 当前仓库已经具备 70% 到 80% 的底层能力，重点是做“入口收口 + 群聊补齐 + 交互定稿”。
- 如果按上面顺序推进，首版应该是一轮中等规模前端收尾改造，而不是新的系统能力建设。
