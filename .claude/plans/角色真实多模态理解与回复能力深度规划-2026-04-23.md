# 角色真实多模态理解与回复能力深度规划

## 0. 2026-04-23 深度复查后的收口执行计划

本节是当前最新执行口径，优先级高于后文早期规划。深度复查结论是：代码层已经具备真实多模态骨架和部分闭环，但还没有达到“当前实例可证明、产品可见、后台可诊断、失败可追踪”的彻底实现标准。

### 0.1 最新现状
- 已真实存在：图片输入可作为 `AiMessagePart.image` 进入聊天主链路，并映射到 `image_url / input_image`。
- 已真实存在：语音、音频文件、视频文件会进入 `MediaInsightJob`，通过转写结果回写 `transcriptText`。
- 已真实存在：文本类文档可抽取正文，Responses API provider 可走原生文件输入。
- 已真实存在：普通单聊可按规划生成文本 + 语音 sibling message，图片回复通过 `ReplyArtifactJob` 异步生成。
- 已真实存在：群聊可通过 `ReplyArtifactJob` 异步生成语音 / 图片回复。
- 未彻底：当前运行实例默认 provider 没有配置 `transcriptionModel`，语音/视频理解不能在当前实例被证明可用。
- 未彻底：图片生成、TTS、数字人没有独立 provider 通道，仍过度依赖默认推理 endpoint 是否兼容。
- 未彻底：后台 contract / API client / UI 没有完整接入 `media-insights`，`ReplyArtifactJob` 也没有后台查询面板。
- 未彻底：PDF / Office 文档被识别为文档任务，但在非 Responses provider 下缺少真实正文抽取。
- 未彻底：视频仍主要是“文件 + 音轨转写”，没有抽帧、OCR、画面摘要和视频播放器级展示。
- 未彻底：多模态健康检查没有独立 smoke 入口，无法一键证明图像输入、转写、TTS、图片生成、数字人 provider 全部可用。

### 0.2 彻底实现定义
- 当前实例必须能在后台诊断页显示每一种能力的真实可用状态：图片输入、语音转写、视频理解、文档抽取、TTS、图片生成、数字人 render。
- 每一种输入模态必须有持久化 insight：状态、provider、错误码、错误信息、结果预览、来源消息、重试次数。
- 每一种输出模态必须有持久化 artifact job：状态、provider、错误码、错误信息、产物消息、取消原因、重试次数。
- 前端必须能真实发送和展示图片、语音、视频、文档及其分析状态，而不是只把附件当作文件名。
- 后台必须能复盘一轮多模态消息：用户输入了什么模态、系统抽取了什么洞察、AI 使用了什么 provider、规划了什么回复模态、最终交付了什么。
- 验收必须包含真实运行 smoke，而不是只通过 typecheck。

### 0.3 总体实施路线
- 第一阶段先修“当前实例不可证明”的问题：provider 能力配置、诊断 smoke、后台可观测。
- 第二阶段补“理解不彻底”的问题：PDF / Office 文档、图片摘要/OCR、视频抽帧/OCR/转写组合。
- 第三阶段补“回复不彻底”的问题：独立 TTS / 图片生成 provider、reply artifact 后台可观测、数字人真实 render job。
- 第四阶段补“产品不可见”的问题：前端视频展示、分析状态、后台多模态工作台、评测集。

### 0.6 提交 3：MediaInsightJob 扩展为真正多模态理解流水线
- 调整 `media_insight_jobs` 唯一性：
  - 当前唯一约束是 `threadType + threadId + sourceMessageId`。
  - 必须改为 `threadType + threadId + sourceMessageId + insightKind`，否则同一视频无法同时跑转写、抽帧、OCR、摘要。
- 扩展 `MediaInsightKind`：
  - `image_vision_summary`
  - `image_ocr`
  - `audio_transcription`
  - `video_transcription`
  - `video_frame_summary`
  - `video_ocr`
  - `document_text_extraction`
  - `document_ocr`
  - `document_summary`
- 增加 `MediaInsightArtifact` 或 `MediaInsightJob.resultPayload` 统一结构：
  - `provider`
  - `text`
  - `summary`
  - `ocrText`
  - `frames`
  - `durationMs`
  - `width`
  - `height`
  - `pageCount`
  - `confidence`
  - `sourceAssetUrl`
- 图片理解：
  - 发送图片后异步生成 `image_vision_summary`。
  - 若 provider 支持原生图片输入，聊天回复仍走原生图片。
  - 若 provider 拒绝图片输入，自动用 `visionSummary + ocrText` fallback，而不是只用文件名。
- 文档理解：
  - `txt / md / csv / json / xml / html` 走本地文本抽取。
  - `pdf` 走 PDF 文本抽取库；空文本 PDF 标记为需要 OCR。
  - `docx` 走 docx 文本抽取库。
  - 非文本可抽取时，走 provider document input 或 OCR fallback。
- 视频理解：
  - 上传视频后识别为视频附件或视频文件。
  - 服务端抽取基础元数据：时长、宽高、mime、文件大小。
  - 抽取关键帧，至少首帧 / 中间帧 / 末帧。
  - 对关键帧跑图像摘要与 OCR。
  - 同时跑音轨转写。
  - 合成 `videoSummary` 进入聊天 prompt。
- 验收：
  - 发图片，后台出现 `image_vision_summary`，聊天可引用图中元素。
  - 发 PDF，后台出现 `document_text_extraction`，聊天可引用正文。
  - 发视频，后台出现 `video_transcription + video_frame_summary`，聊天可同时引用画面和音轨。
  - API build 通过。

## 1. 目标
- 让角色真正具备多模态理解能力，不再只靠附件占位文本或音频转写兜底。
- 让角色真正具备多模态回复能力，不再局限于普通聊天文本回复，能按场景输出语音、图片、视频数字人等结果。
- 保持单世界架构、现有聊天主协议和前端体验连续演进，避免一次性推翻现有消息模型。
- 优先做“真实可运行”的能力，不做只有 UI 壳、没有底层闭环的伪多模态。

## 3. 产品定义

### 3.1 什么叫“真正具备多模态理解能力”
- 角色能消费图片、语音、视频、文档等输入的真实语义，而不是只看占位文案。
- 对视频理解不能只依赖音轨转写，至少要补视觉帧、OCR、场景摘要。
- 对音频理解不能只停在普通听写，至少要保留时长、说话片段、语速或说话人片段等元数据入口。
- 对文档不能只看文件名，至少要有可抽取的正文/OCR摘要进入模型。

### 3.2 什么叫“真正具备多模态回复能力”
- 角色可以按场景决定回复模态，而不是所有场景都强制退回文本。
- 在普通聊天场景，至少支持：
  - 文本
  - 语音
  - 图片
  - 后续扩展到视频卡片/短视频
- 在语音通话场景，默认回复模态至少是：
  - 文本
  - 语音
- 在数字人视频通话场景，默认回复模态至少是：
  - 文本
  - 语音
  - 数字人视频结果或可回退的播放器结果

### 3.3 多模态能力不是单个模型能力，而是系统能力
- 真能力必须拆成四层：
  - 输入接入层
  - 理解预处理层
  - 推理编排与模型路由层
  - 输出渲染与交付层
- 不能把“provider 能看图”直接等同于“系统具备完整多模态能力”。

## 4. 方案原则
- 统一内部消息模型，最后一跳再映射到 provider 协议。
- 理解能力优先做稳定的派生表示，不把能力完全绑死在某一个模型是否原生支持视频/音频。
- 回复能力优先做“规划器 + 渲染器”两段式，不让聊天主链路直接耦合 TTS、画图、数字人 provider。
- 保持消息表兼容，首期不要为了多模态一次性重做整个聊天协议。
- 所有多模态结果必须带可观测元数据和降级原因。

## 5. 北极星架构

### 5.1 统一输入模型
- 新增统一内部模型，例如：
  - `MultimodalTurnInput`
  - `MultimodalMessagePart`
  - `DerivedMediaInsight`
- `AiMessagePart` 扩展为：
  - `text`
  - `image`
  - `audio`
  - `video`
  - `document`
  - `file`
  - `contact_card`
  - `location_card`
  - `note_card`
- 每个 part 既保留原始资产引用，也保留派生洞察字段。

### 5.2 统一输出模型
- 新增统一输出规划模型，例如：
  - `ReplyModalityPlan`
  - `ReplyArtifact`
  - `ReplyDeliveryEnvelope`
- 规划器先决定本轮要产出哪些模态：
  - `text`
  - `speech`
  - `image`
  - `digital_human_video`
  - `stream_player`
- 渲染器再按模态分别执行。

### 5.3 能力注册表
- 引入 `ModelCapabilityProfile` / `ProviderCapabilityProfile`
- 至少声明：
  - `supportsInputImage`
  - `supportsInputAudio`
  - `supportsInputVideo`
  - `supportsInputDocument`
  - `supportsOutputSpeech`
  - `supportsOutputImage`
  - `supportsOutputVideo`
  - `supportsResponsesApi`
  - `requiresPublicAssetUrl`
  - `maxInlineImageBytes`
  - `maxAudioBytes`
  - `maxVideoBytes`
- 不再用零散的正则和字符串猜测能力。

## 6. 多模态理解方案

### 6.1 图片理解
- 保留现有图片直通能力。
- 增补三个兜底能力：
  - 本地或内网 URL 时自动转 data URL / base64
  - 图片 OCR
  - 基础图像摘要缓存
- 当 provider 拒绝原生图像输入时：
  - 自动退回“结构化图像摘要 + OCR 文本”
  - 不直接退回 `[图片] xxx.jpg`

### 6.2 语音理解
- 分两层：
  - `ASR 文本`
  - `音频元数据`
- 首期不强制追求 provider 原生 audio input，但要补：
  - 说话时长
  - 录音 MIME
  - 分段结果
  - 可选说话风格标签
- 语音消息进入聊天主链路后：
  - 不只是 `transcriptText`
  - 还要携带 `voiceMeta`

### 6.3 视频理解
- 不能继续只靠音轨转写。
- 推荐做服务端 `Video Understanding Bundle`：
  - 抽帧
  - OCR
  - 音轨转写
  - 镜头切换摘要
  - 时长 / 分辨率 / 宽高比
- 若 provider 支持原生视频输入：
  - 可直接送原视频
  - 但仍同时保留 `Video Understanding Bundle` 作为 fallback 和可观测物
- 若 provider 不支持原生视频输入：
  - 用抽帧图片 + 转写文本 + OCR 摘要组合喂模型

### 6.4 文档理解
- 为 `pdf/docx/txt/image-pdf` 补抽取链路：
  - 纯文本抽取
  - OCR
  - 标题与摘要
- 文档 part 首期不要求 provider 原生 file input
- 先把正文片段、标题、页数、摘要以结构化文本进入模型

### 6.5 群聊理解
- 群聊要保留发言者标签，但不能破坏多模态 part。
- 当前 `GroupReplyOrchestrator` 已传 `userMessageParts`
- 后续要升级为：
  - 历史消息也保留多模态 part
  - 群聊 planner 可根据图片/视频内容决定谁更适合回复

## 7. 多模态回复方案

### 7.1 引入“回复规划器”
- 在 `AiOrchestratorService` 之外增加 `Reply Modality Planner`
- 先判断：
  - 当前 surface
  - 用户输入模态
  - 角色偏好
  - provider 能力
  - 预算与并发
- 再输出本轮计划，例如：
  - 文本 only
  - 文本 + 语音
  - 文本 + 图片
  - 文本 + 语音 + 数字人视频

### 7.2 文本 + 语音统一回到普通聊天链路
- 现在语音回复只在通话专用接口里真实存在。
- 需要把语音渲染能力从 `voice-calls.service.ts` 抽出来，变成聊天通用能力。
- 普通聊天建议支持两种策略：
  - 被语音消息触发时自动回 `text + voice`
  - 角色配置为“偏语音型”时主动回 `text + voice`

### 7.3 图片回复
- 新增图片生成编排器，而不是直接把画图逻辑塞进聊天服务。
- 适用场景：
  - 用户明确要求“画图”
  - 角色设定本身会返回草图、卡片、示意图
- 输出建议：
  - 先回一条文本说明
  - 再发一条图片消息
  - 两条消息共享 `responseGroupId`
- 首期不建议把图片二进制直接塞回同一条文本消息。

### 7.4 视频数字人回复
- 数字人视频必须从“会话外壳”升级成真实渲染链路。
- 新增持久化 session / render job，而不是只用内存 Map。
- 推荐拆成：
  - `DigitalHumanSessionEntity`
  - `DigitalHumanRenderJobEntity`
- 生成流程：
  - 文本脚本
  - TTS 音频
  - provider render
  - player / stream 回调
- 不再把数字人 provider 仅当作 mock iframe。

### 7.5 多模态回复的消息落库策略
- 当前 `MessageEntity` 只有单 attachment。
- 首期推荐兼容方案：
  - 一轮回复允许生成多条 sibling message
  - 每条共享 `responseGroupId`
  - 文本、语音、图片、视频分别各自落消息
- 这样可以避免一次性改前端所有消息渲染结构。
- 第二阶段再评估是否需要“单消息多附件数组”。

## 9. 模型与 Provider 路由

### 9.1 当前问题
- 现在系统是“一个主推理模型 + 转写模型 + TTS 模型”
- 这不足以支撑真实多模态

### 9.2 目标路由
- 至少拆成四类 provider：
  - `reasoning/text`
  - `speech/transcription`
  - `speech/synthesis`
  - `image/video generation`
- 对同一轮请求允许做跨 provider 编排。

### 9.3 角色级策略
- 角色配置里增加：
  - 默认回复模态偏好
  - 是否允许图片回复
  - 是否允许语音回复
  - 是否允许数字人视频回复
  - 优先使用的模型路由标签
- 不是所有角色都默认开启视频/图片输出。

### 9.4 Surface 级策略
- 普通单聊：
  - 默认 `text`
  - 可升级 `text + voice`
  - 条件触发 `text + image`
- 群聊：
  - 默认 `text`
  - 谨慎开启 `voice`
  - 首期不自动发视频
- 语音通话：
  - 默认 `text + speech`
- 数字人视频通话：
  - 默认 `text + speech + digital_human_video`

## 10. 前端规划

### 10.1 普通聊天页
- 继续复用现有消息列表
- 先支持：
  - 文本气泡
  - 语音气泡
  - 图片气泡
  - 视频/播放器卡片
- 增加对 `responseGroupId` 的弱聚合：
  - 同一轮的文本 + 语音 + 图片在视觉上靠近展示

### 10.2 发送入口
- 图片、音频、视频、文件上传后不只做 UI 预览
- 前端要显示：
  - 分析中
  - 已提取摘要
  - 发送后模型已读图/已读视频
- 避免用户以为系统还在只看文件名

### 10.3 语音与视频通话页
- 语音通话页复用通用 `speech renderer`
- 视频通话页复用通用 `digital human session`
- 不再让视频页把语音页逻辑拷一份后套舞台壳

### 10.4 管理后台
- 新增多模态能力诊断：
  - 当前模型是否支持图像输入
  - 当前转写 provider 是否可用
  - 当前图片生成 provider 是否可用
  - 当前数字人 provider 是否真实接通还是 mock
- 新增多模态回合预览页：
  - 输入一个图片/语音/视频
  - 查看派生洞察
  - 查看规划出的回复模态

## 12. 实施分期

### Phase 0：基线收口
- 修复配置口径不一致：
  - `/api/config/ai-model` 与真实 provider_model 对齐
- 把 `ModelCapabilityProfile` 落地
- 在 `/api/system/status` 暴露真实能力而不是只暴露模型名
- 结果：
  - 系统先能准确说清自己到底具备什么能力

### Phase 1：统一多模态输入模型
- 扩展 `AiMessagePart`
- 单聊 / 群聊历史回放都保留多模态 parts
- 图片 fallback 增加 OCR / 摘要
- 结果：
  - 图片理解不再只依赖 provider 原生路径

### Phase 2：音频 / 视频 / 文档理解
- 建立 `MediaUnderstandingCache`
- 打通：
  - 音频转写 + 元数据
  - 视频抽帧 + OCR + 音轨
  - 文档抽取 + OCR
- 结果：
  - 真正形成“系统级多模态理解”

### Phase 3：回复规划器
- 新增 `Reply Modality Planner`
- 让系统先规划模态，再执行
- 结果：
  - 不再所有请求都直接退回文本

### Phase 4：普通聊天语音回复
- 把现有 TTS 从通话专用链路抽成聊天通用能力
- 支持文本 + 语音 sibling messages
- 结果：
  - 普通聊天首次具备真实多模态回复

### Phase 5：普通聊天图片回复
- 接入图片生成 provider
- 支持角色按意图回图
- 结果：
  - 普通聊天首次具备视觉产出能力

### Phase 6：真实数字人视频回复
- 引入持久化 session / render job
- 接外部数字人 provider
- 支持真实 player / stream 回调
- 结果：
  - 视频通话从 mock 舞台升级为真实数字人链路

### Phase 7：前端与后台收口
- 聊天页多模态组合展示
- 后台诊断页与预览页
- 结果：
  - 多模态不再只在底层存在，产品可见、可调、可诊断

## 14. 风险与依赖
- 转写 provider 429 会直接拖垮音频/视频理解链路，需要独立限流与重试队列。
- 图片和视频输入若依赖公网 URL，必须补 signed URL / data URL fallback。
- 视频生成与数字人渲染延迟高，不能塞回同步聊天主请求，需要 job 化。
- 普通聊天如果默认全量开启语音/图片输出，会带来成本爆炸，需要角色级和 surface 级开关。
- 数字人能力一旦落持久化实体，实施当日必须更新 `AGENTS.md` 里的结构说明。

## 15. 验收标准
- 用户发一张图，角色能稳定引用图中真实元素，即使 provider 原生读图失败也能退回结构化视觉摘要理解。
- 用户发一段视频，角色能同时引用画面内容和音轨内容，而不是只复述转写文本。
- 用户发一份文档，角色能引用正文摘要，而不是只回复文件名。
- 普通单聊里，角色能在符合规则时真实回复语音。
- 普通单聊里，角色能在符合规则时真实回复图片。
- 数字人视频通话页接入真实 render/player 结果，不再只是 mock iframe 舞台。
- 后台能明确看见每轮输入模态、输出模态、fallback 原因与 provider 选择。
