# Web 端桌面版通讯录好友朋友圈入口对齐微信电脑端规划

## 任务目标
- 把 `apps/app` 里桌面版通讯录右侧好友详情中的“朋友圈”从占位文案，改成微信电脑端那种真实可进入的资料入口。
- 对齐微信电脑端已经形成的使用路径：`通讯录好友资料 -> 朋友圈入口行 -> TA 的朋友圈时间线 / 相册视角`。
- 保持当前项目既有桌面工作区架构，不新起一套页面，不把简单入口做成大改路由重构。

## 对标基线

### 本地代码基线
- 通讯录右侧好友详情：`apps/app/src/features/contacts/contact-detail-pane.tsx`
- 通讯录桌面页容器：`apps/app/src/routes/contacts-page.tsx`
- 桌面朋友圈路由：`apps/app/src/routes/moments-page.tsx`
- 桌面朋友圈工作区：`apps/app/src/features/desktop/moments/desktop-moments-workspace.tsx`
- 桌面朋友圈侧栏：`apps/app/src/features/desktop/moments/desktop-moments-sidebar.tsx`
- 联系人资料行组件：`apps/app/src/features/contacts/desktop-contact-profile-blocks.tsx`
- 完整角色资料页：`apps/app/src/routes/character-detail-page.tsx`

### 微信电脑端近期开口径
- 2025-03-21，IT 之家转引微信官网上架的 `4.0.3` 测试版信息，提到电脑端支持：
  - 发表朋友圈
  - 查看朋友的朋友圈相册
  - 朋友圈置顶
- 2025-04-01，IT 之家再次跟进 `4.0.3` 正式版，明确描述了电脑端好友资料详情页的行为：
  - 资料页中间新增“朋友圈”板块
  - 鼠标移入后右侧出现 `>`，点击后进入该好友的朋友圈界面
- 2026-03-19，微信派官宣电脑端 `4.1.8` 更新语音输入、一键撤回、视频倍速。这个信息和朋友圈入口不是同一功能，但它说明微信电脑端已明确走 PC 主端路径，不再只是手机陪衬。

## 当前真实现状

### 已经有的能力
- 桌面端已经有独立“朋友圈”一级入口，路径是 `/tabs/moments`。
- `MomentsPage` 已经区分移动端和桌面端，桌面端走 `DesktopMomentsWorkspace`。
- `DesktopMomentsWorkspace` 内部已经有“按作者聚焦”的状态：
  - `activeAuthorId`
  - `authorSummaries`
  - `focusAuthor(authorId)`
- 朋友圈数据已经能支撑“查看某个作者的时间线”：
  - `Moment.authorId`
  - `Moment.authorName`
  - `Moment.authorAvatar`
- 通讯录详情页已经有“内容与关系”分区，并且“更多资料”是可点击动作行。

### 当前缺口
- 通讯录详情页里的“朋友圈”仍是纯展示文案，不可点击。
- `MomentsPage` 的路由 hash 目前只支持 `moment=<id>`，不支持 `authorId=<id>` 这种“按好友进入”的状态。
- `DesktopMomentsWorkspace` 的 `activeAuthorId` 只存在于组件本地 state，不接受外部路由驱动。
- 完整资料页 `character-detail-page.tsx` 里同样还是“朋友圈 / 后续接入”。
- 聊天页右侧信息面板还没有同步提供“朋友圈”入口，所以现在是三处资料视图口径不一致：
  - 通讯录右侧资料
  - 完整资料页
  - 聊天信息侧栏

## 结论判断

### 这次不应该做成“新功能研发”
- 这不是从 0 到 1 做朋友圈。
- 朋友圈主工作区、作者筛选、动态列表都已经存在。
- 真正缺的是：
  - 入口接线
  - 路由状态补齐
  - 工作区接受“来自联系人资料页的上下文”

### 最合适的改法
- 采用微信电脑端同类路径：
  - 在联系人资料页里把“朋友圈”做成动作行
  - 点击后进入现有桌面朋友圈工作区
  - 进入后直接聚焦该好友作者视角
- 不新增 `/contacts/:id/moments` 之类的新页面。
- 不把联系人详情页直接嵌一个朋友圈预览列表。
- 不在第一阶段做“相册网格”或“独立作者页”；先用现有工作区筛选态完成 80% 的微信感。

## 目标交互

### 微信式目标路径
1. 用户在桌面版通讯录选择一个已添加好友。
2. 右侧资料卡的“内容与关系”分区里看到 `朋友圈` 动作行。
3. 该行默认表现和“更多资料”一致：
  - hover 高亮
  - 右侧箭头
  - 点击进入
4. 点击后跳到 `/tabs/moments`。
5. 朋友圈工作区启动后，直接切到这个好友的作者聚焦态：
  - 主列只展示该好友的朋友圈
  - 右栏切到该作者摘要视图
  - 顶部计数文案显示 `好友名 · N 条`
6. 若该好友当前没有朋友圈：
  - 仍然允许进入
  - 工作区显示“TA 还没有发表朋友圈”的作者级空态
  - 不应回退到全量列表，避免用户误以为点错人

### 与微信电脑端一致的交互细节
- 联系人资料页里的“朋友圈”应该是动作行，不是静态字段行。
- 没有必要在入口行上直接展示“后续接入”“暂未开放”这类产品开发口径。
- 如果后续补悬浮反馈，建议只在 hover 时弱显右箭头，不做过强按钮感。
- 若好友不是通讯录联系人，只是“世界角色”，则先不开放这一入口，避免和微信的“好友资料页”语义错位。

## 方案设计

### 一、路由状态设计

#### 目标
- 让 `/tabs/moments` 支持“从外部带入作者上下文”。

#### 建议方案
- 为朋友圈新增独立 route-state 工具文件，而不是继续把 parse / build 函数塞在 `moments-page.tsx` 底部。
- 建议新增：
  - `apps/app/src/features/desktop/moments/desktop-moments-route-state.ts`

#### 状态结构
```ts
export type DesktopMomentsRouteState = {
  momentId?: string;
  authorId?: string;
};
```

#### 解析规则
- 兼容现有 `moment=<id>` 深链。
- 新增 `authorId=<id>`。
- 两者同时存在时：
  - 优先保证 `authorId` 的作者聚焦
  - 若 `momentId` 属于该作者，则同时选中该条动态
  - 若 `momentId` 不属于该作者，则丢弃 `momentId`

#### 为什么不用 search params
- 当前项目桌面工作区大量使用 hash 维护页内状态。
- 同一模式已经存在于：
  - 搜索
  - 添加朋友
  - 聊天文件
  - 图片查看器
- 这次继续用 hash，最符合现有工程习惯，也最小改。

### 二、通讯录详情改造

#### 当前问题
- `ContactDetailPane` 里“朋友圈”使用的是 `DesktopContactProfileRow`，没有点击能力。

#### 目标改法
- 改成 `DesktopContactProfileActionRow`。
- 新增可选回调：
  - `onOpenMoments?: () => void`

#### 桌面通讯录页责任
- `contacts-page.tsx` 负责知道当前选中的好友是谁，因此跳转逻辑应由页面容器实现。
- 建议新增：
  - `handleOpenSelectedFriendMoments()`
- 行为：
  - `navigate({ to: "/tabs/moments", hash: buildDesktopMomentsRouteHash({ authorId: selectedFriendItem.character.id }) })`

#### 开放条件
- 仅 `selectedFriendItem` 存在时显示可点击入口。
- 若是非好友角色：
  - 仍可保留一行，但显示成弱化态“加为好友后可查看”
  - 或者直接不展示
- 第一阶段建议：
  - 桌面通讯录右侧资料仅对好友显示可点击“朋友圈”
  - 对世界角色不显示该项，避免制造伪入口

### 三、朋友圈工作区接收外部上下文

#### 当前问题
- `DesktopMomentsWorkspace` 的 `activeAuthorId` 完全是内部 state。
- 路由切到 `/tabs/moments#authorId=xxx` 后，工作区不会自动聚焦作者。

#### 目标改法
- 为 `DesktopMomentsWorkspace` 增加受路由驱动的初始 / 同步参数：
  - `routeSelectedAuthorId?: string | null`

#### 同步规则
- `routeSelectedAuthorId` 变化时：
  - 若该作者存在于 `authorSummaries`，则设置 `activeAuthorId`
  - 若不存在，但 route 里明确带了 `authorId`，则进入“指定作者空态”
- 手动在工作区中切换作者、清空作者时：
  - 反向更新 hash
  - 保证刷新、复制地址、浏览器回退时状态不丢

#### 为什么要双向同步
- 微信电脑端虽然不是浏览器，但其行为是“上下文切过去以后仍然稳定停在这个朋友的朋友圈”。
- Web 端如果只做一次性跳转，不更新路由：
  - 刷新会丢状态
  - 返回会混乱
  - 分享当前地址也不可复现

### 四、作者级空态

#### 当前问题
- `DesktopMomentsWorkspace` 里只有全局空态，没有明确的“指定作者但暂无内容”空态。

#### 需要新增的空态分支
- `routeSelectedAuthorId` 存在，但 `authorSummaries` 里找不到该作者时：
  - 顶部仍显示作者名占位或“该好友”
  - 主区显示作者级空态
  - 行动按钮可提供：
    - `查看全部朋友圈`
    - 若作者是世界主人自己，可显示 `发朋友圈`

#### 文案建议
- 好友无朋友圈：
  - `TA 还没有发表朋友圈`
  - `后续有新动态时，会直接出现在这里。`
- 作者不可见或已被过滤：
  - `当前无法查看这个联系人的朋友圈`
  - 仅当真的出现屏蔽 / 数据异常时才走这条分支

### 五、完整资料页与聊天信息侧栏的统一策略

#### 这次先不一起做完，但要预留同一路径
- `character-detail-page.tsx` 目前也写着“朋友圈 / 后续接入”。
- 后续如果继续对齐微信电脑端，应该统一成同一个动作：
  - 去 `/tabs/moments#authorId=xxx`

#### 第二波建议同步的入口
- 完整好友资料页
- 单聊右侧聊天信息面板
- 群聊成员资料弹出层

#### 原则
- 不要三处各自发明一套“看 TA 的朋友圈”逻辑。
- 统一跳同一路由，统一用同一个 route-state。

## 分阶段实施

### Phase 1：最小可用接线
目标：把桌面通讯录好友详情里的“朋友圈”真正接通。

前端改动：
- 新增 `desktop-moments-route-state.ts`
- `moments-page.tsx` 接入 `authorId` 路由状态解析
- `DesktopMomentsWorkspace` 接受并同步 `routeSelectedAuthorId`
- `contact-detail-pane.tsx` 把“朋友圈”从静态行改成动作行
- `contacts-page.tsx` 增加跳转逻辑

交互结果：
- 可从好友详情进入 TA 的朋友圈
- 可复制当前地址并还原同一作者视图
- 浏览器回退可回到通讯录

这一阶段不做：
- 相册网格
- 朋友圈预览缩略
- 聊天信息侧栏同步入口
- 移动端资料页改造

### Phase 2：作者视角细化
目标：让“看 TA 的朋友圈”更像微信，而不是只是筛选列表。

可补能力：
- 作者头部摘要更明显
- 作者级空态
- 右栏增加“返回全部朋友圈”
- 顶部面包屑或返回按钮
- 进入作者视图时自动滚到顶部

这一阶段仍然不做后端改表。

### Phase 3：统一所有好友资料入口
目标：把通讯录、完整资料页、聊天侧栏的朋友圈入口收口成一套。

改动范围：
- `character-detail-page.tsx`
- 聊天页右侧信息面板相关组件
- 可能涉及头像弹出卡片

结果要求：
- 只要是“看某个好友资料”的地方，进入朋友圈都跳同一路径
- 不再保留“后续接入”残留文案

### Phase 4：再决定是否补微信完整版相册能力
目标：从“按作者筛选的时间线”升级到更接近微信的朋友相册。

这阶段才考虑：
- 作者相册网格
- 置顶朋友圈
- 多图 / 视频
- 评论表情 / 图片

这已经超出这次入口接线任务，不要混进首轮实现。

## 验收标准
- 在桌面版通讯录中选择一个好友，右侧资料区能点击“朋友圈”进入该好友朋友圈。
- 进入后主区不是全量朋友圈，而是该好友的作者聚焦态。
- 刷新页面后仍停留在该好友朋友圈视图。
- 浏览器回退返回通讯录时，不出现明显状态错乱。
- 该好友没有朋友圈时，出现作者级空态，而不是误展示全量列表。
- 现有 `/tabs/moments#moment=...` 深链不回归。
- 不新增测试文件，首轮通过手动联调验收。

## 风险与注意点
- 最大风险不是跳转本身，而是 route-state 和工作区内部 state 打架。
- 若只把 `authorId` 当初始值，不做双向同步，后面会出现刷新丢状态的问题。
- 若作者级空态不补，用户点进好友朋友圈但看到全量列表，会误判为“入口没生效”。
- 当前联系人详情和完整资料页都存在“朋友圈 / 后续接入”，实施时要避免只改一处导致口径继续分裂。

## 建议执行顺序
1. 先补朋友圈 route-state，确保工作区可以被 `authorId` 驱动。
2. 再接通通讯录右侧好友详情里的“朋友圈”动作行。
3. 补作者级空态和 hash 双向同步。
4. 最后收掉完整资料页里的“后续接入”文案，并评估是否同步到聊天侧栏。
