# 多模态未完成能力真实实现收口规划

## 1. 本规划的判断口径

这份规划只解决“还没有彻底真实实现”的多模态能力，不重复规划已经真实可用的文本与图片输入基础链路。

“彻底真实实现”必须同时满足以下条件：

- 有正式 provider 配置，不依赖 mock、不依赖主推理 endpoint 偶然兼容。
- 有真实运行时调用，不能只靠字段、按钮、提示词或静态状态声明。
- 单聊和群聊都能进入同一套理解与回复语义。
- 输入资产、派生洞察、回复工件都有持久化状态和失败原因。
- 前台可见、后台可诊断、失败可重试、清空/删除/撤回可取消。
- 有最小端到端 smoke，能证明 provider 真实返回了文本、音频、图片、视频或结构化洞察。

## 2. 当前真实基线

基于 2026-04-23 的代码审计和运行时诊断：

- `text`：真实可用，诊断 `ok / real=true`，当前模型 `gpt-4.1`。
- `image_input`：真实可用，诊断 `ok / real=true`，图片能进入 `image_url / input_image`。
- `transcription`：当前实例不可用，缺独立语音转写 endpoint / API Key / model。
- `tts`：当前实例不可用，缺独立 TTS endpoint / API Key，虽然存在默认 TTS model 字段。
- `image_generation`：当前实例不可用，缺独立图片生成 endpoint / API Key / model。
- `digital_human`：当前是 `mock_digital_human`，不算真实数字人视频能力。
- `video_understanding`：代码口径仍为 `supportsNativeVideoInput=false`，当前只做音轨转写，不做帧级视觉理解。
- `document_understanding`：文本类文档可本地抽取；PDF / DOCX / 扫描件没有完整真实解析和 OCR 兜底。

## 3. 最终目标形态

### 3.1 角色多模态理解

角色必须真实理解以下输入：

- `text`：已具备，继续保持。
- `image`：已具备，补端到端回归和后台可观测。
- `audio`：上传语音或音频文件后，必须真实转写并把转写内容写入本轮上下文。
- `video`：上传视频后，必须同时理解音轨和画面，至少包含转写、关键帧摘要、OCR、场景摘要。
- `document`：上传文本、PDF、DOCX、扫描 PDF 后，必须抽取可引用正文或 OCR 内容。
- `structured_card`：名片、位置、笔记、表情包继续作为结构化文本上下文进入模型。

### 3.2 角色多模态回复

角色必须真实产出以下回复：

- `text`：已具备。
- `voice`：用户要求语音或语音场景触发时，生成真实可播放音频附件。
- `image`：用户要求发图/画图时，生成真实图片附件。
- `digital_human_video`：数字人通话中，真实外部 provider 返回可播放视频/流，不再把 mock stage 算作完成。

可后续扩展但不是当前首要 blocker：

- `document_artifact`：角色生成文件、笔记、卡片型附件。
- `video_generation`：角色直接生成短视频。当前先完成数字人视频，而不是泛视频生成。

## 4. 总体架构收口

### 4.1 Provider 能力分层

保持 `InferenceProviderAccount` 的多通道思路，但继续收口为明确能力通道：

- 主推理：`endpoint / apiKey / defaultModelId / apiStyle`
- 图片输入：由主推理模型能力和 smoke 共同证明。
- 语音转写：`transcriptionEndpoint / transcriptionApiKey / transcriptionModel`
- TTS：`ttsEndpoint / ttsApiKey / ttsModel / ttsVoice`
- 图片生成：`imageGenerationEndpoint / imageGenerationApiKey / imageGenerationModel`
- 数字人：新增独立 provider 配置，不再混在 mock adapter 中。

新增或强化规则：

- `supportsTranscription` 只由转写三件套加 smoke 成功证明。
- `supportsSpeechSynthesis` 只由 TTS 三件套加 smoke 成功证明。
- `supportsImageGeneration` 只由图片生成三件套加 smoke 成功证明。
- `supportsNativeVideoInput` 不能靠模型名启发式默认打开，必须由 provider catalog 或 smoke 证明。
- 系统状态不能再把 mock 或 fallback 当作 ready。

### 4.2 多模态输入流水线

统一所有附件理解为 `MediaInsightJob` 流水线：

- 原始附件先进入消息表。
- 每个附件按类型产生一个或多个 insight job。
- job 成功后把结果回写到附件 payload。
- AI 回复链路使用“原始附件 + 洞察结果 + provider 原生 payload”三层上下文。
- job 失败不阻塞基础消息发送，但必须在后台和前端显示失败原因。

### 4.3 多模态回复流水线

统一所有非文本回复为 `ReplyArtifactJob`：

- 文本回复先保存。
- 语音、图片、数字人视频作为独立 artifact job 生成。
- job 产物最终落为普通消息或数字人 turn artifact。
- 新消息、撤回、清空、删除必须取消仍未交付的 job。
- 单聊和群聊使用同一套 job 状态语义。

### 4.4 回复模态规划器

当前 regex planner 只能覆盖“语音回复”和“画图”关键词，后续需要升级为两层 planner：

- 规则层：低成本识别明确请求，比如“用语音回复”“画一张图”。
- LLM JSON planner：识别隐含多模态意图，输出结构化 `ReplyModalityPlan`。

计划结构：

```ts
type ReplyModalityPlan = {
  text: { enabled: true };
  voice?: { requested: boolean; reason: string; styleHint?: string };
  image?: { requested: boolean; prompt: string; reason: string };
  digitalHumanVideo?: { requested: boolean; reason: string };
  blocked?: Array<{ modality: string; reason: string }>;
};
```

planner 输出必须持久化到消息或 artifact job，方便后台复盘。

## 5. 分阶段执行计划

## 阶段 A：运行态能力证明与配置闭环

目标：先让当前实例能明确配置、诊断并证明每个 provider 通道。

执行项：

- 增加 `POST /api/admin/inference/diagnostics/run-all`，一次跑完所有诊断。
- 增加诊断历史表或系统配置快照，记录最近一次 `ok / unavailable / failed`、耗时、provider、model。
- 管理后台推理页增加“运行所有诊断”和最近诊断时间。
- `/api/system/status` 改为读取最近 smoke 结果，而不是只看字段是否存在。
- 增加 `GET /api/admin/multimodal/overview`，集中返回输入/输出能力矩阵。

验收：

- 未配置转写/TTS/出图时，系统明确显示不可用。
- 配置后必须真实调用 provider，并把结果标成 `real=true`。
- mock 数字人永远不能显示为真实可用。

## 阶段 B：语音理解真实实现

目标：用户语音消息、音频文件、视频音轨都能真实转写并进入角色上下文。

执行项：

- 在管理后台补完整转写 provider 配置向导和 smoke 上传。
- `MediaInsightJob` 对 `voice`、`audio/*`、`video/*` 都创建 `audio_transcription` 或 `video_transcription`。
- 转写结果写入 `attachment.transcriptText`，并显示 provider、耗时、错误码。
- 前端消息气泡展示“转写中 / 转写失败 / 转写内容”。
- 语音通话页如果 `speechReady=false`，禁止进入真实通话或给出明确配置提示。
- 为 `tryTranscribeMediaFromUrl` 增加明确 size、mime、下载失败、provider 失败分类。

验收：

- 上传一段真实语音后，角色回复必须引用语音内容。
- 上传音频文件后，角色回复必须引用转写内容。
- 转写失败时，消息仍发送成功，但 UI 和后台必须显示失败原因。

## 阶段 C：TTS 语音回复真实实现

目标：用户要求语音回复时，角色生成真实音频附件。

执行项：

- TTS provider 配置必须包含 endpoint/key/model/voice。
- `synthesizeSpeech` 只走 TTS 通道，不回退主推理 key。
- 单聊 `createAssistantReplyMessages` 保持文本 sibling + 语音 sibling。
- 群聊继续走 `ReplyArtifactJob`，失败可重试。
- 前端语音消息必须支持播放、时长、转写文本回看。
- 管理后台展示 voice artifact job 状态和失败原因。

验收：

- 输入“用语音回复我：今天适合做什么”后，返回文本消息和可播放语音消息。
- 断开 TTS key 后，不应生成假语音，后台显示失败。

## 阶段 D：图片生成回复真实实现

目标：用户要求发图/画图时，角色生成真实图片附件。

执行项：

- 图片生成 provider 独立配置和 smoke。
- 单聊图片回复由 `ReplyArtifactJob` 异步生成，前端显示“正在生成图片”状态。
- 群聊图片回复同样使用 job。
- 图片生成结果落入聊天附件存储，保存 mime、size、provider、revisedPrompt。
- 失败时补一条系统提示或在原消息旁显示失败状态，避免用户以为角色已发图。
- planner 支持更稳的图片意图识别，而不只靠少量 regex。

验收：

- 输入“帮我画一张杭州雨夜的头像”，最终必须出现一张真实图片消息。
- 清空会话或新消息覆盖旧轮后，未完成图片 job 不得继续补发。

## 阶段 E：视频理解真实实现

目标：视频不是“文件名 + 音轨转写”，而是真实理解画面。

执行项：

- 引入视频探针与处理器：
  - 使用 `ffprobe` 获取 duration、width、height、codec。
  - 使用 `ffmpeg` 抽取关键帧。
  - 默认抽取首帧、中间帧、尾帧；长视频按间隔抽帧。
- 扩展 `MediaInsightKind`：
  - `video_transcription`
  - `video_frame_summary`
  - `video_ocr`
  - `video_scene_summary`
- 扩展 attachment payload：
  - `durationMs`
  - `width`
  - `height`
  - `frameInsights`
  - `ocrText`
  - `sceneSummary`
- provider 支持原生 video input 时走原生。
- provider 不支持原生 video input 时走“抽帧图片 + 音轨转写 + OCR + 汇总摘要”。
- 前端视频消息支持播放、封面、理解状态。

验收：

- 上传一个无音轨但画面包含明显物体/文字的视频，角色能回答画面内容。
- 上传一个有音轨的视频，角色能同时引用画面和音轨。
- 当前没有视频 provider 时，至少通过抽帧 + 图片理解完成真实画面理解。

## 阶段 F：文档理解真实实现

目标：PDF/DOCX/扫描件都能被角色真实理解。

执行项：

- 本地解析器：
  - `txt/md/csv/json/xml/html` 继续走当前文本抽取。
  - `pdf` 使用 PDF 文本抽取库。
  - `docx` 使用 DOCX 文本抽取库。
- OCR 兜底：
  - 扫描 PDF 或图片型文档走 OCR provider。
  - OCR 结果写入 `document_ocr` insight。
- 文档分块：
  - 按页码/段落分块。
  - 限制进入 prompt 的长度。
  - 在 prompt 中保留文件名、页码、摘要。
- Responses provider 支持 `input_file` 时优先原生文件输入。
- 非 Responses provider 使用抽取文本摘要。

验收：

- 上传文本 PDF，角色能回答文档里的具体事实。
- 上传 DOCX，角色能总结内容。
- 上传扫描 PDF，OCR 成功后角色能引用识别内容。

## 阶段 G：真实数字人视频回复

目标：数字人视频不再是 mock iframe，而是真实 render provider。

执行项：

- 新增数字人 provider 配置：
  - `digitalHumanProviderKind`
  - `digitalHumanEndpoint`
  - `digitalHumanApiKey`
  - `digitalHumanAvatarId`
  - `digitalHumanCallbackToken`
  - `digitalHumanMode`
- 抽象 provider adapter：
  - `createSession`
  - `submitTurn`
  - `getRenderStatus`
  - `handleCallback`
  - `cancelTurn`
- 新增 render job：
  - `DigitalHumanRenderJob`
  - status: `pending / rendering / ready / failed / cancelled`
  - 保存 audio url、subtitle、video url、stream url、provider job id。
- 数字人通话流程：
  - 用户语音转写。
  - AI 文本回复。
  - TTS 生成音频。
  - 音频提交数字人 provider。
  - provider 返回视频/流。
  - 前端播放器展示真实视频/流。
- mock 只保留为 dev fallback，并在诊断中永远标 `real=false`。

验收：

- 后台诊断 `digital_human` 返回真实 provider `ok / real=true`。
- 发起视频通话后，至少一轮 AI 回复由真实 provider 返回视频或 stream。
- provider 失败时前端明确显示失败，不把海报 + 音频当成真实数字人。

## 阶段 H：统一回复模态 planner

目标：角色不是被硬编码关键词驱动，而是能根据用户意图和能力矩阵规划回复模态。

执行项：

- 新增 `ReplyModalityPlannerService`。
- 输入：
  - 用户消息文本。
  - 附件类型和洞察结果。
  - 当前 provider capability profile。
  - 场景：单聊、群聊、语音通话、视频通话。
- 输出：
  - `ReplyModalityPlan`。
  - `blocked` reason，例如“用户要图片，但图片生成 provider 未配置”。
- 先规则后 LLM：
  - 明确意图由规则快速判断。
  - 模糊意图交给小模型输出 JSON。
- 持久化 planner 结果到消息 metadata 或 artifact job metadata。
- 管理后台 reply logic 页展示 planner 输入、输出、被阻断原因。

验收：

- “你直接发语音跟我说”触发语音。
- “把这个视频里出现的招牌画成头像”需要等待视频 insight 后触发图片生成。
- 能力缺失时，角色不假装发图/发语音，而是自然说明当前做不到或给出文本替代。

## 7. 数据结构变更清单

需要新增或扩展：

- `InferenceProviderAccount`：数字人真实 provider 字段。
- `MediaInsightJob`：允许同一消息多个 insight kind。
- `MediaInsightJob.resultPayload`：统一结构化结果。
- `ReplyArtifactJob`：增加 digital human video 类型。
- `DigitalHumanRenderJob`：真实数字人渲染任务。
- `MessageAttachment`：
  - image: OCR / vision summary 可选字段。
  - file: page count / extracted chunks / OCR text。
  - voice: transcript status。
  - video: duration / dimensions / frame insights / OCR / transcript。
- `Message` 或 metadata：保存 `ReplyModalityPlan` 摘要。

结构变更落地后必须同步更新 `AGENTS.md`。

## 8. 主要风险

- Provider 能力差异大，OpenAI-compatible 并不代表支持 TTS、转写、图片生成或文件输入。
- 视频抽帧和 OCR 会引入 CPU/磁盘成本，需要 size、duration、frame count 限制。
- PDF/DOCX 解析库会增加依赖和安全面，需要限制文件大小和解析超时。
- 数字人 provider 的回调、鉴权、异步状态复杂，必须把 render job 设计成可恢复。
- 过度自动多模态回复可能打扰用户，planner 需要明确“用户请求优先，不主动滥发音频/图片”。
