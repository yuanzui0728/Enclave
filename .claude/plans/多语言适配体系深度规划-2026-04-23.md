# 多语言适配体系深度规划

日期：2026-04-23

## Summary

- 这次不能只做“前端接一个 i18n 库”，否则后续新功能还是会继续把中文或英文硬编码写进页面、接口错误、时间格式、语音语言和 AI 提示词里。
- 目标不是“把现有文案翻译一遍”，而是建立一条默认生效的多语言流水线：
  - 开发时默认用可提取消息写文案
  - 提交时自动提取新增消息
  - CI 自动阻止新的硬编码文案漏进仓库
  - 构建时自动编译语言包
  - 新功能只要按规范写，就会自动进入多语言体系
- 结合当前仓库结构，我建议：
  - 前端三套 React 应用统一采用 `Lingui`
  - 日期/数字/排序统一收口到 `Intl` 包装层
  - 后端错误返回从“直接吐中文 message”改成“`code + params + fallbackMessage`”
  - 运营配置和系统种子文案引入 `LocalizedText` 模式
  - AI 生成链路补上 `preferredLocale / contentLocale`
- 不建议现在把整个仓库强行切成单一英文 source locale。更实际的做法是把“源码语言”和“对外展示语言”拆开处理，并用错误码和共享格式化层把前后端粘起来。

## 1. 调研结论

## 1.1 仓库现状不是“未接 i18n”，而是“语言逻辑散落在全栈”

本地扫描结果：

- `apps/app/src` 含中文字符源码行数约 `7087`
- `apps/admin/src` 约 `6091`
- `apps/cloud-console/src` 约 `10`
- `api/src` 约 `8353`
- `apps/cloud-api/src` 约 `231`
- `packages/ui/src` 约 `18`

这说明问题分布在 5 层：

1. React 页面文案
2. React 组件属性文案
3. 工具函数里的时间/状态/错误描述
4. NestJS service/controller 的异常消息
5. AI 与语音相关默认语言

典型例子：

- `apps/app/src/lib/format.ts`
  - 直接写死了 `刚刚 / 昨天 / 星期一...`
  - `Intl.DateTimeFormat("zh-CN")` 被多处直接使用
- `apps/app/src/features/chat/use-speech-input.ts`
  - 浏览器识别默认语言固定为 `zh-CN`
- `packages/contracts/src/client.ts`
  - 请求错误会直接把后端 `message` 透传给前端
- `apps/admin/src/lib/admin-api.ts`
  - 管理后台继续直接把服务端 message 当成 UI 文案使用
- `api/src/modules/chat/chat.service.ts`
  - 直接抛 `只能撤回自己发送的消息。`
- `api/src/modules/real-world-sync/real-world-sync.constants.ts`
  - 默认 locale 固定 `zh-CN`
- `api/src/modules/inference/inference.service.ts`
  - 语音转写语言固定 `zh`

结论：

- 只在 React 层接 `t()` 不够
- 只翻译页面文案也不够
- 必须同时治理：
  - 文案声明方式
  - 格式化方式
  - API 错误协议
  - 结构化内容存储
  - AI 内容目标语言

## 1.2 三个前端入口都还没有语言 Provider

入口文件现状：

- `apps/app/src/main.tsx`
- `apps/admin/src/main.tsx`
- `apps/cloud-console/src/main.tsx`

三者目前都只挂了：

- `QueryClientProvider`
- `RouterProvider`

没有：

- 全局 i18n provider
- locale 检测
- `document.documentElement.lang/dir` 同步
- 语言包动态加载

## 1.3 当前世界主人模型里没有语言偏好字段

现状：

- `packages/contracts/src/world.ts`
- `api/src/modules/auth/user.entity.ts`
- `apps/app/src/store/world-owner-store.ts`

都还没有：

- `preferredLocale`
- `contentLocale`
- `autoDetectLocale`

这意味着：

- 用户切换语言无法持久化
- UI 语言和 AI 回复语言无法分开
- 后续云世界 / 本地世界 / 桌面壳之间也没有统一语言来源

## 1.4 云平台和主产品的“源码主语言”并不一致

主产品现状：

- `apps/app`
- `apps/admin`
- `api`

明显以中文为主。

云平台现状：

- `apps/cloud-console`
- `apps/cloud-api`

明显以英文为主，但夹杂少量中文错误提示。

这意味着：

- 不适合把整个 monorepo 暴力统一成单一 source locale
- 更适合：
  - UI 统一一套运行时
  - catalog 可以按应用分域
  - API 层尽量改成错误码协议，避免源码语言影响客户端展示语言

## 1.5 现有多语言相关“事实基础”是有的，但没有被产品化

已存在的基础：

- README 已有 `README.en.md / README.ja.md / README.ko.md`
- 部分契约/实体已有 `locale` 字段
- 项目是现代浏览器 + React 19 + Vite + TypeScript 5.9
- Web / Desktop / Android / iOS 共用同一套 `apps/app`

这意味着：

- 前端运行时统一做 i18n 的收益极高
- `Intl.DateTimeFormat / Intl.NumberFormat / Intl.Locale` 可以直接作为格式化底座
- Web 侧语言切换一旦做对，四端业务层基本一起受益

## 2. 官方资料对比后的技术选型建议

我查了官方文档，重点看了 4 类能力：

- React + Vite 接入成本
- 消息自动提取能力
- 类型与 lint 约束能力
- 按语言拆包/动态加载能力

## 2.1 推荐：前端使用 `Lingui`

推荐原因：

1. 非常适合当前仓库这种“用户文案大量直接写在 TSX/TS 里”的项目。
2. 官方提供：
   - CLI：提取、编译、校验 catalog
   - Vite plugin：按语言动态加载、构建集成
   - ESLint plugin：直接阻止未本地化字符串
3. 宏写法对现有 React 代码侵入较低：
   - `<Trans>...</Trans>`
   - `t\`...\``
   - `defineMessage / msg`
4. comment/context 对翻译和 AI 辅助翻译都很关键，适合你后续让模型自动草拟翻译。
5. 官方明确支持 Vite 和动态 import，比较贴合这个仓库。

为什么不是第一选择：

- `react-intl / FormatJS`
  - 能做，也很成熟
  - 但当前仓库如果大规模改造，`formatMessage({ id, defaultMessage })` 这类描述符写法会更重
  - 官方文档还特别提示要处理其 ESM/transpile 要求，维护成本更高一些
- `i18next`
  - 生态最强，也有新的官方 CLI 和 selector API
  - 但它更适合“key 驱动 / JSON 资源驱动”的团队
  - 你现在最大的痛点是海量现有文案和未来新文案自动进入流水线，Lingui 的 message extraction 模式更顺手

结论：

- React 客户端首选 `Lingui`
- `i18next` 作为备选方案，不是当前最优
- `FormatJS` 适合作为 `Intl`/polyfill/标准参考，不建议当这次主框架

## 2.2 日期、数字、相对时间统一基于 `Intl`

不要再自己手写：

- `昨天`
- `星期一`
- `3月4日 12:30`
- 金额/数字格式

统一原则：

- 文案走 catalog
- 数据格式走 `Intl`
- 任何 `"zh-CN"`、`"en-US"` 字面量都不应在业务页面零散出现

## 2.3 后端不建议继续直接产出最终展示文案

后端多语言最容易踩的坑是：

- service 里直接 `throw new BadRequestException("中文")`
- 前端直接展示 `error.message`

这种模式会导致：

- 多客户端很难统一
- 云平台英文、主产品中文的源码现实会互相污染
- 后续新增接口天然会继续漏翻

因此推荐：

- 协议层以 `errorCode + params + fallbackMessage` 为主
- 客户端按当前 locale 自己翻译
- 只有必须由服务端直接生成给用户看的文本，才在服务端做真正 i18n

## 3. 目标口径

## 3.1 这次多语言要覆盖的不是一个层，而是四个层

### A. UI 壳层

- 页面标题
- 按钮
- 标签
- 空态
- Toast / Notice
- 表单说明
- aria-label

### B. 协议错误层

- API 错误
- 权限失败
- 文件上传失败
- 云平台认证失败

### C. 结构化业务内容层

- 后台运营可配置文案
- 默认系统种子文案
- 固定入口卡片说明
- 游戏目录等平台内容

### D. AI 与语音层

- 语音识别默认语言
- TTS / STT 语言标签
- AI Prompt 目标语言
- 内置角色/真实世界同步的 locale 默认值

## 3.2 不在第一阶段做的事

第一阶段不做：

- 路由 path 本身多语言化
- 用户生成内容自动翻译入库
- 全部历史数据库内容一次性补齐多语言
- 所有外部内容源的机器翻译展示

原因：

- 这些都不是“让后续功能自动进入多语言流水线”的必要条件
- 先做会显著扩大改造面，而且收益不如先把框架立住

## 3.3 第一批支持语言建议

建议先支持：

- `zh-CN`
- `en-US`
- `ja-JP`
- `ko-KR`

原因：

- 你仓库里已有中、英、日、韩 README 基础
- 这 4 个语言包足够验证：
  - CJK 排版
  - 英文长度膨胀
  - 日期/数字/排序差异

## 4. 总体架构设计

## 4.1 新增 `packages/i18n`

建议新增共享包：

- `packages/i18n`

职责：

- locale 常量与能力声明
- 语言检测与 fallback
- `Lingui` 初始化
- 语言包加载器
- 通用格式化器
- 错误码文案映射
- 伪本地化和审计脚本入口

建议结构：

- `packages/i18n/src/locales.ts`
- `packages/i18n/src/runtime/`
- `packages/i18n/src/format/`
- `packages/i18n/src/errors/`
- `packages/i18n/src/provider/`
- `packages/i18n/catalogs/shared/{locale}.po`
- `packages/i18n/catalogs/app/{locale}.po`
- `packages/i18n/catalogs/admin/{locale}.po`
- `packages/i18n/catalogs/cloud-console/{locale}.po`
- `packages/i18n/catalogs/api-errors/{locale}.po`

为什么 catalog 粒度按“应用”而不是“功能目录”：

- 这样新功能只要落在现有应用目录里，就会被自动提取
- 不需要每加一个模块都去手动改 catalog 配置
- 先保证自动接入，再视包体量决定是否二次拆分

## 4.2 `packages/ui` 尽量不拥有最终产品文案

`packages/ui` 目前少量组件自己带文案。

建议原则：

- UI 包只提供视觉壳和基础交互
- 最终产品文案尽量由调用方传入
- 共享组件内部如果必须保留文案，放到 `shared` catalog

这样可以避免：

- 中文主产品和英文云平台都依赖同一份“源语言 UI 组件文案”

## 4.3 每个前端入口统一挂 `I18nProvider`

入口改造点：

- `apps/app/src/main.tsx`
- `apps/admin/src/main.tsx`
- `apps/cloud-console/src/main.tsx`

统一增加：

- `I18nProvider`
- `loadLocaleCatalogs(locale)`
- `document.documentElement.lang = locale`
- `document.documentElement.dir = resolveTextDirection(locale)`

## 4.4 语言来源优先级

建议优先级：

1. 用户显式设置
2. 本地缓存
3. 浏览器 `navigator.languages`
4. 默认 locale

主 App 额外增加：

- 世界主人资料持久化 `preferredLocale`
- 可选 `contentLocale`

Admin / Cloud Console 可以先只存在本地存储，后续再决定是否做账号级持久化。

## 4.5 API 协议改成“错误码优先”

建议新增统一错误响应结构：

```ts
type ApiErrorResponse = {
  code: string;
  message?: string;
  params?: Record<string, string | number | boolean | null>;
  requestId?: string | null;
};
```

规则：

- `code` 是稳定协议
- `params` 用于插值
- `message` 只作为兼容 fallback，不再作为前端最终展示事实来源
- `requestId` 用于排障

例子：

```json
{
  "code": "chat.recall.only_own_message",
  "message": "只能撤回自己发送的消息。",
  "params": {},
  "requestId": "req_xxx"
}
```

前端收到后：

- 优先按 `code` 本地翻译
- 没命中再回退 `message`

这样做的收益：

- 主产品、云平台可以展示不同语言
- 后端源码语言不再绑死前端语言
- 新接口天然进入统一多语言协议

## 4.6 时间/数字/排序/相对时间全部走共享格式化层

新增统一格式化模块，不允许业务代码再直接写：

- `new Intl.DateTimeFormat("zh-CN")`
- `localeCompare(..., "zh-CN")`
- `星期一 / 刚刚 / 昨天`

建议抽象：

- `formatTimestamp`
- `formatConversationTimestamp`
- `formatRelativeTime`
- `formatNumber`
- `formatCurrency`
- `getCollator(locale)`
- `getSpeechRecognitionLocale(locale)`

落地后替换：

- `apps/app/src/lib/format.ts`
- 搜索排序
- 联系人排序
- 管理后台表格排序
- 云平台列表排序

## 4.7 结构化内容统一引入 `LocalizedText`

对后续新功能，凡是“运营可维护、系统预置、需要跨语言展示”的内容，不要再只存一份字符串。

建议共享类型：

```ts
type LocalizedText = {
  defaultLocale: string;
  values: Partial<Record<SupportedLocale, string>>;
};
```

适用场景：

- 游戏目录标题/简介
- 固定运营入口说明
- 后台模板文案
- 系统默认种子卡片

不建议第一阶段强推到所有现有表。

建议口径：

- 新功能从 day 1 就用 `LocalizedText`
- 老功能逐步迁移高价值字段

## 4.8 AI 与语音链路要引入“展示语言”和“内容语言”

新增建议：

- `preferredLocale`
  - UI 展示语言
- `contentLocale`
  - AI 回复 / 推荐 / 运营生成内容目标语言

原因：

- 有人希望 UI 英文，但角色仍说中文
- 也有人希望 UI 中文，但训练英语学习助手时内容语言是英文

需要改造的点：

- `apps/app/src/features/chat/use-speech-input.ts`
- `api/src/modules/inference/inference.service.ts`
- `api/src/modules/ai/ai-orchestrator.service.ts`
- `api/src/modules/system/system.service.ts`
- `api/src/modules/real-world-sync/*`
- `api/src/modules/need-discovery/*`
- `api/src/modules/action-runtime/*`

原则：

- UI copy 跟 `preferredLocale`
- AI 输出默认跟 `contentLocale`
- STT/TTS 默认语言从 `contentLocale` 推导

## 5. 自动化方案

## 5.1 新功能“自动进入多语言体系”的核心不是翻译自动，而是约束自动

必须承认：

- 新功能不可能无审核自动产出高质量多语言文案

但工程上可以做到：

- 新功能不会漏接入
- 新文案会自动被提取
- 新硬编码会被 CI 卡住
- 未翻译条目会自动暴露

## 5.2 具体流水线

### 开发时

- 开发者写：
  - `<Trans>...</Trans>`
  - `t\`...\``
  - `msg({...})`
- 不再手写 key JSON

### 提交前

- 自动执行 `lingui extract`
- 只要改了文案，catalog 自动更新

### CI

- `lingui extract --clean`
- `lingui compile`
- eslint 检查 `no-unlocalized-strings`
- 自定义 `i18n audit` 检查：
  - 新增业务硬编码字符串
  - 新增 `zh-CN` / `en-US` 业务直写
  - 新增 `throw new Error("中文/英文用户文案")`

### 发布前

- 缺失 source locale 禁止构建
- 目标语言可先允许缺失并回退 source locale
- release 分支可逐步收严为“核心语言缺失即失败”

## 5.3 必须补一层 repo 级审计脚本

只靠 ESLint 不够，因为：

- TS 工具函数
- Nest service
- contracts client
- 非 JSX 字符串

都可能绕过 UI 规则。

建议新增：

- `scripts/i18n-audit.mjs`

检查项：

- React/TS 中新出现的 CJK/英文用户可见硬编码
- 新出现的 `Intl.DateTimeFormat("zh-CN")`
- 新出现的 `localeCompare(..., "zh-CN")`
- 新出现的 `recognition.lang = "zh-CN"`
- 新出现的 `BadRequestException("...")`

策略：

- 第一阶段只做 report + allowlist
- 第二阶段开始“新增违规 fail CI”

## 5.4 让 AI 帮忙补翻译，但不是直接无审上线

推荐增加一个内部脚本：

- `scripts/i18n-draft-translate.mjs`

输入：

- 新提取的未翻译 message
- 文件路径
- comment/context

输出：

- `en-US / ja-JP / ko-KR` 草稿翻译

规则：

- 只能生成草稿
- PR 中 review 后入库
- 不允许覆盖已人工确认翻译

这样可以让“新增功能多语言适配”从人工搬运，变成“AI 草拟 + 人审确认”。

## 6. 迁移策略

## 6.1 不建议一口气改全仓

风险太大：

- 主 App 路由极多
- Admin 页面体量很大
- API 错误透传链路很长
- 还有 Web / Desktop / Mobile / Cloud 四条面

推荐分 5 个阶段。

## 6.2 Phase 0：立规范和骨架

范围：

- 新建 `packages/i18n`
- 接入 `Lingui` 基础配置
- 接入 `packages/tooling` 的 i18n lint 能力
- 新增 repo 级 `i18n-audit`
- 定义 locale 常量、错误协议、格式化层

文件：

- `packages/i18n/*`
- `packages/tooling/src/eslint-react.mjs`
- 根 `package.json`
- `tsconfig.base.json`

交付标准：

- 三个 React 应用都能切换 `zh-CN/en-US`
- 能动态加载语言包
- 能自动提取 catalog
- CI 能检测新增硬编码

## 6.3 Phase 1：先做 `packages/ui + apps/admin` 试点

为什么先选 admin：

- 文案量大，但 runtime 复杂度比主 App 低
- 能完整验证：
  - 表单
  - 统计卡
  - 空态
  - Toast
  - 错误展示
  - 表格排序

范围：

- `packages/ui`
- `apps/admin`
- `apps/admin/src/lib/admin-api.ts`

重点：

- 页面级文本迁移到 `Trans/t`
- 后端 message 先兼容 fallback
- 排序和格式化收口

## 6.4 Phase 2：主 App 先改“基础骨架 + 设置入口”

范围：

- `apps/app/src/main.tsx`
- `apps/app/src/store/world-owner-store.ts`
- `packages/contracts/src/world.ts`
- `api/src/modules/auth/user.entity.ts`
- `api/src/modules/auth/world-owner.service.ts`
- `profile/settings`
- `welcome-page`
- `splash-page`
- 公共格式化层

目标：

- 用户能切换并保存语言
- 主 App 基础壳和常用格式化不再写死中文

## 6.5 Phase 3：主 App 按业务域平推

建议顺序：

1. 聊天域
2. 通讯录域
3. 发现域
4. 公众号/视频号/游戏
5. 桌面工作区

每个域都做三件事：

- 文案迁移
- 错误展示收口
- 日期/排序/语音语言收口

## 6.6 Phase 4：API 错误码和云平台收口

范围：

- `api`
- `apps/cloud-api`
- `apps/cloud-console`
- `packages/contracts/src/client.ts`

目标：

- 前端不再直接依赖服务端自然语言 message
- 主产品和云平台都改成 `errorCode + params`
- 云平台补齐自己的 catalog

## 6.7 Phase 5：AI/运营/种子内容本地化

范围：

- 角色预设
- 真实世界同步默认 prompt
- Need Discovery / Followup / Reminder 默认提示
- 系统固定模板
- 运营配置型文案

目标：

- UI 多语言之外，内容层也能按 locale 输出
- 后续新增内容型功能直接使用 `LocalizedText`

## 7. 关键实现细节

## 7.1 两个语言概念必须拆开

- `preferredLocale`
  - UI 语言
- `contentLocale`
  - 生成内容语言

这两个字段不要混成一个。

## 7.2 路由 path 不本地化

当前仓库：

- Web / Desktop / Mobile 共用大量深链
- 桌面与移动有很多兼容跳转
- 云平台也有不少 scoped permalink

所以：

- path 保持稳定
- 只本地化页面标题、面包屑、菜单、按钮、通知

## 7.3 `packages/contracts` 不要承载“最终展示文案”

契约层应该承载：

- `code`
- `params`
- `LocalizedText`
- locale 字段

不要承载：

- 客户端最终展示语言句子

## 7.4 老代码迁移要配合 codemod

人工一个个包太慢。

建议写 AST codemod 分三类迁移：

1. JSX 纯文本
2. `title/label/placeholder/aria-label` 这类静态 prop
3. 明显的 `new Error("...")` / `BadRequestException("...")`

复杂插值和长文案再人工收尾。

## 7.5 默认 source locale 不强求仓库唯一

建议策略：

- `apps/app + apps/admin` 以 `zh-CN` 为 source locale 更现实
- `cloud-console` 是否保留英文 source locale，取决于你对云平台默认语言的产品定位
- 但 API 层最终都应转向错误码协议，这样就不会被源码语言卡死

如果你希望整个产品最终默认都用中文，也可以把 `cloud-console` 逐步迁到中文 source locale，但不建议把这件事塞进第一阶段。

## 8. 风险与规避

## 8.1 最大风险：只改页面，不改协议

后果：

- 页面文案看似多语言了
- 一到报错又冒出中文或英文原文

规避：

- API 错误协议必须尽早进 Phase 0 设计、Phase 4 落地

## 8.2 第二大风险：只做翻译，不做 lint/CI

后果：

- 老页面能切语言
- 新页面继续硬编码

规避：

- 必须上 `Lingui` ESLint 规则
- 必须上 repo 级 `i18n-audit`
- 必须让“新增违规 fail CI”

## 8.3 第三大风险：把 UI 语言和 AI 内容语言绑死

后果：

- 英文界面下角色被迫只说英文
- 做语言学习类角色会被系统限制死

规避：

- `preferredLocale` 和 `contentLocale` 必须拆开

## 8.4 第四大风险：把多语言推进成全量重写

后果：

- 周期过长
- 中途业务开发全被卡住

规避：

- 先建骨架
- 先卡新增
- 再渐进迁移存量

## 9. 里程碑和验收标准

## M1：框架立住

- 三个 React 应用接入 provider
- `packages/i18n` 可工作
- 可切 `zh-CN/en-US`
- `lingui extract/compile` 跑通
- lint 能拦新增未本地化字符串

## M2：Admin 试点完成

- `packages/ui + apps/admin` 核心页面完成迁移
- 错误提示、排序、时间格式不再写死中文

## M3：主 App 基础链路完成

- 用户设置里可切换语言
- 主 App 壳层、欢迎页、设置页、公共格式化层完成迁移

## M4：协议收口

- `api/cloud-api` 核心错误都带 `code`
- 前端按 code 翻译，不再直接透传服务端自然语言

## M5：新功能默认自动进入多语言体系

达到以下条件才算真正完成：

- 新功能开发时无需手写 key 文件
- 提交后 catalog 自动更新
- 新硬编码会被 CI 卡住
- 未翻译条目能自动暴露
- AI 可自动草拟目标语言翻译

## 10. 建议的首轮实施清单

如果下一步开始做，我建议第一轮直接做这些：

1. 新建 `packages/i18n`
2. 根脚本补：
   - `i18n:extract`
   - `i18n:compile`
   - `i18n:check`
   - `i18n:audit`
3. `packages/tooling` 接入 `Lingui` ESLint 规则
4. `apps/admin` 先做 provider 和 catalog
5. 把 `apps/app/src/lib/format.ts` 收口到共享格式化层
6. `packages/contracts` 新增 `ApiErrorResponse`
7. `api` 和 `cloud-api` 设计统一错误码返回
8. `UserEntity / WorldOwner` 增加 `preferredLocale`、`contentLocale`

## 11. 官方资料要点

本规划涉及的外部结论主要来自官方文档：

- Lingui 官方文档说明它提供 CLI、Vite plugin、ESLint plugin，并支持 React、动态语言包加载、message extraction、comment/context。
- i18next 官方文档说明新版官方 CLI 已覆盖 key extraction、lint、locale sync、type generation，且 TypeScript selector API 在新版本里是重点方向。
- FormatJS 官方文档说明 `@formatjs/cli` 支持 message extraction/compile/verify，并提供 ESLint 规则与 pseudo locale 能力，但在 React + Vite 现有仓库里接入写法和构建维护成本更高。
- MDN 官方文档说明 `Intl.DateTimeFormat`、`Intl.NumberFormat`、`Intl.Locale` 都已是现代浏览器可用基础能力，足够作为项目格式化底座。
