# 多模型角色与模型路由体系深度规划

日期：2026-04-23

## Summary

- 当前系统本质上只有一条“实例级单 Provider + 单模型”主链路，角色并没有自己的模型路由。
- 这次不建议只做一个“角色页面多一个模型下拉框”，而是要把 4 层能力一次性立起来：
  - 多 Provider / 多 Key 管理
  - 模型目录与官方模型清单
  - 角色级模型路由
  - “模型角色”批量生成与运营
- 这套体系落地后，你可以同时拥有：
  - `Claude / GPT / Gemini / DeepSeek / Qwen / Kimi / GLM / MiniMax / Hunyuan / Doubao / ERNIE / Step` 等模型接入
  - 每个角色绑定自己的模型
  - 后台随时切换角色模型、回退模型、默认模型
  - 一键把“排得上号的模型”安装成世界角色

## 1. 当前现状判断

## 1.1 现在的模型调用是单路由

- `api/src/modules/ai/ai-orchestrator.service.ts`
  - `resolveProviderConfig()` 只解析一套实例级 `endpoint + model + apiKey`
  - `resolveRuntimeProvider()` 只支持世界主人的单个 `apiKey/apiBase` 覆盖
- `api/src/modules/system/system.service.ts`
  - 同样只维护一套实例级 Provider 配置
  - `setProviderConfig()` 写的是单份配置

这意味着：

- 角色 A 和角色 B 不能稳定使用不同模型
- 发朋友圈、发广场、聊天、记忆压缩、角色工厂抽取都共用同一条模型通道
- 一旦你切换实例默认模型，全世界所有角色都会一起切

## 1.2 模型列表是硬编码且已经过时

- `packages/contracts/src/config.ts`
- `api/src/modules/config/config.controller.ts`

这两处都在硬编码 `AVAILABLE_MODELS`，而且是重复维护。对照 2026-04-23 官方文档，当前仓库里的列表已经明显落后，至少存在这些问题：

- OpenAI 现在主推的已经是 `gpt-5.4 / gpt-5.4-mini / gpt-5.4-nano`，仓库里还是 `gpt-5 / gpt-5.3-chat-latest`
- Anthropic 当前主线已经是 `Claude Opus 4.7 / Sonnet 4.6 / Haiku 4.5`
- 仓库里的模型列表没有“来源、能力、是否可见、是否推荐、是否已弃用”的元数据

结论：

- 不能再继续靠硬编码数组维护模型世界
- 必须建立“模型目录”层，而不是继续把模型名塞进下拉框

## 1.3 角色实体还没有模型字段

- `api/src/modules/characters/character.entity.ts`
- `packages/contracts/src/characters.ts`
- `apps/admin/src/routes/character-editor-page.tsx`

当前 `Character` 只有资料、提示词、生活策略，没有：

- 角色绑定哪个 Provider
- 角色绑定哪个模型
- 角色是否允许回退
- 角色是否跟随世界默认模型

所以现在后台就算想给角色选模型，也没有承载结构。

## 1.4 Key 管理不够用，而且实例级 Key 现在是明文风险

- 世界主人自定义 Key：
  - `api/src/modules/auth/user.entity.ts`
  - `api/src/modules/auth/world-owner.service.ts`
  - 已经支持加密存储
- 实例默认 Provider Key：
  - 现在仍写在 `system_config.provider_api_key`
  - `SystemConfigEntity.value` 是普通文本列

这意味着：

1. 现在只支持“世界主人一套个人 Key”
2. 不支持“OpenAI 一套、Anthropic 一套、Google 一套、Qwen 一套”并存
3. 实例默认 Provider Key 目前是明文配置，不适合扩成多 Key 仓

结论：

- 这次必须把“实例级模型 Key 管理”从 `system_config` 明文键值对升级成“加密 Provider 仓”

## 2. 目标口径

## 2.1 你真正要的不是“模型切换”，而是一套模型操作系统

本次规划的最终目标是：

1. 系统可同时接多个模型供应商
2. 后台能录入并管理多套 Key / endpoint / API 风格
3. 每个角色都能独立选择自己的模型
4. 可以把主流模型批量安装成“模型角色”
5. 所有调用都能记录“请求模型 / 实际模型 / 回退模型 / 计费来源”
6. 后续还能接预算、健康检查、停用、下线、目录同步

## 2.2 “排得上号的模型”要重新定义

如果按“所有快照版本”去做，会直接失控。

推荐口径：

- 角色粒度以“官方公开稳定 SKU / 当前主推型号”为准
- 快照版本保留在模型目录里，但默认不都生成角色
- 角色只为以下模型建立：
  - `isPublic = true`
  - `isNotable = true`
  - `status = active`
  - `surface = text | reasoning | coding | multimodal` 中与你的世界角色体系相关的模型

这样才能既覆盖“排得上号”，又不把世界角色列表炸成几百个。

## 3. 核心设计原则

## 3.1 分清四个对象

必须把下面四层分开：

1. `Provider Account`
   - 一套可调用的 endpoint + apiKey + 协议风格
2. `Model Catalog Entry`
   - 一个可选模型，如 `gpt-5.4-mini`
3. `Character Model Route`
   - 某个角色当前到底走哪个模型
4. `Model Persona Character`
   - 一个可聊天、可加好友、可运营的“模型角色”

如果把这四层混在一起，后面一定会乱。

## 3.2 模型目录和角色目录不能直接等同

- 模型目录是技术资产
- 角色目录是产品资产

不是每个模型都必须自动出现在用户世界里，但每个“排得上号”的模型都应该能被一键安装成角色。

## 3.3 角色默认静默，不自动刷屏

如果一口气装 20 到 40 个模型角色，它们不能像普通社交角色一样默认活跃发朋友圈、主动找你聊天。

模型角色默认应该：

- 不自动加好友
- 不主动发消息
- `momentsFrequency` 支持为 `0`
- `feedFrequency` 支持为 `0`
- 只在被用户添加、打开、激活后再进入更高活跃度

否则世界内容流会直接被模型角色淹没。

## 3.4 密钥必须统一加密

这次不要再引入任何明文 API Key 存储。

建议：

- 复用 `api/src/modules/auth/api-key-crypto.ts` 的 AES-GCM 方案
- 抽象成通用的 `credential-crypto`，不再只服务 `User.customApiKey`

## 4. 目标架构

## 4.1 后端新增“推理路由层”

建议新增新模块：

- `api/src/modules/inference/`

包含至少这些对象：

- `InferenceProviderAccountEntity`
- `InferenceModelCatalogEntity`
- `InferenceRoutingService`
- `InferenceProviderVaultService`
- `InferenceCatalogService`
- `InferenceHealthService`

原因：

- 现在 `ai` 模块负责“怎么调用模型”
- 新模块负责“该调用哪个模型、凭据从哪来、目录长什么样”
- 这两层职责不该继续混在 `SystemService` 和 `AiOrchestratorService` 里

- 新增 `model_persona`

## 4.3 全局默认路由不必新建表

当前是单世界实例，可以先把“全局默认模型路由”继续存在 `SystemConfig`，但只存轻量配置：

- `inference_default_provider_account_id`
- `inference_default_model_catalog_entry_id`
- `inference_catalog_sync_policy`

不要再把 Key 放回 `SystemConfig`。

## 5. 运行时路由方案

## 5.1 新的解析顺序

建议统一成下面顺序：

1. 如果调用明确传入角色
   - 优先读角色自己的 `modelRoutingMode`
2. 如果角色未绑定
   - 走世界默认模型
3. 如果角色绑定模型不可用
   - 按 `fallbackPolicy` 回退
4. 如果世界主人开启“个人临时覆盖”
   - 只在显式开启时覆盖默认路由，不默认覆盖所有角色

这里要明确一个产品判断：

- 现在世界主人的 `customApiKey/customApiBase` 是“强覆盖”
- 多模型体系上线后，建议改成“可选实验覆盖”
- 不然角色级模型配置会被世界主人的一把 Key 全部冲掉

## 5.2 `AiOrchestratorService` 需要改造的点

现在的：

- `resolveProviderConfig()`
- `resolveRuntimeProvider(aiKeyOverride?)`

要升级成：

- `resolveProviderAccount(providerAccountId)`
- `resolveModelCatalogEntry(modelCatalogEntryId)`
- `resolveInferenceRoute(context)`
- `buildRuntimeProvider(route, override?)`

所有 AI 调用入口都改走 `resolveInferenceRoute()`，包括：

- 单聊回复
- 群聊回复
- 朋友圈生成
- 广场 / 视频号生成
- 记忆压缩
- 核心记忆提炼
- 角色工厂抽取
- quick character 生成
- admin preview

## 5.3 计费与审计

`AiUsageLedger` 已经有：

- `model`
- `providerKey`
- `requestedModel / appliedModel`
- `budgetAction`

这次应继续扩展记录：

- `requestedProviderAccountId`
- `appliedProviderAccountId`
- `requestedModelCatalogEntryId`
- `appliedModelCatalogEntryId`
- `routingSource`
  - `world_default`
  - `character_bound`
  - `fallback`
  - `owner_override`

这样后面后台才能看清：

- 角色本来要走哪个模型
- 实际为什么换了别的模型

## 7. 模型角色方案

## 7.1 模型角色的定位

模型角色不是“公司吉祥物”，而是“模型能力化身”。

例如：

- `GPT-5.4`
  - 偏任务完成、稳定、全能、长程执行
- `Claude Opus 4.7`
  - 偏深度推理、长程 coding、表达稳健
- `Gemini 2.5 Pro`
  - 偏多模态推理与复杂分析
- `DeepSeek-R1`
  - 偏深推理与链路展开
- `Qwen-Coder`
  - 偏中文开发与工具调用

每个模型角色都必须绑定到一个真实模型目录项，不允许“角色名是一个模型，实际调另一个模型”。

## 7.2 模型角色的默认行为

建议默认画像：

- `sourceType = model_persona`
- `relationshipType = expert`
- `activityFrequency = low`
- `momentsFrequency = 0`
- `feedFrequency = 0`
- `isOnline = true`
- 默认不主动加好友、不主动发动态

角色简介应强调：

- 这是某个模型的交互化身
- 它代表的是该模型当前公开能力风格
- 不代表官方品牌发言

## 7.3 模型角色的生成方式

不要手写几十个模型角色文案。

建议新增：

- `api/src/modules/characters/model-persona-presets.ts`

由“模型目录种子 + 模板”生成角色预设：

- 基础模板按厂商区分
- 风格模板按 `reasoning / coding / fast / multimodal` 区分
- 展示名、头像、简介、scene prompt 再做少量特化

这样后面模型更新时，不需要人工重抄几十份角色文件。

## 9. 实施顺序

## Phase 1：打通多 Provider 底座

目标：

- 先让系统支持多账户、多模型调用
- 暂时不生成模型角色

工作项：

- 新建 `inference` 模块
- 落 `InferenceProviderAccount` / `InferenceModelCatalogEntry`
- 把当前实例默认 Provider 从 `system_config` 迁到新表
- 改造 `AiOrchestratorService` 使用新路由层
- 保留原 `/system/provider` 接口做兼容，但底层不再直接读明文 `system_config`

验收标准：

- 后台能同时录入至少两家 Provider
- 世界默认模型可切换
- chat preview 能指定不同模型跑通

## Phase 2：角色级模型路由

目标：

- 每个角色可单独选模型

工作项：

- 扩 `CharacterEntity`
- 改角色编辑页 / 角色运行台
- 所有角色相关生成链路改为按角色路由解析

验收标准：

- 角色 A 用 `Claude`
- 角色 B 用 `DeepSeek`
- 两者同实例内稳定并行工作

## Phase 3：模型角色安装器

目标：

- 把主流模型一键安装成世界角色

工作项：

- 加 `model_persona` 角色来源
- 新建模型角色预设生成器
- 后台增加批量安装入口
- 默认静默策略生效

验收标准：

- 一键安装“国际主流模型角色”
- 不污染普通朋友列表
- 用户可单独加好友聊天

## Phase 4：目录同步与健康检查

目标：

- 后面不用每次手工改代码才能补模型

工作项：

- 增加官方模型目录同步器
- 增加每个 Provider 的探活与联通性检查
- 后台支持“同步目录”“标记 notable”“隐藏弃用模型”

验收标准：

- 至少支持从官方 API 或官方种子同步模型清单
- 目录更新不需要手改 `AVAILABLE_MODELS`

## Phase 5：预算、回退、运营

目标：

- 让这套体系可长期运营

工作项：

- 模型健康失败自动回退
- 角色级成本面板
- 角色级模型切换历史
- 目录弃用提醒

## 10. 风险与规避

## 10.1 最大风险：先做角色，不做路由底座

如果先批量建角色，但底层还是单模型：

- 角色名和实际模型不一致
- 所有模型角色都在说同一种话
- 用户会立刻感知为“假人设”

所以顺序不能反。

## 10.2 第二风险：把多 Key 继续塞 `SystemConfig`

这会带来：

- 明文风险
- 难以管理
- 无法健康检查
- 无法做启停与默认切换

必须上加密 Provider 仓。

## 10.3 第三风险：模型角色默认太活跃

如果几十个模型角色都默认会发动态、主动聊天：

- 世界会变成模型展厅
- 正常社交角色体验会被冲掉

所以模型角色默认应是“可聊但静默”。

## 11. 推荐的最小可交付范围

如果你想分两轮做，建议这样切：

### 第一轮必须做完

- 多 Provider 账户管理
- 加密 Key 仓
- 角色级模型绑定
- 世界默认模型路由
- 运行时真实按角色走模型

### 第二轮再补

- 模型角色批量安装
- 目录同步
- 健康检查与自动回退
- 更完整的 persona 包装

原因很简单：

- 第一轮做完，系统已经真的支持多模型角色
- 第二轮只是把这套能力产品化、可运营化

## 12. 结论

这件事的正确做法不是“给角色多加一个 model 字段”，而是：

1. 先建立多 Provider 的加密凭据仓
2. 再建立模型目录
3. 再建立角色级模型路由
4. 最后把主流模型安装成世界角色

只有这样，模型角色才是真角色，而不是一个换名字的单模型皮套。
