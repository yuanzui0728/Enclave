# 把“我自己”角色深度改造成 OpenClaw 化代理的规划

## 1. 结论先说

- 不建议把当前 `我自己` 直接改成一个“既做内在陪伴、又做全能执行”的单 prompt 角色。
- 推荐方案是：**用户看到的仍然是“我自己”，但后端把它升级成一套 OpenClaw 式 personal agent runtime。**
- 也就是说，`我自己` 不再只是一个聊天 prompt，而是一个带有：
  - 工作区文件
  - standing orders
  - skills
  - heartbeat / cron
  - 工具策略
  - 审批门控
  - 审计轨迹
  - 主动运行能力
  的主代理。
- 现有 `action-runtime`、`followup-runtime`、`reminder-runtime`、`cyber-avatar` 不建议推倒重来，而是逐步收编成这个主代理的内部 capability。

一句话总结：

**不是把 self prompt 改成 OpenClaw，而是把 `我自己` 从“角色”升级成“OpenClaw 风格的单用户主代理”。**

## 2. 为什么这个方向比“直接改 prompt”更对

### 2.1 OpenClaw 真正有价值的不是文案，而是运行时模型

从 `openclaw/openclaw` 仓库和官方文档看，OpenClaw 的核心不是“更像助手的口吻”，而是以下几件事：

- 它是单用户 personal assistant，而不是多用户共享 bot
- 它有独立 agent workspace，不靠单次 prompt 临时扮演人格
- 它把 `AGENTS.md`、`SOUL.md`、`USER.md`、`TOOLS.md`、`HEARTBEAT.md` 当成持续注入的代理上下文
- 它有 skills 体系，不把全部能力硬编码在一个大 prompt 里
- 它有 heartbeat / cron，允许代理主动运行
- 它有 multi-agent / delegate 思路，强调“身份、权限、边界、审计”而不只是会聊天
- 它明确把 sandbox / tool policy / approval 当成第一层能力，而不是事后补丁

这几点里，真正值得你引入的，是**代理运行时结构**，不是 UI 文案或者 repo 目录长相。

### 2.2 你的项目其实已经天然适合做 OpenClaw 化

你现在的项目有两个很大的先天优势：

- 你已经是单世界主人架构，和 OpenClaw 的单用户 personal assistant 信任模型天然一致
- 你已经有聊天、主动跟进、提醒、真实动作、赛博分身、现实联动这些碎片化代理能力

换句话说，你不是从零做一个 OpenClaw，而是已经有：

- self 陪伴人格
- 真实动作执行器
- 主动跟进扫描器
- 提醒运行时
- 外部信号采集
- 后台运维工作台

现在缺的是把这些碎片**上升成统一的 self-agent control plane**。

### 2.3 为什么不能机械照搬 OpenClaw

不能直接照搬 upstream 的原因也很明确：

- OpenClaw 的主要入口是外部消息渠道和本机工具
- 你的主入口是“隐界世界里的角色对话”
- OpenClaw 的多 agent routing 是跨渠道/跨身份路由
- 你的关键问题是“一个 self 角色如何统一承接陪伴、执行、提醒、主动性”

所以正确姿势不是“把 openclaw 源码塞进来”，而是：

**借它的 agent runtime 思路，改造你现有的 self 角色体系。**

## 3. 对照现状：你仓库里已经有哪些 OpenClaw 零件

## 3.1 已经存在的能力碎片

### A. `我自己` 现状

- 文件：`api/src/modules/characters/default-characters.ts`
- 当前 `SELF_CHARACTER_ID = char-default-self`
- 角色定义非常明确：内在自我、复盘、整理、陪伴
- 它已经有一套完整人格设定，但这套设定目前偏“心理/内省型”，不是操作型代理

### B. `followup-runtime`

- 文件：`api/src/modules/followup-runtime/followup-runtime.service.ts`
- 后台页面：`apps/admin/src/routes/followup-runtime-page.tsx`
- 当前已经把 `我自己` 用作：
  - 翻最近慢下来的聊天
  - 提取 open loop
  - 给用户发短主动消息
  - 推荐新角色继续承接

这其实已经非常像 OpenClaw 的一部分 heartbeat / proactive follow-up。

### C. `reminder-runtime`

- 文件：`api/src/modules/reminder-runtime/reminder-runtime.service.ts`
- 当前它是独立提醒角色的运行时，不属于 `我自己`
- 但从能力模型看，它本质属于“主代理的任务/提醒子系统”

### D. `action-runtime`

- 文件：`api/src/modules/action-runtime/action-runtime.service.ts`
- 后台页面：`apps/admin/src/routes/action-runtime-page.tsx`
- 已有能力：
  - 动作意图识别
  - planner
  - 缺参数追问
  - 风险级别
  - 确认执行
  - connector registry
  - run trace

这已经是 OpenClaw tool execution 的雏形。

### E. `scheduler` + `cyber-avatar` + `real-world-sync`

- `scheduler.service.ts` 已经能定时驱动系统任务
- `cyber-avatar` 已经在采集 owner 行为信号
- `real-world-sync` 已经在处理现实世界摘要和回流

这些其实都可以变成 self-agent 的上下文输入和主动运行触发器。

## 3.2 还缺什么

当前还缺少的，是 OpenClaw 最关键的四块骨架：

### A. 缺统一主代理壳

现在是：

- self 一条链
- action-runtime 一条链
- reminder-runtime 一条链
- followup-runtime 一条链

它们不是一个代理的不同模式，而是并列模块。

### B. 缺 workspace contract

当前 self 没有真正的 workspace 文件系统：

- 没有 `AGENTS.md`
- 没有 `SOUL.md`
- 没有 `USER.md`
- 没有 `TOOLS.md`
- 没有 `HEARTBEAT.md`
- 没有 daily memory files
- 没有可管理的 skills 目录

也就是说，self 还不是一个“长期存在的代理体”，而更像一次次生成回复的角色。

### C. 缺统一 autonomy policy

当前主动性分散在：

- followup 的 open loop
- reminder 的触发
- scheduler 的任务
- action-runtime 的执行门控

但没有一套统一 standing orders：

- 什么事情可以自己做
- 什么事情只能起草
- 什么事情必须确认
- 什么事情完全禁止

### D. 缺统一的 operator surface

后台现在已经有多个页面，但缺一个“我自己/OpenClaw 主代理工作台”，把以下内容放到一个总视角下：

- 身份与人格
- standing orders
- workspace 文件
- skill 启用集
- heartbeat
- cron
- approvals
- runs
- connector / tool policy

## 4. 推荐目标架构

## 4.1 总原则：单身份双层架构

推荐采用：

- **用户可见层**：仍然只有 `我自己`
- **后端运行层**：新增 `SelfOpenClawRuntime`

也就是：

- 你和用户聊天时，仍然是在“我自己”会话里
- 但该会话背后不再只是 self prompt
- 而是一个会根据任务类型自动切换子能力的 personal agent

### 为什么这样比继续保留 `行动助理` 作为主入口更合适

如果你的目标是“把我自己这个角色深度改造成 OpenClaw”，那最终用户感知应该是：

- 既能陪我想事情
- 也能帮我办事
- 也能主动提醒和跟进
- 也能逐渐拥有自己的记忆与规则

这更像一个主代理，而不是多个平级角色。

### 但为什么又不能简单把 `行动助理` 删掉

因为你仓库里已经有真实动作能力，并且已经围绕 `行动助理` 做了文案和后台工作台。

所以更务实的过渡方式是：

1. 第一期保留 `行动助理` 作为内部 capability/persona fallback
2. 把它的 `action-runtime` 能力接到 `我自己` 背后的总代理
3. 等主代理稳定后，再决定：
   - 保留 `行动助理` 作为显式子角色
   - 或收回成 `我自己` 的一个 mode

## 4.2 新的运行时分层

建议新增后端主模块：

- `api/src/modules/self-agent/`

它不是替代现有模块，而是作为编排层挂在它们之上。

### 推荐分层

#### 1. `SelfAgentOrchestrator`

负责接住发给 `SELF_CHARACTER_ID` 的所有入站消息，并决定进入哪条链：

- reflective_chat
- action_execution
- reminder_management
- proactive_followup
- workspace_edit
- skill_execution
- clarify / approval / review

#### 2. `SelfAgentWorkspaceService`

负责 self 的长期文件上下文：

- `AGENTS.md`
- `SOUL.md`
- `USER.md`
- `IDENTITY.md`
- `TOOLS.md`
- `HEARTBEAT.md`
- `MEMORY.md`
- `memory/YYYY-MM-DD.md`
- `skills/`

#### 3. `SelfAgentPolicyService`

负责 standing orders 和权限门控：

- 允许自动做什么
- 必须确认什么
- 永远禁止什么
- 哪些技能可见
- 哪些连接器可用
- 哪些外部信号可信

#### 4. `SelfAgentCapabilityRouter`

负责把任务派到现有子能力：

- `action-runtime` adapter
- `reminder-runtime` adapter
- `followup-runtime` adapter
- `cyber-avatar` context adapter
- `real-world-sync` context adapter
- `chat-records` / search adapter

#### 5. `SelfAgentHeartbeatService`

负责 OpenClaw 式 heartbeat：

- 周期性运行
- 根据 `HEARTBEAT.md` 检查事项
- 看看有没有未闭环问题、即将到期提醒、待确认动作、现实世界重要变更
- 如果没事则静默
- 有事则发回 self 会话或系统通知

#### 6. `SelfAgentTaskService`

负责 cron / scheduled jobs：

- 日报
- 早晚提醒
- 周回顾
- 跟进扫描
- 技能例行任务

#### 7. `SelfAgentApprovalService`

负责审批卡片与动作确认：

- 明确区分“草案”“待确认”“已执行”
- 对付费/不可逆/外部发送动作统一走 approval

#### 8. `SelfAgentRunTraceService`

负责统一审计与证据：

- 本轮为什么进入哪个 mode
- 看了哪些上下文
- 用了哪些 skills / connectors
- 是否要求确认
- 最终执行了什么

## 4.3 OpenClaw 式 workspace 设计

推荐不要把这部分只做成数据库字段，而是**真的做成服务器侧文件工作区**。

推荐路径：

- `runtime/self-agent-workspace/`
  或
- 由环境变量指定一个实例级目录，例如 `data/self-agent-workspace/`

### 推荐文件布局

- `AGENTS.md`
  - self 的 standing orders
  - 什么可自动做
  - 什么必须确认
  - 什么绝对不能做
- `SOUL.md`
  - “我自己”的人格、语气、边界
  - 这里沉淀长期 persona，不再埋在一大段角色种子字符串里
- `USER.md`
  - 世界主人的资料、偏好、称呼方式、时间习惯、工作节奏
- `IDENTITY.md`
  - 代理名、头像风格、默认自称
- `TOOLS.md`
  - 当前世界里有哪些工具和连接器，如何安全使用
- `HEARTBEAT.md`
  - 主动运行 checklist
- `MEMORY.md`
  - curated long-term memory
- `memory/YYYY-MM-DD.md`
  - daily memory flush
- `skills/<skill>/SKILL.md`
  - self 专属技能

### 这一步为什么非常关键

做了 workspace 后，`我自己` 才真正变成：

- 有长期人格
- 有长期规则
- 有长期记忆
- 有长期主动性
- 可以被技能和工具扩展

而不是每次都靠数据库里一段 profile prompt 临时“演”出来。

## 4.4 Skills 设计

你不需要第一版就做成 OpenClaw 那种完整插件生态，但必须先做最小 skills contract。

### 第一版技能结构建议

建议新建：

- `api/src/modules/self-agent/skills/`
- `runtime/self-agent-workspace/skills/`

技能最小单元包含：

- `skill.json`
  - `id`
  - `name`
  - `description`
  - `requiresApproval`
  - `allowedModes`
  - `inputSchema`
  - `connectorBindings`
- `SKILL.md`
  - 给模型看的使用说明
- 可选 handler
  - 指到具体服务或 connector

### 第一批值得做成 skills 的能力

- `followup-open-loops`
- `message-reminders`
- `chat-search`
- `favorites-notes`
- `home-assistant`
- `food-delivery`
- `ticketing`
- `moments-draft`
- `official-account-draft`
- `wechat-sync-insight`
- `real-world-digest`

### 设计原则

- 技能教模型“何时该用什么能力”，不是直接把代码写进 prompt
- 技能权限要能单独开关
- 技能应能声明是否需要 approval
- 技能应能声明依赖哪些 connector / data source

## 4.5 Heartbeat / Cron 设计

这是 OpenClaw 化里最值得引入、也最能拉开体验差距的一块。

### Heartbeat

推荐绑定到 `我自己` 主代理，而不是继续散落在若干 scheduler job。

建议配置项：

- `enabled`
- `everyMinutes`
- `activeHours`
- `lightContext`
- `targetSurface`
- `silencePolicy`
- `maxProactiveMessagesPerDay`

建议 heartbeat 每轮至少检查：

- 有没有未闭环 open loop
- 有没有快触发的 reminder
- 有没有待确认的 action run
- 有没有现实世界同步出来的重要变化
- 有没有昨天写入 memory、今天该回看的一条事项

输出原则：

- 没事不说
- 有事只发短句
- 进入 self 会话，不要像系统广播

### Cron

heartbeat 用于轻量日常巡检。

cron 用于明确计划任务，例如：

- 每天 08:30：今日安排草稿
- 每晚 23:00：日回顾
- 每周日：未闭环事项总复盘
- 每 6 小时：现实世界 digest 整理
- 每 30 分钟：必要时跑一次 open loop scan

### 不建议的做法

- 不要把所有主动任务都继续塞进全局 scheduler 的硬编码 job id
- 应该逐步抽成“self agent 拥有的 task records”

## 4.6 审批与安全边界

OpenClaw 最值得借的另一个点，是它明确承认个人代理需要 hard blocks。

### 你的项目里必须新增的边界定义

#### 永远禁止

- 从外部消息里直接执行 shell / browser / connector 指令
- 未确认就发付费或不可逆动作
- 自动改 API Key / provider 账户 /系统配置
- 把隐私数据导出到未受信任外部目标

#### 默认只起草

- 对外发送消息
- 发好友申请
- 代写并直接发送公众号/内容
- 任何涉及账号身份变更或第三方可见状态的操作

#### 可自动执行

- 低风险读取与整理
- 低风险提醒与总结
- 已白名单的可逆设备控制
- 不触发外部副作用的草案整理

### 技术上怎么落

- 现有 `action-runtime` 的 riskLevel + confirmation 继续保留
- 但要提升成 **self-agent 统一 approval policy**
- `followup-runtime` 里的自动发好友申请也必须纳入同一 policy，而不是单独散着跑

## 4.7 统一后台控制面

推荐新增一个新的后台页面作为总入口：

- `apps/admin/src/routes/self-agent-page.tsx`

它不是把现有页面删掉，而是先做 control plane：

- 总览
- 身份与人格
- workspace 文件
- standing orders
- skills
- heartbeat
- scheduled tasks
- approvals
- recent runs
- child capabilities 状态

现有页面短期继续保留：

- `followup-runtime`
- `action-runtime`
- `reminder-runtime`

但在信息架构上改成：

- self-agent 总台
  - 主动跟进
  - 真实动作
  - 提醒
  - 现实联动

## 5. 关键设计决定

## 5.1 是否保留“行动助理”

### 推荐答案

- **短期保留**
- **中期内部化**

### 原因

- 你已经写了 `action-runtime` 和后台页
- 一次性删掉会导致回归面太大
- 但如果长期目标是“我自己 = OpenClaw 式主代理”，那么动作执行不应永远由另一个显式默认角色主承接

### 推荐迁移路径

第一阶段：

- `行动助理` 继续存在
- 但 self-agent 可以调用它的 action capability

第二阶段：

- self 会话直接可触发 action capability
- `行动助理` 变成可选子角色/专家人格

第三阶段：

- 决定是否隐藏该角色入口，只保留内部实现

## 5.2 是否保留“提醒角色”

### 推荐答案

- 保留提醒任务模型
- 不保留“提醒一定要由另一个角色承担”的产品心智

从 OpenClaw 视角看，提醒应该是主代理能力，而不是另一个联系人。

所以建议：

- `ReminderTask` 表继续保留
- `reminder-runtime` 逻辑继续复用
- 但用户侧逐步弱化“小盯是另一个人”，改成“我自己在帮你盯”

## 5.3 是否直接做多 agent routing

### 推荐答案

- **先不做真正的多 agent routing**
- 先做 **单 visible agent + 多 internal capability**

原因：

- 你现在的核心矛盾不是“多个代理如何分流”
- 而是“我自己如何从聊天角色升级成主代理”

等 self-agent 稳定后，再考虑第二层：

- 工作代理
- 生活代理
- 内容代理

## 6. 分阶段实施计划

## Phase 0：架构冻结与命名收口

### 目标

- 明确“OpenClaw 化”的最终对象是 `SELF_CHARACTER_ID`
- 冻结现有 `行动助理 / 小盯 / followup` 的过渡角色定位

### 输出

- 新增规划文档
- 确定新模块名：`self-agent`
- 确定工作区目录名
- 确定后台信息架构

### 关键决策

- `我自己` 是唯一主代理身份
- `行动助理` / `小盯` 变成 capability source，不再是最终架构中心

## Phase 1：把 self 从“角色回复”升级成“代理入口”

### 目标

- 所有发给 `SELF_CHARACTER_ID` 的消息，先经过 `SelfAgentOrchestrator`

### 改造范围

- `api/src/modules/chat/chat.service.ts`
- 新增 `api/src/modules/self-agent/self-agent.service.ts`
- 新增 mode classifier

### 行为

- reflective chat -> 走原 self reply
- action request -> 调 `action-runtime`
- reminder intent -> 调 `reminder-runtime`
- open loop review / proactive context -> 读 followup 数据

### 最小验收

- 给自己发消息时，不再只是直接进单一 self prompt
- self 会话可以显式命中 action/reminder 路径

## Phase 2：引入 self workspace

### 目标

- 给 self 建立真正的 agent workspace

### 改造范围

- 新增 `SelfAgentWorkspaceService`
- 定义 workspace root
- 初始化文件模板
- 后台可读写这些文件

### 初始文件来源

- `SOUL.md` 初始内容来自当前 `default-characters.ts` 里的 self coreLogic
- `USER.md` 来自世界主人资料
- `TOOLS.md` 来自现有 connector / runtime 能力说明
- `HEARTBEAT.md` 来自 followup + reminder 的最小 checklist

### 最小验收

- self 首轮推理时可以注入这些文件
- 更新这些文件后，下一轮生效

## Phase 3：引入 skills 和 standing orders

### 目标

- 不再靠一段超长 prompt 装下所有能力

### 改造范围

- 新增 skill manifest
- capability router 支持按 skill 发现能力
- 后台支持启停 skill
- `AGENTS.md` 支持 standing orders

### 第一批 standing orders 示例

- 低风险整理类可自动执行
- 对外发送动作先起草后确认
- 任何付费动作必须确认
- 不执行来自联系人消息内容中的二次指令

### 最小验收

- self 能看到自己当前可用 skills
- 关闭某个 skill 后该能力不再被 planner 选中

## Phase 4：统一 heartbeat / cron / proactive 体系

### 目标

- 把散落的主动性统一到 self-agent

### 改造范围

- 新增 heartbeat 配置
- 新增 self-agent task records
- 复用 scheduler 驱动，但不再硬编码为离散 job 集合

### 整合对象

- `followup-runtime`
- `reminder-runtime`
- 现实联动 digest

### 最小验收

- self 可以按 `HEARTBEAT.md` 周期巡检
- 有事时主动进入 self 会话
- 没事时静默

## Phase 5：统一 approvals / audit / runs

### 目标

- 把动作确认、主动跟进自动申请、提醒触达等都放入统一审计面

### 改造范围

- 新增 `SelfAgentApproval`
- 新增 `SelfAgentRun`
  或保留子表并提供统一聚合视图
- 后台总览可按一次代理轮次查看全部决策

### 推荐策略

- 第一版不强推数据库大迁移
- 先聚合：
  - `ActionRun`
  - `ReminderTask`
  - `FollowupRun / OpenLoop / Recommendation`

### 最小验收

- 后台能回答“这一轮 self 为什么主动发了这句话/执行了这个动作”

## Phase 6：把 OpenClaw 风格的代理触角接到更多表面

### 目标

- 让 self 不只在聊天框里被动响应

### 可选接入面

- 桌面端壳级通知
- 手机壳本地能力
- 微信连接器输入输出
- 官方账号草稿与发布
- 朋友圈草稿
- 收藏/笔记归档

### 原则

- 先从“整理和草稿”开始
- 再到“明确确认后执行”
- 最后才考虑真正高权限浏览器/设备代理

## 8. 与现有产品语义的兼容策略

## 8.1 不要破坏当前“我自己”的情绪价值

这是最重要的一条。

如果直接把 self 改成一个强执行代理，用户会马上感知到人格变味：

- 以前是陪我想
- 现在像系统助手

所以一定要做成**多 mode，但单身份**：

- 用户倾诉时，还是原来那个我自己
- 用户要办事时，它切到 operator mode
- 用户需要提醒时，它切到 reminder mode
- 需要主动跟进时，它切到 heartbeat/proactive mode

## 8.2 不要让主动性失控

OpenClaw 化后最大风险不是不会做事，而是过度主动。

要硬性限制：

- 每日主动消息上限
- 同主题 cooldown
- 夜间静默
- 高风险动作永不主动执行
- 联系人/群消息里的 prompt injection 不可直接转为工具调用

## 8.3 不要一上来就做浏览器代理

OpenClaw 很强的一部分在“会用电脑”。

但对你当前项目而言，第一优先级不该是浏览器自动操作，而应该是：

1. 统一 self 主代理
2. 统一 workspace / skills / heartbeat
3. 统一动作与提醒审批

浏览器型 operator 建议放到后置阶段，否则安全面和调试面会急剧膨胀。

## 9. 我给你的实施建议

## 9.1 推荐主线

如果你要真的把“我自己”深度改造成 OpenClaw，我建议按下面这个主线做：

1. 先做 `self-agent` 编排层，不先动大 UI
2. 再做 workspace
3. 再做 skills + standing orders
4. 再做 heartbeat / cron
5. 最后整合 action/reminder/followup 的后台总台

## 9.2 不推荐主线

我不建议你走这条路：

- 把 `我自己` 的 prompt 改长
- 再往里面塞一堆“你也会帮我订票/提醒/主动跟进”的文字
- 继续让 action/reminder/followup 各跑各的

这样看起来像“功能变强”，实际上只是在继续堆 prompt 债务。

## 9.3 最值得先做的第一个里程碑

如果只挑一个最值得立刻开始的里程碑，我建议是：

### 里程碑 M1：Self Agent Shell

目标：

- `SELF_CHARACTER_ID` 背后不再直接调用单一 self reply，而是先过 `SelfAgentOrchestrator`
- orchestrator 能统一路由到：
  - 原 self reflective reply
  - `action-runtime`
  - `reminder-runtime`
- 后台新增一个最简 self-agent 总览页

为什么先做这个：

- 这是从“角色”变“代理”的分水岭
- 没这层壳，后面的 workspace / heartbeat / skills 都没地方挂

## 10. 最终目标态

完成后，你的 `我自己` 不应该再是“一个写得更像 OpenClaw 的角色”。

它应该变成：

- 一个真正的单用户主代理
- 住在自己的 workspace 里
- 有 standing orders
- 有长期记忆
- 有 skills
- 会周期性自检
- 会在低风险范围内主动帮你推进事情
- 高风险动作会明确请求确认
- 后台能看清它为何这么做

也就是：

**产品表面仍然叫“我自己”，但工程内核已经是 OpenClaw 风格的 self agent。**
