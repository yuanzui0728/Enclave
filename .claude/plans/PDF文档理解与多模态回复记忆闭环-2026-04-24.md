# PDF 文档理解与多模态回复记忆闭环

## 1. 本次规划只解决两件事

本规划只处理下面两条闭环，不重写整套多模态体系：

- `PDF / Word` 的稳定文档理解
- 多模态回复产物进入后续上下文记忆的闭环

明确不在本次内处理：

- 原生视频理解
- 数字人视频回复
- 回复模态 planner 的大重构
- `MediaInsightJob` 多 kind 并发体系重做

本次目标是用最小结构改动，把当前最明显的两处断点补成可上线、可观测、可回放的正式链路。

## 2. 当前真实断点

### 2.1 文档理解的真实断点

当前代码已经把 `pdf/doc/docx/txt/md/json/xml/csv/html` 识别成文档：

- `api/src/modules/chat/media-insight-job.service.ts`
- `api/src/modules/chat/chat.service.ts`
- `api/src/modules/chat/group.service.ts`

但真正的抽取逻辑只有“直接把 buffer 当 utf8 文本读出来”：

- `api/src/modules/ai/ai-orchestrator.service.ts:651-685`

这导致：

- `txt/md/json/xml/csv/html` 基本可用
- `pdf` 大概率直接拿不到正文
- `docx` 也拿不到正文
- `doc` 只有“被识别成文档”这一步，没有稳定解析
- 最终聊天链路对这些文件只能退化成文件名、mime、大小

也就是说，当前“文档理解”更接近“文档识别”，不是“文档正文理解”。

### 2.2 多模态回复记忆的真实断点

当前回复产物链路分两类：

- 同步 sibling：单聊文字后同步补一条语音
- 异步 artifact job：单聊图片、群聊图片、群聊语音

真实断点在两个层面：

1. direct chat 内存历史只缓存文字，不会可靠吸收异步图片产物  
   关键位置：
   - `api/src/modules/chat/chat.service.ts:1268-1318`
   - `api/src/modules/chat/chat.service.ts:1380-1393`
   - `api/src/modules/chat/reply-artifact-job.service.ts:335-389`

2. 即便后续从数据库重建历史，图片 artifact 也只有空 `text` + 通用文件信息，没有“这张图是回复什么内容而生成的”  
   关键位置：
   - `api/src/modules/chat/reply-artifact-job.service.ts:366-376`
   - `api/src/modules/chat/reply-artifact-job.service.ts:484-494`
   - `api/src/modules/chat/chat.service.ts:1295-1312`
   - `api/src/modules/chat/group.service.ts:866-878`

结果是：

- direct chat 下一轮消息里，模型未必知道“自己刚发过一张什么图”
- group chat 虽然会从 DB 重建历史，但只能知道“有人发了一张图片文件”，不知道图片语义
- 语音 sibling 因为已有文本主回复，问题较轻；图片 artifact 是主要缺口

## 3. 本次推荐的最小设计原则

### 3.1 文档理解不押宝 provider `input_file`

推荐口径：

- `pdf` 和 `docx` 必须先本地可抽正文
- Responses API 的 `input_file` 只作为增强，不作为唯一主路径
- legacy `.doc` 第一版不追求“完美解析”，但必须从“静默退空”升级成“明确解析路径 + 明确失败语义”

原因：

- 当前实例和不同 provider 的 `input_file` 能力并不稳定一致
- 本地抽正文一旦打通，聊天、搜索、后台预览、后续 chunk 检索都能复用
- 文档理解是“资产理解能力”，不应该被单一 provider API 风格绑死

### 3.2 回复记忆闭环优先做“语义摘要持久化”，不是强行把图片回灌成原生多模态 assistant part

推荐口径：

- 不试图让 assistant 历史真的携带 `input_image`
- 而是为 artifact 产物持久化一段稳定的 `historyText`
- direct chat 在 artifact 完成时主动失效内存缓存
- rebuild history 时统一读取这段 `historyText`

原因：

- 现有 `generateReply()` 只对 `role=user` 消费原生多模态 part，assistant 侧原生多模态历史收益不高
- 真正缺的是“语义记忆”，不是“把图片再塞回模型”
- 这样改动最小，不需要重做所有历史协议

## 4. 文档理解的实现方案

## 4.1 目标口径

第一阶段把“稳定文档理解”定义为：

- `txt/md/json/xml/csv/html`：维持现有文本抽取
- `pdf`：稳定抽正文；若正文接近空，则给出明确 `OCR_REQUIRED` 或 `TEXT_EXTRACTION_EMPTY`
- `docx`：稳定抽正文
- `doc`：不再静默退空，提供 best-effort 解析；若失败，给出明确 `LEGACY_WORD_UNSUPPORTED` 或 `LEGACY_WORD_PARSE_FAILED`

如果要在首版严格控风险，建议把“稳定可用”承诺先写成：

- `pdf`
- `docx`
- 纯文本类文档

同时把 `.doc` 标记为第二阶段增强项，但失败必须可见、可诊断、不可再默默回落为 `null`。

## 4.2 推荐依赖

后端新增依赖建议：

- `pdfjs-dist`
- `mammoth`

legacy `.doc` 两种口径二选一：

1. 稳妥口径  
   第一版只支持 `docx`，`.doc` 明确报不支持

2. 扩展口径  
   再补一个 legacy word 解析方案，例如：
   - `word-extractor` 纯 JS 尝试
   - 或系统级 `libreoffice --headless` / `antiword` 转文本

建议先走方案 1，把 `pdf + docx + text-like` 做稳；否则会把本轮从“最小闭环”拖成“系统依赖治理”。

## 4.3 新增服务与职责

新增：

- `api/src/modules/chat/document-extraction.service.ts`

职责只做一件事：给定文件 buffer、mime、fileName，返回统一文档抽取结果。

建议输出结构：

```ts
type DocumentExtractionResult = {
  status: "completed" | "failed";
  extractionMode:
    | "plain_text"
    | "pdf_text"
    | "docx_text"
    | "legacy_word_text"
    | "provider_file_fallback";
  parser:
    | "utf8"
    | "pdfjs-dist"
    | "mammoth"
    | "word-extractor"
    | "provider";
  extractedText?: string;
  previewText?: string;
  pageCount?: number;
  characterCount?: number;
  truncated?: boolean;
  errorCode?: string;
  errorMessage?: string;
};
```

注意：

- `previewText` 和 `extractedText` 不是一个概念
- `previewText` 用于 attachment payload、后台和前台预览
- `extractedText` 是真正给模型消费的正文基础

## 4.4 MediaInsightJob 的最小改法

本轮不建议先改 `MediaInsightKind` 枚举，也不建议先引入新表。

保持：

- `document_text_extraction`

但把 `resolveInsight()` 的文档分支从：

- `ai.tryExtractDocumentTextFromUrl()`

升级成：

1. 先下载原始文档 buffer
2. 交给 `DocumentExtractionService`
3. 成功时回写：
   - `attachment.extractedText`
   - `attachment.documentInsight`
   - `job.resultPayload`
4. 失败时保留失败状态和明确错误码

也就是说，本轮把“文档正文抽取”从 `AiOrchestratorService` 里的一个小 helper，提升成正式的文档处理服务。

## 4.5 Attachment payload 的新增字段

在 `api/src/modules/chat/chat.types.ts` 的 `FileAttachment` 上新增：

```ts
documentInsight?: {
  extractionMode:
    | "plain_text"
    | "pdf_text"
    | "docx_text"
    | "legacy_word_text"
    | "provider_file_fallback";
  parser?: string;
  previewText?: string;
  pageCount?: number;
  characterCount?: number;
  truncated?: boolean;
};
```

保持兼容：

- 老消息没有这个字段，不影响读取
- 现有 `extractedText` 继续保留
- 现有 `insight` 继续承载状态、provider、错误

这样改的目的，是避免这次就引入新的 attachments table 或新实体。

## 4.6 大文档与 prompt 注入策略

这是本次必须一起规划的点，否则“能抽正文”仍然不等于“能稳定理解”。

推荐规则：

- `<= 8k chars`：可以直接进入 `attachment.extractedText`
- `> 8k chars`：仍然保存完整抽取结果，但 prompt 里只注入：
  - `previewText`
  - 文档头部片段
  - 命中关键词的相关片段

第一版不要引入复杂向量检索，直接做轻量 lexical chunk selection：

1. 按段落 / 页码切 chunk
2. 用当前用户消息做关键词切分
3. 选 top K 重叠 chunk
4. 拼到 `buildMessagePromptText()` 里

建议新增 helper：

- `api/src/modules/chat/document-chunk-selection.ts`

这样可以避免大 PDF 直接把 prompt 撑爆。

## 4.8 验收标准

最小验收样本：

- 一份文本型 PDF
- 一份扫描或空文本 PDF
- 一份 `docx`

通过标准：

- 文本型 PDF：`MediaInsightJob.status=completed`，`attachment.extractedText` 非空
- 扫描或空文本 PDF：不能静默成功；要么明确失败，要么明确进入 OCR 待办语义
- `docx`：`attachment.extractedText` 非空
- 聊天回复能引用正文，而不是只引用文件名
- 后台 `chat-records` 能看到 `previewText / parser / extractionMode`

## 5. 多模态回复记忆闭环的实现方案

## 5.1 本轮真正要补的不是“所有多模态”，而是“artifact image 的语义记忆”

当前语音 sibling 已有文字主回复，所以语义基本存在。

真正断掉的是：

- 单聊异步图片回复
- 群聊异步图片回复

因此这轮的主目标是：

- 任何异步图片回复都要留下稳定的 `historyText`
- 后续下一轮对话必须能知道“刚才发的图是什么”

## 5.2 最小数据设计

不改消息表结构，直接在 attachment JSON 里加一层：

```ts
generatedContext?: {
  sourceReplyArtifactJobId?: string;
  sourceMessageId?: string;
  historyText?: string;
  imagePrompt?: string;
};
```

这个字段建议加到 `ImageAttachment`，必要时也可加到 `VoiceAttachment`，但本轮先服务图片。

`historyText` 的构造规则建议固定：

- direct image：`随后发来一张图片，内容大致是：${trimmedImagePrompt}`
- group image：`[${characterName}] 随后发来一张图片，内容大致是：${trimmedImagePrompt}`

长度控制：

- `imagePrompt` 保留原始 prompt，用于后台复盘
- `historyText` 只保留 120-180 字左右的摘要，避免污染历史

## 5.3 ReplyArtifactJob 在完成时要做的事情

在：

- `processConversationImageJob()`
- `processGroupImageJob()`

新增 3 件事：

1. 生成 attachment 时写入 `generatedContext`
2. 落库消息后，为 direct conversation 失效内存历史缓存
3. artifact job 的 `resultPayload` 同时记录 `historyText` 和 `imagePrompt`

这样可以同时解决：

- DB 重建历史丢语义
- 运行中 conversation cache 不刷新
- 后台无法复盘图片产物来自哪个 prompt

## 5.4 direct chat 的历史重建规则调整

在 `api/src/modules/chat/chat.service.ts`：

- `ensureConversationHistory()`
- `shouldIncludeAssistantMessageInHistory()`

做两类调整：

### 调整 A：assistant attachment 不能再只看 `text`

如果满足：

- `senderType === 'character'`
- `attachment.generatedContext.historyText` 存在

则即使 `message.text === ""`，也必须纳入 history。

### 调整 B：assistant attachment 的 history content 用 `historyText`

优先级建议：

1. `attachment.generatedContext.historyText`
2. `message.text`
3. 通用 fallback

不要再让异步图片回复只剩：

- `发来一张图片，文件名：chat-reply-image.png`

这种无语义文本。

## 5.5 group chat 的历史构造规则调整

group 没有长期内存 map，但每轮会从 DB 重建历史。

因此要改：

- `api/src/modules/chat/group.service.ts`
- `buildAiHistoryMessage()`
- `buildMessagePromptText()`

规则同样是：

- 如果 attachment 里有 `generatedContext.historyText`
- 优先使用这段语义文本
- 不再使用纯文件名 fallback

这样群聊的下一轮跟进才知道前一位角色发的图是什么。

## 5.6 direct conversation 的运行时缓存处理

这是 direct chat 特有问题。

当前 artifact job 直接改 DB，不会同步更新：

- `ChatService.conversationHistory`

推荐最小改法：

- 在 `ChatService` 新增 `invalidateConversationHistory(conversationId: string)`
- `ReplyArtifactJobService.processConversationImageJob()` 完成后调用它

不建议本轮直接让 `ReplyArtifactJobService` 手动 append history entry，原因是：

- direct conversation cache 的格式和 rebuild 逻辑已经存在
- 失效再重建更简单
- 更不容易出现重复 append 或 turn 顺序错乱

如果出现循环依赖，可以抽一个极小的 runtime service，例如：

- `conversation-history-cache.service.ts`

但第一选择仍然是直接在 `ChatService` 暴露失效方法。

## 5.7 后台和前台可见性

本轮最小补充建议：

后台：

- `chat-records` 里图片附件可显示：
  - `generatedContext.historyText`
  - `generatedContext.imagePrompt`

前台：

- 前台不用新 UI
- 但消息详情 / 调试日志至少要能通过现有数据验证 payload 正确

因为这轮核心目标是“模型记住”，不是先做新的 UI 表达。

## 5.9 验收标准

最小验收：

1. 用户发送“给我画一张海边夜景图”
2. 角色先发文字
3. 异步图片落库
4. 用户紧接着发送“把刚才那张图改成下雨版本”

通过标准：

- direct chat 下一轮里，模型明确知道“刚才那张图”的语义是海边夜景，而不是只知道有一张 png
- group chat 中，其他角色下一轮也能知道前一位角色发图的大意
- 后台能看到该图片 artifact 的 `imagePrompt` 和 `historyText`

## 7. 风险与取舍

### 7.1 这次不建议先上 OCR

原因：

- OCR 一旦进来，马上会把依赖、耗时、队列策略都抬高
- 当前最紧的断点不是 OCR，而是“pdf/docx 静默退空”

因此建议：

- 第一版把扫描 PDF 明确标记成未抽到正文
- 不再伪装成“理解成功”
- OCR 作为下一轮独立补丁

### 7.2 这次不建议先新建表

原因：

- 这两件事情都可以通过 attachment JSON additive 方式先闭环
- 一旦动新表和迁移，节奏会明显变慢
- 当前主要缺的是 runtime semantics，不是表不够用

后续如果文档 chunk 和 artifact metadata 明显变大，再考虑拆表。

### 7.3 legacy `.doc` 必须明确产品语义

如果首版不支持 `.doc` 真解析，必须满足两件事：

- 后台有明确错误码
- 前端或接口返回里不再假装“理解中 / 已理解”

最差的情况不是“不支持”，而是“静默退空，模型胡猜”。

## 8. 本轮完成定义

本轮真正完成，必须同时满足：

- `pdf` 稳定抽正文
- `docx` 稳定抽正文
- 文档内容能进入聊天回复上下文
- 单聊异步图片回复能进入下一轮 direct 历史
- 群聊异步图片回复能以语义文本进入后续群历史
- 后台能复盘文档抽取模式和图片 artifact 记忆文本

如果只做到“能解析 PDF/DOCX”但没有补后续上下文记忆，不算完成。  
如果只补了图片 artifact 记忆，但文档仍然静默退空，也不算完成。
