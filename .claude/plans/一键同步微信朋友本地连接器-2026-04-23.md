# 一键同步微信朋友本地连接器完整实现规划

日期：2026-04-23

## 目标

把管理后台「一键同步微信朋友」从当前的“预留 HTTP 连接器 + 手动 JSON 导入”推进到可本地真实使用的完整能力：

- 本地启动一个 `127.0.0.1:17364` 微信连接器服务。
- 管理后台可探测连接器、扫描本地微信数据源、列出联系人、批量生成 `WechatSyncContactBundle[]`。
- 用户确认后复用现有 `/api/admin/wechat-sync/preview` 与 `/api/admin/wechat-sync/import`，生成隐界角色并建立好友关系。
- 全链路默认本地化、显式授权、可审计、可回滚，不把原始聊天记录上传到远端。

## 现状判断

当前仓库已有：

- `apps/admin/src/lib/wechat-local-connector.ts`：前端 HTTP 客户端，默认连接 `http://127.0.0.1:17364`。
- `apps/admin/src/routes/wechat-sync-page.tsx`：管理后台页面，支持连接器读取和手动 JSON 导入。
- `api/src/modules/admin/wechat-sync-admin.service.ts`：接收联系人快照，生成角色预览、导入角色、建立好友关系、记录导入快照和回滚。
- `packages/contracts/src/wechat-sync.ts`：`WechatSyncContactBundle` 契约。

当前仓库缺失：

- 没有真正监听 `17364` 的本地连接器服务。
- 没有微信数据源发现、解密、导出、摘要、联系人聚合逻辑。
- 没有本地连接器启动脚本、状态管理、日志、测试夹具。

## 外部项目取舍

参考项目：

- `ylytdeng/wechat-decrypt`：微信 4.x 本地数据库解密工具，可从运行中的微信进程内存提取 SQLCipher 4 密钥，解密数据库并提供 Web UI / HTTP API。
- `hellodigua/ChatLab`：本地聊天记录分析桌面 App，Electron/Vue 架构，支持 WeChat 等多平台导入与统一数据模型。

取舍结论：

- 不再把许可证作为实施阻塞项：在合作前提下，`wechat-decrypt` 与 `ChatLab` 均可作为上游组件、代码来源或能力来源。
- `wechat-decrypt` 作为微信本地数据采集主路径：负责微信路径发现、密钥提取、数据库解密、联系人/标签/会话/消息读取；连接器支持 HTTP 调用、Python bridge、文件读取和后续 vendor/adapter 四种接法。
- `ChatLab` 作为统一聊天数据模型与导入 pipeline 的重要参考：优先复用/改写 ChatLab JSON、JSONL、WeFlow parser，而不是复刻完整桌面应用。
- 本项目最终边界仍是 `WechatSyncContactBundle`：所有第三方来源都先归一到隐界连接器协议，再进入 Core API 预览与导入链路。

## 产品边界

必须做：

- 本地服务只绑定 `127.0.0.1`，不监听局域网地址。
- 首次使用必须明确告知：读取的是用户本机微信数据或用户自己导出的聊天数据。
- 默认只输出联系人画像快照和代表性样本，不把全量原始消息写入 Core API。
- 支持 dry-run：只扫描、只预览、不导入。
- 支持数据最小化：用户可选择样本条数、时间范围、排除群聊、排除敏感联系人。
- 支持删除连接器缓存、清理本地索引。

明确不做：

- 不做后台静默读取。
- 不做未授权账号数据读取。
- 不把微信密钥、解密数据库、原始聊天库上传到服务端。
- 不默认绕过系统权限；需要管理员权限或外部工具时必须由用户显式执行。

## 本地连接器安全模型

### 绑定与 CORS

- 默认 `host = 127.0.0.1`，默认 `port = 17364`。
- CORS 只允许：
  - `http://127.0.0.1:5181`
  - `http://localhost:5181`
  - 后续桌面壳内置管理后台 origin
- 禁止 `*`。

### Pairing Token

首次启动连接器生成本地 token：

```text
%APPDATA%/Yinjie/wechat-connector/config.json
```

`GET /health` 可匿名返回：

```json
{
  "ok": true,
  "version": "0.1.0",
  "requiresPairing": true,
  "pairingHint": "请在本地终端复制配对码"
}
```

敏感接口必须带：

```text
X-Yinjie-Connector-Token: <token>
```

管理后台增加“连接器配对码”输入，保存到 localStorage。

### 本地缓存

缓存目录：

```text
%APPDATA%/Yinjie/wechat-connector/cache/
```

缓存内容：

- 联系人索引摘要。
- 消息采样摘要。
- 最近扫描状态。

默认不缓存：

- 解密密钥。
- 原始数据库副本。
- 全量原始聊天记录。

## 数据源 Provider 设计

统一接口：

```ts
interface WechatDataProvider {
  key: string;
  label: string;
  detect(): Promise<ProviderDetectionResult>;
  scan(options: ScanOptions): Promise<ScanResult>;
  listContacts(query: ContactQuery): Promise<ConnectorContactSummary[]>;
  buildBundles(usernames: string[]): Promise<WechatSyncContactBundle[]>;
}
```

### Provider 1：manual-json

目的：

- 保留现有“选择本地 JSON / 粘贴 JSON”能力。
- 让连接器也能读取本地 `WechatSyncContactBundle[]` 文件，并统一提供 `/api/contacts` 与 `/api/contact-bundles`。

输入：

- `WechatSyncContactBundle[]`
- 或更宽松的本地导出 JSON，经过 normalize 后转为 bundle。

优先级：

- P0 必做。
- 最快让 `17364` 服务真实可达。

### Provider 2：chatlab-export

目的：

- 支持用户从 ChatLab 或类似工具导出的标准化聊天数据。
- 可复用/改写 ChatLab parser，重点支持 ChatLab JSON、ChatLab JSONL、WeFlow 导出和通用聊天 JSON/CSV。

输入候选：

- JSON/CSV 会话导出。
- 字段映射配置：
  - `contactId`
  - `displayName`
  - `sender`
  - `timestamp`
  - `text`
  - `isGroup`

处理：

- 流式读取大文件。
- 聚合联系人。
- 为每个联系人生成消息计数、双方发言比例、最近消息时间、代表性样本。

优先级：

- P1。适合合法导出路径，风险低。

### Provider 3：wechat-decrypt-http

目的：

- 适配用户外部运行的 `wechat-decrypt`。
- 作为第一阶段低耦合集成路径，优先通过 `wechat-decrypt` 的本地 HTTP/Web 能力读取用户授权产生的数据；后续同一 provider contract 可切换到 Python bridge 或内置 adapter。

外部预期：

- 用户另行运行 `python main.py`，默认 Web UI/API 为 `localhost:5678`。
- README 中公开的 API 包括 `/api/history`、`/api/tags`、`/stream`。

适配策略：

- 连接器配置 `wechatDecryptBaseUrl = http://127.0.0.1:5678`。
- `scan()` 拉取最近会话/历史与标签，构建联系人索引。
- 如果外部 API 无法直接列联系人，则从历史消息中的 chat/session 反推联系人摘要。
- 对大数据量采用分页和增量 `since`。

风险：

- `wechat-decrypt` API 不一定稳定。
- 其核心能力涉及微信数据库解密和进程内存读取，需要用户自己显式运行。
- 不在本项目内隐藏或自动执行这一步。

优先级：

- P2。是“一键本地真实读取微信”的关键适配器，但必须清楚边界。

### Provider 4：wechat-decrypt-files

目的：

- 读取 `wechat-decrypt` 生成的解密输出目录或导出文件。

输入：

```text
decrypted_dir/
all_keys.json
config.json
```

策略：

- 只读取用户指定目录。
- 只解析必要表或中间导出。
- 先做只读连接，不复制数据库。

优先级：

- P3。数据库 schema 变化风险高，放在 HTTP adapter 后面。

## `WechatSyncContactBundle` 生成规则

已有契约：

```ts
interface WechatSyncContactBundle {
  username: string;
  displayName: string;
  nickname?: string | null;
  remarkName?: string | null;
  region?: string | null;
  source?: string | null;
  tags: string[];
  isGroup: boolean;
  messageCount: number;
  ownerMessageCount: number;
  contactMessageCount: number;
  latestMessageAt?: string | null;
  chatSummary?: string | null;
  topicKeywords: string[];
  sampleMessages: WechatSyncMessageSample[];
  momentHighlights: WechatSyncMomentHighlight[];
}
```

生成策略：

- `username`：来源平台稳定 ID；没有则使用 `wechat:<hash(displayName + source)>`。
- `displayName`：优先备注名，其次昵称，其次会话名。
- `tags`：来自微信标签、ChatLab 标签、文件夹/导出标签，统一去重。
- `messageCount`：总有效文本消息数。
- `ownerMessageCount` / `contactMessageCount`：按方向聚合。
- `sampleMessages`：默认最多 16 条，覆盖最近消息、高频话题、情绪强度、问答样本。
- `chatSummary`：P0 使用启发式摘要；P1 可接本地 LLM 或 Core API 预览阶段生成。
- `topicKeywords`：先用 TF-IDF/词频启发式，后续可接本地模型。
- `momentHighlights`：除非数据源明确提供，否则为空。

敏感信息处理：

- 默认过滤手机号、身份证、银行卡、地址、验证码、URL token。
- UI 提供“显示原文样本 / 使用脱敏样本”的开关。
- 默认导入到 Core API 的样本为脱敏版本。

## 关键风险

### 微信数据来源风险

风险：

- 微信版本变化导致数据库 schema 或加密方式变化。
- 外部工具需要管理员权限。
- 用户误以为项目偷偷读微信。

处理：

- 默认不内置内存读取。
- 只做本地显式配置和外部工具适配。
- UI 强提醒“你正在读取本机数据”。

### 上游集成维护风险

风险：

- `wechat-decrypt` 和 ChatLab 的上游结构、字段、导出格式可能变化。
- 本地 vendor/adapter 后需要持续跟踪上游 bugfix 和微信版本变化。
- 深度集成后调试边界会变复杂：问题可能来自微信版本、密钥提取、数据库 schema、parser、连接器聚合或 Core API 导入。

处理：

- 记录每次同步的上游 repo、commit hash、改写文件和本地 patch 说明。
- 连接器 provider 层保持可替换：HTTP、Python bridge、files、vendor adapter 都输出同一个 `ParsedChatSession` / `WechatSyncContactBundle`。
- 为每个上游格式保存最小 fixtures，升级上游或微信版本时先跑 parser/normalizer/bundle-builder 的最小验证。
- UI 诊断页明确标出当前数据来源、provider、上游版本、最近 scanId 和失败阶段。

### 隐私风险

风险：

- 原始聊天记录、敏感信息进入 Core API。

处理：

- 默认摘要化和脱敏。
- 导入前展示样本。
- 允许用户删除样本或改成只导入联系人资料。
- 本地缓存可一键清理。

## 推荐实现顺序

1. 先做 P0，解决当前 `17364` 不存在的问题。
2. 再做 P1，让用户通过本地导出文件完成真实扫描，不依赖解密工具。
3. 再做 P2，适配外部 `wechat-decrypt`，实现真正接近“一键”的本机微信读取。
4. 最后做 P3，把任务进度、脱敏、缓存、审计做扎实。

## 最终用户流程

### 无解密工具路径

1. 启动管理后台和本地连接器。
2. 在连接器配置中选择 ChatLab/通用聊天导出文件。
3. 点击“扫描本地数据”。
4. 在联系人列表选择要导入的人。
5. 生成角色预览。
6. 编辑确认。
7. 导入并建立好友关系。

### 外部 wechat-decrypt 路径

1. 用户自行启动 `wechat-decrypt`。
2. 启动 `pnpm dev:wechat-connector`。
3. 管理后台配对连接器。
4. 填入 `wechat-decrypt` 本地地址。
5. 点击“扫描微信联系人”。
6. 选择联系人并生成预览。
7. 导入。

## 成功标准

- 用户不再看到“服务根本不存在”的死路。
- 没有外部解密工具时，也能通过文件导入完成真实本地同步。
- 有外部 `wechat-decrypt` 时，可以通过本地连接器完成联系人列表扫描和批量导入。
- Core API 只接收经过用户确认的 `WechatSyncContactBundle[]`。
- 整个链路有清晰权限、日志、缓存清理和回滚能力。

## 关键上游能力映射

### wechat-decrypt 可直接复用的能力

从当前上游结构看，可复用/适配的关键文件：

- `main.py`：一键入口，负责检测微信进程、确保密钥、启动 Web UI 或解密全部数据库。
- `config.py`：自动检测 `db_storage` 路径，Windows 读取 `%APPDATA%\Tencent\xwechat\config\*.ini` 并定位 `xwechat_files\*\db_storage`。
- `find_all_keys.py` / `find_all_keys_windows.py` / `find_all_keys_linux.py` / `find_all_keys_macos.c`：平台密钥扫描入口。
- `decrypt_db.py`：SQLCipher 4 解密逻辑，AES-256-CBC、HMAC-SHA512、page size 4096、reserve 80。
- `monitor_web.py`：Web API、SSE、联系人标签、实时消息、WAL patch、表情/图片处理。
- `mcp_server.py`：更适合我们复用的数据查询抽象，包含联系人、会话、消息搜索、标签、图片等工具函数。

## Provider 细化

### `manual-json.provider`

输入：

- config.filePath。
- 或 `POST /api/manual-json/load` 上传/传入 JSON 文本（可选后续）。

流程：

1. 读取 JSON。
2. 校验是否为 `WechatSyncContactBundle[]`。
3. 每个 bundle 写入 `contact_summaries`。
4. `contact_bundle_cache` 直接保存原 bundle。

用途：

- 让 `17364` 端口 P0 即可可用。
- 也作为测试 fixture provider。

### `chatlab-json.provider`

输入：

- ChatLab `.json` 文件。

映射：

- `meta.name` -> group/direct display fallback。
- `members[].platformId` -> `username`。
- `members[].accountName` / `groupNickname` -> `displayName`。
- `messages[].sender` -> `senderPlatformId`。
- `messages[].timestamp` 秒级时间戳。
- `messages[].content` 文本。

直接聊天策略：

- 如果 `meta.type === direct`，联系人为非 owner 成员。
- 如果没有 ownerId，用发言比例和成员数推断，仍不确定则要求 UI 指定“哪个是我”。

群聊策略：

- 默认 `isGroup = true`，可作为群聊导入候选但现有后端会跳过群聊角色导入。
- 后续可扩展为“从群成员中抽取个人联系人”。

### `chatlab-jsonl.provider`

输入：

- ChatLab `.jsonl` 文件。

流程：

1. 第一行读取 `_type=header`。
2. `member` 行建立成员索引。
3. `message` 行流式聚合，不把全量读入内存。
4. 每 5000 条消息 flush 聚合进 cache。

优势：

- 支持 GB 级文件。
- 是大规模聊天记录的优先推荐格式。

### `weflow.provider`

输入：

- WeFlow `.json`。

字段映射：

- `session.wxid` -> 会话 ID。
- `session.displayName` / `session.nickname` / `session.remark` -> 联系人显示。
- `session.type === '群聊'` -> `isGroup = true`。
- `messages[].isSend === 1` -> direction `owner`。
- `messages[].isSend === 0` -> direction `contact` 或 `group_member`。
- `messages[].isSend === null` -> direction `system`。
- `messages[].senderUsername` -> sender id。
- `messages[].senderDisplayName` -> sender display。
- `messages[].createTime` -> timestamp。
- `messages[].type` -> typeLabel。
- `messages[].content` -> text。

个人聊天 bundle：

- `username = session.wxid`
- `displayName = session.displayName || session.remark || session.nickname || session.wxid`
- `sampleMessages` 从 messages 采样。

群聊 bundle：

- P1 只显示为群聊候选，不默认导入角色。
- P2 增加“从群聊抽取成员”功能：按 `senderUsername` 聚合群成员发言，生成个人候选。

### `wechat-decrypt-http.provider`

前提：

- 用户已运行 `python main.py`，上游默认 `http://localhost:5678`。

可用上游 API：

- `GET /api/history?limit=...`
- `GET /api/history?chat=...&since=...&limit=...`
- `GET /api/tags`
- `GET /stream`，暂不作为扫描主路径。

缺口：

- 上游 `monitor_web.py` 的 `/api/history` 更偏实时消息日志，未必包含完整联系人。
- 上游 `mcp_server.py` 有更适合的 `get_contacts` / `get_chat_history` / `get_contact_tags`，但它是 MCP stdio，不是 HTTP。

实现两层：

1. 轻量 HTTP 适配：从 `/api/history` 和 `/api/tags` 构建最近联系人索引。
2. 深度 Python 适配：直接调用/封装 `mcp_server.py` 的联系人和聊天查询函数。

P1 先做轻量 HTTP：

- `GET /api/tags` -> tags。
- `GET /api/history?limit=2000` -> 最近联系人。
- 适合“最近聊过的人”导入。

P2 做深度 Python：

- 使用同一 `config.json`、`all_keys.json`、`decrypted_dir`。
- 调用 Python 脚本输出 JSON：

```bash
python yinjie_bridge.py contacts --query "" --limit 5000 --json
python yinjie_bridge.py history --chat wxid_xxx --limit 500 --json
python yinjie_bridge.py tags --json
```

## wechat-decrypt 数据库映射细化

联系人：

- DB：`contact/contact.db`
- 表：`contact`
- 字段：
  - `username`
  - `nick_name`
  - `remark`
  - `extra_buffer`
- 映射：
  - `username` -> contact username。
  - `remark || nick_name || username` -> displayName。
  - `remark` -> remarkName。
  - `nick_name` -> nickname。

标签：

- DB：`contact/contact.db`
- 标签定义表：`contact_label`
- 字段：
  - `label_id_`
  - `label_name_`
  - `sort_order_`
- 联系人关联：
  - `contact.extra_buffer` protobuf field #30，逗号分隔 label id。
- 映射：
  - label id -> label name。
  - contact tags -> `WechatSyncContactBundle.tags`。

最近会话：

- DB：`session/session.db`
- 表：`SessionTable`
- 字段：
  - `username`
  - `unread_count`
  - `summary`
  - `last_timestamp`
  - `last_msg_type`
  - `last_msg_sender`
  - `last_sender_display_name`
- 映射：
  - `username` -> 联系人 id。
  - `last_timestamp` -> latestMessageAt。
  - `summary` -> sampleSnippet fallback。

消息：

- DB：`message/message_0.db` 到 `message/message_4.db`，以及资源库。
- 索引表：`Name2Id`
- 做法：
  - 先构建 `username -> message db keys` 映射。
  - 对指定 username 找消息表。
  - 读取最近 N 条文本/富文本可读消息。
  - 群聊内容若包含 `sender:\ntext`，拆出群成员 sender。

消息类型：

- `1` 文本。
- `3` 图片。
- `34` 语音。
- `43` 视频。
- `47` 表情。
- `49` 链接/文件/小程序/引用。
- `10000` 系统。
- `10002` 撤回。

P0/P1 只导入可读文本摘要：

- 文本消息。
- appmsg 中可解析 title/description 的卡片。
- 引用消息转成简短文本。
- 图片/语音/视频只保留类型提示，不进入 `sampleMessages.text`，除非有 OCR/ASR 后续能力。

## 聚合算法细化

### 联系人评分

用于列表默认排序：

```ts
score =
  log10(messageCount + 1) * 20
  + recencyScore(latestMessageAt)
  + bidirectionalScore(ownerMessageCount, contactMessageCount)
  + tagBonus
  - groupPenalty
  - sparsePenalty;
```

### 采样策略

目标：少量样本能代表“说话方式”和“关系上下文”。

每个联系人默认 16 条样本：

- 最近样本 6 条。
- 高频话题样本 4 条。
- 联系人主动发言 4 条。
- 用户和联系人连续问答 2 组。

过滤：

- 去掉系统消息。
- 去掉过短消息：长度 < 2 的纯表情/语气词可降权。
- 去掉重复消息。
- 去掉明显敏感内容或脱敏后保留。

### 摘要策略

P0 启发式摘要：

- “你和 X 最近主要聊 A/B/C，消息量 N，最近互动在 T。”
- 不调用 AI，保证连接器独立。

P1 可选 AI 摘要：

- 管理后台调用 Core API preview 时，由现有 `wechat-sync-admin.service.ts` 的 AI 生成角色草稿。
- 连接器不直接依赖远端模型。

### 关键词提取

P0：

- 中文分词可先不用重依赖，采用字符 n-gram + 停用词 + 高频名词短语启发式。
- 保留 8 个关键词。

P1：

- 引入轻量分词库或自维护中文停用词。
