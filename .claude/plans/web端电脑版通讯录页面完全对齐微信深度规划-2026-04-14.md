# Web 端电脑版通讯录页面完全对齐微信深度规划

## 任务目标
- 将桌面 Web 端 `通讯录` 从当前“项目化双栏联系人页”收敛为“微信电脑端通讯录工作区”。
- 对齐的重点不是单个按钮，而是整页的信息架构、视觉层级、操作肌肉记忆、工作区连续性。
- 保留隐界业务语义，但不让隐界扩展入口破坏微信主心智。
- 不直接复制腾讯素材、插画、品牌图标，只复刻结构、节奏、交互和排版逻辑。

### 对标结论
- 微信桌面端通讯录的重点不是“做出联系人列表”。
- 真正需要抄的是四层：
  - 微信标准入口组与联系人目录分层
  - 右侧联系人详情页的克制排版
  - 通讯录内部各二级能力仍停留在同一工作区语境内
  - 选择、回退、搜索、双击发消息这类桌面行为的一致性

## 当前实现现状

### 已经有的能力，不要误判成缺失
- 朋友列表已支持：
  - 备注名优先展示
  - 拼音首字母分组
  - 搜索匹配昵称、地区、来源、标签
  - 星标状态展示
- 桌面右侧详情已支持：
  - 发消息
  - 朋友圈入口
  - 共同群聊入口
  - 备注和标签编辑
  - 星标、置顶、免打扰、黑名单、删除联系人
- 关联能力已具备基础数据：
  - 好友申请
  - 已保存群聊
  - 公众号
  - 世界角色
  - 会话置顶与免打扰
  - 黑名单状态

### 现状问题，按代码落点拆解

#### 1. `contacts-page.tsx` 过重，桌面和移动逻辑耦在一起
- 当前文件同时负责：
  - 六类 query
  - 多组 mutation
  - 桌面和移动两套 UI
  - 列表构造
  - 桌面详情路由动作
  - 快捷入口配置
- 这会让“照微信重构桌面通讯录”时每一步都被移动端兼容拖住。

#### 2. 桌面态选择状态只有本地 state，没有工作区级深链
- `desktopSelection` 仍是局部 state，刷新和回退不能稳定恢复当前联系人。
- 当前 `新的朋友 / 星标朋友 / 群聊 / 标签 / 公众号` 也都是跳新路由，不是统一的通讯录内工作区状态。
- 结果是：
  - 视觉上像一个模块
  - 交互上却像五六个散页拼起来

#### 3. 微信标准入口和隐界扩展入口混在同一主入口层
- 当前桌面顶部入口组里同时出现：
  - 添加朋友
  - 新的朋友
  - 星标朋友
  - 群聊
  - 标签
  - 公众号
  - 世界角色
- `世界角色` 现在和微信标准联系人入口同权，破坏了“这是微信通讯录”的第一屏心智。

#### 4. 左侧视觉仍然是项目风格，不是微信电脑端
- 当前快捷入口使用高饱和渐变色块、厚圆角和卡片阴影。
- 当前好友列表也被包进一整块大圆角卡片。
- 微信电脑端的通讯录更平、更轻、分割线更明确，不靠彩色模块建立信息层级。

#### 5. 右侧联系人详情还是“角色资料卡”逻辑，不是微信联系人详情
- 现在的右侧结构虽然已比早期更接近微信，但本质还是：
  - 大头像头部
  - 多段 section 卡片
  - 较重的“资料展示感”
- `contact-detail-pane.tsx` 仍然在用一个通用 profile shell 拼接多个资料区，更像“通用档案详情页”而不是微信联系人详情页。

#### 6. 桌面通讯录二级能力实现分散，缺乏统一工作区
- `星标朋友 / 群聊 / 标签 / 公众号 / 好友申请` 都各自有桌面版页面。
- 这些页面各自维护自己的：
  - 搜索框
  - 左右布局
  - 选中状态
  - 空态
- 用户感知会是“我离开了通讯录，再进入另一个页面”，这和微信桌面端连续工作区感不一致。

#### 7. 搜索模型还是“联系人页搜索”，不是“通讯录工作区搜索”
- 当前桌面搜索只服务当前联系人与世界角色目录。
- 标准入口项、群聊、标签、公众号、好友申请并没有被纳入统一搜索结果。
- 微信桌面端的真实感来自：
  - 你在通讯录里搜到的是通讯录语境中的所有东西
  - 而不是“只搜索当前列表里的人”

#### 8. 缺少桌面态选择与操作的肌肉记忆
- 当前联系人只支持：
  - 单击选中
  - 双击发消息
- 还缺少更微信式的桌面细节：
  - 上下键切换联系人
  - `Enter` 打开聊天
  - 右键快捷菜单
  - 搜索后 `Esc` 清空或关闭结果态

#### 9. 联系人详情与聊天信息侧栏还不是同一套人设
- 通讯录右侧能看到：
  - 备注
  - 标签
  - 地区
  - 来源
  - 朋友圈
  - 共同群聊
- 聊天里的联系人信息侧栏则走另一套展示节奏。
- 如果通讯录要“完全像微信”，两个地方必须看起来像同一个联系人系统，而不是两个团队各写一套。

## 目标产品定义

### 一句话目标
- 进入桌面 `通讯录` 后，用户感知应该是“微信电脑端联系人工作区”，而不是“隐界联系人功能集合页”。

### 目标结构
- 仍保留桌面应用外层左侧一级导航。
- `通讯录` 内部收敛为统一双栏工作区：
  - 左侧：微信式通讯录导航与联系人目录
  - 右侧：当前选中入口对应的工作区内容

### 目标原则
- 微信标准入口优先，隐界扩展语义降权。
- 所有桌面二级联系人能力尽量回收到同一工作区，不到处跳散页。
- 右侧内容优先用“资料页”思路，而不是“运营卡片”思路。
- 只在必要处使用色彩，主要靠排版、留白、分割线和选中态建立层级。

## 目标信息架构

### 左侧结构

#### 1. 顶部固定栏
- 标题：`通讯录`
- 右上不再放彩色功能卡
- 搜索框保留，但样式改成微信式扁平搜索框
- 搜索框只承担当前工作区检索，不承担全局搜一搜

#### 2. 微信标准入口组
- 建议统一为扁平入口行，而不是大卡片按钮：
  - 新的朋友
  - 群聊
  - 标签
  - 公众号
- `星标朋友` 处理建议：
  - 不再作为彩色高权入口卡
  - 改成通讯录内部的一个标准分组入口或次级入口
  - 如果保留独立入口，也必须是和其他标准入口同一扁平风格

#### 3. 联系人目录区
- 按拼音首字母分组
- 支持选中态、hover、键盘切换
- 选中态不再用绿色描边卡片，而是微信式浅灰底 + 左侧轻强调
- 备注名优先，昵称作为副文案

#### 4. 隐界扩展区
- `世界角色` 必须下沉，不再进入微信标准入口组。
- 建议两种方案二选一：
  - 方案 A：作为目录底部一个折叠区，标题显式标注 `世界角色（隐界扩展）`
  - 方案 B：完全从主通讯录剥离，只保留在 `添加朋友` 或专门工作区中进入
- 推荐方案：先做方案 A，风险最低，迁移最平滑。

### 右侧工作区类型
- `friend-detail`：好友详情页
- `new-friends`：新的朋友工作区
- `starred-friends`：星标朋友工作区
- `groups`：群聊工作区
- `tags`：标签工作区
- `official-accounts`：公众号工作区
- `world-characters`：世界角色工作区
- `empty`：默认空态

### 关键结论
- 对桌面端来说，`/tabs/contacts` 应该是一个工作区路由，不应该继续让二级入口主要依赖多个平行页面拼装体验。
- 移动端可以继续保持当前独立路由，因为手机端本来就是层级跳转式体验。

## 路由与状态模型

### 当前问题
- 现在桌面通讯录没有工作区内深链模型。
- `desktopSelection`、搜索词、子工作区都不在可还原路由态里。

### 目标模型
- 为桌面通讯录引入独立 route-state：
  - `apps/app/src/features/desktop/contacts/desktop-contacts-route-state.ts`

### 建议结构
```ts
export type DesktopContactsRouteState =
  | { pane: "friend"; characterId: string }
  | { pane: "new-friends" }
  | { pane: "starred-friends"; characterId?: string }
  | { pane: "groups"; groupId?: string }
  | { pane: "tags"; tag?: string; characterId?: string }
  | { pane: "official-accounts"; accountId?: string }
  | { pane: "world-characters"; characterId?: string };
```

### 路由策略
- 桌面态优先把子页面收回到 `/tabs/contacts#...` 内部。
- 移动端维持现有：
  - `/contacts/starred`
  - `/contacts/groups`
  - `/contacts/tags`
  - `/contacts/official-accounts`
  - `/friend-requests`
- 这样不会伤到移动端既有导航，同时桌面端能获得微信式一体工作区。

### 好处
- 刷新后仍可回到当前选中的联系人或子工作区。
- 浏览器回退符合用户预期。
- 从聊天、朋友圈、好友详情回跳时，可以精确落回刚才那个联系人。

## 视觉与交互目标

### 左侧视觉
- 去掉大面积彩色渐变入口块
- 去掉整块圆角大卡片包裹式联系人列表
- 分层改为：
  - 顶部搜索
  - 标准入口行
  - 分组标题
  - 联系人列表行
- 分割线和背景深浅变化应成为主要层级手段

### 右侧视觉
- 头部更紧凑
- 字段行更像微信资料页，不像后台表单
- 朋友圈、共同群聊、更多资料、标签等入口统一成“行级动作”
- 危险操作区放底部，弱化但清晰

### 交互细节
- 单击：选中右侧详情
- 双击：打开聊天
- 上下键：切换联系人
- `Enter`：打开聊天
- `Esc`：
  - 搜索态时清空搜索或关闭结果态
  - 子工作区详情态时回到列表默认选中
- 鼠标右键联系人建议补：
  - 发消息
  - 设为星标 / 取消星标
  - 查看资料
  - 删除联系人

## 二级工作区的统一策略

### 新的朋友
- 桌面态改成通讯录右侧工作区之一，不再让用户感觉离开通讯录。
- 展示申请列表、待处理数量、接受后自动切回新好友详情。

### 星标朋友
- 不再作为独立视觉系统的二级页。
- 保持左侧可选入口，右侧仍用相同的联系人目录 + 详情工作区骨架。

### 群聊
- 桌面群聊页现有左右布局可复用，但要接入统一的通讯录工作区壳层。
- 左侧切换为 `群聊目录`，右侧显示群资料与进入会话动作。

### 标签
- 标签页也需要接入统一壳层。
- 推荐右侧采用“双层选择”：
  - 左侧标签组
  - 右侧联系人详情
- 但整体仍然包在同一个桌面通讯录工作区里。

### 公众号
- 现有公众号工作区保留内容能力，但入口和外壳统一归口到通讯录工作区。
- 用户感知应是“我还在通讯录里看公众号”，不是切到另一个产品区。

### 世界角色
- 不进入标准入口组。
- 作为通讯录底部折叠扩展或添加朋友后的次级入口。
- 右侧依旧可以展示角色资料，但文案和布局要明确这是隐界扩展能力。

- 搜索结果不直接把所有类型混成一锅。
- 建议分组展示：
  - 联系人
  - 群聊
  - 标签
  - 公众号
  - 世界角色

## 数据与接口规划

### 前端第一阶段可直接复用现有接口
- `getFriends`
- `listCharacters`
- `getFriendRequests`
- `getSavedGroups`
- `getBlockedCharacters`
- `getConversations`

### 第一阶段不要急着改后端
- 先通过前端工作区重构把桌面通讯录体验做对。
- 现有数据已经足够支撑 80% 的微信化收口。

### 第二阶段建议新增桌面聚合数据层
- 如果桌面通讯录在重构后仍存在明显请求分散和状态拼装复杂度，可考虑新增：
  - 前端聚合 hook：`useDesktopContactsBootstrap`
  - 或后端聚合接口：`GET /api/contacts/desktop-bootstrap`

### 聚合接口建议返回
- 好友列表
- 已保存群聊摘要
- 好友申请未处理数
- 公众号摘要
- 黑名单 id 集
- 会话摘要
- 世界角色摘要

### 是否必须做新接口
- 不是必须。
- 推荐顺序：
  - 先前端工作区拆分
  - 再看是否真的需要新接口

## 与聊天信息侧栏的协同

### 原则
- 通讯录详情和聊天信息侧栏最终必须共用一套联系人资料表达。

### 本次规划建议
- 桌面通讯录先做主样板。
- 第二波将：
  - `contact-detail-pane.tsx`
  - `desktop-chat-details-panel.tsx`
  共同抽到统一的联系人资料块组件。

### 收益
- 同一个人，在：
  - 通讯录
  - 单聊信息侧栏
  - 群成员资料入口
  看到的字段顺序、动作顺序和视觉节奏一致。

## 分阶段实施方案

### Phase 1：搭工作区骨架，不先碰后端
目标：
- 把桌面通讯录从一个巨型页面拆成工作区壳。

要做的事：
- 新增桌面通讯录 route-state
- 抽离 `DesktopContactsWorkspace`
- 把桌面态从 `contacts-page.tsx` 中迁出
- 保持当前功能都还能用

完成标准：
- 桌面端进入 `/tabs/contacts` 后仍能正常浏览好友与详情
- 刷新页面可以保留当前选中的 pane

### Phase 2：左侧完全微信化
目标：
- 把左侧从“彩色卡片集合”改成微信式联系人工作区。

要做的事：
- 快捷入口改为标准入口行
- 联系人列表去卡片化
- 选中态和 hover 态重写
- `世界角色` 下沉
- 搜索态支持工作区分组结果

完成标准：
- 首屏不再出现大块彩色快捷卡
- 用户第一眼能感知到这是微信式通讯录

### Phase 3：右侧联系人详情完全微信化
目标：
- 把右侧从 profile 卡收口成联系人资料页。

要做的事：
- 头部重排
- 信息区、动作区、设置区、危险区分层
- 朋友圈 / 共同群聊 / 更多资料做成标准动作行
- 去掉不必要的角色说明感

完成标准：
- 右侧详情页和微信电脑端联系人详情在阅读节奏上高度一致

### Phase 4：把二级页并回统一工作区
目标：
- 不再让桌面通讯录像多个平行页面的集合。

要做的事：
- `新的朋友`
- `星标朋友`
- `群聊`
- `标签`
- `公众号`
  逐步接入统一 `DesktopContactsWorkspace`

完成标准：
- 桌面端切换这些入口时，不再是“换一个页面风格”，而是“通讯录右侧切换工作内容”

### Phase 5：补足桌面肌肉记忆
目标：
- 把微信味从布局推进到操作细节。

要做的事：
- 键盘导航
- `Enter` 打开聊天
- `Esc` 行为
- 右键菜单
- 更稳的空态与错误态

完成标准：
- 鼠标和键盘都能高效操作通讯录

### Phase 6：统一联系人系统
目标：
- 让通讯录、聊天信息侧栏、好友详情页共享同一套联系人资料系统。

要做的事：
- 抽共享组件
- 统一字段顺序
- 统一动作顺序
- 删除桌面重复实现

完成标准：
- 联系人资料在桌面各入口保持同一个人设

## 风险与约束

### 风险 1：桌面和移动共用一个路由文件会拖慢重构
- 解决：
  - 尽早把桌面工作区抽出到独立文件

### 风险 2：一次性并回所有二级页改动面太大
- 解决：
  - 先收骨架和左侧
  - 再逐个并回子工作区

### 风险 3：隐界业务语义完全消失会损伤产品识别
- 解决：
  - 不删除 `世界角色`
  - 只把它从微信标准入口层降权

### 风险 4：直接改后端会扩大范围
- 解决：
  - 第一阶段只做前端工作区重构
  - 后端聚合接口后置

## 验收清单
- 进入桌面 `通讯录`，首屏感知接近微信电脑端，而非彩色卡片页
- 标准入口、联系人目录、隐界扩展区三层关系清晰
- `世界角色` 不再污染微信标准入口组
- 右侧联系人详情页结构收敛为微信式资料页
- `新的朋友 / 星标朋友 / 群聊 / 标签 / 公众号` 在桌面端具有统一工作区语义
- 刷新、回退、从其他页返回时，能稳定恢复当前通讯录上下文
- 通讯录详情与聊天信息侧栏后续可共用同一套联系人资料组件

## 建议开工顺序
1. 先拆 `DesktopContactsWorkspace` 和 route-state。
2. 再重写左侧结构，完成微信标准入口组与联系人目录。
3. 然后重排右侧联系人详情。
4. 再把 `新的朋友 / 星标朋友 / 群聊 / 标签 / 公众号` 逐个并回工作区。
5. 最后补键盘、右键、搜索分组和聊天信息侧栏统一。

## 最终结论
- 这次不是简单换皮，而是桌面通讯录模块级重构。
- 真正的收口方向不是继续给 `contacts-page.tsx` 叠功能，而是把桌面通讯录提升为独立工作区模型。
- 只要工作区模型对了，后续：
  - 星标朋友
  - 标签
  - 群聊
  - 公众号
  - 好友详情
  - 聊天信息侧栏
  都能沿着同一套微信桌面范式持续收敛。
