# web端电脑版聊天页搜索聊天记录微信化规划

## 1. 目标

- 把 `apps/app` Web 端电脑版聊天页里的“搜索聊天记录”能力，做成接近微信桌面版当前稳定心智的实现。
- 这轮规划只覆盖桌面聊天页内搜索，不覆盖全局 `搜一搜`，也不覆盖独立的“聊天记录管理”工具页。
- 交互上按微信桌面版抄：
  - 从当前会话进入
  - 在当前聊天工作区内完成
  - 群聊支持按日期、群成员筛选
  - 搜索结果能回到原消息上下文
- 不抄微信品牌资源；抄的是入口位置、面板层级、筛选节奏、结果排版、键盘操作和回跳方式。

## 2. 结论先行

- 当前仓库已经有两套“聊天记录搜索”雏形，但都不适合直接作为微信桌面版最终实现：
  - `apps/app/src/features/desktop/chat/desktop-chat-history-panel.tsx`
    - 是右侧历史面板
    - 但本质还是“最近消息 + 前端本地过滤”
  - `apps/app/src/features/chat/chat-message-search-panel.tsx`
    - 是移动/详情页风格的完整搜索页
    - 交互和桌面聊天页工作流不一致
- 因此本轮不建议继续在现有 UI 上修修补补，而应直接做“桌面聊天搜索专用面板 + 服务端检索 + 原消息上下文定位”。
- 首版不建议上新表或 FTS 虚拟表。先基于现有 `messages` / `group_messages` 做服务端搜索与分页，确认体验后再决定是否补 SQLite FTS5。

## 3. 当前现状

### 3.1 现有入口

- 顶部聊天头部按钮：
  - `apps/app/src/features/desktop/chat/desktop-chat-header-actions.tsx`
  - 已有 `查找聊天记录` 按钮
  - 当前只是切 `rightPanelMode === "history"`
- 聊天信息面板入口：
  - `apps/app/src/features/desktop/chat/desktop-chat-details-panel.tsx`
  - 单聊和群聊都已有 `搜索聊天记录 / 查找聊天记录` 行
  - 当前同样只是打开现有历史面板

### 3.2 现有桌面历史面板的问题

- 文件：
  - `apps/app/src/features/desktop/chat/desktop-chat-history-panel.tsx`
- 现状能力：
  - 拉最近 `80` 条消息，按步长继续扩大
  - 前端本地过滤
  - 结果最多只显示 `60` 条
  - 直接在面板里塞了：
    - 关键词输入
    - 类型 chips
    - 日期 chips
    - `input[type=date]`
    - 群成员 `select`
    - 统计 pill
    - `完整搜索` 按钮
- 这和微信桌面版差距很大：
  - 它是“过滤器工具栏”思路，不是微信的“查找聊天记录”面板思路
  - 它搜的不是整段历史，只是当前已加载消息
  - 它需要跳去另一套 `完整搜索` 页面，割裂了桌面聊天心流

### 3.3 现有完整搜索页的问题

- 文件：
  - `apps/app/src/routes/chat-message-search-page.tsx`
  - `apps/app/src/routes/group-message-search-page.tsx`
  - `apps/app/src/features/chat/chat-message-search-panel.tsx`
- 现状能力：
  - 支持关键词、日期、类型、群成员
  - 还支持图片/文件/链接分类
- 但问题是：
  - 这是移动端详情页式布局，不是桌面聊天页内工作流
  - 它把搜索做成了独立页面，不符合微信桌面版“仍在当前聊天上下文里查”的习惯
  - 桌面页如果继续复用它，只会得到“功能更多，但更不像微信”的结果

### 3.4 后端当前不支持真正的聊天记录检索

- 契约层：
  - `packages/contracts/src/client.ts`
  - `getConversationMessages(id, { limit })`
  - `getGroupMessages(id, { limit })`
- 控制器：
  - `api/src/modules/chat/chat.controller.ts`
  - 当前只收 `limit`
- 服务层：
  - `api/src/modules/chat/chat.service.ts`
  - `api/src/modules/chat/group.service.ts`
  - 当前只支持按会话取消息，不支持：
    - 关键词搜索
    - 日期范围搜索
    - 群成员过滤
    - 文件 / 图片 / 链接分类搜索
    - 搜索结果分页
    - 结果上下文窗口拉取

### 3.5 当前线程页其实已经有可复用基础

- 单聊：
  - `apps/app/src/features/chat/conversation-thread-panel.tsx`
- 群聊：
  - `apps/app/src/features/chat/group-chat-thread-panel.tsx`
- 已有能力：
  - `highlightedMessageId`
  - 定位到 `#chat-message-${messageId}`
  - 若目标消息未加载，会继续加载更早消息直到命中
- 结论：
  - “定位并高亮目标消息”这块不需要推倒重来
  - 真正缺的是“如何搜到目标”和“如何按目标消息加载上下文窗口”

## 4. 微信桌面版基线

### 4.1 可确认的稳定能力

- 2017-10-28，微信 Windows `2.6` 正式版已支持：
  - 搜索本地聊天记录
  - 按类型查看聊天消息
- 2023-05-17，微信 macOS `3.8.0` 官方更新日志已明确支持：
  - 按群成员和日期搜索聊天记录
- 2023-07-22，电脑报转述微信 Windows `3.9.6` 实测：
  - 在聊天页面点右上角 `...`
  - 进入查找聊天记录页面
  - 新增 `日期` 和 `群成员` 筛选

### 4.2 从这些信息反推，桌面版必须满足的产品心智

- 搜索是“当前会话能力”，不是全局工具页跳转。
- 群聊检索必须有：
  - `日期`
  - `群成员`
- 搜索不应只停留在文本关键字，还要有：
  - 类型/分类浏览
  - 原消息上下文回跳
- 排版必须比当前项目更克制：
  - 更像白灰桌面工作面板
  - 不是彩色统计 pill + 表单控件堆砌

## 5. 目标产品定义

## 5.1 入口

- 桌面聊天页保留两个入口，但都落到同一个搜索面板：
  - 聊天头部 `查找聊天记录`
  - 聊天信息面板里的 `查找聊天记录`
- 追加快捷键：
  - Windows / Linux：`Ctrl + F`
  - macOS：`Cmd + F`
- 快捷键只在当前聊天工作区生效，不劫持全局 `搜一搜`

## 5.2 面板层级

- 不跳全页，不再跳 `/chat/$conversationId/search`、`/group/$groupId/search`
- 不保留当前“历史面板 + 完整搜索页面”的双轨结构
- 目标结构改成微信式单轨：
  - 右侧面板就是“查找聊天记录”
  - 从聊天信息面板点进来时，属于二级面板下钻
  - 从头部按钮 / 快捷键进来时，直接打开搜索面板

## 5.3 排版定义

- 顶部：
  - 单行标题区
  - 左侧返回或关闭
  - 中间标题 `查找聊天记录`
- 搜索区：
  - 单个扁平搜索框
  - 自动聚焦
  - 不显示当前项目现有的统计 pill、筛选计数块、彩色标签块
- 默认内容区：
  - 纵向列表，按微信桌面版思路展示“搜索入口项”
  - 单聊建议：
    - 日期
    - 图片与视频
    - 文件
    - 链接
  - 群聊建议：
    - 日期
    - 群成员
    - 图片与视频
    - 文件
    - 链接
- 结果区：
  - 统一列表
  - 按日期分段
  - 每条结果展示：
    - 发送者
    - 时间
    - 摘要
    - 类型辅助信息
  - 文件结果附加：
    - 文件名
    - 扩展名 / 大小（有则显示）
  - 图片 / 视频结果附加：
    - 缩略提示或媒体标签

## 5.4 交互定义

### 进入态

- 从头部进入：
  - 直接打开搜索面板
  - 焦点进入搜索框
- 从聊天信息面板进入：
  - 面板切换到“查找聊天记录”
  - 顶部左侧应有返回到“聊天信息”的能力

### 默认态

- 搜索框为空时，不显示复杂筛选器
- 先展示微信式“可点的搜索维度列表”
- 点击某一项后进入对应二级筛选视图：
  - `日期`
    - 打开日期选择视图
  - `群成员`
    - 打开成员选择视图
  - `文件 / 图片与视频 / 链接`
    - 直接进入该分类的结果列表

### 搜索态

- 输入关键字后实时触发搜索，但要做 `250ms ~ 300ms` 防抖
- 按 `Enter` 立刻搜索并将当前词记作本次检索条件
- 空关键字但有分类 / 日期 / 群成员条件时，也允许展示结果

### 结果态

- 点击结果：
  - 关闭搜索面板
  - 返回中间聊天线程
  - 把消息定位到可视区域中心
  - 高亮目标消息
- 如果当前线程未加载到目标消息：
  - 先按目标消息锚点拉上下文窗口
  - 再滚动定位
- `Esc`：
  - 若在二级筛选视图，先回到搜索主视图
  - 若在搜索主视图，关闭搜索面板

## 6. 不建议沿用的现有实现

- `DesktopChatHistoryPanel` 不建议继续作为最终搜索 UI：
  - 可以保留其中的消息归一化 / 摘要构造思路
  - 但不应再保留它的表单堆叠布局
- `ChatMessageSearchPanel` 不建议直接塞进桌面聊天页：
  - 它更适合移动端详情页或兼容路由
  - 桌面聊天页应改成专用组件

## 7.1 组件拆分

- 新增：
  - `apps/app/src/features/desktop/chat/desktop-chat-search-panel.tsx`
  - `apps/app/src/features/desktop/chat/desktop-chat-search-main-view.tsx`
  - `apps/app/src/features/desktop/chat/desktop-chat-search-filter-view.tsx`
  - `apps/app/src/features/desktop/chat/desktop-chat-search-results.tsx`
  - `apps/app/src/features/desktop/chat/desktop-chat-search-route-state.ts`
- 现有文件改造：
  - `apps/app/src/features/desktop/chat/desktop-chat-workspace.tsx`
  - `apps/app/src/features/desktop/chat/desktop-chat-side-panel.tsx`
  - `apps/app/src/features/desktop/chat/desktop-chat-header-actions.tsx`
  - `apps/app/src/features/desktop/chat/desktop-chat-details-panel.tsx`
  - `apps/app/src/features/chat/conversation-thread-panel.tsx`
  - `apps/app/src/features/chat/group-chat-thread-panel.tsx`

## 7.2 面板状态模型

- 当前 `rightPanelMode` 只有：
  - `history`
  - `details`
- 建议升级成更明确的桌面右侧工作流状态：
  - `details`
  - `search`
- 搜索面板内部再维护子视图：
  - `main`
  - `date-filter`
  - `sender-filter`
  - `results`
- 如果从 `details` 跳入 `search`，搜索面板顶部需要知道“返回到 details”

## 7.3 路由/状态同步

- 桌面聊天搜索不走独立路由页面
- 但建议保留 hash 级别的轻状态同步，避免刷新/窗口还原时状态丢失过多：
  - 当前会话 ID
  - 是否打开搜索面板
  - 选中的搜索分类
  - 关键字
  - 群成员 ID
  - 日期
- 不要求把搜索结果列表完整序列化到 URL

## 7.4 线程回跳

- 当前线程已支持 `highlightedMessageId`
- 但还缺“按消息锚点拉上下文”的稳定能力
- 需要在线程层增加一个明确入口：
  - `openAnchorMessage(messageId, contextWindow)`
- 行为定义：
  - 如果目标消息已在当前消息数组里，直接高亮滚动
  - 如果不在，先向后端请求目标消息附近窗口，再替换/拼接线程消息，再滚动高亮

## 8. 后端与契约方案

## 8.1 首版新增搜索接口

- 单聊：
  - `GET /api/conversations/:id/message-search`
- 群聊：
  - `GET /api/groups/:id/message-search`

建议查询参数：

- `keyword`
- `category=all|media|files|links`
- `messageType=text|image|file|voice|sticker|contact_card|location_card|system`
- `senderId`，仅群聊
- `dateFrom`
- `dateTo`
- `cursor`
- `limit`

建议返回结构：

- `items`
  - `messageId`
  - `createdAt`
  - `senderId`
  - `senderName`
  - `messageType`
  - `category`
  - `previewText`
  - `matchedText`
  - `attachmentMeta`
- `total`
- `nextCursor`
- `hasMore`

## 8.2 上下文窗口接口

- 不建议再靠“不断扩大 limit 直到撞到目标消息”做定位
- 建议扩展消息接口，支持按锚点取上下文：
  - `GET /api/conversations/:id/messages?aroundMessageId=...&before=20&after=20`
  - `GET /api/groups/:id/messages?aroundMessageId=...&before=20&after=20`

这样可以同时满足：

- 搜索结果回跳
- 聊天记录管理页未来从命中消息回跳
- 文件页 / 图片页回跳原消息

## 8.3 服务层实现策略

- 首版直接基于现有表实现：
  - `messages`
  - `group_messages`
- 搜索条件：
  - 关键词命中 `text`
  - 文件名 / 链接 / 附件描述从 `attachmentPayload` 中解析
  - 分类由 `type + attachmentKind + urlPattern` 归一
  - 日期按 `createdAt`
  - 群成员按 `senderId`
- 搜索范围必须遵守：
  - 单聊 `lastClearedAt`
  - 群聊 `lastClearedAt`
- 结果排序：
  - 默认按 `createdAt DESC`
- 分页：
  - 用 `cursor(createdAt,id)`，不要再用“不断加大 limit”模拟分页

## 8.4 首版不做的事

- 不先引入新物理表
- 不先做 SQLite FTS5 虚拟表
- 不先做跨会话全局聊天记录搜索

原因：

- 单世界实例规模可控
- 先把微信式交互打通，比先做重索引更重要
- 后续如果真实数据量上来，再把搜索服务切到 FTS，不影响前端协议

## 9. 视觉与交互约束

- 桌面搜索面板必须去掉当前项目里这些“不像微信”的元素：
  - 彩色统计 pill
  - 大片圆角卡片拼接
  - 多排表单化筛选控件
  - “完整搜索”跳转按钮
- 目标视觉方向：
  - 白灰底
  - 细边界线
  - 行级 hover
  - 结果列表主次层级清楚
  - 强调色只用于选中态和高亮词

## 10. 推荐实施阶段

## 阶段 1：重做桌面搜索壳层

- 把当前 `history` 面板从“过滤器工具栏”改成微信式搜索面板壳
- 打通入口：
  - 头部按钮
  - 聊天信息面板入口
  - `Ctrl/Cmd + F`
- 完成面板主视图 / 二级筛选视图 / 返回逻辑

## 阶段 2：补契约与服务端搜索

- 新增：
  - `message-search` 契约
  - 控制器
  - 服务层查询
- 支持：
  - 关键词
  - 日期
  - 群成员
  - 图片与视频
  - 文件
  - 链接

## 阶段 3：补结果列表与上下文定位

- 搜索结果分组展示
- 结果项点击定位到原消息
- 线程支持按锚点取上下文窗口
- 目标消息高亮与滚动复用现有逻辑

## 阶段 4：收口旧实现

- 桌面聊天页彻底停止跳 `/chat/$conversationId/search`、`/group/$groupId/search`
- `DesktopChatHistoryPanel` 退役或仅保留内部工具函数
- 移动端继续保留现有独立搜索页，不混用桌面 UI

## 11. 验收标准

- 桌面单聊点击“查找聊天记录”后，不离开当前聊天页。
- 桌面群聊支持：
  - 按日期搜索
  - 按群成员搜索
- 搜索不是“最近 80 条本地过滤”，而是整段会话历史服务端检索。
- 文件 / 图片与视频 / 链接可作为独立检索入口使用。
- 点击任意结果后，能稳定回到原消息上下文并高亮目标消息。
- `Ctrl/Cmd + F`、`Esc` 工作正常。
- 最终桌面 UI 不再出现：
  - 当前统计 pill
  - `完整搜索` 跳转按钮
  - 内联 `date input + select + chips` 混搭样式
