# 手机版好友朋友圈对齐微信规划

## 任务目标
- 把移动端好友资料里的“朋友圈”从当前的桌面端占位逻辑，改成真正可用的手机版能力。
- 参考微信手机版的信息架构，明确区分：
  - `发现 -> 朋友圈主页`
  - `好友资料 -> 好友朋友圈独立页`
- 避免继续沿用“大朋友圈里按作者筛选”的产品表达；即使首期数据层暂时复用 `getMoments()`，页面层也必须是独立页。

## 先说结论
- 移动端应和桌面端一样，为“好友朋友圈”提供独立页面，而不是继续显示“请在桌面端查看”。
- 但移动端不应该直接复刻桌面独立页，也不应该把入口做成“进入发现页朋友圈后自动筛到某个作者”。
- 更接近微信手机版的实现方式是双页模型：
  - `/discover/moments`：全局朋友圈主页，承接发现页入口。
  - `/friend-moments/$characterId`：好友朋友圈独立页，承接好友资料入口。
- 单聊详情页首期不增加平行“朋友圈”行，保持微信式主路径：
  - `聊天详情 -> 成员头像/资料 -> 好友资料 -> 朋友圈`

## 微信手机版参考基线
- Apple App Store 当前微信 iOS 版本页显示，`2026-03-21` 上线的是 `8.0.70`；产品描述仍把“分享自己的生活到朋友圈”作为移动端核心能力之一。
  - 来源：https://apps.apple.com/cn/app/%E5%BE%AE%E4%BF%A1/id414478124
- IT之家 `2025-08-12` 报道微信安卓版 `8.0.62` 时提到：“朋友资料页中‘朋友权限’与‘朋友资料’重新独立分开”，说明移动端好友资料页仍是社交能力的主信息页。
  - 来源：https://www.ithome.com/0/874/841.htm
- IT之家 `2025-09-10` 报道微信安卓版 `8.0.63` 时提到：“好友资料页微调，‘朋友权限’相关选项移至资料页同级页面”，进一步说明微信手机版持续在收紧好友资料页的信息层级，而不是把社交能力并回发现页。
  - 来源：https://www.ithome.com/0/881/971.htm
- IT之家 `2025-03-18` 对微信 iOS `8.0.57` 的实测提到朋友圈封面支持更长视频，说明手机版朋友圈并不是一个只有列表的纯轻量页，好友朋友圈页需要为封面/头图类展示留出结构余量。
  - 来源：https://www.ithome.com/0/838/802.htm

## 关键推断
- 基于上面的资料页改版报道，以及微信手机版长期稳定的产品习惯，可以推断：
  - 朋友资料页依然是进入“看这个朋友相关内容”的主入口。
  - 好友朋友圈更像“朋友的朋友圈相册/独立页”，而不是发现页大流中的临时筛选态。
  - 移动端聊天详情首期不需要再做一条平行入口，否则会偏离微信手机版常见路径，并让入口层级重复。

## 目标产品结构

### 1. 页面模型
- 保留 `发现 -> 朋友圈主页` 这一条移动端路径，不动它的一级定位。
- 新增 `好友资料 -> 好友朋友圈独立页` 这一条移动端路径。
- 两者关系不是“主页 + 作者筛选”，而是两个不同目的的页面：
  - 朋友圈主页：看所有好友的动态，带发布入口。
  - 好友朋友圈页：只看这个好友的时间线，强调资料感和连续浏览。

### 2. 路由模型
- 新增移动端独立路由：`/friend-moments/$characterId`
- 保留桌面端独立路由：`/desktop/friend-moments/$characterId`
- 不建议让移动端直接复用 `/desktop/friend-moments/$characterId`，避免平台语义混乱。
- 不建议把移动端新页挂在 `/discover/moments/...` 下面，这会再次把“好友朋友圈”做成发现页子态。

### 3. 入口模型
- 首期只改移动端好友资料页：
  - 好友状态下，“朋友圈”从禁用文案改成可点击动作。
  - 点击后进入 `/friend-moments/$characterId`。
- 单聊详情页首期保持不变：
  - 继续通过成员头像进入好友资料页。
  - 不额外新增一条“朋友圈”设置行。
- 后续其他移动端资料触点如果出现同类入口，也统一跳 `/friend-moments/$characterId`，不要再长出第二套路由。

## 页面形态建议

### 1. 顶部结构
- 顶部使用微信式居中标题栏，标题可以直接显示好友名称，副标题使用“朋友圈”或“好友朋友圈”。
- 左上只保留返回；右上首期不强行加分享/更多，避免先做成工具页气质。

### 2. 头部信息区
- 页面顶部应有独立的好友头部区，而不是直接把时间线列表顶到最上面。
- 建议结构：
  - 头图/封面占位区
  - 头像
  - 备注名或昵称
  - 个性签名
  - 轻量资料文案，如“最近发布了 X 条朋友圈”或最近更新时间
- 即使首期还没有真实封面字段，也要预留视觉结构，不要把页面做成简单列表壳。

### 3. 时间线区
- 只展示这位好友的朋友圈，不出现任何“当前筛选作者”或“查看全部朋友圈”式文案。
- 列表项可继续复用现有动态卡片的内容区、点赞、评论交互，但页面语义必须是独立页。
- 不在好友页顶部放“发一条朋友圈”主动作，避免把“看别人朋友圈”和“发自己的朋友圈”混成一个任务流。

### 4. 状态页
- 非好友：显示“加为好友后可查看”，并提供返回资料页。
- 好友但暂无内容：显示“这位好友还没有发表朋友圈”。
- 已被拉黑或内容不可见：显示明确权限文案，不回退到全量朋友圈页。
- 加载失败：提供重试。

## 数据与状态策略

### 1. 首期实现
- 首期允许继续复用 `getMoments(baseUrl)`，在移动独立页里按 `authorId === characterId` 过滤。
- 这只是数据复用，不是产品复用；页面和路由都必须已经独立。
- 需要新增移动端自己的路由状态工具，最多只承载：
  - `momentId`
  - `source`
- 不再给移动端大朋友圈引入 `authorId` hash。

### 2. 第二阶段
- 增加专用接口，例如：
  - `GET /api/moments/characters/:characterId`
- 这样可以把“好友朋友圈”从全量拉流中过滤，升级成真正的按人取数。
- 如果后续需要封面、发布时间聚合、相册摘要，也应挂在专用接口或专用返回字段上，而不是继续挤在全局流接口里。

## 建议拆分方式
- 不建议继续把移动端真实页面塞进当前 [apps/app/src/routes/friend-moments-page.tsx](/home/ps/claude/yinjie-app/apps/app/src/routes/friend-moments-page.tsx) 的 `if (!isDesktopLayout)` 占位分支里。
- 更稳的拆法是：
  - 抽一个共享数据 hook，例如 `useFriendMomentsData`
  - 桌面继续用 `DesktopFriendMomentsWorkspace`
  - 新增移动端页面主体组件，例如 `MobileFriendMomentsPageBody`
  - 新增移动端路由页，例如 `mobile-friend-moments-page.tsx`
- 这样可以避免一个页面文件里同时堆桌面工作区和手机壳逻辑，后续也更容易继续对齐微信手机版细节。

## 分阶段实施

### Phase 1：移动端独立页落地
- 新增 `/friend-moments/$characterId`
- 新增移动端好友朋友圈独立页骨架
- 复用现有朋友圈数据做作者过滤
- 完成加载、空态、非好友态、失败态

### Phase 2：好友资料页接线
- 移动端好友资料页“朋友圈”改成可点击
- 文案从“请在桌面端查看”改成“查看这位好友最近的朋友圈”
- 非好友继续显示“加为好友后可查看”

### Phase 3：共享逻辑收口
- 抽离桌面/移动共用的数据查询和权限判断
- 让桌面页、移动页共享同一份 friend moments data hook
- 去掉当前 `friend-moments-page.tsx` 里只负责占位的移动端逻辑

### Phase 4：接口与体验升级
- 新增专用好友朋友圈接口
- 视需要补朋友圈封面、发布时间聚合、资料摘要
- 再决定发现页中作者点击是否跳资料页或直接跳好友朋友圈页

## 验收标准
- 移动端好友资料页不再出现“请在桌面端查看”。
- 移动端点击好友资料里的“朋友圈”后，进入的是独立页面，不是发现页朋友圈的筛选态。
- 页面第一眼像“这个朋友的朋友圈页”，而不是“全局朋友圈的一个子列表”。
- 非好友、无内容、不可见、失败等状态都有明确反馈，不会偷偷回退到大朋友圈。
- 单聊详情到好友资料再到朋友圈这条路径完整可用，符合微信手机版的主路径习惯。

## 风险与注意事项
- 直接复用桌面页视觉会让移动端看起来像缩小版工作区，这是首要风险。
- 如果继续把移动路由挂在 `/discover/moments/...` 下，产品感知上还是“发现页子态”，不符合本次目标。
- 首期继续用全量 `getMoments()` 过滤会有性能与语义欠账，因此要在规划里明确这是过渡层，不是最终设计。
- 当前移动端全局朋友圈卡片头部没有成熟的“作者点击”模型，首期不要顺手发明一套半成品跳转，先把好友资料主路径做对。

## 推荐实施顺序
1. 先建移动端独立路由和页面骨架。
2. 再接通好友资料页入口。
3. 然后抽共享数据逻辑，清掉当前占位分支。
4. 最后再考虑专用接口和更细的微信式头部表现。
