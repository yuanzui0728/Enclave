# Web 端电脑版广场动态对齐朋友圈排版规划

## 目标

把 `apps/app` 的桌面 Web / 桌面壳 `广场动态` 页面，从现在这套“左侧说明卡 + 右侧内容列”的发现子页排版，调整成和桌面 `朋友圈` 一致的工作区结构。

这次规划的核心不是把广场动态做成朋友圈副本，而是：

- 复用朋友圈已经验证过的桌面工作区骨架
- 保留广场动态的公开流语义
- 让用户在桌面端切到 `广场动态` 时，交互节奏和 `朋友圈` 一致，但内容边界明显不同

这次规划只覆盖：

- `apps/app` 的桌面 Web / 桌面壳形态
- `/tabs/feed` 与 `/discover/feed` 的桌面呈现
- 广场动态主列表、发帖入口、作者入口、详情侧栏、桌面筛选

这次不覆盖：

- 移动端广场动态
- 图片 / 视频广场动态增强
- 推荐算法、热度排序、分页重构
- 举报、屏蔽、审核面板
- 新测试文件

## 设计原则

### 1. 结构参考朋友圈，语义必须分开

桌面端的广场动态应该和朋友圈一样，采用：

- 顶部轻工具栏
- 中间连续内容流
- 右侧上下文栏

但必须在文案、筛选和右栏信息里持续强调：

- `朋友圈 = 好友私域`
- `广场动态 = 世界居民公开流`

不能出现“只是把朋友圈文案改一改”的感觉。

### 2. 先做工作区统一，再做功能增强

当前桌面 `朋友圈` 已有较完整的工作区组件：

- [desktop-moments-workspace.tsx](/home/ps/claude/yinjie-app/apps/app/src/features/desktop/moments/desktop-moments-workspace.tsx)
- [desktop-moments-toolbar.tsx](/home/ps/claude/yinjie-app/apps/app/src/features/desktop/moments/desktop-moments-toolbar.tsx)
- [desktop-moments-feed.tsx](/home/ps/claude/yinjie-app/apps/app/src/features/desktop/moments/desktop-moments-feed.tsx)
- [desktop-moments-sidebar.tsx](/home/ps/claude/yinjie-app/apps/app/src/features/desktop/moments/desktop-moments-sidebar.tsx)
- [desktop-moment-row.tsx](/home/ps/claude/yinjie-app/apps/app/src/features/desktop/moments/desktop-moment-row.tsx)
- [desktop-moment-detail-panel.tsx](/home/ps/claude/yinjie-app/apps/app/src/features/desktop/moments/desktop-moment-detail-panel.tsx)

广场动态第一阶段不应该继续堆 `AppSection` 卡片页，而应该直接进入“广场工作区”。

### 3. 公开流要比朋友圈多一层“世界视角”

朋友圈右栏更适合承接“我的资料 / 好友作者时间线 / 某条动态详情”。

广场动态右栏则需要额外承担：

- 当前公开范围说明
- 世界主人 / 居民身份切换感知
- 居民作者列表与作者画像摘要
- 当前流向是“全部居民”还是“只看某类作者”

## 对标朋友圈后的页面定位

### 页面定位

桌面 `广场动态` 应定义为：

- 一个固定一级入口
- 一个连续公开流工作区
- 一个承接“世界居民正在说什么”的桌面浏览面板

它和朋友圈的区别不在于排版结构，而在于浏览心智：

- 朋友圈看“熟人圈”
- 广场看“世界面”

### 总体结构

推荐继续沿用朋友圈桌面三段结构：

- 顶部轻工具栏
- 左中主内容流
- 右侧上下文栏

推荐宽度保持一致，降低用户切换成本：

- 主内容流：`680px - 760px`
- 右栏：`300px - 340px`
- 两栏间距：`20px - 24px`

## 页面规划

### 1. 顶部轻工具栏

广场动态的顶栏直接复用朋友圈的交互节奏，但文案和筛选改成公开流语义。

建议保留：

- 左侧标题：`广场动态`
- 右侧轻操作：`刷新`、`回到顶部`
- 主操作：`发动态`
- 搜索框：搜索作者、正文、评论

建议新增或替换的筛选：

- `全部`
- `只看我`
- `只看居民`

不要直接沿用朋友圈的 `只看角色` 文案。广场场景下更自然的是“居民”。

顶栏说明文案要明确：

- 这里是世界公开流
- 不只看朋友
- 居民和世界主人都会出现在流里

### 2. 主内容流

主内容流直接参考朋友圈的连续流排版，不继续使用大圆角营销卡。

每条广场动态建议固定为：

- 左侧头像
- 右侧正文区
- 作者名 + 身份标签
- 发布时间
- 正文
- 互动汇总
- 点赞 / 评论轻操作
- 点赞与评论预览盒

和朋友圈不同的地方：

- 身份标签优先强调 `世界主人` / `居民`
- 可以保留 `AI 已参与回应` 这类广场特有摘要
- 作者头像、作者名点击后进入“作者公开流”

视觉要求：

- 降低 [social-post-card.tsx](/home/ps/claude/yinjie-app/apps/app/src/components/social-post-card.tsx) 当前的卡片感
- 更接近 [desktop-moment-row.tsx](/home/ps/claude/yinjie-app/apps/app/src/features/desktop/moments/desktop-moment-row.tsx) 的连续流密度
- 默认点击帖子主体进入右栏详情
- 评论输入默认折叠，点击评论再展开

### 3. 右侧上下文栏

右栏继续复用朋友圈“上下文容器”的思路，但内容换成广场语义。

分三种状态：

- `summary`：默认概览态
- `author`：作者公开流态
- `detail`：动态详情态

#### `summary` 默认概览态

建议内容：

- 当前世界主人摘要
- `发一条广场动态` 快捷入口
- 当前公开流说明
- 最近活跃居民列表
- 当前筛选摘要

这个状态承担“解释这是哪里”和“从哪继续逛”的作用。

#### `author` 作者公开流态

点击作者头像或作者名后：

- 主内容流切成该作者的公开动态
- 右栏展示该作者摘要
- 顶栏出现轻量筛选状态，例如 `只看 林秋的公开动态`

作者态建议展示：

- 头像、名字、身份
- 最近发言时间
- 动态数、评论数
- `查看全部公开流`
- `发动态`
- 最近几条公开动态

这里不能再写成“TA 的朋友圈”，而应该改成“TA 的公开动态”或“TA 的广场动态”。

#### `detail` 动态详情态

点击某条帖子后，右栏切到详情态，承接完整上下文：

- 作者摘要
- 完整正文
- 发布时间
- AI 回应状态
- 点赞列表
- 完整评论列表
- 评论输入框
- `查看 TA 的公开动态`

这一步让桌面端形成和朋友圈一致的“流里浏览，右栏承接”的体验。

### 4. 发动态入口

桌面广场动态不建议继续保留“左栏大输入框 + 右侧流”的发现页布局。

建议沿用朋友圈方案：

- 顶栏主按钮打开 `Compose Panel`
- 右栏概览态保留一个快捷发帖入口

`Compose Panel` 可以直接复用朋友圈的交互骨架，但文案换成：

- `发广场动态`
- `发给世界里的居民看`

第一阶段依旧只支持文本。

### 5. 收藏入口

广场动态已有收藏能力，桌面工作区里应该保留，但不喧宾夺主。

建议位置：

- 帖子行尾操作区保留 `收藏`
- 详情态中也保留 `收藏 / 取消收藏`

这样不会因为改成工作区而丢掉现有桌面特色能力。

## 筛选模型建议

广场动态建议定义新的桌面筛选类型，而不是硬复用朋友圈的 `owner / character` 语义。

建议：

```ts
type DesktopFeedFilter = "all" | "owner" | "resident";
```

映射规则：

- `all`: 世界主人 + 所有居民公开流
- `owner`: 只看世界主人公开动态
- `resident`: 只看居民公开动态

作者态过滤优先级高于总筛选：

- 如果已选作者，则主流只展示该作者
- 顶栏通过一个轻量胶囊告诉用户当前在作者视角

## 组件拆分建议

广场动态不要继续全写在 [discover-feed-page.tsx](/home/ps/claude/yinjie-app/apps/app/src/routes/discover-feed-page.tsx)。

建议新增一套和朋友圈平行的桌面组件：

- `apps/app/src/features/desktop/feed/desktop-feed-workspace.tsx`
- `apps/app/src/features/desktop/feed/desktop-feed-toolbar.tsx`
- `apps/app/src/features/desktop/feed/desktop-feed-list.tsx`
- `apps/app/src/features/desktop/feed/desktop-feed-row.tsx`
- `apps/app/src/features/desktop/feed/desktop-feed-sidebar.tsx`
- `apps/app/src/features/desktop/feed/desktop-feed-detail-panel.tsx`
- `apps/app/src/features/desktop/feed/desktop-feed-compose-panel.tsx`

原因：

- 广场动态和朋友圈的骨架相似，但语义、字段和右栏文案不同
- 如果强行复用 `desktop-moments-*`，很快会堆满 `variant="feed"` 分支
- 单独拆一套桌面 `feed` 组件更容易维护，也更符合当前代码结构

可以复用的东西：

- 交互模式
- 容器宽度
- 侧栏状态机
- 评论输入折叠方式

不建议直接共用的东西：

- 文案
- 作者标签
- 右栏默认态内容
- 筛选枚举
- “朋友圈 / 公开流”上下文说明

## 页面级路由口径

- `/tabs/feed`：桌面固定主入口，直接进入广场工作区
- `/discover/feed`：移动端发现二级页

桌面下建议两者统一复用同一个 `DesktopFeedWorkspace`，不要维护两套桌面布局。

也就是和朋友圈一样：

- 路由可以保留两条
- 但桌面渲染逻辑收口到一套

## 分阶段实施

### Phase 1：桌面广场工作区落地，不改接口

目标：

- 只用现有 `feed` 接口，把桌面页从卡片内容页改成工作区
- 不动数据库实体
- 不改移动端广场动态

具体改动：

- 在 [discover-feed-page.tsx](/home/ps/claude/yinjie-app/apps/app/src/routes/discover-feed-page.tsx) 桌面态接入 `DesktopFeedWorkspace`
- 新增 `features/desktop/feed/*` 组件
- 把当前 `SocialPostCard` 桌面主流替换成连续流行项目
- 加入 `全部 / 只看我 / 只看居民` 桌面筛选
- 加入 `selectedPostId`、`activeAuthorId`、`searchText` 三个核心状态
- 右栏实现 `summary / author / detail` 三态切换
- 收藏能力保留在新工作区里

验收标准：

- 桌面 `广场动态` 和 `朋友圈` 一样有稳定工作区心智
- 用户能在当前页内完成发帖、点赞、评论、点作者、看详情
- 一眼能看出这里不是朋友圈，而是世界居民公开流

### Phase 2：补齐广场特有信息层

目标：

- 让广场动态不只是“换文案的朋友圈”

建议补齐：

- 顶栏加入排序或推荐模式占位
- 右栏加入居民活跃摘要
- 详情态加入 `AI 已参与回应` 的更明确展示
- 发帖区预留媒体入口占位

这一阶段仍可不改后端实体，只做前端信息层增强。

### Phase 3：考虑与桌面发现体系联动

等桌面 `朋友圈` 和 `广场动态` 都稳定成工作区后，再判断是否需要：

- 在桌面发现总览里增加“最近居民发言摘要”
- 或进一步把 `发现` 做成聚合工作区

这一步不建议和当前改造绑定。
