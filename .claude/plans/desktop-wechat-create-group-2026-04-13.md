# Web 桌面版发起群聊对齐微信电脑版实现规划

日期：2026-04-13

## 目标

把 `apps/app` 的 Web 桌面版“发起群聊”体验收敛到微信电脑版的交互模型：

- 桌面端以弹层为主，不走自定义工作台式大页面
- 从单聊邀请人建群时，默认勾选当前聊天对象
- 单聊场景支持“分享聊天内容”，但作为次级能力，不抢首屏主流程
- 创建完成后直接进入新群会话，不额外出现群资料表单步骤
- 视觉、文案、层级、空态、按钮位置、搜索与勾选反馈都向微信电脑版靠拢

说明：

- 这里的“完全抄”按结构、交互与视觉节奏对齐处理，不直接复用微信商标、原始图标素材或像素级资源。
- 不新增测试文件，采用手工验收清单。

## 现状结论

现有能力已经具备 70% 以上，但实现分裂，且桌面端 UI 明显比微信更重：

- 已有桌面弹层实现：`apps/app/src/features/desktop/chat/desktop-create-group-dialog.tsx`
- 已有桌面独立页面实现：`apps/app/src/routes/create-group-page.tsx`
- 已有桌面入口：
  - 聊天工作区左上角快捷菜单：`apps/app/src/features/desktop/chat/desktop-chat-workspace.tsx`
  - 单聊详情侧栏“发起群聊”：`apps/app/src/features/desktop/chat/desktop-chat-details-panel.tsx`
  - 群聊通讯录页“发起群聊”：`apps/app/src/routes/group-contacts-page.tsx`
- 后端建群能力已够用：
  - 创建群：`api/src/modules/chat/group.service.ts`
  - 从单聊带聊天记录进新群：`copySharedConversationMessages(...)`

当前主要问题：

1. 桌面端存在两套建群 UI
   - 一个是居中弹层
   - 一个是 `/group/new` 下的双栏大页面
   - 两套布局、文案、字段、操作顺序不一致

2. 当前桌面弹层信息密度过高
   - 有大量辅助说明、快捷键说明、目录索引提示
   - “聊天上下文”区域占据首屏大块空间
   - 整体更像内部工具，不像微信桌面端的轻量联系人选择器

3. `/group/new` 的桌面页偏离微信
   - 右侧单独放“群资料”
   - 允许先填群名
   - 这是明显不符合微信电脑版创建群聊首屏心智的

4. 入口体验不统一
   - 聊天工作区用本地 state 开弹层
   - 其他桌面入口走路由页
   - 导致同一个功能在不同地方长得不一样

## 参考口径

- 微信电脑版发起群聊的主流程是“进入选择联系人页 -> 勾选联系人 -> 确定 -> 进入群聊”
- 微信 4.1.2 已支持“单聊中邀请人时分享聊天记录”，且该能力位于建群界面的右下角次级位置

- 全局 `+` 菜单进入建群
- 单聊详情页发起群聊
- 默认勾选当前聊天对象
- 搜索联系人
- 已选联系人超过一行

- 无联系人与无搜索结果

## 目标体验

### 1. 入口统一

桌面端所有入口都进入同一套建群弹层：

- 聊天工作区快捷菜单“发起群聊”
- 单聊详情侧栏“发起群聊”
- 群聊通讯录页“发起群聊”
- 兼容路由 `/group/new`，但桌面端不再呈现独立大页面

### 2. 主流程统一为“选择联系人”

首屏只做一件事：选人。

页面结构建议收敛为：

- 顶部标题栏：`选择联系人`
- 顶部搜索框
- 已选联系人横向头像/标签区
- 联系人列表
- 底部操作区：`取消` / `完成`

不在首屏出现：

- 群名称输入框
- 群资料说明区
- 大段引导文案
- 明显偏工具化的快捷键帮助文本

### 3. 单聊邀请人建群的特殊逻辑

从单聊发起时：

- 默认勾选当前聊天对象
- 在已选区优先展示该联系人
- 仅在该场景显示“分享聊天内容”入口
- “分享聊天内容”是次级按钮或次级开关，不占据首屏大区块

### 4. 分享聊天内容改成二级流程

当前能力可以保留，但表现必须改：

- 不再作为首屏中段的大面板
- 改为底部右下角次级入口，符合微信 4.1.2 的信息层级
- 用户点击后，再展开聊天记录选择区或进入二级轻弹层
- 未点击时，主流程只聚焦“选人并完成”

### 5. 桌面视觉方向

按微信电脑版的桌面感收敛：

- 更克制的白灰底
- 更窄的弹层宽度与更轻的阴影
- 列表 hover / checked / focused 状态更弱，不要“控制台面板感”
- 选中联系人区以微信式头像流或轻标签流为主
- 操作按钮位置回归底部右侧完成
- 空态与错误态保持简洁，不做大块说明卡片

## 实现策略

### Phase 0：先定对照稿

产出一份实现前对照表，明确以下内容：

- 弹层尺寸、边距、标题、搜索框尺寸
- 联系人行高、头像尺寸、勾选框位置
- 已选联系人区换行规则
- 完成按钮文案与禁用态
- 分享聊天内容的出现条件、位置、展开方式

目标：

- 后续实现不再边做边猜

### Phase 1：合并桌面端两套实现

目标是“一套状态，一套 UI”。

建议做法：

- 抽离桌面建群共享状态 hook
  - 例如：`useDesktopCreateGroupFlow`
- 抽离桌面建群共享视图组件
  - 例如：`DesktopCreateGroupModal`
- `desktop-create-group-dialog.tsx` 与 `create-group-page.tsx` 不再各自维护一份完整实现

处理原则：

- 移动端 `CreateGroupPage` 保留原路由页模型
- 桌面端 `CreateGroupPage` 改成兼容壳，只复用同一套弹层组件
- `DesktopChatWorkspace`、`GroupContactsPage` 都调用同一组件，不允许再出现第二套桌面建群 UI

### Phase 2：按微信桌面端重做首屏结构

重点改造 `DesktopCreateGroupDialog`：

- 标题改为微信式“选择联系人”
- 去掉当前“聊天上下文”“联系人目录”“快捷键说明”等显性说明块
- 把联系人列表改成更直接的单栏选择器
- 已选联系人区收敛成微信式头像流/标签流
- 底部只保留取消、完成、计数

当前这些内容应删除或弱化：

- 大段说明文案
- 右侧字母索引的存在感
- 首屏大块上下文分享卡片
- 过强的 ring / shadow / badge 装饰

### Phase 3：把“分享聊天内容”改成微信式次级入口

保留后端与选择消息能力，但重构表现层：

- 仅单聊来源显示
- 入口位于底部右侧次级区域
- 点击后再展开消息选择区
- 默认不抢占首屏高度

交互约束：

- 没有可分享消息时，入口可点击但展示简洁空态
- 勾选消息后，完成按钮附近给出轻量计数反馈
- 关闭该入口后，不应影响已选联系人

### Phase 4：入口联动与关闭逻辑统一

需要统一以下行为：

- `Esc` 关闭弹层
- 点击遮罩关闭
- 完成后进入新群会话
- 取消后回到原桌面上下文
- 从单聊详情进入时，重新打开仍能记住默认种子成员逻辑

### Phase 5：手工验收

按桌面 Web 场景逐项走：

1. 从聊天工作区快捷菜单进入
2. 从单聊详情页进入
3. 从群聊通讯录页进入
4. 搜索联系人
5. 勾选/取消勾选联系人
6. 默认带出当前单聊对象
7. 分享聊天内容入口展开与选择
8. 创建完成后跳转到新群
9. 群会话列表刷新
10. 空联系人、无搜索结果、接口报错

## 明确不做

- 不做移动端建群重构
- 不新增测试文件
- 不新增群名称首屏填写步骤
- 不把桌面建群做成新的独立工作区产品页

## 验收标准

满足以下条件后可视为完成：

- 桌面端所有“发起群聊”入口视觉与交互统一
- 桌面端不再出现独立双栏建群大页面
- 建群首屏与微信电脑版的“选择联系人”模型一致
- 单聊来源默认勾选当前对象
- 单聊来源支持“分享聊天内容”，且为次级入口
- 创建后直接进入新群聊，无额外中间页

## 建议的实施顺序

1. 先把桌面独立页与桌面弹层合并成同一套实现
2. 再重做视觉与层级，收掉非微信式信息块
3. 最后调整“分享聊天内容”的入口位置与展开方式

这样可以先消除重复实现，再做体验对齐，返工最少。
