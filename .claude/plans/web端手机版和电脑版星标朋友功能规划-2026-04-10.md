# web端手机版和电脑版星标朋友功能规划

## 1. 目标

在 web 端的手机版和桌面版里补齐“星标朋友”能力，行为尽量对齐微信：

- 只有已经在通讯录里的好友才能设为星标朋友
- 从联系人资料页可开启/关闭“设为星标朋友”
- 通讯录里出现“星标朋友”入口
- 点击“星标朋友”进入星标好友列表
- 星标好友仍然保留在普通通讯录 A-Z 列表里，不从原联系人目录移除
- 取消星标后立即从星标列表消失
- 不影响现有聊天置顶、消息排序、好友申请、世界角色目录

这套方案更接近微信当前习惯，而不是把星标好友直接改造成“通讯录顶部固定分组”。

## 3. 微信对齐口径

本次按下面这套口径实现，避免做成“自定义收藏夹”：

- 星标是好友关系上的一个布尔属性，不是独立实体
- 星标入口放在“联系人资料”里，不放在聊天详情页里作为主入口
- 通讯录展示单独的“星标朋友”入口
- “星标朋友”是一个独立视图，不替代普通联系人列表
- 星标列表按最近星标时间优先，再按姓名排序
- 非好友、黑名单对象、世界角色目录中的未加好友角色不能设星标

## 4. 数据与契约变更

### 4.1 Friendship 实体

在 [friendship.entity.ts](/home/ps/claude/yinjie-app/api/src/modules/social/friendship.entity.ts) 增加：

- `isStarred: boolean`
- `starredAt?: Date | null`

约束：

- 默认 `false`
- 取消星标时 `starredAt = null`
- `status = blocked` 时强制视为不可星标

说明：

- 当前 `api` 使用 TypeORM `synchronize: true`，新增列会在启动时自动同步；不需要单独迁移脚本
- 旧数据默认回填为未星标，不需要额外数据清洗

### 4.2 共享契约

在 [social.ts](/home/ps/claude/yinjie-app/packages/contracts/src/social.ts) 扩展：

- `Friendship.isStarred: boolean`
- `Friendship.starredAt?: string`
- 新增 `SetFriendStarredRequest { starred: boolean }`

在 [client.ts](/home/ps/claude/yinjie-app/packages/contracts/src/client.ts) 增加：

- `setFriendStarred(characterId, { starred }, baseUrl?)`

## 5. 后端接口规划

### 5.1 保留现有读取接口

继续使用：

- `GET /api/social/friends`

返回的 `friendship` 中附带 `isStarred` 和 `starredAt`，前端自己拆出星标列表即可。

这样改动最小，通讯录页、聊天详情页、角色资料页都能直接共享同一份好友缓存。

### 5.2 新增写接口

新增：

- `POST /api/social/friends/:characterId/star`

请求体：

```json
{ "starred": true }
```

服务规则：

- 目标必须已经是好友
- 目标不能处于 `blocked`
- `starred=true` 时写入 `isStarred=true` 与 `starredAt=now`
- `starred=false` 时写入 `isStarred=false` 与 `starredAt=null`
- 找不到好友关系时直接报错，不自动创建好友关系

对应改动文件：

- [social.controller.ts](/home/ps/claude/yinjie-app/api/src/modules/social/social.controller.ts)
- [social.service.ts](/home/ps/claude/yinjie-app/api/src/modules/social/social.service.ts)

如果实施时新增了路由，需要同步更新 `AGENTS.md` 里的“主 App 页面”说明。

## 8. 排序与展示规则

### 普通通讯录

- 仍按现有 A-Z 分组
- 星标好友继续保留在原分组中
- 可在联系人行右侧补一个很轻的星标标识，但这不是第一优先级

### 星标朋友列表

排序建议：

1. `starredAt` 倒序
2. 同时间再按姓名拼音
3. 再按 `character.id`

原因：

- 更符合“最近设星标的优先看到”
- 比单纯按姓名更符合用户最近操作记忆

## 9. 查询与缓存策略

前端统一继续以 `["app-friends", baseUrl]` 为主缓存源：

- 通讯录页读取全部好友
- 星标页前端过滤 `isStarred`
- 切换星标后统一失效 `["app-friends", baseUrl]`
- 若详情页依赖该缓存，自动同步更新

这样可以避免再造一套 `app-starred-friends` 查询键和专门接口。

[contacts-page.tsx](/home/ps/claude/yinjie-app/apps/app/src/routes/contacts-page.tsx) 已同时承担移动端、桌面端、快捷入口、索引导航、世界角色目录、双栏详情。星标朋友再直接堆进去，后续可维护性会继续恶化。

### 11.3 默认好友

[social.service.ts#L126](/home/ps/claude/yinjie-app/api/src/modules/social/social.service.ts#L126) 会自动补默认好友。星标逻辑必须保证：

- 默认好友可以设星标
- 自动补默认好友时不要自动设星标

## 12. 最终建议

建议按“后端字段 + 联系人资料开关 + 独立星标页”这条线落地，不建议只在通讯录主列表顶部塞一个分组。

原因：

- 更接近微信
- 对移动端和桌面端都容易复用
- 不会破坏现有联系人 A-Z 主目录
- 对现有代码侵入最小
