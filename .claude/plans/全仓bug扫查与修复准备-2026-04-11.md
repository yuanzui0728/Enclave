## 扫描结论

### 可直接确认的现状

- `pnpm typecheck`：通过
- `pnpm build`：通过，但主 App 有分包失效与大包警告
- `pnpm lint`：失败，集中在 `apps/app`
- `api`：`build` 与 `tsc --noEmit` 均通过
- `api`：去掉 `prettier` 噪音后仍有 324 条 ESLint 问题，主要集中在类型安全塌陷、未处理 Promise、误导性接口语义

### 当前最重要的事实

- 仓库不是“编不过”，而是“能编、能出包，但存在一批行为 bug 和大量被 `any` 掩盖的后端风险”。
- 后端最大根因不是某一个模块单点失误，而是 `WorldOwnerService` 返回类型未显式收口，导致单世界主人链路把 `any` 扩散进聊天、群聊、社交、公众号、动态等多个服务。
- 前端最大根因是若干页面把“初始路由状态”当成“持续状态”使用，或者在 `useEffect` 中重置本地交互态，导致页面复用或查询刷新后发生状态回滚。

## 关键问题清单

### P0 行为 bug

1. `apps/app/src/features/chat/mobile-reminder-toast-host.tsx:82`
本地通知只在“提醒首次成为 active 且页面已经 hidden”时触发。若提醒在前台变为 active，用户之后再切到后台，`useEffect` 不会因 `visibilitychange` 重跑，这条提醒就永远不会发本地通知。

2. `apps/app/src/features/chat/mobile-group-call-screen.tsx:124`
`joinedMemberIds` 在依赖 `members` / `routeState` 的初始化 effect 中被重置。只要群成员查询刷新或路由状态变化，用户手动切换的“已加入成员”状态就会被覆盖，继而把 `activeCount`、同步消息和结束通话摘要一起带偏。

3. `apps/app/src/routes/mini-programs-page.tsx:86`
`selectedMiniProgramId` 和 `launchContext` 只在首次挂载时从 `window.location.search` 读取。若同一路由在不卸载组件的情况下切换不同 `miniProgram`、`sourceGroupId`、`sourceGroupName`，页面会继续使用旧上下文，群接龙回填目标可能发错群。

4. `api/src/modules/chat/chat.controller.ts:593`
`triggerAiReplies()` 被直接 fire-and-forget 调用，没有 `void` 也没有外围错误处理。若它在定时器调度前阶段抛错，例如群访问校验、成员查询、规则加载失败，会形成未处理 Promise 拒绝，并直接丢失群聊 AI 回复流程。

5. `api/src/modules/system/system.service.ts:561`
`exportDiagnostics()` 和 `restoreBackup()` 对外返回 `success: true`，但实际未执行导出与恢复。管理后台若依赖这两个接口，会得到“操作成功”的假信号，这是运维级误导，不应继续暴露为成功态。

### P1 根因级问题

1. `api/src/modules/auth/world-owner.service.ts:43`
`ensureSingleOwnerMigration()` 与 `getOwnerOrThrow()` 都没有显式返回类型，且二者递归调用。TypeScript 在这里把返回推断塌成了 `any`，这正是 `main.ts`、`group.service.ts`、`social.service.ts`、`feed.service.ts`、`official-accounts.service.ts` 等大量 `no-unsafe-*` 问题的上游根因。

2. `api/src/modules/characters/character-blueprint.service.ts:477`
`summarizeValue()` 对对象直接 `String(value)`，会把复杂字段渲染成 `[object Object]`。角色工厂 diff、修订记录、字段摘要会出现不可读内容，实际影响后台可运营性。

3. `apps/app/src/runtime/runtime-config-store.ts:88`
这里为了切换运行时地址动态 `import("../lib/socket")`，但 `socket.ts` 已被多个页面静态导入，Vite 明确提示不会形成独立 chunk。当前设计让运行时配置更新和聊天室主包耦合，分包意图已经失效。

### P2 清理与回归风险

1. `apps/app/src/components/chat-composer.tsx`
当前 lint 失败里既有未使用变量，也有多处 `useEffect` 缺依赖。未使用项本身偏清理问题，但 Hook 依赖遗漏需要逐条判断是否会导致截图编辑、语音面板和 mention 光标状态失真，不能只做机械修 lint。

2. `apps/admin/src/routes/reply-logic-page.tsx:286`
后台回复逻辑页存在多处 effect 依赖缺失。由于该页同时管理角色草稿、规则草稿和 preview 结果，依赖遗漏会导致旧 mutation/error 状态残留到新角色或新会话。

3. `apps/app/src/features/search/use-search-index.ts:84`
全局搜一搜在输入关键字后会对所有会话抓消息并构建消息索引，且依赖键只按会话列表和活动时间收口。现在还没炸，但数据量一上来会成为明显的 UI 卡顿点。

## 修复批次规划

### 批次 1：先止血 P0 行为 bug

目标：
- 修正用户可感知的错误行为
- 给后续 lint/type 修复降低噪音

范围：
- `apps/app/src/features/chat/mobile-reminder-toast-host.tsx`
- `apps/app/src/features/chat/mobile-group-call-screen.tsx`
- `apps/app/src/routes/mini-programs-page.tsx`
- `api/src/modules/chat/chat.controller.ts`

改法：
- 提醒 host 监听 `visibilitychange` 或把 visibility 纳入 store / state，确保 active reminder 在前台变成 due 后，切后台时仍能补发通知。
- 群通话页把“初始化 joinedMemberIds”的逻辑限定在真正的首屏建立时机，不再在成员查询刷新时覆盖用户操作态。
- 小程序页改为使用 router search 或 location state 驱动，而不是只在初次挂载时从 `window.location.search` 读一次。
- `triggerAiReplies()` 外围加明确的 `void` 和错误记录，或改为 service 内部吞吐并记录错误，禁止未处理 rejection。

提交建议：
- `fix(app): stabilize reminder, group call, and mini-program route state`
- `fix(api): guard async group ai reply trigger`

### 批次 2：收口后端单世界主人类型边界

目标：
- 消除 `any` 级扩散
- 让 API ESLint 真正暴露剩余真实问题

范围：
- `api/src/modules/auth/world-owner.service.ts`
- 所有直接调用 `getOwnerOrThrow()` / `ensureSingleOwnerMigration()` 的服务

改法：
- 给 `ensureSingleOwnerMigration(): Promise<UserEntity>`、`getOwnerOrThrow(): Promise<UserEntity>`、`getOwnerProfile()` 等核心方法补全显式返回类型。
- 清理因此联动暴露的 `no-unsafe-assignment`、`no-unsafe-member-access`、`no-unsafe-call`。
- 重新跑一次 `api` ESLint，拿到真实剩余清单。

提交建议：
- `fix(api): restore typed world owner service boundary`

### 批次 3：修正运维与后台误导语义

范围：
