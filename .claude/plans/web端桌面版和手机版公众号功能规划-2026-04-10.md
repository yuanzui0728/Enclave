# Web端桌面版和手机版公众号功能深度规划

## 结论先行

这块能力不是从零开始做。

当前仓库里已经有一版“阅读链路骨架”：
- 后端已有 `official-accounts` 模块
- 共享契约已有公众号与文章类型
- 前端已有：
  - `通讯录 -> 公众号列表`
  - `公众号主页`
  - `文章详情`
  - 桌面三栏工作区

所以接下来的实现重点不是“先把公众号页面搭出来”，而是：
- 补齐现有阅读链路缺口
- 把公众号真正接入“消息”体系
- 明确区分 `订阅号聚合` 和 `服务号会话`
- 在不污染现有 `direct / group` 聊天域的前提下完成首版

## 关键现实约束

### 1. 当前 chat 域高度假设只有 `direct / group`

现有以下代码都写死了这个假设：
- `packages/contracts/src/chat.ts`
- `api/src/modules/chat/chat.types.ts`
- `api/src/modules/chat/chat.service.ts`
- `apps/app/src/routes/chat-list-page.tsx`
- `apps/app/src/features/desktop/chat/desktop-chat-workspace.tsx`
- `apps/app/src/features/chat/conversation-thread-panel.tsx`
- `apps/app/src/features/chat/use-conversation-thread.ts`
- `apps/app/src/routes/chat-room-page.tsx`
- `apps/app/src/routes/chat-details-page.tsx`
- `apps/app/src/routes/chat-message-search-page.tsx`
- 聊天 Socket / 发送消息 / AI 回复链路

如果直接把 `ConversationType` 从：
- `direct | group`

硬扩成：
- `direct | group | official_account`

那会连带改动：
- 会话列表
- 会话详情
- 会话背景
- 搜索
- 消息线程
- Socket 协议
- 发送消息逻辑
- AI 自动回复逻辑

这个爆炸半径明显过大，不适合作为公众号功能首轮接入方案。

### 2. 当前公众号模块更像“内容域”，还不是“消息域”

现在的 `official-accounts` 模块本质上只提供：
- 内容账号
- 文章
- 关注关系

它还没有承载：
- 推送投递
- 服务型消息
- 消息聚合排序
- 未读计数

### 3. 产品目标要求“像微信”，但当前代码更适合走增量路线

所以这次实现不建议先重构整个 chat 域。

更稳的路线是：
- 保持 `chat` 继续只负责 `direct / group`
- 在 `official-accounts` 域内补齐“订阅号消息 / 服务号消息”
- 前端消息列表做“合并展示”

这能满足产品目标，同时把风险控制在可实施范围内。

## 核心技术决策

### 决策 1：首轮不扩 `packages/contracts/src/chat.ts` 的 `ConversationType`

首轮不把公众号消息硬塞进 `Conversation`。

原因：
- 当前 direct chat 线程与 AI 回复强耦合
- 当前 group chat 线程与群实体强耦合
- 服务号消息和订阅号聚合都不是这两套语义
- 先扩 chat 类型会让公众号实现被聊天存量代码牵着走

首轮方案：
- 新增 `official-accounts` 自己的消息契约
- 在 App 的“消息页 / 桌面消息工作区”做统一展示层
- 后续如果公众号消息规模继续扩大，再决定是否抽象统一 message entry 契约

### 决策 2：订阅号和服务号从数据层开始拆开

必须从接口和实体层就分开：

- `subscription`
  - 使用“投递记录”模型
  - 在消息页中显示为一个聚合入口：`订阅号消息`
  - 进入后看到各订阅号最近推送

- `service`
  - 使用“服务消息线程”模型
  - 在消息页中显示为独立消息项
  - 点击进入服务号消息页

### 决策 3：消息列表统一展示，但后端仍分域提供数据

不让 `getConversations()` 一口气承载所有公众号语义。

改为：
- `GET /api/conversations`
  - 继续只返回普通私聊 / 群聊
- `GET /api/official-accounts/message-entries`
  - 返回：
    - `subscriptionInbox`
    - `serviceConversations`

前端消息页负责把两类结果合并排序展示。

这样改动最小，也最符合当前代码结构。

### Phase 1B：引入订阅号消息聚合

这是首个真正进入“消息页”的阶段。

### 新增实体

新增：
- `api/src/modules/official-accounts/official-account-delivery.entity.ts`

字段建议：
- `id`
- `ownerId`
- `accountId`
- `articleId`
- `deliveryKind`
  - `subscription_digest`
- `deliveredAt`
- `readAt`

说明：
- 一条订阅号文章推给世界主人时，产生一条 `delivery`
- 这层是“消息已投递给谁”的记录，不是文章本体
- 单世界模型下仍要保留 `ownerId`，因为语义上这是“世界主人收到了什么”

### 共享契约新增

新增到 `packages/contracts/src/official-accounts.ts`：
- `OfficialAccountDeliveryItem`
- `OfficialAccountSubscriptionInboxGroup`
- `OfficialAccountSubscriptionInboxEntry`
- `OfficialAccountMessageEntries`

建议结构：
- `subscriptionInbox`
  - 特殊聚合入口摘要
  - 字段包含：
    - `unreadCount`
    - `lastDeliveredAt`
    - `preview`
- `subscriptionGroups`
  - 订阅号聚合页用
  - 每个公众号一组最近投递文章

### 后端接口新增

新增：
- `GET /api/official-accounts/message-entries`
  - 返回消息页需要的公众号消息入口
- `GET /api/official-accounts/subscription-inbox`
  - 返回订阅号聚合流
- `POST /api/official-accounts/deliveries/:deliveryId/read`
  - 标记某条订阅投递已读
- `POST /api/official-accounts/subscription-inbox/read`
  - 一键标记聚合页已读

### 后端服务逻辑

`OfficialAccountsService` 新增职责：
- 为已关注的订阅号生成投递摘要
- 查询聚合页所需的文章投递流
- 汇总未读数量

首版生成方式：
- 仅基于已关注公众号的历史文章倒排生成投递记录
- 如果历史库里还没有 delivery，可在首次访问时回填

这样能先把体验做通，不阻塞后续真正的投递生产链路。

## 实施顺序

推荐严格按下面顺序做，小步提交：

1. 先补阅读链路
   - 文章已读
   - 桌面搜索 / 关注
   - 文章阅读器补全

2. 再补订阅号聚合
   - `delivery` 实体
   - 订阅号聚合接口
   - `/chat/subscription-inbox`
   - 消息页插入 `订阅号消息`

3. 再补服务号会话
   - 服务号消息实体
   - 服务号会话摘要接口
   - 服务号线程页
   - 消息页插入服务号消息

4. 最后补菜单
   - `menuPayload` 契约化
   - 菜单点击回写消息

## 当前实现准备结论

已经可以开始实现，但实现口径要从原始规划调整为下面这条：

- 阅读链路不是从零做，而是补全现有公众号模块
- 消息接入不要先重构整个 chat 域
- 首轮用 `official-accounts` 自己的消息实体和接口承载：
  - `订阅号聚合`
  - `服务号会话`
- App 消息页做联合展示层

这是和当前仓库最匹配、返工风险最低、又能继续向微信靠齐的方案。
