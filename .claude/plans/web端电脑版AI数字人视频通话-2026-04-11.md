# Web 端电脑版 AI 数字人视频通话规划

## 1. 目标
- 在 `apps/app` 的 Web 桌面端落地单聊 `AI 数字人视频通话`。
- 产品口径不是真人 RTC，也不是本地摄像头双向推流，而是：
  - 用户在桌面单聊里发起视频通话
  - 左侧进入桌面通话工作台
  - 远端接入 `AI 数字人` 舞台
  - 用户通过麦克风按轮次说话
  - 后端完成 STT -> AI 回复 -> TTS -> 数字人口型/视频播报
  - 当前轮的文本、语音、视频状态同步写回聊天上下文
- 首版只做：
  - Web 桌面端
  - 单聊
  - 半双工
  - AI 数字人
- 本次不做：
  - 群视频
  - 真人对真人 WebRTC
  - 本地摄像头上行给远端 AI 做视觉理解
  - 完整 RTC 基础设施

## 2. 现状结论

### 2.1 前端已经有“视频通话壳”，但还不是真正的桌面数字人能力
- `apps/app/src/features/chat/conversation-thread-panel.tsx`
  - 桌面单聊已支持点击“视频通话”。
- `apps/app/src/features/desktop/chat/desktop-direct-call-panel.tsx`
  - 现在 `kind === "video"` 只显示“视频暂未开放”。
  - 该面板已经有桌面工作台布局，可以继续复用。
- `apps/app/src/features/chat/mobile-ai-call-screen.tsx`
  - 手机端 `mode="video"` 已有完整页面形态：
    - 远端数字人头像舞台
    - 本地摄像头预览
    - 轮次文本摘要
    - 语音播放状态
  - 但这里本质仍然是 `语音通话 + 数字人视觉包装`，不是实时数字人视频流。
- `apps/app/src/features/chat/use-self-camera-preview.ts`
  - 已有本地摄像头预览 hook，可直接复用到桌面版。
- `apps/app/src/features/chat/use-voice-call-session.ts`
  - 已封装单轮录音、上传、自动播放、消息刷新逻辑。

### 2.2 后端只有语音回合链路，没有数字人视频链路
- `api/src/modules/chat/voice-calls.service.ts`
  - 已有 `createTurn()`，流程是 `STT -> 发消息 -> AI 回复 -> TTS -> 返回音频 URL`。
- `api/src/modules/ai/ai.controller.ts`
  - 已有：
    - `POST /api/ai/transcriptions`
    - `POST /api/ai/speech`
- 当前仓库没有：
  - 数字人 provider 抽象
  - 数字人会话创建/关闭接口
  - 数字人口型驱动或视频流地址
  - 视频播报状态同步协议

### 2.3 结论
- 如果只想“尽快在桌面端可用”，可以先把手机端那套 `数字人视觉包装 + 语音回合链路` 搬到桌面。
- 如果要求“接的是 AI 数字人”且要有真正的数字人视频输出，就必须补一层新的后端数字人服务，不适合继续只靠 `/chat/voice-calls/turns`。
- 因此建议拆成两个层次：
  - `MVP`：桌面版数字人视频通话工作台先落地，远端舞台先用数字人卡面/动画承载，链路仍复用现有语音回合。
  - `正式版`：补数字人 provider 接入，把远端舞台替换成真实数字人视频源。

## 3. 推荐产品口径

### 3.1 对用户的表述
- 入口名称继续保留 `视频通话`。
- 桌面工作台标题改成 `AI 数字人视频通话`。
- 副标题统一成：
  - `当前为半双工数字人视频通话，你说一轮，TA 回一轮。`
- 远端区域明确区分：
  - `数字人舞台`
  - `我的摄像头预览`
- 如果数字人 provider 未接通或降级：
  - 明确提示 `当前降级为语音驱动的数字人舞台，不是实时数字人视频流`

### 3.2 首版必须收口的点
- 不宣称真人接听。
- 不宣称实时双向视频。
- 不做“等待对方接听”“对方已加入”这类真人 RTC 文案。
- 把现有桌面端转手机接力逻辑从单聊视频通话链路移除。

## 4. 技术方案

### 4.1 Phase A：先把桌面版数字人视频工作台做出来
- 目标：
  - 桌面端点击“视频通话”后，不再跳 `/desktop/mobile`。
  - 直接在当前聊天工作区打开桌面视频通话面板。
- 复用文件：
  - `apps/app/src/features/desktop/chat/desktop-direct-call-panel.tsx`
  - `apps/app/src/features/chat/use-voice-call-session.ts`
  - `apps/app/src/features/chat/use-self-camera-preview.ts`
  - `apps/app/src/features/chat/mobile-ai-call-screen.tsx`
- 建议拆分：
  - `apps/app/src/features/desktop/chat/desktop-digital-human-call-stage.tsx`
  - `apps/app/src/features/desktop/chat/use-desktop-digital-human-call.ts`
- 具体改法：
  - 将 `DesktopDirectCallPanel` 拆成 `voice` / `video` 两套视图，而不是把 `video` 当占位。
  - `video` 模式直接复用 `useVoiceCallSession` 作为回合链路。
  - `video` 模式接入 `useSelfCameraPreview`，提供桌面右上角本地小窗。
  - 远端主舞台先用：
    - 角色头像 / 半身图
    - 播报中 / 思考中动效
    - 播报字幕与最近一轮文本
  - 先完成“桌面端可用的视频通话感”。

### 4.2 Phase B：新增数字人 provider 抽象层
- 目标：
  - 后端不要把数字人 provider 写死在页面逻辑里。
- 新增后端模块建议：
  - `api/src/modules/ai/digital-human/`
- 建议拆分：
  - `digital-human.types.ts`
  - `digital-human.service.ts`
  - `digital-human.provider.ts`
  - `digital-human.controller.ts`
- 数字人服务职责：
  - 创建单次数字人通话 session
  - 根据文本和音频驱动数字人播报
  - 返回可播放的视频流地址、播放器 token 或渲染资源
  - 查询当前播报状态
  - 销毁 session
- 推荐契约：
  - `POST /api/ai/digital-human/sessions`
  - `POST /api/ai/digital-human/sessions/:id/turns`
  - `GET /api/ai/digital-human/sessions/:id`
  - `DELETE /api/ai/digital-human/sessions/:id`
- 契约字段建议：
  - `conversationId`
  - `characterId`
  - `mode: "desktop_video_call"`
  - `provider`
  - `streamUrl` 或 `playerUrl`
  - `posterUrl`
  - `sessionStatus`
  - `lastTurn`

### 4.3 Phase C：把现有语音回合升级成“数字人回合”
- 不建议前端自己串三段请求。
- 更稳的方式是后端新增统一编排接口，例如：
  - `POST /api/chat/digital-human-calls/turns`
- 这一轮后端内部流程：
  - 接收用户录音
  - `transcribeAudio`
  - `chatService.sendMessage`
  - `synthesizeSpeech`
  - 调数字人 provider 发起播报
  - 返回：
    - 用户转写文本
    - AI 文本
    - AI 音频 URL
    - 数字人视频播放信息
    - 本轮耗时
- 这样桌面端视频通话仍然保持“一次请求代表一轮”的简单模型。

### 4.4 前端状态机建议
- 新增 `video call session` 状态：
  - `idle`
  - `requesting_permission`
  - `recording`
  - `uploading`
  - `transcribing`
  - `thinking`
  - `synthesizing`
  - `rendering_avatar`
  - `playing`
  - `error`
- 比当前 `useVoiceCallSession` 多出来的关键状态是：
  - `rendering_avatar`
  - `session_connecting`
  - `session_disconnected`

### 4.5 前端页面改造点
- `apps/app/src/features/chat/conversation-thread-panel.tsx`
  - 桌面单聊点“视频通话”时保留在桌面工作区，不再走 `/desktop/mobile`。
- `apps/app/src/features/desktop/chat/desktop-chat-workspace.tsx`
  - 单聊视频通话不要再导航到手机接力页。
- `apps/app/src/features/desktop/chat/desktop-chat-header-actions.tsx`
  - 菜单文案维持不变，但打开后进入真实桌面面板。
- `apps/app/src/features/desktop/chat/desktop-direct-call-panel.tsx`
  - 正式承接视频工作台。
  - 左侧放远端数字人舞台 + 顶部状态。
  - 右侧放：
    - 本轮字幕
    - 用户转写
    - AI 回复
    - 本地摄像头预览
    - 静音 / 摄像头开关 / 重播 / 挂断

### 4.6 契约层改造点
- `packages/contracts/src/speech.ts`
  - 新增数字人 session / turn 类型。
- `packages/contracts/src/client.ts`
  - 新增：
    - `createDigitalHumanSession()`
    - `createDigitalHumanTurn()`
    - `getDigitalHumanSession()`
    - `closeDigitalHumanSession()`
- 保持现有 `createVoiceCallTurn()` 不动，继续给纯语音通话使用。

## 5. 推荐实施顺序

### Step 1：桌面 UI 先去占位
- 目标：
  - 桌面单聊视频通话不再显示“暂未开放”。
  - 不再跳手机接力。
- 产出：
  - 真正可交互的桌面视频通话工作台。

### Step 2：复用现有语音链路做桌面视频首版
- 目标：
  - 用 `useVoiceCallSession` 跑通桌面视频通话首版。
  - 本地摄像头预览、字幕、远端数字人舞台先全部到位。
- 产出：
  - “可用但仍是语音驱动”的桌面数字人视频通话。

### Step 3：补后端数字人模块
- 目标：
  - 接通真实 AI 数字人 provider。
  - 让远端主舞台从“头像动画”升级到“数字人视频输出”。
- 产出：
  - 后端 provider 抽象
  - 数字人 session API
  - 数字人 turn API

### Step 4：把桌面视频工作台切到真实数字人接口
- 目标：
  - 前端从 `useVoiceCallSession` 升级到 `useDesktopDigitalHumanCall`。
  - 统一消费数字人 session / turn 契约。
- 产出：
  - 真正“接的是 AI 数字人”的桌面视频通话。

## 6. 风险和取舍

### 6.1 最大风险
- 当前仓库没有任何数字人 provider 接入。
- 所以“视频通话”如果要求是真数字人视频，不是一个纯前端任务，核心阻塞在后端和第三方能力。

### 6.2 取舍建议
- 如果本周目标是先上线：
  - 先做 Phase A，桌面端立即可用。
- 如果目标是对外宣称“AI 数字人视频通话已接通”：
  - 必须做 Phase B / C，不建议只用头像动画冒充。

### 6.3 兼容性
- Web 桌面端需要处理：
  - 浏览器麦克风权限
  - 摄像头权限
  - 自动播放限制
  - provider 断流/超时
- 所以 UI 必须提供明确的降级提示和重试按钮。

## 7. 本次建议的实施边界
- 先按下面的边界推进最稳：
  - 只做桌面单聊
  - 先去掉“视频暂未开放”
  - 先做桌面数字人视频工作台
  - 前端先复用现有语音回合链路
  - 同时把后端数字人模块接口骨架预留好
- 这样不会把桌面首版卡死在第三方数字人接入上，也不会把后面的真实接入路线堵死。

## 9. 结论
- 这件事不能只理解成“把桌面端视频按钮点亮”。
- 当前仓库已经具备 `桌面工作台 + 语音回合 + 本地摄像头预览` 三个关键积木，先做桌面首版并不难。
- 真正的难点在于 `AI 数字人 provider 接入`。
- 因此最合理的推进方式是：
  - `先上线桌面数字人视频工作台`
  - `再接真实数字人后端`
  - `最后切到正式数字人流`
