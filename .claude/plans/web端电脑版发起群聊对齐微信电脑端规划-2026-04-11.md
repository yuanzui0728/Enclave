# Web 端电脑版发起群聊对齐微信电脑端规划

## 任务目标
- 只盘 `Web 端 + 电脑版 + 发起群聊` 这条链路，不泛化到整个群聊页。
- 对照微信电脑端当前聊天页里的发起群聊操作，找出我们还差什么。
- 给出一条“先做最像微信的动作，再补需要后端配合的能力”的实施顺序。

## 本次核对范围

### 本地代码基线
- 桌面聊天工作区快捷入口：`apps/app/src/features/desktop/chat/desktop-chat-workspace.tsx`
- 桌面单聊右侧聊天信息面板：`apps/app/src/features/desktop/chat/desktop-chat-details-panel.tsx`
- 桌面发起群聊页：`apps/app/src/routes/create-group-page.tsx`
- 发起群聊路由状态：`apps/app/src/lib/create-group-route-state.ts`
- 移动端单聊详情已做好的 seed 逻辑：`apps/app/src/routes/chat-details-page.tsx`
- 后端建群契约：`packages/contracts/src/chat.ts`
- 后端建群实现：`api/src/modules/chat/group.service.ts`
- 桌面现有“加群成员”弹层：`apps/app/src/features/desktop/chat/desktop-group-member-picker.tsx`

### 微信电脑端对标口径
- 2025-10-22，36 氪转载“见实”文章提到：微信 `4.1.2` 在单聊里邀请他人建群时，已支持“分享聊天记录”，并且是在聊天页右上角 `...` 菜单里继续发起。
- deepin 官方在介绍微信 Linux `4.0` 时明确写到：新版聊天体验、通讯录和联系人管理已全面对齐 Windows / Mac 主线版本。
- 这两个信号足够说明一件事：
  - 微信电脑端“发起群聊”已经不是单纯的联系人选择器。
  - 它是一条嵌在当前聊天上下文里的连续操作流。

## 当前已经接近微信的部分

### 1. 桌面聊天页里已经有“发起群聊”入口
- 左上角快捷 `+` 菜单里已有“发起群聊”。
- 单聊右侧聊天信息面板的成员区也已有 `+` 入口。
- 这说明入口数量不是主要矛盾，主要矛盾是入口后的交互和上下文保留不对。

### 2. 已经有联系人查询、搜索和默认群名能力
- 桌面发起群聊页会拉取通讯录联系人并支持搜索。
- 已支持根据已选成员生成默认群名。
- 所以数据基础并不缺，问题在于表现形态和操作顺序不像微信。

### 3. 已经有一个可复用的桌面弹层基础件
- `desktop-group-member-picker.tsx` 已经是“遮罩 + 左侧联系人 + 右侧已选结果 + 底部确认”的桌面弹层。
- 这意味着桌面端不需要从零做“微信式选人弹层”，更合理的做法是把这个组件抽象成“建群 / 加人”共用选择器。

## 关键差异判断

### 结论
- 现在的桌面发起群聊更像一个单独的“群聊创建页面”。
- 微信电脑端更像“从当前聊天页里拉起的轻量建群弹层”。
- 差异不在“能不能建群”，而在：
  - 是否保留当前聊天上下文
  - 是否尽量少打断用户
  - 是否把“单聊延续成群聊”做成连续动作

## 还和微信电脑端有明显差异的地方

## P0：这些先补，不然用户一上手就会觉得不像微信

### 1. 桌面单聊信息面板发起群聊时，没有默认带上当前聊天对象
- 现在桌面端右侧聊天信息面板点击“发起群聊”只是直接跳 `/group/new`，没有把当前单聊对象带过去。
- 代码位置：`apps/app/src/features/desktop/chat/desktop-chat-details-panel.tsx:324`
- 但移动端这条链路已经做了：
  - 传 `source=chat-details`
  - 传 `conversationId`
  - 传 `seedMemberIds`
- 代码位置：`apps/app/src/routes/chat-details-page.tsx:297`
- 这会导致桌面端比移动端还退一步，不符合微信电脑端“从当前人继续拉群”的直觉。

### 2. 桌面快捷菜单发起群聊是无上下文跳页，不是微信那种当前页内继续操作
- 当前桌面聊天工作区 `+` 菜单点击后直接 `navigate("/group/new")`。
- 代码位置：`apps/app/src/features/desktop/chat/desktop-chat-workspace.tsx:541`
- 微信电脑端更像在当前聊天主界面里继续完成联系人选择，而不是跳去一个新的大页面。
- 用户感知上，这会把“聊天操作”变成“页面跳转”，肌肉记忆完全不一样。

### 3. 桌面端建群是完整页面，不是微信电脑端的轻量弹层
- 当前桌面建群页是一个独立的大工作区：
  - 顶部整条 header
  - 左侧联系人栏
  - 右侧群资料栏
  - 底部操作栏
- 代码位置：`apps/app/src/routes/create-group-page.tsx:189`
- 这更像后台式创建流程。
- 微信电脑端更像：
  - 从聊天页右上角操作继续下钻
  - 弹出“选择联系人”窗口
  - 确认后直接回到新群会话

### 4. 桌面端强制把“起群名”前置了，顺序不像微信
- 当前桌面发起群聊第一页就要求用户看见并可编辑群名称。
- 代码位置：`apps/app/src/routes/create-group-page.tsx:305`
- 微信电脑端通常是：
  - 先选人
  - 先把群建出来
  - 群名按成员自动生成
  - 后续再到聊天信息里改群名
- 现在把群名编辑前置，会让整个流程更重，也更不像微信。

### 5. 桌面联系人选择区排法不像微信电脑端
- 当前桌面端左侧联系人是纯列表，右侧是大卡片式“已选成员”。
- 代码位置：`apps/app/src/routes/create-group-page.tsx:227`、`apps/app/src/routes/create-group-page.tsx:326`
- 微信电脑端更接近：
  - 顶部搜索
  - 已选成员条
  - 主列表连续选择
  - 选中反馈更轻、更紧凑
- 现在的右侧大卡片确认区太重，不像微信这种高频工具操作。

### 6. “单聊延续建群并分享聊天记录”完全缺失
- 微信 `4.1.2` 已公开有：
  - 在单聊中邀请他人时可分享聊天记录
  - 选中的历史消息会直接进入新群
  - 这是“把单聊延续成群聊”的核心体验
- 我们当前只有 seed member，没有任何聊天上下文延续能力。
- 这会让“从当前讨论继续拉第三人进来”这件事断掉。

### 7. 后端建群契约太薄，支撑不了微信式“延续建群”
- 当前契约只有：
  - `name`
  - `memberIds`
- 代码位置：`packages/contracts/src/chat.ts:104`
- 当前服务端建群也只是：
  - 建 `Group`
  - 建 owner member
  - 循环写入成员
- 代码位置：`api/src/modules/chat/group.service.ts:111`
- 缺少这些关键字段：
  - `sourceConversationId`
  - `shareHistory`
  - `sharedMessageIds`
  - `originMessageSnapshot` 或服务端复制后的新群首批消息

## P1：不影响建群成功，但会影响“完全像微信”

### 8. 桌面发起群聊页没有明显复用聊天页的视觉语义
- 当前建群页有更强的圆角、渐变、投影和“卡片工作台”感。
- 微信电脑端的联系人选择弹层会更克制、更平、更贴近系统窗口。
- 如果目标是“完全抄微信”，这块要跟桌面聊天主界面一起收敛视觉 token。

### 9. 桌面选择联系人还缺更顺手的键盘工作流
- 现有实现以鼠标点击为主，没有明显的：
  - 上下方向键切换
  - Enter 选中
  - Backspace 快速移除已选成员
  - Esc 关闭弹层
- 微信电脑端这类选择器天然应该更偏键盘友好。
- 这不是首批阻塞项，但会影响“像桌面工具而不是网页表单”的质感。

### 10. 桌面建群和“添加群成员”是两套近似实现，后续容易继续分叉
- 现在：
  - 建群走 `create-group-page.tsx`
  - 已有群加人走 `desktop-group-member-picker.tsx`
- 两边结构已经很像，但没有收成同一个桌面选择器。
- 如果继续各写各的，后面越改越难和微信统一。

## 建议的实施顺序

## Phase 1：先把桌面端入口改成“从当前聊天继续发起”
- 目标：先修最明显的操作断裂。
- 要做：
  - 桌面单聊信息面板点击“发起群聊”时，补上和移动端一致的 route hash seed
  - 桌面 `+` 菜单发起群聊时，若当前处于单聊，默认附带当前聊天对象
  - `create-group-route-state.ts` 增补桌面来源枚举，避免后面弹层化时上下文丢失
- 产出标准：
  - 从单聊里发起群聊时，默认已经勾上当前对象
  - 返回时能回到原聊天上下文

## Phase 2：把桌面建群从“独立页面”改成“聊天内弹层”
- 目标：把核心交互改成微信电脑端路径。
- 要做：
  - 基于 `desktop-group-member-picker.tsx` 抽象出 `DesktopConversationPickerDialog`
  - 支持两种模式：
    - `create-group`
    - `add-group-member`
  - 桌面端发起群聊时优先弹层，不再跳完整 `/group/new` 大页
  - 桌面模式下把“群名称输入”从首屏移除，改成创建后默认名
- 主要改动面：
  - `apps/app/src/features/desktop/chat/desktop-group-member-picker.tsx`
  - `apps/app/src/features/desktop/chat/desktop-chat-workspace.tsx`
  - `apps/app/src/features/desktop/chat/desktop-chat-details-panel.tsx`
  - `apps/app/src/routes/create-group-page.tsx`

## Phase 3：补齐“单聊延续成群聊”的微信关键体验
- 目标：把最能拉开差距的微信新能力补上。
- 要做：
  - 建群前新增“分享聊天记录”开关
  - 从当前单聊拉取可分享消息列表
  - 支持勾选若干条消息带入新群
  - 新群创建成功后，把共享内容作为系统归档消息或真实转发消息落到新群
- 主要改动面：
  - `packages/contracts/src/chat.ts`
  - `packages/contracts/src/client.ts`
  - `api/src/modules/chat/chat.controller.ts`
  - `api/src/modules/chat/group.service.ts`
  - `apps/app/src/features/chat`
  - `apps/app/src/features/desktop/chat`

## Phase 4：最后收“微信桌面工具感”
- 目标：让细节也不出戏。
- 要做：
  - 桌面联系人选择支持键盘操作
  - 弹层头部、列表密度、按钮文案、取消/确认顺序按微信收口
  - 收敛圆角、阴影、渐变，让它更像微信电脑端而不是项目自定义工作台

## 建议的最小落地版本
- 第一刀不要同时碰后端和整个聊天页。
- 最稳妥的顺序是：
  1. 桌面单聊入口补 seed
  2. 桌面建群改弹层
  3. 去掉首屏群名输入
  4. 再补“分享聊天记录”后端协议

## 风险与注意点
- 现有工作区里已有未提交改动：
  - `apps/admin/src/routes/evals-page.tsx`
  - `apps/app/src/features/chat/chat-message-search-panel.tsx`
  - `apps/app/src/features/chat/group-chat-thread-panel.tsx`
- 后续实现时应只触发起群聊链路相关文件，避免误碰这些改动。
- 若“分享聊天记录”要做成真实消息复制，需要提前定口径：
  - 是写成系统消息
  - 还是写成带来源说明的转发消息
  - 还是仅写入一次性摘要卡片

## 结论
- 现在最像微信的方向，不是继续堆更多群聊设置项。
- 而是先把“从当前单聊里顺手拉人建群”这件事做得像微信：
  - 默认带上当前对象
  - 不跳大页
  - 不强迫先起群名
  - 能把当前聊天延续进新群
